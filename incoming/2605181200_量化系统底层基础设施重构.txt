# 一、核心原则（非常重要）

你现在的数据库系统，实际上已经形成：

```text
数据层
↓
因子层
↓
方法层
↓
策略层
↓
回测层
↓
知识层
```

因此：

> 数据库迁移 ≠ 文件移动
> 而是“系统引用关系重构”

所以必须遵循：

```text
先建立新架构
→ 双写/并行验证
→ 引用切换
→ 最后清理旧库
```

而不是：

```text
先删旧库
→ 再修代码
```

后者会炸。

# 四、墨涵指出的关键问题（逐项分析）

# 问题1：readonly/test_settlement 不该先删

完全正确。

因为：

```text
readonly 很可能承担：
- 快照
- 并行验证
- 风险隔离
```

你未来会做：

```text
旧系统结果 vs 新系统结果
```

这必须依赖 readonly 快照。

所以：

# readonly 先保留

但：

```text
移出主工作路径
```

例如：

```text
runtime/readonly/
→ archive/runtime_snapshot/
```

不是删除。

---

# 问题2：schema mapping 缺失

这是目前最大风险。

因为：

```text
oil_daily
stock_daily
```

很可能：

```text
字段名不同
类型不同
索引不同
日期格式不同
```

一旦直接 merge：

后面：

```python
df["turnover_rate"]
```

可能直接 NaN。

最危险的是：

# 不报错，但结果全错。

所以必须先做：

# Schema Registry

建立：

```text
schema_registry.yaml
```

例如：

```yaml
oil_daily:
  trade_date: TEXT
  close_price: REAL

stock_daily:
  trade_date: TEXT
  close: REAL
```

然后建立：

# ETL Mapping

例如：

```yaml
oil_daily.close_price
→ market_daily.close
```

这一步必须自动化。

---

# 问题3：漏了 file_registry.db

这个问题非常专业。

# “数据资产目录系统”

```text

```

都依赖 registry。

所以：

# registry 是基础设施

不是附属文件。
