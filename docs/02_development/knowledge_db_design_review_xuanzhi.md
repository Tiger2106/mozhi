## 审查结论：**有条件的通过（需解决战略级盲点后方可进入实施）**

设计方案工程层面质量合格，且墨萱已识别全部 P0/P1 技术问题。但知识库系统若孤立看待，可能偏离"为报告流水线服务"的根本目标。以下从战略视角补充审查。

## 一、整体架构完整性

### 覆盖情况

| 场景 | 是否覆盖 | 评价 |
|:-----|:---------|:------|
| 回测结果存档与查询 | ✅ | 完整，五表模型支持按策略/标的/时段灵活查询 |
| 参数溯源与对比 | ✅ | `params_snapshot` 含 `diff_from_prev` 字段 |
| 绩效沉淀与置信度 | ✅ | `validity_grade` + `confidence` 双层判定 |
| 知识增量累积 | ✅ | `knowledge_entries` 聚合表 |
| 自动衰减 | ✅ | `decay_check()` + `deprecated_at` |
| 报告历史对比 | ⚠️ 部分 | 设计中提及 KnowlegeExtractor / WeeklyReportExtractor 改查 DB |

### 战略级遗漏（1处）

**知识增益的回馈闭环缺失。** 当前设计是单向流水线：

```
回测 → 存档 → 聚合知识 → 报告引用
```

缺少反向回路：

```
报告结论 → 写入 knowledge_entries 补充 → 下一轮回测参考
```

- 日/周报中 Step5 产出的分析结论（如"某标的处于趋势衰竭期"）具有一定战略价值，但没有被沉淀回 knowledge.db
- 长此以往，knowledge.db 仅存储回测硬数据（夏普、回撤等），缺少"宏观判断"类软知识的积累
- **建议**：在 knowledge_entries 中增加 `insight_source` 枚举值 `report_step5`，允许 Step5 的分析结论以结构化方式回写入库

## 二、与报告流水线（Step0→Step7）集成的深度审查

### 2.1 当前设计覆盖的流水线阶段

| 流水线步骤 | 知识库关联 | 状态 |
|:-----------|:-----------|:------|
| Step0（市场扫描） | 无直接关联 | ❌ 遗漏 |
| Step1（数据采集）/ Step11 | 无直接关联 | ❌ 遗漏 |
| Step2（情绪/策略编排） | 无直接关联 — 但 `list_runs()` 可作为编排参考 | ⚠️ 潜在 |
| Step3（回测执行） | ✅ `_persist_result()` 扩展写入 | 已设计 |
| Step3.5（战略复核） | 无直接关联 — 但可补充历史回测对比依据 | ❌ 遗漏 |
| Step4（汇总渲染） | 无直接关联 | ❌ 遗漏 |
| Step5（定稿） | ⚠️ 部分 — 引用的 `query_from_db()` 在第二阶段 | 设计中 |
| Step6（推送） | 无直接关联 — DB 不在推送范围内 | 合理 |
| Step7（异步审阅） | 无直接关联 | ❌ 遗漏 |

**战略结论**：当前设计仅覆盖 Step3 后的写路径和 Step5 的读路径。**Step0、Step1、Step3.5、Step7 均未被集成**。

### 2.2 关键遗漏分析

#### 遗漏 1：Step0/Step1 — 历史同市况参照缺失

Step0 的板块轮动判断、Step1 的"今日最强板块"解读本质上是**将当前市场状态与历史回测进行对标**。

- 如果 knowledge.db 存有历史回测在"震荡市"、"快速上涨"等市况下的表现（通过 `market_context` 表的 `market_regime` 字段），Step0 的判断可以更精准
- 但当前设计没有说明 Step0/Step1 如何读取 knowledge.db 作为参照基准

**建议：** 在第二阶段增加 `KnowledgeDB.query_by_regime(market_regime, symbol)` 接口，供 Step0 的宏观判断作为历史参照。

#### 遗漏 2：Step3.5 / Step7 — 战略审阅的数据支撑

玄知的 Step3.5 战略复核和 Step7 异步审阅本质是"用历史数据验证当前结论"。

- 缺少一个明确的查询模式：给定当前标的 + 当前市场状态 → 返回历史相似回测表现区间 → 作为审阅的量化依据
- `aggregate_knowledge()` 的粒度太粗（仅返回聚合均值），不够支撑"审阅偏差"级别的判断

**建议：** 新增查询模式 `get_historical_benchmark(symbol, regime)` 返回 25%/50%/75% 分位数的绩效指标范围，而非仅返回均值。

#### 遗漏 3：无 "回测知识 → 报告引用" 的标准接口

设计中 KnowlegeExtractor 的 `query_from_db()` 是第二阶段，但没有定义**数据格式标准**——即 knowledge.db 的数据以什么格式被报告引用？

- 如果格式不统一，Step3（回测执行）的回测结果是 pandas DataFrame，Step5 需要的是 markdown 片段，格式转换将引入额外复杂度
- 建议在 `query_from_db()` 的返回值中内嵌 markdown 渲染能力：

```python
def query_from_db(self, ...) -> dict:
    """
    Returns:
        raw: List[Dict]       # 原始数据
        markdown: str          # 可直接嵌入报告的 markdown 片段
    """
```

## 三、盲点分析

### 3.1 盲点：缺乏元数据全局视图

当前知识库的设计是"以回测为中心"，缺失一个**整体资产视图**：

- A50 回测覆盖了哪些标的？覆盖了多少次？
- 某标的上次回测是什么时候？是死数据还是活数据？
- 整个 knowledge.db 的健康度如何（数据新鲜度、样本量分布）？

**建议：** 增加 `KnowledgeDB.health_check()` 方法，返回如下元信息：

```python
{
  "total_backtests": 152,
  "active_entries": 23,
  "stale_symbols": ["601857.SH"],  # 超过 45 天未更新
  "coverage_rate": 0.85,            # 已覆盖标的 / A50 全部标的
}
```

此信息可嵌入 Step3.5 审阅报告中，帮助战略方判断知识库数据是否足够可信。

### 3.2 盲点：尚无 知识熵值 / 知识冗余检测

多条相似的参数配置会产出一条以上的回测结果，`aggregate_knowledge()` 能聚合但无法检测冗余。如果 30 次回测中 25 次是微小参数变化，样本量虚高，结论的置信度评估将失真。

**建议：** 在 `params_snapshot` 的 `diff_from_prev` 基础上增加"参数量化相似度"阈值（如 cosine similarity < 0.95 才计入独立样本），避免样本量膨胀。

### 3.3 盲点：`market_context` 自动填充的时间延迟

设计中 `market_context` 的自动填充依赖 T6（市场状态分析）的完成——而 T6 是流水线中的固定步骤，仅在日报/中报生成时执行。

- 如果一次回测是在非报告时段触发的（如盘中手动回测），`market_context` 将缺失或延迟填充
- 建议在 `store_run()` 中增加快速市场状态估算（基于简单的价格平均线 + 波动率），不等待 T6 就完成初步写入；T6 产出后再 UPSERT 覆盖

## 四、风险分析（非技术风险）

### 4.1 技术风险（墨萱已覆盖，补充 1 条）

| 风险 | 补充说明 |
|:-----|:---------|
| 运行时对回测引擎的影响 | `_persist_result()` 中追加 DB 写入可能延长执行时间（特别在 DB 首次初始化时），设计文档未给出预期延迟范围。如果回测任务有严格的时间预算（如 5 分钟的 cron 窗口），DB 写入的 0.5-2s 延迟可能需要纳入工时估算 |

### 4.2 维护性风险

| 风险 | 等级 | 描述 |
|:-----|:-----|:------|
| 对 `_persist_result()` 的侵入式修改 | 中 | 三个策略文件各追加至少 2 行代码（import + 调用），如果未来某个策略的 `_persist_result()` 被重构，DB 写入可能被忽略 |
| 知识库与实际回测行为不同步 | 低 | 如果 DB 写入失败（事务回滚）但 JSON 写入成功，回测结果存在但知识库缺失——这种"数据不一致"不易被发现 |
| 回填脚本与当前结构耦合 | 中 | 回填脚本 `backfill_knowledge_db.py` 需要理解 JSON 文件结构，如果未来 JSON 格式变化，脚本需同步更新 |

### 4.3 三阶段时间线的节奏评估

| 阶段 | 预计工时 | 评估 |
|:-----|:---------|:------|
| 第一阶段（P0 + 写路径） | 4-5h | **合理**。核心写路径优先，先跑通再优化。但建议将 `_find_project_root()` 修复和 FK 修复纳入此阶段（约 +0.5h），避免带着 P0 问题上线 |
| 第二阶段（读路径 + 市场上下文） | 3-4h | **条件合理**。依赖 T6 完成，意味着第二阶段的时间线受 T6 排期约束。如果 T6 延期，整个知识库的读端能力也延期。建议设定 T6 的硬截止日期 |
| 第三阶段（衰减） | 2-3h | **过早评估**。"30+次回测后"的节点不可预测，实际可能在 1-2 周内达到也可能 1 个月。建议改为"回测记录 >= 30 条 或 已上线 >= 4 周"双条件触发，而非单一数量条件 |

**总体节奏判断**：三阶段时间线基本合理，但第二阶段应与 T6 解耦（见 3.3 盲点建议）。

## 五、战略补充建议

### 建议 1：优先解决 P0 后再实施，但不要等待 T6

墨萱识别的 2 处 P0 问题（FK 语义错误 + `_find_project_root()` 健壮性）都位于**写路径**，应在第一阶段修复后立即上线。但第二阶段不应等待 T6 完成——`market_context` 的快速估算（见 3.3）可作为并行路线。

### 建议 2：增加"知识闭环"的指标追踪

将以下指标纳入 Dashboard / 信号文件：

- `knowledge_entries` 总量 & 近 7 天增量（知识库是否在增长）
- 报告引用 knowledge.db 的次数 & 引用条目 ID（知识库是否被使用）
- 衰减率（deprecated 占比，过高说明策略老化快）

### 建议 3：增加"回测知识集成"到 Step3.5 的标准流程

Step3.5 的审阅检查项应增加一条标准查询：

> 本次报告涉及的标的，历史相似市况下的平均绩效范围是什么？

对应的实现由 knowledge.db 提供 `get_historical_benchmark()` 接口。此集成可以写入 `reports/morning|midday/YYYYMMDD/strategic_review_*.md` 中作为背景数据。

## 六、总结

| 维度 | 结论 |
|:-----|:------|
| 技术设计 | 整体良好，墨萱已识别全部 P0/P1，建议在实施前修复 P0 |
| **战略盲区** | **报告流水线的回馈闭环缺失**——知识应反向回流给报告和策略配置 |
| **集成缺口** | Step0/1/3.5/7 均缺乏与 knowledge.db 的直接集成 |
| 时间线 | 三阶段节奏合理，但第二阶段应与 T6 适当解耦 |
| 维护性 | 侵入式修改带来的耦合风险可接受，但需文档化 DB 写入的预期延迟 |
| **知识熵值** | 缺少冗余检测和覆盖度评估，建议在 `aggregate_knowledge()` 中增加相似度过滤 |

**一句话结论**：本方案可以实施，但不应止步于"回测存档工具"。知识库真正的战略价值在于成为报告流水线的"历史记忆"——这意味着 Step0→Step7 的绝大多数环节都需要与之对话。墨萱的 P0 修复是上线前提，而战略层面的集成规划应在第一阶段实施前就明确路线图。
