一、现状vs目标
现状：文件即状态
─────────────────────────────────────────────
任务触发  →  写 trigger_*.json
任务运行  →  写 heartbeat/*.json
任务完成  →  写 *.done
任务失败  →  写 *.failed
下游轮询  →  检查文件是否存在
结果      →  文件永久堆积，无人清理

目标：数据库即状态
─────────────────────────────────────────────
任务触发  →  INSERT pipeline_tasks (status=pending)
任务运行  →  UPDATE status=running
任务完成  →  UPDATE status=done
任务失败  →  UPDATE status=failed, error_msg=...
下游轮询  →  SELECT WHERE status=done
结果      →  零中间文件，全程可查询可追溯

二、数据库设计
新建pipeline.db，独立于knowledge.db和trade_engine.db，职责单一：管理所有任务状态。
核心表：pipeline_tasks
sqlCREATE TABLE pipeline_tasks (
    task_id      TEXT PRIMARY KEY,
    -- 格式: {pipeline}_{step}_{date}_{agent}
    -- 例如: evening_step1_pnl_20260516_moheng

    pipeline     TEXT NOT NULL,
    -- 'morning' | 'evening' | 'backtest' | 'knowledge'

    step         TEXT NOT NULL,
    -- 'step1_pnl' | 'step2_winrate' | 'knowledge_extract' 等

    agent        TEXT NOT NULL,
    -- 'moheng' | 'mohan' | 'moxuan'

    run_date     TEXT NOT NULL,
    -- YYYYMMDD，每日任务的自然分区键

    status       TEXT NOT NULL DEFAULT 'pending',
    -- 'pending' | 'running' | 'done' | 'failed' | 'skipped'

    triggered_by TEXT,
    -- 触发来源: 'cron' | 'manual' | 'upstream_task_id'

    started_at   TEXT,
    finished_at  TEXT,
    duration_sec REAL,           -- 自动计算：finished_at - started_at
    error_msg    TEXT,           -- 失败时写入
    retry_count  INTEGER DEFAULT 0,
    payload      TEXT DEFAULT '{}',  -- 任务参数（JSON）
    result       TEXT DEFAULT '{}',  -- 任务产出摘要（JSON）

    created_at   TEXT DEFAULT (datetime('now', 'localtime')),
    updated_at   TEXT DEFAULT (datetime('now', 'localtime')),

    UNIQUE(pipeline, step, run_date, agent)
);

CREATE INDEX idx_tasks_pipeline_date ON pipeline_tasks(pipeline, run_date);
CREATE INDEX idx_tasks_status        ON pipeline_tasks(status);

四、核心模块：PipelineDB
Python# mozhi_platform/src/pipeline/pipeline_db.py

class PipelineDB:
    """流水线任务状态管理，替代所有 *.done / *.failed 文件"""

    def create_task(self, pipeline, step, agent, run_date,
                    triggered_by=None, payload=None) -> str:
        """
        创建任务记录，返回 task_id。
        替代：写 trigger_*.json 文件
        """

    def start_task(self, task_id) -> None:
        """
        标记任务开始运行。
        替代：心跳文件生成
        """

    def complete_task(self, task_id, result=None) -> None:
        """
        标记任务完成。
        替代：写 *.done 文件
        """

    def fail_task(self, task_id, error_msg) -> None:
        """
        标记任务失败。
        替代：写 *.failed 文件
        """

    def is_done(self, pipeline, step, run_date, agent=None) -> bool:
        """
        查询任务是否完成。
        替代：检查 *.done 文件是否存在
        """

    def wait_for(self, upstream_task_id, timeout_sec=300) -> bool:
        """
        轮询等待上游任务完成。
        替代：轮询文件系统
        """

    def get_daily_summary(self, run_date) -> dict:
        """
        获取指定日期所有任务的执行摘要。
        供墨涵 02:00 日报使用
        """

五、现有代码的迁移方式
最小化原则——只改任务的首尾两端，中间业务逻辑不动：
Python# 改动前（文件模式）
def run_evening_step1():
    trigger_file = "signals/tasks/evening_step1_moheng.json"
    done_file    = "signals/tasks/evening_step1_moheng.done"
    failed_file  = "signals/tasks/evening_step1_moheng.failed"

    if not os.path.exists(trigger_file):
        return
    try:
        # ... 业务逻辑不变 ...
        open(done_file, 'w').close()
    except Exception as e:
        open(failed_file, 'w').close()
    finally:
        os.remove(trigger_file)


# 改动后（数据库模式）
def run_evening_step1():
    db = PipelineDB()
    task_id = "evening_step1_pnl_{date}_moheng"

    if not db.is_done("evening", "step0", today()):
        return                          # 上游未完成，等待

    db.start_task(task_id)
    try:
        # ... 业务逻辑完全不变 ...
        db.complete_task(task_id, result={"rows": 42})
    except Exception as e:
        db.fail_task(task_id, error_msg=str(e))
每个任务函数头尾各约3行，业务逻辑零关联。

六、过渡期策略
不整体建议切换，分三步走：
第一步：双写期（1周）
  现有文件逻辑保留不动
  同时写入数据库
  验证数据库记录与文件状态一致

第二步：数据库主导期（2周）
  下游改为查询数据库
  文件仍然生成但不再被读取
  确认系统稳定

第三步：纯数据库期
  停止生成所有中间文件
  墨涵02:00任务清理存量文件
  文件系统彻底干净

七、墨涵02 :00任务的变化
引入pipeline.db后，02 :00任务大幅简化：
改造前：
  扫描 signals/tasks/*.done     → 统计 + 清理
  扫描 pipeline/heartbeat/      → 统计 + 清理
  扫描 pipeline/signals/audit/  → 统计 + 清理
  生成日报（从多个目录拼凑）

改造后：
  SELECT * FROM pipeline_tasks WHERE run_date = yesterday
  → 直接生成日报，无需扫描文件
  → 无需清理文件（根本不生成）
日报生成从“扫描文件系统拼凑”变成一条SQL：
sqlSELECT
    pipeline,
    COUNT(*) AS total,
    SUM(CASE WHEN status='done'   THEN 1 ELSE 0 END) AS done,
    SUM(CASE WHEN status='failed' THEN 1 ELSE 0 END) AS failed,
    AVG(duration_sec) AS avg_duration_sec
FROM pipeline_tasks
WHERE run_date = '20260516'
GROUP BY pipeline;

八、唯一需要保留为文件的东西
并非所有东西都应该入库，以下两类仍建议保留为文件：
保留为文件原因最终报告（*.md，人类阅读）文件更详细查看和分享归档压缩包（*.tar.gz）长期保存，数据库不适合存大文件
其余全部入库。

九、一句话总结

pipeline.db是模拟的神经系统，knowledge.db是知识的长期记忆，文件系统只存人类要读的最终波形。三者职责分离后，中间文件问题从根本上消失，而不是靠清理脚本对抗。
