# R003 — 回测方案先行规则（v2）

> v2变更：补充验收标准量化引导、确认流程定义、执行偏差记录机制、结构化模板字段。

## 一、规则陈述

**任何回测系统实验启动之前，必须先完成书面试验方案，并经审查确认后方可执行。**

未完成试验方案即开始回测的，视为违规操作，结果不可采信。

### 背景

2026-05-23 TSI 复权数据事故中：
- TSI回测在数据源从未验证的情况下直接运行（adj_factor全NULL未被发现）
- 三个标的的多标回测无方案，结果被覆盖丢失
- 回测结论（"+0.80%，不适合实盘"）基于错误数据

事后复盘表明：如果事先有试验方案，这些问题本可完全避免。

## 三、确认流程

### 3.1 填写审查

方案填写完成后，需经至少一名非拟定人的团队Agent审查：

| 角色 | 可审查内容 | 审查重点 |
|:----|:-----------|:---------|
| 墨涵🐘 | 所有方案 | 问题定义一致性、归档完整性 |
| 墨衡⚙️ | 执行方案 | 参数合理性、数据源正确性、可行性 |
| 墨萱🔍 | 测试方案 | 验收标准可验证性、结果捕获方式 |
| 玄知🔒 | 技术门槛方案 | 架构一致性、边界风险 |

**审查标准**：
- ✅ 数据源验证项全部完成（无未填项）
- ✅ 验收标准包含至少3项量化检查项
- ✅ 风险/已知问题如实描述

**审查记录方式**：方案文件内填写「填写审查记录」区块，或群消息确认。

**超时机制**：审查人限时2小时反馈，无反馈视为自动通过（事后可追认）。

### 3.2 执行后QA（墨萱专用，非必须但推荐）

- 回测执行完成后，可选由墨萱验证结果与方案一致性
- 如有执行偏差，偏差记录写入方案「执行后确认」区块
- QA结论不以方案通过与否为结果，而是记录「结果可采信/不可采信」

---

## 四、方案锁定规则

方案经审查确认后即为"已锁定"状态。锁定后**数据源、复权方式、标的**三者不可变更：

| 变更类型 | 处理方式 |
|:---------|:---------|
| 参数数值微调（如ma周期5→10） | 方案中追加变更说明即可 |
| 标的增删 | **需新方案** |
| 数据源/复权方式变化 | **需新方案** |
| 策略逻辑变化 | **需新方案 + R002** |

---

## 五、执行偏差处理

### 5.1 偏差定义

方案确认后实际运行的参数与方案不一致的，视为执行偏差。

### 5.2 偏差分级

| 偏差类型 | 级别 | 结果处理 |
|:---------|:----:|:---------|
| 参数微调（参数数值变化≤1个） | 轻度 | 记录偏差，结果可采信 |
| 日期偏移≤2个交易日 | 轻度 | 记录偏差，结果可采信 |
| 标的增减 | **重大** | 结果部分可采信（按标的独立评估） |
| 数据源/复权方式不一致 | **致命** | 结果不可采信，需重填方案重跑 |
| 策略逻辑变化 | **致命** | 走R002，需新方案 |

### 5.3 偏差记录方法

方案文件末尾追加「执行后确认」区块，如实填写偏差项。

偏差记录本身不构成违规（方案是计划，偏差是现实）。但**故意隐瞒偏差不记录**的，构成违规。

## 六、适用场景

| 场景 | 是否适用 | 说明 |
|:----|:--------:|:-----|
| 首次运行未经验证的策略 | ✅ **必须** | 数据源、参数、输出方式均需方案 |
| 更换数据源/复权方式后的重跑 | ✅ **必须** | adj_factor验证+Buy&Hold基准验算 |
| 多标的同时回测 | ✅ **必须** | 需明确结果捕获方式 |
| 参数微调后的重复回测 | ✅ | 可简化为变更说明，见附录A |
| 已有成熟策略的自动化日常运行 | ❌ | 成熟判断：连续运行≥3个月且未触发数据源变更 |
| 紧急故障验证（hotfix验证） | ❌ | 事后24h内补方案并标记HOTFIX |

### hotfix例外约束

- hotfix验证的结果产物**必须显式标记为 `HOTFIX`**（文件名/run_id/存档路径均加前缀）
- 事后24h内补填方案，内容至少覆盖：事发原因、做了什么修复、验证了什么
- hotfix结果不得与正式结果混用

## 七、正例（✅ 推荐）

| 正例 | 效果 |
|:----|:-----|
| TSI首次回测前：方案写明"数据源data/analysis.db，adj_factor未验证，需先验算Buy&Hold" | 提前发现问题 |
| 多标回测：方案写明"3标的，结果通过ArchiveManager归档至backtest_back_202605.db" | 结果可追溯 |
| 参数网格扫描：方案写明"10组参数×3标的×2周期，结果存入grid_results表" | 结果结构化 |

## 八、反例（🚫 禁止）

| 反例 | 问题 |
|:----|:-----|
| TSI回测直接查表，未验证adj_factor | ❌ 用未复权数据跑出错误结论 |
| 三标的同时回测无方案，backtest.db只保留最后一个 | ❌ 结果丢失 |
| 数据源变更后直接重跑，不验算Buy&Hold | ❌ 数字变化不可归因 |
| 方案与实际运行有偏差但不记录 | ❌ 失去可复现性 |
| hotfix结果混入正式结果库 | ❌ 污染数据 |

## 九、与其他规则的关系

- **R001（问题定义优先）**：本规则是 R001 在回测场景的具体化。R001 聚焦"这个回测要解决什么问题"，本规则聚焦"用什么数据、什么参数、如何记录结果"
- **R002（先需求后设计）**：如需开发新的回测功能或策略逻辑变化，先走 R002 需求分析，再执行本规则
