# T+21全量运行三层根因分析

基于现象描述文档 v5 中可观测事实，对三个问题进行客观的三层根因分析。

---

## 问题1：管线运行时被 SIGKILL 杀死

### 第一层（直接原因）

管线进程被操作系统 OOM killer 强制终止。系统总内存 15.6GB，事后查询可用内存仅 1.76GB（约11%），两次 SIGKILL 均发生在启动后 ~15-17 分钟，模式符合 OOM killer 典型行为特征。

### 第二层（间接原因）

系统内存资源被其他进程（尤其是 vLLM 推理服务）大量占用，导致管线进程启动时可用内存已不足，管线实际内存需求超出剩余可分配额度。两次运行的间隔（27分钟）足以释放前一次管线进程的 RSS，但不足以使系统整体可用内存恢复到安全水平——这表明内存瓶颈来自管线之外的常驻进程。

### 第三层（系统性原因）

1. **缺乏预运行环境检查**：启动管线前不做可用内存检查，OOM 发生后才能从状态推断而非预防。
2. **缺乏资源隔离**：管线与其他进程共享同一物理内存空间，无 memory cgroup / 容器机制隔离资源预算，一个进程的 OOM 可以影响其他进程。
3. **缺乏内存画像（profiling）**：管线进程自身在峰值时具体需要多少 RSS 未知——被杀时无实时内存快照，事后无法精确复盘。
4. **缺乏运行期监控/告警**：系统内存水位下降过程中无告警触发，直到 OOM killer 动手才被发现。Owner 也指出"无杀死时刻的实时内存快照"是核心证据缺口。

**交叉关系**：问题1 的数据产出（415个有效窗口）是问题2 的数据基础。SIGKILL 导致的覆盖不完整影响问题2 的验证范围，详见问题2 第三层。

## 问题2：黄金基线 verdict=FAIL（动量20日）

### 第一层（直接原因）

momentum_20d 因子在 415 个有效窗口上计算的 5 项基线指标中，仅 1 项通过阈值（mean_ic_abs=0.221≥0.02），其余 4 项均未通过：positive_ratio=0.487<0.50、ic_std=0.271>0.20、significant_ratio=0.248<0.30、half_life=0.47周∉[4,12]。Verdict=FAIL 是验证脚本基于数据的正确判定。

### 第二层（间接原因）

momentum_20d 因子在 A50 截面 2007~2026 近 20 年区间上的 IC 特征不符合黄金基线门槛——方向性仅略低于随机（positive_ratio=0.487）、波动性高（ic_std=0.271）、显著比例不足（significant_ratio=0.248）、IC 半衰期极短（0.47周 vs 期望 4~12 周）。这些指标反映的是因子本身的统计特性，而非数据质量问题。

### 第三层（系统性原因）

1. **无数据覆盖完整性预检**：验证阶段在执行 `compute_baseline_metrics()` 之前，没有检查 DB 中的数据是否覆盖了预期的完整日期范围。SIGKILL（问题1）导致第2次运行只产出到 2026-05-19，缺少最新几个月的截面，但这不影响 20 年尺度的统计结论（文档已确认 415 个有效窗口样本量充足）。
2. **黄金基线阈值校准依据缺失**：positive_ratio ≥ 0.50、ic_std ≤ 0.20、significant_ratio ≥ 0.30、half_life ∈ [4,12] 周的阈值来自 req_draft_v2 §5.3，但缺乏实证依据说明这些阈值对 A50 截面 IC 的合理性。对于动量因子而言，0.487 的 positive_ratio 是否在同类研究报告的典型范围内？half_life=0.47周 是否说明动量因子在 A50 上确实衰减极快而非计算错误？这些问题在现有流程中无环节验证。
3. **FAIL 后的升级路径缺失**：验证脚本判定 FAIL 后只是记录结果，没有触发进一步的诊断流程（如：与其他因子对比、检查同窗口期是否有因子通过、报告阈值是否需重新标定）。判定结果"死"在 JSON 文件中，缺乏自动化后续动作。

**交叉关系**：问题1 → 问题2（数据覆盖范围因 SIGKILL 缩小，但不改变结论）。问题2 与问题3 无依赖关系，但两者叠加使候选因子池缩减幅度扩大。

## 问题3：估值因子 IC=0（数据源缺失）

### 第一层（直接原因）

管线计算估值因子 IC 时，stock_daily 表中 pe、pe_ttm、pb 三列定义存在但值全部为 NULL（2007~2026 全部年份），ps_ttm、pcf_ttm 列更是完全不存在于表 Schema 中。因此估值 4 因子在所有截面上 rank_ic 均为 NULL 或 0。

### 第二层（间接原因）

1. **akshare 数据源不提供估值字段**：stock_daily 表的数据入库逻辑将从 akshare 接口接收到的值写入数据库。若接口返回的响应体中不含 pe/pb/pe_ttm 字段（或返回值为空），写入时使用 None 填充——这是数据采集逻辑的选择性缺失，而非采集后处理阶段丢失。
2. **ps_ttm 和 pcf_ttm 的 Schema 缺失**：这属于设计阶段的遗漏——stock_daily 表的列定义在设计时就没有纳入这两个估值字段，与数据采集阶段无关。

### 第三层（系统性原因）

1. **无数据源验证/质量门禁（最主要系统性缺陷）**：因子管线在设计时就依赖 stock_daily 表的 pe/pb/pe_ttm/ps_ttm/pcf_ttm 字段计算估值因子，但从未在数据采集入库后设置一道质量门禁来验证这些字段是否有有效值。如果在首次 ingestion 后立即检查 `SELECT COUNT(*) FROM stock_daily WHERE pe IS NOT NULL LIMIT 1` 返回 0，这个问题在管线开发阶段就会被发现。
2. **因子设计与数据源可用性脱钩**：估值因子的开发（factor_valuation.py 或对应模块）没有与数据源依赖清单同步。管线假设"数据表定义了这些列"= "这些列有数据"，忽略了列定义和数据填充之间的鸿沟。
3. **无数据源替代方案（Backup Data Source）**：akshare 不提供 A50 成分股的历史估值截面数据这一事实应当被提前识别——仅依赖单一数据源是架构风险。未建立替代数据源（如 Wind、tushare pro、聚宽离线数据）的接入通道。
4. **Schema 设计审核缺失**：ps_ttm 和 pcf_ttm 列在 stock_daily 表设计时未纳入，说明因子的数据需求与数据库 Schema 之间没有对齐流程。估值 4 因子需要的 4 个字段中有 2 个连列都没建，属于数据建模阶段的遗漏。

**交叉关系**：问题3 与问题1/问题2 **无依赖关系**，是独立的数据源问题。即使管线完整运行（无 SIGKILL）、动量基线通过（PASS），估值因子 IC 仍为 0。

## 问题间交叉关系总览

| 路径 | 关系类型 | 说明 |
|:----:|:-------:|:----|
| 问题1 → 问题2 | 数据来源影响 | SIGKILL 导致数据覆盖不全（缺少 2026-05-19 后数据），但不改变 verdict=FAIL 的结论 |
| 问题1 → 问题3 | 无关系 | 完全独立 |
| 问题2 → 问题3 | 无关系 | 独立因子集，判定互不干扰，但叠加效应扩大候选因子池缩减幅度 |

**综合建议修复优先级**：
1. **问题1**（SIGKILL/OOM）：最高优先——阻塞所有后续全量运行。需要快速止血：预检查可用内存、设置 swap、或暂停非关键进程（如 vLLM）
2. **问题3**（估值因子数据源）：高优先——因子管线的数据依赖链断裂。需要确认估值数据的获取方案
3. **问题2**（黄金基线 FAIL）：中优先——如果阈值已确认无误，则表明 momentum_20d 在 A50 截面上的表现确实不达标，属于因子淘汰的决策问题而非系统故障
