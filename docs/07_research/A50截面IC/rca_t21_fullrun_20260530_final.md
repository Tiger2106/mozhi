## 问题1：管线运行时被 SIGKILL 杀死

### 第一层（直接原因）【三方共识】

操作系统 OOM killer 判定内存耗尽，向管线进程发送 SIGKILL 信号强制终止。

**可观测证据（三方一致认同）**：
- 系统总内存 15.6GB，事后查询可用仅 1.76GB（约 11%）
- 两次杀死时间点均在启动后 ~15-17 分钟，符合 OOM killer 行为模式
- 第1次（marine-s，16:43~17:00）：被杀
- 第2次（mellow-o，17:27~17:42）：被杀

**⚠️ 核心证据缺口（三方均确认）**：
杀死时刻无实时内存快照。事后 1.76GB 不可代表死前时刻的内存状态。OOM 推断是基于"事后低水位 + 行为模式匹配"的概率性结论，非实证性结论。这是 RCA 自身的置信度约束。

### 第二层（间接原因）

1. **系统内存被常驻进程大量占用**（三方共识）
   - 两次运行间隔 27 分钟，释放前一次管线 RSS 后仍被杀 → 系统本就处于高压状态
   - 高度怀疑 vLLM 推理服务是主要内存消耗源（三方推论一致，但无实证）

2. **管线自身内存需求未知**（三方共识）
   - 管线峰值 RSS 未被记录或 profiling
   - 不确定是管线本身需求大，还是可用内存太小，还是叠加效应

3. **INSERT OR IGNORE 逐截面提交的意外保全效应** [侧重: 墨萱]
   - 墨萱特别指出：每次截面写入后立即 commit 的机制确保杀死时已完成的数据不丢失
   - 这是"幸运的巧合"——如果采用批量写入（全部计算完再 commit），两次 SIGKILL 将导致 0 数据可用
   - 墨衡和玄知未将此作为第二层写入，但均认可此为事实

4. **二次运行是串行而非并发** [独有: 玄知]
   - 玄知明确判断：从时间线看（16:43 被杀 → 17:27 启动，间隔 27 分钟），两次运行是先后串行
   - 排除了并发运行可能导致数据交叉污染的可能性

5. **mellow-o 在"验证阶段"被杀的模糊性** [独有: 玄知补充]
   - 现象描述 v5 未明确该阶段具体含义
   - 若验证在 IC 写库之后 → 428 截面已完成写入，数据可信
   - 若验证是管线内部中间校验 → 428 截面可能不完整
   - 需要玄知或 Owner 确认

### 第三层（系统性原因）

三方对第三层原因的判断高度一致，以下按防御层次合并，并标注各方的侧重：

1. **【预防层】缺乏预运行环境健康检查**（三方共识，均列为P0/P1）
   - 管线启动前不做可用内存/磁盘/CPU 检查
   - 如果启动时检查 `psutil.virtual_memory().available` 并判断可用 < 4GB 则拒绝启动，OOM 可被完全预防
   - [侧重: 墨衡/玄知] 强调此检查缺失是"第一次犯错就被杀死"的根因

2. **【隔离层】缺乏进程级资源约束**（三方共识，均列为P0/P1）
   - 管线与 vLLM 共享同一物理内存空间，无 cgroup/Docker 资源限制
   - [侧重: 玄知] 强调"共享资源无配额"是最根本的系统级缺陷
   - [侧重: 墨萱] 提出应在容器或 systemd scope 内限制管线内存上限

3. **【恢复层】缺乏断点续传/checkpoint 机制**（三方共识）
   - SIGKILL 不可捕获、不可处理
   - 管线没有 checkpoint 机制——被杀后只能从头重跑
   - 第3次使用 `--no-run` 属于手工变通，不是系统自带的恢复能力
   - [独有: 墨萱] 提到自动重试（带更小批次或更低内存模式的降级运行）缺失

4. **【监控层】缺乏运行期内存监控和告警**（三方共识）
   - 系统内存从初始水位持续下降至 1.76GB 的整个过程无任何告警
   - OOM killer 是最后一道防线而非预警机制
   - 死后无 RSS 快照可追溯 → 无法判断修复方向

5. **【审计层】管线进程内存画像缺失**（三方共识）
   - 在 `memory_profiler` 或 `tracemalloc` 缺失的情况下，无法判断峰值内存需求

### 交叉关系【三方统一】

问题1 直接影响问题2 的数据覆盖范围（缺少 2026-05-19 后的最新截面），但对问题3 无影响。

## 问题2：黄金基线 verdict=FAIL（动量20日）

### 第一层（直接原因）【三方共识】

验证脚本 `validate_golden_baseline.py` 基于 momentum_20d 因子的有效窗口计算 5 项基线指标，判定 verdict=FAIL：

| 指标 | 实际值 | 阈值 | 通过? | 偏离程度 |
|:----|:-----:|:----:|:----:|:--------:|
| mean_ic_abs | 0.221 | ≥0.02 | ✅ PASS | +1005%（远超）|
| positive_ratio | 0.487 | ≥0.50 | ❌ FAIL | -2.6%（边界）|
| ic_std | 0.271 | ≤0.20 | ❌ FAIL | +35.5%（超限）|
| significant_ratio | 0.248 | ≥0.30 | ❌ FAIL | -17.3%（不足）|
| half_life | 0.47周 | 4~12周 | ❌ FAIL | -88.3%（极短）|

**共同声明**：FAIL 是脚本基于数据的正确输出，不是软件缺陷。

**⚠️ 关键前置条件（三方共识，但墨衡最高度关注）**：
数据完整性（541 截面是否存在跨运行重复）是 verdict 可信度的前置条件。需执行 SQL 确认去重后，FAIL 才能作为正式判定。详见 §补充2。

**说明**：
- 墨衡v2将此结论标记为「待确认(pending)」，因数据完整性（G2）未验证
- 墨萱和玄知保留了结论但加注前提条件（需数据完整性校验通过）
- [侧重: 墨衡] 认为 positive_ratio=0.487 距阈值 0.50 仅差 2.6%，数据去重可能翻转结论
- [侧重: 墨萱/玄知] 认为即使有微量重复截面也不改变 FAIL 的整体结论，因为 half_life=0.47 与下界 4 周相差一个数量级，非重复数据可以解释

### 第二层（间接原因）

1. **momentum_20d 在 A50 截面上的 IC 特征不达标**（三方共识）
   - positive_ratio=0.487：略低于 0.5 随机线，近乎无方向性
   - ic_std=0.271：IC 波动高，信号不稳定
   - half_life=0.47周（≈3.3交易日）：动量信号衰减极快
   - [共有认识] 这些数字反映 A50 大盘股截面中动量因子的真实表现——大市值股票价格动量普遍弱于小盘

2. **已有数据覆盖范围不足以解释 FAIL**（三方共识但有不同侧重）
   - 415 个有效窗口覆盖 2007~2026 近 20 年，样本量充足
   - [侧重: 墨萱/玄知] 明确认为：数据覆盖完整性与 verdict 无关——即使有全部截面，动量因子表现仍然不达标
   - [侧重: 墨衡] 数据完整性未验证前，不能过早排除"数据失真导致 FAIL 误判"的可能性

3. **415 有效截面的数据血统**（三方共识但有不同清晰度）
   - 三方均引用"415 = 从 428/541 截面中去除 rank_IC=NULL、num_stocks<30、计算异常后得到"
   - 但**三方均承认未独立执行 SQL 去重验证**
   - [独有: 墨萱] 提供了最清晰的字段映射链：`a50_daily_ohlcv → factor_valuation.py → ETL(FIELD_MAP) → stock_daily → akshare`

### 第三层（系统性原因）

1. **黄金基线阈值缺乏实证校准**（三方共识，但分歧点如下）

   **三方一致的判断：**
   - 阈值的来源和实证依据不透明
   - 需要补充 A50 截面 IC 实证研究以校准阈值
   
   **分歧点：**
   - **墨衡**：threshold 需要标定（验证脚本[4,12] vs req_draft_v2 >12 本身就是矛盾的，见 G8）
   - **墨萱**：half_life=0.47 离 4 周阈值很远，即使是放松标准也无法通过；更关注的是为何选择 momentum_20d 作为黄金基线
   - **玄知**：最激进的立场——认为 [4,12] 周半衰期下限可能不适合日频截面，建议改为 [1,4] 周；并提出 z-score/percentile 的相对评判方式替代绝对二值判断

2. **数据覆盖完整性预检缺失**（三方共识）
   - 验证阶段没有检查 DB 数据是否覆盖预期全部日期范围
   - 一个简单的 `SELECT MIN(trade_date), MAX(trade_date) FROM a50_cross_ic_result` 可验证

3. **FAIL 后无诊断/升级路径**（三方共识）
   - 验证脚本判出 FAIL 后仅记录到 JSON，无任何后续动作
   - 需要触发对照因子分析、检查同窗口期其他因子通过率、发出通知

4. **黄金基线因子选择逻辑值得审视** [侧重: 墨萱]
   - 为何选择 momentum_20d 作为黄金基线单一因子？
   - 在 A50 超大盘截面中动量效果历来偏弱，学术界有共识
   - 单一因子作为"门神"存在单点故障风险

5. **half_life 阈值矛盾（验证脚本 [4,12] vs req_draft_v2 >12 周）** [独有: 墨衡 G8]
   - 两个定义不一致，相差整一个区间
   - 本次不影响结论（half_life=0.47 无论按哪个阈值都是 FAIL）
   - 但这是 pipeline 设计层面的系统性风险，重跑前需统一

### 交叉关系【三方部分分歧】

- **三方共识**：问题2 与问题3 无依赖关系
- **分歧点**：问题1 对问题2 的影响程度
  - **墨衡v2**：更谨慎——数据完整性未确认前，FAIL 的正式判定应挂起。SIGKILL 可能造成数据重复或缺失，影响统计量
  - **墨萱**：数据覆盖不完整但**不影响 verdict 稳定性**，415 窗口样本量足够
  - **玄知**：明确表态——数据覆盖范围**不是问题2的根因**，最新几个月截面不会将 positive_ratio 从 0.487 拉过 0.50

## 问题3：估值因子 IC=0（数据源缺失）

### 第一层（直接原因）【三方共识】

估值 4 因子（pe_ttm、pb、ps_ttm、pcf_ttm）计算截面 IC 时，底层数据字段全部不可用：

| 因子 | 问题类型 | 列状态 | 值状态 |
|:----:|:--------:|:------:|:------:|
| pe_ttm / pb | 数据填充缺失 | 列存在 | **全部 NULL**（2007~2026全覆盖）|
| ps_ttm / pcf_ttm | Schema 设计缺失 | **列不存在** | 完全无法写入 |

因此所有截面的估值因子 rank_ic 均为 NULL → IC=0。

### 第二层（间接原因）【三方共识，但墨萱最详细】

按数据链追溯（从 pipeline 向上游逐层排查）：

1. **因子计算层**（pipeline IC 计算模块）：引用 `a50_daily_ohlcv.pe`、`pb`，但 `ps_ttm` 和 `pcf_ttm` 引用不存在于表中的列名 → 返回 NULL 或报错
2. **ETL 层**（`etl_a50_daily.py`）：`FIELD_MAP` 中仅映射 `pe_ttm→pe` 和 `pb→pb`，**无 ps_ttm/pcf_ttm 映射项**
3. **原始数据层**（`market_data.db.stock_daily`）：`pe_ttm` 和 `pb` 全部为 NULL，`ps_ttm` 和 `pcf_ttm` 列**不存在**
4. **数据源层**（akshare）：不返回或返回空值

**三方均认同的两类问题区分**：

| 类别 | 因子 | 性质 | 修复难度 |
|:----:|:----|:----|:-------:|
| A类 | pe_ttm/pb | 列存在但数据源无值 | ⚠️ 需补充数据源 |
| B类 | ps_ttm/pcf_ttm | Schema 本身缺失 | ❌ 需 DDL + ETL + 计算同步 |

[侧重: 墨萱] 提供了最完整的全链路分析——从设计文档→编码→ETL→DB→验证的全链条断裂案例。[侧重: 玄知] 更关注"为什么到全量运行时才发现"的流程缺失。

### 第三层（系统性原因）

1. **数据质量门禁（QA Gate）缺失** ⭐（三方共识，均列为最高优先级系统性缺陷）
   - 管线在计算 IC 之前，不对输入数据做任何"hydration check"
   - 一个简单的 `SELECT pe, pb FROM a50_daily_ohlcv WHERE pe IS NOT NULL LIMIT 1` 即可在启动时 FAIL FAST
   - **当前状态**：管线正常运行 → 输出 18 因子 IC → 4 个全为零 → **没有报错、没有日志警告、没有质量标记**
   - [侧重: 玄知] 提问：估值因子 IC=0 是 T+21 全量运行才首次产出的，还是在之前增量运行中就一直产出 NULL？如果是后者，管线开发阶段的函数调试就应该发现问题——说明单元测试覆盖但可能未检查数据有效性，或根本无单元测试

2. **因子设计与数据源可用性完全脱钩**（三方共识）
   - `design_v2.md` §2.2.1 中 pe_ttm 和 pb_lf 已标注 "⚠️ 字段存在，质量待验证"——设计阶段已识别风险，但**全量运行前未执行验证**
   - ps_ttm 和 pcf_ttm 不在设计文档的任何因子清单中——说明代码自行添加但未同步更新 DDL/ETL/设计文档
   - [独有: 墨萱] 精准指出这是"从设计→开发→ETL→验证全链条断裂"

3. **单一数据源依赖风险**（三方共识）
   - 整个估值因子管线仅依赖 akshare 一个数据源
   - 该数据源不提供 A50 成分股的历史估值截面数据
   - [侧重: 玄知] 建议：建立多数据源策略或至少对每个因子的输入列做 data availability sign-off

4. **Schema 设计审核流程缺失**（三方共识）
   - `a50_daily_ohlcv` DDL 中没有 `ps_ttm`/`pcf_ttm` 列，但计算代码引用了这些列名
   - 需要一个 "Schema ↔ Factor 双向对齐" 的签核流程

5. **摸底调查（Survey）的触发条件过窄** [侧重: 墨萱]
   - `data_survey_report.md` 只调查了 stock_daily 已有字段的缺失率，未调查"因子需要但 stock_daily 不存在的列"
   - 如果摸底流程包含因子需求 Schema 与数据表 Schema 的差集对比，ps_ttm/pcf_ttm 的缺失在摸底阶段就会被发现

### 交叉关系【三方共识】

- 问题3 与问题1、问题2 **完全独立**
- 即使无 SIGKILL 且管线完整运行，估值因子仍然产出 NULL
- 叠加效应：候选因子池从 15 → 10（基线 FAIL 淘汰 1 + 数据缺失废掉 4 个估值因子 — 但其中 momentum_20d 的淘汰结论待数据完整性确认）

## 问题优先级排序

### 三方均有排序，以下合并为统一优先级表

| 优先级 | 系统缺陷 | 关联问题 | 提出方 | 合理性说明 |
|:------:|:--------|:-------:|:------:|:----------|
| **P0** | 数据完整性校验（去重 SQL 检查） | 问题2 前置条件 | **三方一致** | 如果存在重复截面，基线判定推倒重来。一条 SQL 即可确认，是最低成本最高回报的紧急操作 |
| **P0** | 预运行环境健康检查缺失 | 问题1 | **三方一致** | 阻塞所有后续全量运行。不修管线跑不完 |
| **P0** | 数据质量门禁（QA Gate）缺失 | 问题3 | **三方一致** | 4 个因子全 0 无任何告警——静默失败是最危险的缺陷 |
| **P1** | 资源隔离（Docker/cgroup） | 问题1 | **三方一致** | 防止 vLLM 等进程内存泄漏波及其他管线 |
| **P1** | 因子设计与数据源可用性脱钩 | 问题3 | **三方一致** | 从因子清单对齐到 DB Schema、ETL 映射、数据源可用性 |
| **P1** | 运行期监控告警 | 问题1 | **三方一致** | 系统内存下降全程无预警 |
| **P1** | 估值数据源替代方案 | 问题3 | **玄知独有** | 单一数据源单点故障 |
| **P2** | 黄金基线阈值校准 | 问题2 | **三方一致** | 阈值需要实证依据或改为相对评判方式 |
| **P2** | FAIL 后诊断升级路径 | 问题2 | **三方一致** | FAIL 后自动执行对照因子分析+通知 |
| **P2** | 估值因子日志补全 | 问题3 | **墨衡独有(G10)** | 日志不补齐则问题3 的根因分析仍建立在假设上 |
| **P2** | Schema 审核流程 | 问题3 | **墨萱/玄知** | 代码变更需同步更新 DDL/ETL/设计文档 |
| **P2** | half_life 阈值矛盾统一 | 问题2 | **墨衡独有(G8)** | 验证脚本[4,12] vs req_draft_v2 >12，重跑前需统一 |
| **P3** | 摸底调查报告范围扩展 | 问题3 | **墨萱独有** | 增加因子需求与数据表 Schema 差集对比 |
