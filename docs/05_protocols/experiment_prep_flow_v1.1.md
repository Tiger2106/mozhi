## 流程概览

```
Step 0 ─── 试验就绪检查（墨衡）
  │           ├── 回测参数匹配试验方案
  │           ├── 数据库数据一致性
  │           └── 环境/API就绪
  │           ↓
Step 1 ─── 数据QC验证（墨衡）
  │           ├── 数值合理性检查（含数据指纹）
  │           ├── adj_factor/涨跌幅/成交量异常
  │           └── 全量扫描报告
  │           ↓
Step 2 ─── 代码准备+Dry-run（墨衡）
  │           ├── 脚本创建/修改
  │           ├── dry-run 全流程（含审核标准）
  │           └── 产出文件清单
  │           ↓
Step 3 ─── 墨萱独立复核（墨萱，TMPL-003模板）
  │           ├── 检查 Step 0~2 产出
  │           ├── 判定条件通过/不通过
  │           ├── 列出缺失项
  │           └── 时限≥5min
  │           ↓
Step 4 ─── 墨涵门控决策
              ├── 墨萱通过 → 直接放行
              ├── 墨萱条件通过 → 列缺失项 → 退回墨衡补齐 → 循环（差异比对）
              └── 墨萱不通过 → 告警Owner介入 → 暂缓执行
```

## Step 0：试验就绪检查（墨衡）

**执行人**: 墨衡
**时长**: ~5-10 分钟
**前置**: 试验方案已终签、数据源已就绪

### 墨衡检查清单

| 检查项 | 要求 | 判定标准 |
|:-------|:-----|:---------|
| 回测参数 | 是否与终签方案一致 | 标的范围、因子定义、回测时段、持有期、波动率阈值 5 项逐项核对 |
| 数据库数据 | stock_daily 数据覆盖全部标的+时段 | 无缺失期、无异常 NULL |
| adj_factor | 个股 adj_factor 非 NULL | QC-003 验证通过 |
| 交易日历 | 覆盖回测区间 | trading_calendar 齐备 |
| API就绪 | Tushare pro 可用 | API 响应正常、无熔断残留 |
| 环境就绪 | market_data.db / pipeline_cache.db 可写 | 2 个关键库文件是否存在、大小是否合理 |

### 产出文件

```
reports/{task_id}_readiness_check.md
```

### 结果传递

墨衡输出检查报告之后，进入 Step 1。

## Step 1：数据QC验证（墨衡）

**执行人**: 墨衡
**时长**: ~5 分钟
**前置**: Step 0 检查通过

### 墨衡检查清单

| 检查项 | 方法 | 通过条件 |
|:-------|:-----|:---------|
| 数值合理性 | 全量扫描 stock_daily | 无 NULL 值、无超范围值（涨跌幅>±20%、close∉[0.01,5000] 等） |
| adj_factor 异常跳变 | 逐日比较 adj_factor 变化率 | 变化率 < 50% 或确认属除权除息事件 |
| volume/amount 异常 | 成交量 vs 30日均值 | 未超过 50 倍均值 |
| 重复主键 | (ts_code, trade_date) 唯一性 | 无重复 |
| 字段完整性 | 关键字段 + 辅助字段 | 无 NULL |

### 产出

```
reports/{task_id}_qc_validation.md
```

包含：
- 各标的 QC 通卡表、异常点清单（如有）、总 verdict（PASS/REPORT/FAIL）
- 【数据指纹】文件头部记录数据版本信息：
  - 数据获取时间戳
  - 数据范围（最小/最大 trade_date）
  - 总行数、标的数
  - 建议：记录 stock_daily 表全量行数的哈希摘要（MD5/SHA256），使未来回放时有明确的"此时的数据长什么样"参考锚点

## Step 2：代码准备 + Dry-run（墨衡）

**执行人**: 墨衡
**时长**: ~10-15 分钟
**前置**: Step 1 通过

### 墨衡操作

| 步骤 | 说明 |
|:----|:------|
| 2.1 | 根据试验方案创建/修改回测脚本（通常 3-6 个文件） |
| 2.2 | 执行 dry-run 验证全流程不崩溃 |
| 2.3 | 确认输出格式符合预期（IC 表、因子值、回测报告） |
| 2.4 | 列出所有新建文件路径和用途 |

### Dry-run 审核标准（三要素）

墨衡在执行 dry-run 后，必须确认以下三项：

| 审核要素 | 判定标准 | 说明 |
|:---------|:---------|:------|
| ① 无崩溃 | dry-run 无异常堆栈/错误退出 | 非预期异常必须解决，不跳过 |
| ② 产出文件存在且非空 | 预期输出文件路径存在、文件大小 > 0 | 缺文件/空文件说明逻辑断点 |
| ③ 关键指标在合理范围 | 回测报告中年化收益、夏普、最大回撤在合理值范围 | 异常极值说明配置错误或数据异常 |

任何一项不满足 → 修复后重新 dry-run。

### 产出

- 回测脚本文件（通常 3-6 个 .py 文件）
- Dry-run 日志（含三项审核标准确认状态，如 `dry-run: checks [✅/✅/✅]`）

## Step 3：墨萱独立复核（墨萱）

**执行人**: 墨萱
**时长**: **≥5 分钟**（防止走过场）
**前置**: Step 0~2 全部完成并通知墨萱

### 复核模板

Step 3 输出使用 **TMPL-003 清单格式**（不得自由发挥），内容包括：

### 墨萱检查清单

**必检项目**（方案评审时已确定的检验项）和 **补充必检**（每次都必须检查的基础项）：

#### 方案层检验（来自 Stage 2 方案评审的确认清单）

| 检查维度 | 说明 | 通过条件 |
|:---------|:-----|:---------|
| 敏感性分析 | 关键参数变化时结果是否稳定 | 参数范围（如±5%/±10%）已在方案中界定 |
| 多重比较校正 | Bonferroni / FDR BH 是否应用 | 方法已在方案中确定 |
| 稳定性集成 | L3 层检验 | 阈值（如 3/4 或 4/4）已在方案中确定 |
| 前视偏差 | 无未来信息泄漏 | 验证方法（如 Bootstrap 排列因子值/排列IC序列）已在方案中确定 |
| 版本标记 | version_tag 可追溯 | 代码+配置 version_tag 是否已记录 |

> ⚠️ **若方案评审时未确定上述检验项清单**，墨萱应将其标记为**方案级缺失项**，退回 Stage 2（方案评审阶段）补充，不得在试验准备阶段自行定义。

#### 数据层校验（TMPL-003 核心要求）

| 检查项 | 方法 | 通过条件 |
|:-------|:-----|:---------|
| **DB 交易表/持仓表/净值表行数校验** | 逐表比对预期行数与实际行数 | 偏差 = 0 |
| **报告数据与 DB 交叉验证** | 年化收益、夏普、最大回撤 3 项逐项比对 | 偏差在 TMPL-003 容差范围内 |
| **version_tag 与实际代码 hash 比对** | version_tag vs git HEAD/代码内容哈希 | 严格匹配（验证而非仅确认记录） |

#### 输出文件校验

| 检查项 | 方法 | 通过条件 |
|:-------|:-----|:---------|
| Step 0 产出 | `readiness_check.md` 存在且内容完整 | 各检查项有明确 verdict |
| Step 1 产出 | `qc_validation.md` 存在、QC verdict 为 PASS | 包含数据指纹 |
| Step 2 产出 | dry-run 日志三项审核全部通过 | dry-run: checks [✅/✅/✅] |

### 判定

| verdict | 含义 | 处理 |
|:--------|:-----|:-----|
| ✅ **通过** | 无任何问题 | 直接进入 Step 4 |
| ⚠️ **条件通过** | 发现 N 项缺失/问题 | 列出缺失项 → 退回墨衡修复后重新复核 |
| ❌ **不通过** | 严重问题 | 告警Owner介入，暂缓执行 |

### 退回重审规则

- 退回后墨萱重审时，**只检查修复项对应的维度**，已通过的维度不做全量复查（除非修复波及导致连锁影响）
- 墨萱在重审时应在 verdict 中显式标注"本次为差异比对"+"受影响的检查维度"

## Step 4：墨涵门控决策（墨涵）

**执行人**: 墨涵
**前置**: Step 3 完成

### 门控逻辑

```
收到墨萱复核结果
         │
    ┌────┴────┐
    │         │
  ✅ 通过   ⚠️ 条件通过   ❌ 不通过
    │         │              │
    │    ┌────┴────┐         │
    │    │         │         │
    │  缺失项     Owner      │
    │  ≤3项       >3项       │
    │    │         │         │
    │    │    ┌────┴────┐    │
    │    │    │         │    │
    │    │  Owner     Owner  │
    │    │  裁定：   裁定：   │
    │    │  补齐→   暂缓      │
    │    │  循环      │      │
    │    │    │       │      │
    │    └────┘       │      │
    │         │       │      │
    │    ┌────┘       │      │
    │    │            │      │
    └────┴────────────┴──────┘
         │
    宣布放行（群内通知）
         │
    通知墨衡：执行开始
```

### 墨涵决策流程

| 墨萱 verdict | 缺失项数量 | 墨涵行动 | 时间代价 |
|:----------:|:---------:|:---------|:--------:|
| ✅ 通过 | 0 | 直接放行，通知墨衡执行 | 0 |
| ⚠️ 条件通过 | ≤3 | 退回墨衡补齐，通知+要求墨萱重新复核（差异比对） | ~5-10min |
| ⚠️ 条件通过 | >3 | 上报Owner，Owner裁定补齐/暂缓 | 视Owner响应 |
| ❌ 不通过 | — | 上报Owner，暂缓执行，Owner裁定方向 | 视Owner响应 |

### 循环规则

- 退回墨衡补齐后 → 必须回到 Step 3 由墨萱重新复核确认
- 墨萱重审时采用**差异比对**模式（只检查修复项+受波及维度）
- **墨涵不得代修、不得跳过墨萱直接放行**
- 同一试验最多退回 2 次，第 3 次仍不通过 → 告警Owner裁决
