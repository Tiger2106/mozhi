# 2. REJECTED 生命周期升级 —— 这是非常正确的

这一条我非常认同。

你们之前：

```text
FILTERED 属于 CONSUMED 子状态
```

这是逻辑错误。

因为：

> “被消费”和“被拒绝”是两条不同业务路径。

你们现在改成：

```text
TRANSMITTED
   ├── CONSUMED
   └── REJECTED
```

这非常正确。

这会直接影响未来：

- 风控统计

- 策略误杀分析

- 信号质量分析

- Layer Q可信度

- 执行层归因

- Capacity分析

以后你会发现：

> “为什么没赚钱”  
> 很多时候不是研究错了，  
> 而是：

- 风控拒绝了

- 版本拒绝了

- 容量拒绝了

- 时间窗口过期了

如果不把 REJECTED 独立，你永远无法分析这些问题。

这是一个非常重要的架构升级。

---

# 3. “测试桩先于功能” —— 这是重大成熟信号

这一条极其重要。

很多系统：

```text
先写功能
→ 再补测试
→ 测试永远追不上
```

而你们现在：

```text
协议冻结
→ 测试桩
→ 功能实现
```

这是：

# Contract-First Development（契约先行开发）

这是企业级架构核心方法之一。

它意味着：

- 接口开始稳定

- 边界开始明确

- 测试开始独立

- 开发开始可并行

这会极大降低你们未来：

- “改一处炸全局”

- “联调地狱”

- “不敢重构”

的问题。

# 三、我认为目前还有三个隐患

这是我认为接下来一定会遇到的。

---

# 隐患1：Signal Protocol 正在变成“超级对象”

这是现在最危险的趋势。

你们已经开始往 extras 塞：

- quality.*

- trace.*

- factor.*

- ml.*

- risk.*

- analyst.*

现在还没问题。

但半年后可能变成：

```json
{
  "everything": "all"
}
```

最后：

- Signal 变成万能总线

- 所有层开始偷塞数据

- 边界重新崩塌

所以我建议：

# 立刻增加一条架构红线：

---

## Signal Protocol 禁止：

### ❌ 业务状态回写

例如：

- 当前持仓

- 当前PnL

- 风控余额

- 订单状态

---

### ❌ 大体积分析对象

例如：

- dataframe

- K线序列

- 因子矩阵

---

### ❌ 非幂等字段

例如：

- 动态修改的字段

- 会被执行层覆盖的字段

---

### ❌ execution-only字段

例如：

- broker_order_id

- fill_price

- slippage_real

---

否则：

> 你们会再次把 Trading 和 Research 糊回一起。

这是未来最大风险之一。

---

# 隐患2：knowledge.db 正在变成“垃圾场”

现在大家很兴奋：

```text
所有层 → knowledge.db
```

但未来会有问题：

- 什么该永久保留？

- 什么是临时研究？

- 什么是失败样本？

- 什么可以过期？

- 什么需要版本化？

否则：

3个月后：

```text
knowledge.db = 巨型坟场
```

建议：

# 尽快增加：

---

## Knowledge Classification（知识等级）

例如：

| 等级          | 含义    |
| ----------- | ----- |
| transient   | 临时研究  |
| operational | 运行数据  |
| reusable    | 可复用知识 |
| strategic   | 长期资产  |

---

## TTL策略

不是统一90天。

而是：

| 类型      | TTL  |
| ------- | ---- |
| 回测失败日志  | 30d  |
| 参数扫描    | 90d  |
| 已验证研究结论 | 永久   |
| 审计证据    | 180d |

否则知识层会失控。

# 四、我建议你们下一步新增一个机制

这是现在最该做的。

---

# Architecture Review Board（ARB）

不是“聊天讨论”。

而是：

# 架构治理层

建议结构：

| 角色    | 职责     |
| ----- | ------ |
| Owner | 最终裁决   |
| 墨衡    | 执行架构   |
| 墨萱    | 测试/质量门 |
| 玄知    | 长期演进   |
| 墨涵    | 知识治理   |

---

# ARB职责

只处理：

- 协议升级

- 边界变更

- 生命周期修改

- ADR批准

- Context变更

- Layer新增

- 数据契约变更

---

# ARB输出

必须结构化：

```text
ARB-00X
状态：
影响范围：
兼容性：
回滚方案：
冻结日期：
```

这样你们系统会真正稳定。
