# 问题定义流水线 v1.0 — 墨枢定制版

> **作者**：墨涵
> **日期**：2026-05-31
> **来源**：评估 Claude 问题定义流水线后，针对墨枢团队定制的版本
> **状态**：v1.0 最终版

---

## 一、Claude 流水线评估

### 1.1 值得保留的优点

| # | 优点 | 说明 |
|:--|:-----|:-----|
| 1 | 现象与原因分离 | Stage 1 禁止出现原因分析和解决方案，强制团队先看清事实再下结论 |
| 2 | 自检机制 | Q1-Q7 是最低质量门槛，特别是 Q7 数据口径防混用 |
| 3 | 双人查证 | 墨衡执行 + 墨涵复核，防止查证结果被误解 |
| 4 | 边界声明 | 各阶段均要求标注不确定项和不在范围内，防止范围蠕变 |
| 5 | 超时熔断 | 每阶段有明确超时上限，超限即告警 Owner |
| 6 | 结构化交付 | 每阶段有固定模板，减少自由发挥的空间 |

### 1.2 不适合我们团队的地方

| # | 问题 | 处理 |
|:--|:-----|:-----|
| 1 | 阶段过多（8阶段） | 压缩为5阶段（模式A）|
| 2 | 并行 spawn（Stage 2a/2b） | 改为全串行，匹配 spawn_template_v1.0 |
| 3 | 墨萱角色偏移 | 保留"链条审查"，明确与QA角色对齐 |
| 4 | 玄知重复出现（2次） | 合为1次（阶段2 结构审查） |
| 5 | 缺少 Owner 签署 | 新增 |
| 6 | 缺少知识审查 | 阶段5内嵌 |
| 7 | 仅覆盖 Bug/事故类问题 | 新增模式B（Initiative） |
| 8 | 与现有 spawn 协议不兼容 | 全部串行通信 |

### 1.3 适用场景映射

Claude 流水线解决的是编码/测试过程中发现了异常（Incident 类问题）。
我们已有的 4 步流程（现状→后果→根因→陈述）解决的是新启动一个业务任务（Initiative 类问题）。

两者互补，不能互相替代。

## 二、墨枢问题定义流程（双模式）

### 2.1 流程总览

```
                      问题触发
                          │
               ┌──────────┴──────────┐
               │                     │
          模式A：Incident         模式B：Initiative
        （编码/回测异常）       （新策略/新需求）
               │                     │
         Incident流程          4步引导流程
         （5阶段串行）          （含自检清单）
               │                     │
               └──────────┬──────────┘
                          │
                   墨涵最终校验
                          │
                    ├── 通过 → 归档 + 知识审查
                    └── 争议 → Owner 裁决
```

---

### 2.2 模式A：Incident 问题定义（5阶段串行）

**适用场景**：编码/回测/测试过程中发现了异常现象，需要正式定义问题后再进入根因分析。

**总耗时**：≤75min（不含查证重做）

#### 阶段0 — 上下文准备（墨涵）

| 项目 | 内容 |
|:-----|:------|
| 执行者 | 墨涵 |
| 耗时 | ≤5min |
| 产出 | `context_{problem_id}.md` |

**内容**：
- 问题发生的背景（哪个任务、哪个阶段）
- 数据时间线：所有已知数字的来源和时间点
- 问题范围边界初稿

**约束**：上下文准备好后须确认墨衡能否开始（"这个背景够用吗？"），确认后进入阶段1。

#### 阶段1 — 现象还原（墨衡）

| 项目 | 内容 |
|:-----|:------|
| 执行者 | 墨衡（墨涵 spawn，串行） |
| 输入 | 墨涵准备的上下文包 |
| 耗时 | ≤20min |
| 产出 | `phenomenon_{problem_id}.md` |

**约束**：
- 仅记录观察到的事实，禁止原因分析和解决方案
- 每条现象标注类型：实测 / 推断 / 不确定
- 所有数字标注来源和时间点
- 明确列出「不确定项」和「我认为不在范围内的事项」
- 关键数据须标注基准参考线（正常窗口预期值、当前实际值、偏差幅度、比较窗口）

**模板**：

```markdown
## 执行时间线
[按时间顺序还原每一步操作和观察到的现象]

## 基准参考线
- 指标：在正常窗口的预期值：[数值]
- 当前实际值：[数值]
- 偏差幅度：[%]
- 比较窗口：[时间范围]

## 现象清单
| # | 现象 | 类型 | 数据来源 | 时间点 |
|---|------|------|---------|--------|
| P1 | ... | 实测 | DB查询 | HH:MM |

## 不确定项
[自己也不确定的地方，逐条列出]

## 边界外事项
[明确标注不在问题范围内的内容]
```

**自检（墨衡完成，≤10min）**：

| 检查项 | 通过条件 |
|:-------|:---------|
| 无原因分析 | 全文不出现"因为/导致/原因是" |
| 无解决方案 | 全文不出现"应该/建议/修复/补丁" |
| 类型标注完整 | 每条现象有实测/推断/不确定标注 |
| 数字可追溯 | 每个数字有来源和时间点 |
| 不确定项非空 | 若真无不确定项，需说明理由 |

⚠️ 自检不通过 → 打回重做（上限3次 → 告警 Owner）

---

#### 阶段2 — 结构审查（玄知）

| 项目 | 内容 |
|:-----|:------|
| 执行者 | 玄知（墨涵 spawn，串行） |
| 输入 | 阶段1 产出 + 墨涵上下文包 |
| 耗时 | ≤10min |
| 产出 | `structure_review_{problem_id}.md` |

**回答三个问题**：

1. **现象关系**：各现象之间的关系类型（依赖/独立/互斥/未知），标注依据来源
2. **边界判断**：在范围内 / 不在范围内 / 不确定
3. **待查清单**（含优先级）：需要查 DB/日志才能确认的条目

**约束**：
- 看不懂的列入待查清单，不推断填补
- 不做原因分析，只做现象间结构判断
- 发现数据口径混用，直接标注"口径问题"
- **发现阶段1自检中口径混用或数据来源不一致** → 标注并由墨涵决定是否退回阶段1

**玄知在阶段2执行Q7二次防线**：
- 审查阶段1中每条数字的数据口径是否统一
- 不存在自检偏差（墨衡自检通过不代表数据口径无误）
- 发现口径问题 → 标注后由墨涵决定是否退回阶段1

---

#### 阶段3 — 链条审查（墨萱）

| 项目 | 内容 |
|:-----|:------|
| 执行者 | 墨萱（墨涵 spawn，串行） |
| 输入 | 阶段1 产出 + 墨涵上下文包 |
| 耗时 | ≤10min |
| 产出 | `chain_review_{problem_id}.md` |

**回答三个问题**：

1. **链条扫描**：按编码/回测流程各阶段标注异常/正常/未知
2. **首次出现节点**：在哪个阶段首次观察到异常迹象
3. **待查清单**：从文档发现的需要数据验证的条目
4. **[新增] 数据依赖链审查**：追踪关键数据值从原始数据到最终产出的完整路径，标注每步传递是否完整、口径是否一致

**约束**：
- 与阶段2 合并去重待查清单（墨涵合并后发给阶段4）
- 发现数字存疑或口径混用，列入待查，不自行下结论

---

#### 阶段4 — 双人查证（墨衡+墨涵）

| 项目 | 内容 |
|:-----|:------|
| 执行者 | 墨衡执行查证，墨涵复核 |
| 输入 | 阶段2+3 合并去重的待查清单（含优先级） |
| 耗时 | ≤15min（SQL/日志查询每项≤3min，脚本验证类每项≤8min，超时标注待查） |
| 产出 | `verification_{problem_id}.md` |

**执行规则**：

每项查证输出格式：
```markdown
### 待查项 #N
问题：[待查清单原文]
优先级：[高/中/低]
查证方式：[SQL/日志/脚本]
原始结果：[粘贴原始输出，不做解读]
结论：已查证 / 待查（原因）
```

**墨涵复核要点**：
- 这条查证回答了待查的问题吗？
- 结果与阶段1 文档中的对应数字是否一致？
- 不一致 → 标注"存在出入"，决定是否回写修正阶段1
- 没回答问题 → 打回重查

**查证结果回写机制**：
- 查证结果与阶段1 数据不一致 → 在验证报告中标注"差异说明"
- 墨涵判断是否需要回写修正阶段1 文档
- 需回写 → 更新 `phenomenon_{problem_id}.md` 并在文件头标注修订

**约束**：
- 查不到的标注"待查"，不推断填补
- 墨涵发现"查了但没回答问题"，打回重查

---

#### 阶段5 — 收敛定义（墨涵）

| 项目 | 内容 |
|:-----|:------|
| 执行者 | 墨涵 |
| 输入 | 阶段1~4 全部产出 |
| 耗时 | ≤10min |
| 产出 | `problem_definition_v1.0_{problem_id}.md` |

**填写模板**：

```markdown
## 问题定义 v1.0

### 核心现象（一句话）
[不超过50字]

### 严重性评估
评级：[Critical / Major / Minor / Informational]
依据：- [涉及资金数据不一致 → Critical]
       - [仅是报告格式问题 → Minor]
       - [其他参考标准]

### 时间线
[关键节点]

### 确认事实（有查证依据）
| # | 事实 | 依据来源 |
|---|------|---------|

### 未确认项
| # | 待查内容 | 责任人 | 截止时间 |
|---|---------|--------|---------|

### 范围边界
在内：[...]
在外：[...]
不确定：[提交Owner裁决]
```

**同时执行知识审查（墨涵）：**

审查待激活的知识条目（若有）：
1. 回测结论是否与数据一致
2. confidence 定级（high/medium/low）
3. 激活或驳回

**后续动作**：
- 无争议 → 通知墨萱/玄知/墨衡确认 → 归档
- 有争议 → 标注后提交 Owner 裁决（Owner <30分钟未回应 → 墨涵决策升级）
- 归档后 → 知识条目激活（若适用）

---

#### 超时汇总（模式A）

| 阶段 | 执行者 | 超时 | 超限动作 |
|:-----|:------:|:----:|:---------|
| 阶段0 上下文准备 | 墨涵 | 5min | 自我告警 |
| 阶段1 现象还原 | 墨衡 | 20min | 告警 Owner |
| 阶段1 自检 | 墨衡 | 10min | 告警 Owner |
| 阶段2 结构审查 | 玄知 | 10min | 告警 Owner |
| 阶段3 链条审查 | 墨萱 | 10min | 告警 Owner |
| 阶段4 双人查证（每项） | 墨衡+墨涵 | SQL/日志≤3min/项，脚本≤8min/项 | 标注待查 |
| 阶段5 收敛定义+知识审查 | 墨涵 | 10min | 告警 Owner |

### 2.3 模式B：Initiative 问题定义（4步流程 + 自检）

**适用场景**：启动新策略/新需求/新回测任务时的前置问题定义。

**流程**：

| 步骤 | 内容 | 墨涵职责 | 耗时 |
|:----:|:-----|:---------|:----:|
| 1 | 描述现状（有数据支撑） | 引导全员描述 | 5min |
| 2 | 分析后果（影响范围+紧急性） | 引导分析 | 5min |
| 3 | 寻找根因（至少追问三层） | 追问"为什么"直到根因 | 10min |
| 4 | 统一问题陈述（全员确认） | 记录共识，确认后放行 | 5min |

**自检清单（模式B）**：
- [ ] 现象描述是否有具体数据支撑（数据来源）？
- [ ] 后果分析是否量化了影响范围和紧急性？
- [ ] 根因是否至少追问到第三层"为什么"？
- [ ] 问题陈述是否全员确认无异议？

**产出**：`problem_statement_{task_id}.md`（墨涵记录，Owner 确认后放行）

**与设计阶段的关系**：
```
问题定义（模式B）→ Owner确认放行 → 设计（墨衡）→ ...
```

## 三、与现有机制的对接

### 3.1 通信协议

模式A 各阶段使用 spawn_template_v1.0 串行模式（TASK→START→执行→Announce→完成）。

- 无并行 spawn
- 每阶段独立 .done 文件
- 墨涵在群主 session context 回复

### 3.2 产出归档

最终定义文档归档至：docs/06_problem_definitions/{problem_id}_v1.0.md

### 3.3 知识衔接

问题定义过程中发现的可记录 insight → 写入 knowledge_entries/draft/ 目录，由阶段1执行者（墨衡）在阶段1 完成后创建草稿。墨涵在阶段5 执行知识审查后决定激活或驳回。

### 3.4 争议定义

- **争议**：墨涵与任一执行者（墨衡/墨萱/玄知）就关键事实或结论存在不可调和分歧
- **Owner SLA**：告警后 30 分钟内未回应 → 自动升级由墨涵决策

## 四、与 Claude 原版的关键差异

| 维度 | Claude 原版 v1.0 | 墨枢定制版 v1.0 |
|:-----|:----------------|:---------------|
| 模式 | 单模式（Incident 专用） | 双模式（Incident + Initiative） |
| 阶段数 | 8（含 Pre-stage） | 5（模式A）/ 4步（模式B） |
| 并行机制 | Stage 2a/2b 并行 | 全部串行 |
| 墨萱角色 | 流程链条审查 | 链条审查（与 QA 角色对齐） |
| 玄知介入 | 两次（2b+4） | 一次（阶段2 结构审查 + Q7二次防线）|
| Owner 签署 | 无 | 有（争议裁决 + 业务确认） |
| 知识审查 | 无 | 阶段5 内嵌（墨涵知识守门） |
| 数据口径 | Pre-stage 声明 | 阶段1 自检 + 阶段2 玄知二次防线 |
| 查证机制 | 墨衡+墨涵双人 | 墨衡+墨涵双人 + 回写修正机制 |

## 五、实施建议

1. 先试点 Incident 模式：选一次编码异常走完整流程，验证阶段耗时和产出质量
2. 阶段1 自检是关键防线：所有阶段都可以简化，但阶段1 的现象+原因分离不能省
3. Owner 通知策略：告警 Owner 时附上当前产出文件，让 Owner 能看到进展再判断
4. 知识审查后置：阶段5 内嵌但不阻塞签字，知识审查可以稍后独立完成
5. 时效优化：如果问题明显简单（单一现象、来源清晰），墨涵可跳过阶段2/3直接进入阶段4
