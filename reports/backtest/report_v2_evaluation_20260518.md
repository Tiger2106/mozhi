### 1. Signal Distribution Analysis（信号分布分析）

**可行性**: ⚠️需改造

**当前状态**:
- 现有系统回测完成时只输出成交记录（`Trades`），不保存所有中间信号
- SignalFusion 模块在运行时有权重计算和阈值判定逻辑，但这些中间判定结果未被持久化
- 当前仅有 1 笔成交，无法展示有意义的信号分布

**所需改造**:

| 改造项 | 说明 | 工作量 |
|:-----|:----|:----:|
| signal_events 采集 | 每次信号判定（无论是否通过所有过滤器）都记录到事件表 | 1~2 天 |
| 过滤器链日志 | 记录每个信号经过过滤器链（MarketStateFilter / DrawdownGuard / 冷却期等）的通过/拒绝状态 | 1 天 |
| 置信度分桶 | 将 SignalFusion 的加权分数按区间分桶统计 | 0.5 天 |
| 报告展示层 | 新增 Section 渲染信号分布表 | 0.5 天 |

**数据结构建议**:
```json
{
  "event_id": "uuid",
  "timestamp": "2026-01-22T09:30:00",
  "symbol": "601857",
  "signal_type": "grid_trigger",
  "fusion_score": 0.85,
  "filters": {
    "MarketStateFilter": "pass",
    "CoolingPeriod": "pass",
    "DrawdownGuard": "pass"
  },
  "final_decision": "executed"
}
```

**依赖**:
- 无外部依赖，纯内部改造
- 需要定义 `signal_events` 数据结构和写入时机

**预估工作量**: 3~5 天

### 2. Conditional Return Matrix（条件收益矩阵）

**可行性**: ⚠️需改造

**当前状态**:
- 现有系统已具备 Regime 判断（TREND_UP/RANGE/RANGE_SIDEWAYS）和 Volume 数据
- 但仅 1 笔交易，拆到 Regime+Volume 复合条件下几乎全部为空，矩阵无统计意义
- 框架概念就绪，缺乏多标的/多周期数据支撑

**所需改造**:

| 改造项 | 说明 | 工作量 |
|:-----|:----|:----:|
| Regime 判定持久化 | 当前 Regime 序列仅用于展示，需保存为时间序列数据 | 0.5 天 |
| 成交量的条件分桶 | 将成交量按"放量/缩量/正常"分桶 | 0.5 天 |
| 收益矩阵计算 | 按 Regime+Volume 复合条件聚合交易收益 | 0.5 天 |
| 报告展示层 | 渲染收益矩阵表 | 0.5 天 |

**关键问题**:
- 当前单标的、单周期数据无法填充大部分矩阵格子
- 矩阵的价值在跨标的、跨周期后才会体现

**依赖**:
- 需扩展至多标的或多个周期的回测结果
- 多周期数据提取（至少 3~5 年）

**预估工作量**: 2~3 天（框架搭建），实际有价值数据需等 Phase 2 完成

### 3. Capital Efficiency（持仓效率分析）

**可行性**: ⚠️需改造

**当前状态**:
- 现有系统已计算资金利用率（0.20%）、持仓时间（31天）、闲置天数
- **多标的并行模拟是全新的系统能力**，当前为纯单标的设计

**所需改造**:

| 改造项 | 说明 | 工作量 |
|:-----|:----|:----:|
| 单标的持仓效率完善 | 平均持仓时间、空仓时间占比、单位资金收益 / 单位风险收益 | 0.5 天 |
| 多标的并行回测引擎 | 设计支持 N 个标的同时运行回测 | 5+ 天 |
| 资金池分配管理 | 多标的间的资金分配策略（等权重/风险平价/动态分配） | 2 天 |
| 并行效率模拟 | 生成 1/10/50 标的的假设结果 | 1 天 |

**关键问题**:
- 多标的设计是系统架构级改造，涉及回测引擎核心
- 当前代码假设所有资金分配到一个标的，多标的需要引入"资金池"概念
- 标的间的相关性、资金分配策略、再平衡逻辑都需要实现

**依赖**:
- 标的池定义（601857/600519/000001...）
- 回测引擎架构重构
- 外部数据源（多标的的行情数据）

**预估工作量**: 5~10 天（完整的多标的能力）

### 4. Signal Decay Analysis（信号衰减分析）

**可行性**: ⚠️需改造

**当前状态**:
- 现有系统只有"最终持仓 31 天 → 收益 30.18%"，没有不同持有期的收益率回溯
- 需要额外的数据采集点和计算逻辑

**所需改造**:

| 改造项 | 说明 | 工作量 |
|:-----|:----|:----:|
| 信号时间戳采集 | 记录每个信号的精确产生时间 | 0.5 天 |
| 持有期收益回溯 | 对每个信号，计算持有 1/3/5/10/20 天后的收益 | 1~2 天 |
| 衰减曲线拟合 | 绘制平均收益 vs 持有期曲线，识别衰减点 | 1 天 |
| 最优持有期判定 | 基于衰减曲线计算最优持有窗口 | 0.5 天 |

**数据采集点增加**:
1. 信号产生时间（已由回测日志间接记录，需标准化）
2. 信号产生后 N 天的价格快照
3. 当信号间产生冲突（新信号 vs 旧持仓）时的判定逻辑

**依赖**:
- 多标的、多周期数据（衰减分析的统计意义需要大样本）
- 回调数据准备（需按持有期回算收益）

**预估工作量**: 3~5 天

| 改造项 | 说明 | 工作量 |
|:-----|:----|:----:|

| 阶段判定算法 | 基于 TrendQuality + VWAP偏离 + Volume 序列实现状态机 | 1 天 |
| 历史阶段回溯 | 对回测期进行各阶段标注 | 0.5 天 |
| 各阶段收益统计 | 统计各阶段内策略的收益表现 | 0.5 天 |

**阶段定义建议（量化版本）**:

| 阶段 | 判定条件 | 关键指标 |

- TrendQuality 模块已就绪（需确认内部机制）
- VWAP 偏离度已就绪
- Volume 数据已就绪
- OBV 计算（与假突破画像共享）
