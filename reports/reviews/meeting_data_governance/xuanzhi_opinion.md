# 数据录入交叉验证规范 — 玄知意见（技术把关）

## 0. 预估与置信度

- **预估耗时**：本意见输出 15 分钟
- **置信度**：90% — 定位明确，争议点可裁决

---

## 1. 战略价值：这是投研体系的"宪法修正案"

**定性：极高战略价值，不应被视为"数据组琐事"。**

原因有三：

### 1.1 信任链的基石

量化研究建立在"输入 → 计算 → 信号 → 交易"的信任链条上。当前链条中最大的薄弱环节就是**数据录入**——因为这个环节几乎不设防。墨衡和墨萱的方案补上了这个缺口。

用一个比喻：这相当于在数据仓库上装了**空气安全阀**——本来压力（数据错误）大了系统直接爆炸（错误策略 → 亏损），现在有一个预警机制能在压力到达临界值之前提示。

### 1.2 信号质量的生命线

从研究战略角度看，这条规矩直接决定了**因子研究的复现性和置信度**：

- 没有交叉验证 → 今天的回测结果可能是"恰好两个源都错了但错的方向一致"
- 有交叉验证 → 至少能断言"数据在录入时经过多源核验，正确性有据可查"
- 这会让报告（Step5 output）的置信度从"数据未经审计"提升到"数据经多源交叉核验"

### 1.3 降维解决未来纠纷

当前是小团队，未来扩展时，每个人对"数据是否正确"的理解不同。有了一套规范：
- 新人接手的第一个问题不是"你信这个数据吗"，而是"查 audit_log，看 PASS 了多少条"
- 这降低的是**长期的组织认知成本**，不是单个规则的执行成本

## 2. 与现有协议的兼容性

### 2.1 是否需要更新 ADR？

**是，但不紧急。**

这条规矩的核心是**ETL pipeline 的 transform 阶段插入验证步骤**，不改变现有数据流拓扑。属于 ADR 中 **"模块内部改进"** 级别，不需要重新决议架构层面的变更。

但建议：
- 在本会议的会议纪要（由墨涵记录）中明确指出这是一次 ADR 更新，而非仅仅是讨论备忘
- 将这个规则约定为 **墨枢系统第 E-001 号执行规则**，纳入 `docs/03_references/` 下的执行规则清单
- 后续如果验证机制扩展到**跨 Agent 通信**级别（比如墨衡输出的信号也需要交叉验证），则需要更新 Signal Protocol

### 2.2 是否需要更新 Signal Protocol？

**不需要。**

Signal Protocol 定义 Agent 间通信和数据交换格式。数据录入交叉验证是 ETL 层面的内部质量保障，不改变 Agent 之间的信号内容或格式。

但如果未来验证规则扩展到**Agent 输出信号**（比如"墨衡的买入信号必须经墨萱的因子库交叉验证"），那就需要 Signal Protocol 更新。目前无此需求。

### 2.3 与墨枢流水线的兼容性

| 流水线步骤 | 兼容性 | 说明 |
|-----------|--------|------|
| Step0 市场扫描（玄知） | 完全兼容 | 玄知读的是 stats 和 signals，不直接触 ETL |
| Step1 数据采集（玄知） | 完全兼容 | 玄知读的是行情数据和已清洗结果，不涉及原始录入 |
| Step3.5 战略复核（玄知） | 完全兼容 | 同上 |
| Step5 报告定稿（墨涵） | 间接受益 | 报告中的数据引用将附带更高质量保证 |
| Step7 异步审阅（玄知） | 间接受益 | 审阅时可引用"数据经交叉验证"提升结论置信度 |

**结论：零冲突，纯增益。**

## 3. 全局影响评估：会不会过度约束？

### 3.1 开发节奏影响

**短期微增（~4-7 小时一次性投入），中期解耦，长期无负担。**

看墨衡的成本评估：
- Phase 1（核心字段，本周）：2-3 小时
- Phase 2（扩展字段，本月）：2-3 小时
- Phase 3（制度化，下季度）：持续但低频率

即使加上墨萱的测试套件：
- 四级测试金字塔第一轮实现：+1-2 小时
- 预生产统计模式：+0.5 小时配置时间，主要耗时在跑一周观察

**总一次性投入：约 5-9 小时。** 对于一个非紧急但战略重要的基础设施，这是极低的投入。

### 3.2 运营摩擦

**低。** 验证函数的执行时间（~5-10ms/次）对 ETL 管线几乎没有可见影响。映射表的维护是变更驱动的，不是周期性的。

唯一可能产生摩擦的是：
- 新数据源接入时的映射表配置（10-20 分钟/次）
- 极少数"合理差异"需要人工覆盖（预计频率：每周 0-2 次，且非关键字段）

### 3.3 会不会约束研究灵活性？

**不会。** 验证规范针对的是原始数据录入阶段——正确性是硬约束。研究层面的灵活性（比如选择用前复权还是后复权、不同复权方式下的回测差异）属于分析和处理阶段，不受此规则影响。

### 3.4 风险反转：如果不实施

**这个风险我建议量化一下：**

如果一条 K 线数据错误（比如 volume 单位误用）没有被早期捕获，它可能：
1. 进入因子计算 → 错误的成交量因子 → 错误的分组回测
2. 进入信号生成 → 错误的交易信号 → 回测绩效失真
3. 进入报告结论 → 报告结论质量存疑

以墨枢 Pipeline 当前运转节奏，**一次未被捕获的数据错误的理论传播链**可以覆盖所有步骤。交叉验证是这条传播链上**最低成本、最高影响力**的检查点。

## 4. 争议点裁决：墨衡 vs 墨萱

### 4.1 核心张力：弹性比对 vs 强制测试

先拆解：这不是"要不要测试"的争议，而是**实施顺序和优先级分配**的争议。

- **墨衡的立场**：先跑通流程（Phase 1 核心字段），测试可以并行或稍后
- **墨萱的立场**：测试必须作为 Phase 1 的硬性交付物，否则验证系统本身不可信

**我站在墨萱一边，但有修饰。**

### 4.2 裁决逻辑

**墨萱指出的风险是真实的：** 一个没有测试的验证系统，它的可靠性等于验证引擎的可靠性——而验证引擎的开发者也写 Bug。这是"吃自己的狗粮"问题，也是我作为技术把关者不能忽视的。

**但墨衡的"先跑再说"策略也有合理成分：** 如果先花 2 小时写全测试套件再跑验证，但阈值不对导致 80% 的合法数据被阻断——测试通过了，但验证无用。墨衡的核心观点是"阈值需要统计预生产的数据才能定"。

**我的裁决方案：** 两者不矛盾，按墨萱的方案执行，但墨衡的"先跑统计预生产"应**先于测试套件完备**：

```
Phase 1a（本周前半段，0.5h）：预生产统计模式配置
 → 只记录不阻断，跑一天观察实际跨源差异分布
 → 输出：阈值建议值（基于 P50/P95/P99）

Phase 1b（本周后半段，4-5h）：验证引擎实现
 → 写验证函数 + 核心单元测试（不是完整四级金字塔）
 → 映射表 + audit_log schema 落地
 → 单元测试仅覆盖 Phase 1a 统计出的阈值

Phase 1c（下周，2-3h）：测试套件完善
 → 补全墨萱的四级测试金字塔
 → 验收红线的 CI 集成
```

### 4.3 另一个值得裁决的点：diff_reason 枚举化

**墨萱坚持枚举，墨衡写的是自由文本。** 这里我选择墨萱的方案，且强调一个工程理由：

**枚举化不仅是为测试服务的，更是为后续自动化分析服务的。** 如果 diff_reason 是自由文本：
- 机器无法汇总 "本周有多少次 UNIT_CONVERTED" 的统计
- 每周的数据质量报告写不出 "阈值调整建议" —— 因为不知道哪些"合理差异"在积累
- 墨衡的长期改进机制（统计违规频率→调整阈值）会因为 diff_reason 不可分析而无法实施

枚举化的实现成本几乎为零（定义 7 个枚举值 + 校验约束），收益却很大。**强制枚举化。**

### 4.4 墨萱补充的关键字段：trade_date + symbol

**完全同意，提升到 P0。** 数值验证通过但时间错位是真正的"验证盲区"，没有 date 和 symbol 校验的验证系统是有漏洞的。

这不需要复杂的数值比对逻辑，只需要：
- 日期格式归一化后比较
- 代码格式归一化后比较
- 跨源同一字段的日期一致性检查

实现成本极低（基本上是字符串比对），风险覆盖率提升明显。

## 5. 最终建议

### 5.1 是否立即实施？

**建议有条件立即实施。** 条件如下：

**必须满足的三项条件（否决权）：**
1. ✅ 墨萱的验收红线 E.01-001~004（验证机制存在性）在 Phase 1 交付时通过（这个范围我接受，不要求全四级金字塔在开头）
2. ✅ Phase 1a 预生产统计模式必须首先执行，阈值由数据驱动而非拍定
3. ✅ diff_reason 必须枚举化

**推荐但非强制的优化：**
- trade_date + symbol 交叉验证在 Phase 1 内（而非 Phase 2）实现
- 映射表完备性自动化断言（墨萱的 test_unit_conversion_table_completeness）

### 5.2 实施路线（经过争议裁决后的版本）

```
Phase 1a（0.5h，本周）：
  配置预生产统计模式 → 收集跨源差异分布 → 确定阈值

Phase 1b（4-5h，本周）：
  验证引擎核心实现（验证函数 + 单位归一化 + 映射表）
  审计日志 schema（按墨萱的 15 字段版）
  核心单元测试（覆盖 Phase 1a 定出的阈值）
  字段覆盖：volume, amount, open, high, low, close, trade_date, symbol
  验收：E.01-001~004 通过

Phase 1c（2-3h，下周）：
  补全四级测试金字塔（墨萱方案）
  CI 集成验收红线
  数据源 mock 性能测试

Phase 2（本月）：
  扩展字段：total_shares, negotiable_shares, market_cap
  接更多数据源

Phase 3（下季度）：
  制度化纳入 Check-List
  月度数据质量报告
```

### 5.3 风险警示

**最大的风险不是实施成本，而是"验证疲劳"。** 当验证规则每天都在触发、每次触发都产生 diff_reason_DELAYED_SOURCE 时，团队会麻木。我建议：

1. **只对 P0 字段做阻写验证**，P1+ 只标记 WARN
2. 每次 valid 失败自动触发告警，但只有频率 > 阈值时才提升到人工处理
3. **每季度评估一次验证规则的误报率**，如果误报率 > 10% 则规则需要修正
