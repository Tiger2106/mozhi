# 回测试验方案评审会流程 v2.0

## 总体结构

```
Stage 1         Stage 2                Stage 3
  修改  ──→      会签  ──→             归档

  提案人起草      墨萱技术签（§2/3/6）   墨涵推进状态
  墨涵主持R001   墨涵知识签（§0+R001）   version_status→准予执行
                  Owner终签（准予执行/终止）
                  退回循环（≤2次，附修改要求）
```

## 流程说明

本流程覆盖试验方案执行前的评审阶段（模板 §0 ~ §0.5）。执行完成后的结果评审走已存在的《回测总结会4阶段流程 v1.1》。

### Stage 1：修改

**起草人：** 提案人（A/B/C类试验对应不同起草人）
**主持人（R001）：** 墨涵
**参与者：** 按需

**两个前置条件必须满足后方可进入会签：**

1. **R001 问题定义已共识**——墨涵主持四步流程（描述现状→分析后果→找根因→统一陈述），全员确认后写入模板§0
2. **模板§1~§7已填写完整**——无空白、无缺段；失败案例库自查已执行（cold_start可放行）

### Stage 2：会签

**原则：顺序固定，不可并联。**

#### 第1轮：墨萱技术签

**核验范围：**

| 核验项 | 对应节 | 通过判定 |
|:------|:-----|:---------|
| 标的排除标准合规（上市≥2年、free_float≥0.5、复权质量） | §2.1 | 全部满足或已标注例外说明 |
| 数据精度检验规则明确 | §2.3 | data_qc 脚本路径已指定，assert 已替换为 raise 硬阻断 |
| 因子权重配置无循环依赖 | §3.1 | factor_weight 与 factor_signal 已分离 |
| A股硬约束已启用（最小交易单位1手、印花税、价格笼子） | §3.2 | 已标注启用/禁用状态 |
| 停复牌运行规则已定义 | §2.2 | 已完整填写 |
| IC/IR 验收阈值已设定 | §4.1 | >50% 硬约束，>53% 优秀线，边界区间单调性已规划 |
| P2 自洽性执行时间已嵌入 | §6 | 已验证可执行 |
| 数据窗口划分合理 | §2 | IS/OOS 分界点有依据 |
| 失败案例库已自查 | §1 | 提案人查询记录已写入 |
| 参数扫描范围已明确 | §3 | 全局扫描或指定区间已标注 |
| H0/H1 判定规则已写清 | §1 | 原假设与备择假设明确 |
| 环境版本已锁定 | §5 | Python/依赖库版本已记录 |

**退回规则（附修改要求）：**
- 第1次退回：定位缺陷 + 规则引用 + 修改方向（三要素缺一不可）
- 第2次退回：加注更具体修改要求
- 第3次仍不通过 → 强制触发 Owners 裁决

**退回时间上限：** 每次退回修改≤2个工作日。

#### 第2轮：墨涵知识签

**核验范围：**
- R001 问题陈述完整（对象+偏差+可观测指标）
- 模板§0已填写（问题记录、根因分析、全员确认签名）
- 会签结果与历史知识无矛盾（独立确认knowledge.db）
- 修改方向符合要求

**退回规则（不代修原则）：**
- 不合格 → 退回原起草人，注明不合格范围 + 修改方向
- 覆盖范围：知识层面问题（非技术细节）
- 格式/结构错误可顺手修正

#### 第3轮：Owner 终签

**判定：**
- **准予执行** → 进入 Stage 3
- **终止** → version_status 标记为"已终止"，按失败归档流程处理

#### 快速通道

当以下条件全部满足时跳过 Stage 1 直接进入 Stage 2：
- R001 已共识且有书面记录（含至少两层层根因追问）
- 核心逻辑不变（非"仅数值调整"——明确排除条件变更/设计重构）
- 无跨团队依赖项

**快速通道下的补偿检查：**
- 墨萱抽检 R001 书面记录三要素合规性（对象+偏差+可观测指标）
- 墨涵确认根因至少追到第二层

### Stage 3：归档

**执行人：** 墨涵

**标准步骤：**

1. version_status 从"评审中"推进至"准予执行"
2. 试验编号 EXP-{YYYY}-{XXX} 正式激活
3. 会签结果写入 `backtest_run.notes`
4. 方案文档归档至试验目录
