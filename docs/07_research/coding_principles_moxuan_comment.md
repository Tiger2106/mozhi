## 二、可测性分析：哪些原则最难验证？

### 最难验证 Top 3

**🥇 P-001 三层分离 — 可测性评级: C级（需端到端集成验证）**

三层分离不是能用单元测试覆盖的。它的正确性必须通过"同一组信号，三层分离前后结果一致"来验证。问题是，如果三层分离和前视偏差修复（P-012）同步进行，我们无法区分"结果变化是因为分层导致的逻辑错误"还是"修复导致的预期变化"。

**建议**: 三层分离的回滚路径必须预先定义。首次回归失败时应能快速回退到旧架构确认问题归属。

**🥈 P-013 TimeAlignmentGuard — 可测性评级: C级（需专项测试套件）**

TimeAlignmentGuard的难点不在实现，而在测试覆盖。前视偏差的引入方式太多：
- 显式 `data[di]` 访问（容易检出）
- 隐式滚动窗口 `min_periods=1` 泄漏当前值（难以检出）
- 下游函数 `shift(-1)` 重对齐（极其隐蔽）

我要求在 CI 中部署 **前视偏差注入测试**：构造已知含前视偏差的输入，TimeAlignmentGuard 必须正确捕获。这个测试套件由我来设计，墨衡只需要实现 Guard 接口。

**🥉 P-006 策略接口标准化 — 可测性评级: B级（静态检查+契约测试）**

固定函数签名本身不难验证（mypy 静态类型检查即可），但"策略函数是纯函数"这个约束需要运行时契约测试。我会为每个注册的策略函数执行"相同输入→相同输出"的可重复性测试。

### 相对容易验证的

| 原则 | 验证方式 | 评级 |
|:-----|:---------|:----:|
| P-003 数据一次性加载 | profile 检查 I/O 调用次数 | A级 |
| P-004 循环内零新分配 | 内存分配追踪 + code review | B级 |
| P-007 数据合约固定 | dataclass 类型检查 | A级 |
| P-009 确定性执行 | CI 两次运行结果比对 | A级 |
| P-010 交易日志完整可审计 | 日志格式 schema 验证 | A级 |

## 三、评审强度：哪几条最容易在 CR 漏掉？

基于我过往的审查经验（审计报告、EXP-003 QA、解耦验证），**CR 漏报率排名**：

### ⚠️ 高漏报风险

**1. P-004 循环内零新分配 — 漏报风险: 高**

原因：它藏在性能优化范畴，CR时 Reviewer 容易默认"作者既然写了应该没问题"。而且 Python 中一次性分配和循环内分配的语法差异很细微。需要一个显式的 CR 检查项专门扫描循环体内的 `np.array(...)`, `dict(...)`, `list([...])`。

**2. P-012 契约传递 — 漏报风险: 高**

P-012 本身的契约容易理解，但"契约传递"规则最容易被忽略。下游函数（runner、策略、分析脚本）在收到信号矩阵后可能无意识地再做一次 `shift(-1)`，导致信号变成 t-2。这在 CR 中几乎不可能被肉眼发现，唯一的保障是 P-013 的运行时检测。

**3. P-009 确定性执行 — 漏报风险: 中→高（浮点数陷阱）**

`random`/`np.random` 的直接调用较容易检出，但浮点数比较和序列化的非确定性更难发现。`float64 → str → float64` 的精度丢失在 CR 中很少被注意。建议将此纳入 P-008 回归对比的自动检查。

### 🟢 低漏报风险

| 原则 | 理由 |
|:-----|:------|
| P-002 向量化 | 逐行循环有明显性能特征，profile 可检出 |
| P-008 回归对比 | CI 自动检查，无需人工 |
| P-010 交易日志 | 日志输出有 schema 约束，测试可自动验证 |
| P-007 数据合约 | Python dataclass + mypy 静态检查 |

## 四、我的担忧：被遗漏的编码原则

### ⚠️ 遗漏1：错误处理与断言密度（建议新增 P-014）

当前13条原则中**完全没有错误处理**的约束。从我的审计经验看，回测引擎中最隐蔽的问题发生在"边界情况未处理"——比如交易日少于计算窗口时的静默失败、数据长度为0时的除零隐式传播。

**建议增加原则**：**"关键路径必须有前置断言"**

- 函数入口必须包含输入合法性断言（shape、dtype、非NaN）
- 失败必须抛出具体异常类型，不允许静默失败
- 回溯过程中的中间结果不一致必须立即中断

这条原则不增加测试工作量，但显著降低调试成本。

### ⚠️ 遗漏2：测试覆盖率门限（建议新增 P-015）

13条原则中 P-010/P-011 涉及验证，但没有对**测试覆盖率**提出约束。从我在 EXP-003 的审查经验看（Q1→Q2 验证中发现 TrendQuality 样本量数字偏差），**报告与数据不一致的原因往往是测试不完整**。

**建议增加原则**：**"P0模块单测覆盖率≥90%，集成测试覆盖所有约束组合"**

这不是 KPI 式的数字要求——90% 不是终点，而是发现未被测试覆盖的边缘情况。约束的任意组合（停牌+T+1、涨停+T+1、复牌首日、同日买卖）必须有独立测试用例。

### ⚠️ 遗漏3：重构工具化（建议增加 CR 规则）

原则列表中缺失对**验证工具**的约束。分层重构期间，`diff` 不能只对比最终结果——需要对比逐层中间结果。如果重构后计算层输出和重构前不一致，故障定位会很困难。

**建议**: 在 P-008 回归对比中增加"中间层验证"层级——不只在最终结果层面对比，也在数据层→计算层、计算层→模拟层的接口层面设立回归检查点。这样重构失败时可以快速定位是哪一层出问题。

## 五、对 P-008 回归对比的补充建议（基于 EXP-003 经验）

### 5.1 EXP-003 的 WARN→PASS 经历

EXP-003 Q2 验证中，我经历了一个典型的 WARN→PASS 闭环：

**发现问题**: Q2 报告中 TrendQuality 样本量声称 "n≈136,000"，但 JSON 原始数据中训练期实际 n=146,362，偏差约 7%。

**分析过程**:
1. 这不是计算错误——无论是 136K 还是 146K，都不影响结论
2. 也不是数据错误——JSON 原始数据正确，错误发生在"报告撰写→数字简化→人为近似"环节
3. 最终判定 WARN（非阻塞），因为偏差不影响任何统计结论，但暴露了**报告自动化的缺失**

**教训**: 如果 P-008 回归对比只关注"资金曲线"和"交易次数"之类的高层指标，会漏掉"报告数字与原始数据不一致"这一类缺陷。验证不能只停留在计算结果的复现，还要延伸到**报告的每个数字都可以追溯到数据源**。

### 5.2 对 P-008 的补充建议

**建议1：增加"报告数字可追溯性"检查**

回归对比框架应增加一个验证维度：报告中出现的所有 IC 值、样本量、衰减率、收益等数字，必须能从原始数据中自动推导，不能有手工设定的数字。

```python
# 示例：回归对比框架应包含
def validate_report_numbers(report: Report, raw_data: Dict) -> List[ValidationEntry]:
    """对报告中每个断言性数字进行可追溯性验证"""
    ...
```

**建议2：增加"回归粒度分级"**

| 层级 | 名称 | 对比内容 | 频次 | 举例 |
|:----:|:-----|:---------|:----:|:-----|
| L1 | 终值回归 | 最终业绩指标 | 每次提交 | 夏普、年化、回撤 |
| L2 | 时序回归 | 逐日 NAV 序列 | 每次 P0 提交 | full equity curve |
| L3 | 中间层回归 | 计算层输出矩阵 | 重构时 | calc 输出信号矩阵，sim 输出日志 |
| L4 | 报告回归 | 报告数字与原始数据 | 每次输出 | 所有断言数字可追溯 |

P-008 当前主要覆盖 L1+L2。我建议在**分层重构期间强制 L3**，否则出现问题只能盲查。

**建议3：全等回归的"完全一致"定义需要明确边界**

P-008 的全等回归容差要求：
- IC: |Δ| < 1e-6
- NAV: max |Δ| / NAV_max < 0.01%

这对浮点数计算来说是合理的。但需要注意：**重构后的浮点数运算顺序改变会导致不可避免的微小差异**（向量化改变了加法的结合顺序）。建议验证框架能自动区分"浮点顺序差异"和"真正的逻辑差异"，避免 false positive。
