## 一、盲点清单（本阶段被忽略或关注不足的方向）

### 盲点 1：用户视角缺失 — 如何定义"好用"？

**严重程度：** 🔴 高

本阶段全部精力投向**架构重构**（契约定义 + 迁移验证 + 测试覆盖），但始终无人从**终端用户（量化研究员/策略开发者）** 视角评估新系统的使用体验：
- 插件系统的开发者体验（DX）如何？从写一个 Method 到跑通需要几步？
- KnowledgeAnalyzer 虽然纯 pandas/numpy 实现，但**输出结果如何被用户理解和使用**？
- 没有用户测试、没有原型验证、没有使用流程文档。1059 个测试覆盖的是"系统正确运行"，而非"人能用起来"。

**为什么这是盲点：** 架构再优雅，如果用户觉得比旧系统更复杂、更慢、更不透明，就会产生"回退冲动"——这是项目落地最大的隐形杀手。

### 盲点 2：新旧过渡期的"并行运维成本"未被量化

**严重程度：** 🟡 中

`run_trend.py` 等旧入口仅标记 DEPRECATED（未物理删除），新旧 Runner 双模式并存。这个过渡状态**有明确的运维成本**：
- 两个代码路径需要同时维护
- 新策略用新入口、旧策略跑旧入口，运维复杂度翻倍
- 团队成员需要同时理解两套运行逻辑
- **没有明确的旧入口下线时间表**（报告中仅标注"规划中"）

本阶段完成了"能否跑通"的技术验证，但没有回答"什么时候能切换到新系统单轨运行"。

### 盲点 3：R3/R4 的战略建议沦为"悬空待办"

**严重程度：** 🟡 中

评审会上我（玄知）提出的 4 项战略建议，R3 和 R4 至今是"待确认"占位。这导致：
- R1（Runner 预检）和 R2（KB 前端）已落地，但系统性的战略提升仅完成了 50%
- R3/R4 的内容在本报告中**没有被明确记录**（不确定是由于遗失还是尚未确认）
- 建议立刻追索评审会纪要，补全 R3/R4 内容并评估是否仍适用

### 盲点 4：BitableSync 真实 API 的"一日游"风险

**严重程度：** 🟡 中

BitableSync 真实 API 升级在时间线上是一个**单独的事件点**而非持续机制：
- 凭证已配置、E2E 测试通过、三方会签完成
- 但缺乏**定期刷新 token 的告警机制**（飞书 API token 有时效性）
- 缺乏**API 变更时的自动化通知接入**
- 如果飞书侧 API 发生向后不兼容变更（ruthless as they are），没有任何预警机制

这里的风险不在于"现在能不能用"，而在于"三个月后还能不能自动跑通"。

## 二、风险清单（可能影响后续推进的隐患）

### 风险 1：「0% 偏差」可能产生虚假安全感

**影响层面：** 方法论隐患 / 战略级

10 个 Method 新旧对比 0% 偏差，看似完美，但存在一个根本性问题：**新旧对比验证的 Method 样本量是多少？占全部 Method 的比例？**
- 如果只有 10 个 Method 被验证，而系统中存在更多（未迁移或未验证的）Method，0% 偏差结论不具备外推性
- **核心风险：** 后续新增 Method 可能未经新旧对比验证就直接进入生产，此时"0% 偏差"的历史记录会降低团队的警惕性

**建议：** 明确标注"已验证 Method 清单"和"待验证 Method 清单"，设定新增 Method 必须经过对比验证的强制流程。

### 风险 2：知识库（KnowledgeBridge）与回测系统的**耦合度缺乏衡量**

**影响层面：** 架构级 / 中期

KB Phase 1-3 打通了因子/策略知识到回测系统之间的桥梁，这是一个进步。但：
- **KB 是下游扩展示例，还是核心架构要素？** 如果 KB 被视为"旁挂"功能，其与回测系统的耦合度设计就是合理的；如果 KB 未来要成为核心数据基础设施的一部分，当前设计可能过于"轻量"
- 报告中提及 Phase 1c 的"Runner 接入 + KB 升级"通过 1015/1015 测试，但没有说明 KB 与 Runner 之间的**接口稳定性承诺**——如果 KB 接口变化，是否会影响 Runner？

### 风险 3：复盘循环机制的缺失

**影响层面：** 流程级 / 战略级

本阶段做了一件非常重要的事情——回测方法体系重构。但**缺少一个机制来回答"这个重构是否正确"**：
- 重构后的回测结果与旧版对比为 0% 偏差，这验证了"语义不变"，但没有验证"语义本身就是对的"
- 回测方法与现实市场表现之间缺乏定期比对机制
- 如果某个老 Method 本身就有逻辑缺陷，0% 偏差只是把缺陷移植到了新架构中

**建议：** 在下阶段建立"回测 vs 实盘"的定期校准机制，让重构的正确性有外部验证锚点。

### 风险 4：监控盲区 — 回测系统在无人值守时如何表现？

**影响层面：** 运维级 / 中期

报告中提到"MethodBacktestRunner 运行监控/告警未接入"仅被列为低优先级技术债务，但从战略角度看这不是"可有可无"：
- 如果回测系统是量化策略的生命线，一次无声的失败可能导致一整天/一周的策略信号缺失
- 当前系统的设计重点是"重构的正确性"，而非"运行的可靠性"
- 建议**将监控接入从"技术债务"升级为"下一阶段 P0"**

### 风险 5：团队知识单点依赖

**影响层面：** 人力级 / 长期

本阶段工作细节极为繁杂：Phase 1-5 的 5 个阶段拆解、958 个测试的统计口径、BitableSync 的 3 种模式切换——这些知识高度集中在墨衡/墨萱/墨涵三人的头脑中。如果任何一人离开（一周假期、病假、或其他），"回测系统怎么改的"的知识将难以快速传递。

这不是危言耸听——缺乏文档（报告中已承认"插件系统开发指南尚未整理"）使得知识传递只能靠口口相传。

## 四、总体评价

### 工程完成度：✅ 优秀
本阶段实现了预设的全部工程目标：插件化改造、
0 回归迁移、BitableSync 真实 API 对接。墨衡的交付质量过硬。

### 战略完整性：⚠️ 存在缺口

本阶段是一流的架构改造，但**少了"人"和"时间"两个维度**：
- **人**：没有评估终端用户是否能轻松上手新系统，DX 缺失
- **时间**：旧系统下线无排期，缺乏让系统在数月后仍可自动稳定运行的监控/告警/API 变更预警

### 对本阶段打分：7.5/10（方向正确，执行力强，需补全后端保障）

| 扣分项 | 扣分 | 理由 |
|--------|:----:|------|
| 架构与测试 | -0 | 基础扎实 |
| 用户视角缺失 | -1.0 | 无 DX 评估、无原型验证 |
| 过渡期无时间表 | -0.5 | 旧入口下线无承诺 |
| 文档/知识缺失 | -0.5 | 开发指南未整理 |
| 监控/告警未接入 | -0.5 | 运维盲区 |
