# R002 — 先需求后设计规则

## 一、规则陈述

**任何技术方案设计之前，必须先完成完整的需求分析，并经过三方共识确认。**

需求分析未完成的标志：存在未被 Owner 确认的需求分歧。

## 二、适用场景

| 场景 | 是否适用 | 说明 |
|:----|:--------:|:-----|
| 新功能开发（如 ingest_analysis.py） | ✅ | 必须完成需求→评审→定稿→再设计 |
| 已有功能的修复/优化 | ✅ | 小修复可走简化流程，但影响范围涉及多Agent时仍需需求讨论 |
| 数据库DDL变更 | ✅ | 先说明变更原因和影响范围，然后出方案 |
| 纯配置变更 | ❌ | 无需需求分析，直接执行 |
| 紧急故障修复（hotfix） | ❌ | 应急处理，事后补需求记录 |

## 三、标准流程

```
问题描述 → 需求讨论（多方会诊） → 需求文档定稿 → Owner确认 → 开始设计
    ↑            ↑                   ↑              ↑            ↑
  发现需求   墨衡/墨萱/玄知       墨涵汇总         Owner签     墨衡出方案
            各提需求建议          成v1.0定稿
```

## 四、反例（🚫 禁止）

| 反例 | 问题 |
|:----|:-----|
| 直接写代码，不做需求分析 | ❌ 开发结果可能偏离实际需求 |
| 需求就一句话"做XXX"，没有文档 | ❌ 无法验证完成度 |
| 需求讨论只有一个人发言 | ❌ 缺少多方视角，容易遗漏关键要求 |
| Owner确认前就开始设计 | ❌ 设计可能基于未确认的假设，返工风险高 |

## 五、正例（✅ 推荐）

| 正例 | 效果 |
|:----|:-----|
| ingest_analysis.py 需求流程 | ✅ 墨衡/墨萱/玄知三方各提需求→墨萱评审→玄知把关→墨涵汇总→Owner确认→开始设计 |
| 回测流程设计前的需求讨论 | ✅ 明确各方职责边界，达成共识后出方案 |

## 六、与R001的关系

- **R001（问题定义优先）**：适用于所有任务启动阶段，聚焦"我们到底要解决什么问题"
- **R002（先需求后设计）**：适用于需要出技术方案的场景，聚焦"这个方案要满足什么条件"

两者互补：R001把问题框定好 → 然后进行需求分析（R002）→ 再出设计方案
