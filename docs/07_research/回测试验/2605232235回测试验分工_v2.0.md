## 一、角色定位与核心职责

基于四人能力特征与已有职责，不做平均主义分配。

### 墨衡 — 试验执行人

**定位：** 所有需要"跑"的步骤的执行方。

**核心职责：**
- 模板 §2 数据与标的定义的实际操作
- 模板 §3 因子配置与变量矩阵的编码与运行
- 模板 §5 试验流程 Step1~Step8 的各阶段执行
- 模板 §6 数据库写入（backtest_run / validation_check / 废弃标记）
- 脚本编写与维护（data_qc_check.py / mark_abandoned.py / generate_signal_defined.py 等）
- 代码版本冻结与 version_tag 记录
- 预置脚本清单附录C中对应脚本的维护

### 墨萱 — 验收与质量门禁

**定位：** 质量门禁的执行者——有人真的拦，不能让人自己给自己打钩。

**核心职责：**
- 模板 §4 验收标准执行（IC/IR 阈值、P2 自洽性、参数邻域劣化等）
- B级及以上试验的交叉复核（非提案人确认通过条件）
- Layer Q 评级准确性核验（每轮抽选≥1个评级为B级以上的策略，手动复核置信度评分）
- price_rmse / factor_pearson 等数据精度检验的独立验证
- validation_check 入库结果的独立复核
- 发现问题时触发退回流程

### 玄知 — 架构合规与风险预判

**定位：** 跑完之前的"能不能跑"判断，而非跑完之后的结果复核。玄知在前，墨萱在后。

**核心职责：**
- 模板附录B检查清单中架构层条目的审查
- 数据源合规确认（来源为 market_data.db，非 analysis.db）
- A股硬约束是否正确启用（最小交易单位、印花税、价格笼子）
- 参数孤峰风险的预判评估
- 新试验是否与历史失败案例存在同类风险（knowledge.db 自查结果评估）
- schema 变更的 migration 脚本合规审查

### 墨涵 — 问题定义把关与知识归档

**定位：** 流程的归口人——问题定义不对不开工，沉淀没到位不归档。

**核心职责：**
- 模板 §0 R001 问题定义全程主持（四步流程引导+共识确认）
- 确认问题陈述符合"对象+偏差+可观测指标"格式
- 争议超过30分钟时裁定
- 模板 §7 知识沉淀的最终归档确认
- 管理版本状态推进（草案→评审中→准予执行→已归档）
- 不承担代修职责（详见第五节）

## 二、三级会签流程

**原则：** 顺序固定，不可并联。前一环节未完成不得进入下一环节。

### 会签链

```
Step A  墨萱技术签
        核验内容：数据层合规 + 代码层合规 + validation_check 完整性
        核验范围：§2 数据定义、§3 因子配置、§4 验收执行、§6 数据库写入
        输出：技术核验报告

     ↓ 墨萱通过 → 进入下一级
     ↑ 墨萱退回 → 退回起草人修改（≤2次，第3次Owner裁决）

Step B  墨涵知识签
        核验内容：R001 问题定义已确认 + 知识沉淀已到位于 + 归档文件完整
        核验范围：§0 问题定义记录、§7 知识写入确认
        输出：知识归档确认

     ↓ 墨涵通过 → 进入下一级
     ↑ 墨涵退回 → 退回起草人修改（不代修，详见第五节）

Step C  Owner终签
        核验内容：业务方向确认 + 方案可执行性确认
        输出：准予执行 或 终止
```

### 各签字方判定标准

| 签字方 | 通过条件 | 退回条件 | 退回后处理 |
|:-----:|:---------|:---------|:----------|
| 墨萱 | 数据与代码层全部合规，无未解决的问题 | 存在数据层/代码层不合规项，或 validation_check 不完整 | 退回起草人修改，≤2次，第3次Owner裁决 |
| 墨涵 | 问题定义有记录、归档完整、knowledge.db已同步 | R001 记录缺失、知识沉淀未完成或修改不符合要求 | 退回原起草人，注明不合格范围及修改方向 |
| Owner | 业务方向正确、方案可执行 | 方向偏离或可行性存疑 | 终止或要求重新设计 |

### 后续备案工作

- 会签通过后，墨涵负责将 version_status 从"评审中"推进至"准予执行"
- 试验完成后回填阶段，墨涵推进至"已归档"
- 每轮会签结果写入 `backtest_run.notes` 字段

### 阈值变更流程

```

        ↓

        ↓

        ↓
生效

三步缺一不可：Owner 批准 → 墨衡编码 → 墨萱复核
```

## 四、协作触发条件

定义三类协作模式，按需触发。

### 4.1 例行复审

**触发时机：** 每完成一个 EXP 试验后

**主持：** 墨涵

**参与：** 全员（墨衡、墨萱、玄知、Owner）

**议程：**
1. 墨衡简述执行过程与结果（≤5分钟）
2. 墨萱呈报验收结论（≤5分钟）
3. 玄知评估架构合规与风险（≤5分钟）
4. 自由讨论（≤15分钟）
5. 墨涵总结并推进状态（归档/退回/终止）

**时长约束：** 不超过30分钟。超时未结论的争议项进入"船东裁决"通道。

---

### 4.2 履约修改触发（退回重做）

**触发条件：** 任一签字方（墨萱/墨涵/Owner）在会签中发现质量问题

**动作：**
1. 签字方直接退回起草人（墨衡/墨萱），撤回召集全员
2. 退回须附具体修改要求，**不允许空退回**（"有问题请"不算有效退回语句）
3. 起草人修改后重新提交，签字方再次判定

**退回上限：** 同一试验 ≤2 次退回。第3次仍不通过 → 强制触发船东裁决。

---

### 4.3 船东裁决

**触发条件：** 以下任一情况发生：
- 退回满2次后第3次提交仍未通过
- 例行复审中超时30分钟仍无结论
- Owner 认为需要直接介入

**裁决流程：**
1. Owner 给出结论：**"通过"** 或 **"终止"**
2. 结论不可再裁决（终局判定）
3. 裁决结论写入 `backtest_run.notes` 字段
4. 判定"终止"的试验 → version_status 标记为"已终止"，按失败归档流程处理

## 五、不代修原则

### 原则定义

**墨涵在任何环节不承担代修职责。**

具体含义：
- 墨涵在 §0.5 会签及 §7 归档中，职责为**判定通过/退回**，不是修改
- 退回时须注明：**不合格范围 + 修改方向**（实现方式由起草人自行决定）
- 修改由原起草人（墨衡/墨萱）负责，修改后重新提交
- 墨涵对最终结果再次判定，但**不对修改内容承担连带责任**

### 适用范围

| 场景 | 墨涵行为 | 不可为 |
|:----|:---------|:-------|
| §0.5 会签发现不合格 | 退回+注明范围和方向 | 直接在文件上修改 |
| §7 归档发现沉淀不完整 | 退回+注明缺失内容 | 替起草人补写 |
| 墨衡/墨萱产出质量不符合要求 | 退回+附修改要求 | 代他们重做 |
| 2次退回后Owner裁决"修改后通过" | 继续判定 | 直接动手改 |

### 例外（允许代修的场景）

不代修原则不适用于以下情况：
- 格式类错误（缩进/标点/拼写）—— 墨涵可顺手修正
- 模板结构性问题（章节顺序/标记错误）—— 墨涵作为文档维护人可调整
- 其他经 Owner 明确授权的修改
