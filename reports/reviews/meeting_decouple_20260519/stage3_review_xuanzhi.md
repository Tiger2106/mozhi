## 一、Stage 3 裁决审阅

### 1.1 架构框架总体评价：合理，完整度超出预期

Owner 将四个方案定位为"同一架构的不同时间尺度层"而非竞争方案，这个判断本身就是正确的。四方案不是选择题而是拼图。

具体来说：

- **墨衡主线**选型正确。Level 3 作为 Core Architecture，聚焦"Research → Signal → Consumer → Simulator → Validation"完整链路，比单纯物理拆分（Level 1）和"仅产出不可验证"（Level 2）都更具工程完备性。
- **墨萱与墨涵作为约束层**的定位非常好——没有约束层，墨衡的主线架构在执行过程中必然衰减（"不做契约先行、不做环境隔离"是前车之鉴）。
- **玄知Q3路线图延至约束层**是合理的自我约束。Stage 2 我已在报告中明确提出"现在不是做扩展的时机"，Owner 的 Level 5 冻结决策与此一致。

### 1.2 Level 3 冻结线：✅ 可接受，战略上正确

**核心理由：Level 3 是架构 Rescue 的最小完整方案。**

| 维度 | 评价 |
|:-----|:------|
| 完整性 | Research → Signal → Consumer → Simulator → Validation 五步闭环，缺一步都不算"解耦" |
| 风险可控 | 双系统并行验证+5类偏差阈值，迁移途中不会失控 |
| 可回滚 | 旧系统保持不变，新系统偏差超阈值可安全回切 |
| 足够"痛" | 3周、15-18天是刚好能感知压力的时间盒，防止scope蔓延 |

**补充建议**：Level 3 的验证闭环（Validation）在实际执行中容易被先做"主要迁移"后做"对比验证"的顺序所腐蚀。建议墨萱在 Phase 0 就把 Validation 框架的测试桩立好——不要等到 Phase 3 才做对比。**验证框架的进度应当在 Phase 0 就有测试桩，Phase 2 就有可运行的对比能力。**

### 1.3 Q3 延期预期：足够明确，但需要更具体的"触发条件"

Q3 延期方向明确，Level 5 冻结清晰，但有一个风险：

> **现状："Q3 再评估" 没有具体的触发条件。**

当前表述："当策略 > 10 时" 或 "当有跨市场需求时" 是模糊的。如果到了 Q3 策略数还是 4 个，是否意味着 Level 5 永远不做？这会让系统在中期面临新的瓶颈——knowledge.db 的简单 TTL 在 Q3 可能不够用，Signal Store 的设计决策也会一直悬而未决。

**补充建议**：
1. **为 Q3 设定具体的时间锚点**：例如 2026-09-01 为 Q3 路线图的启动评估日，不论策略数是否变化，都要做一次架构健康检查。
2. **为天时（timing）条件设置量化标准**：
   - 策略数 ≥ 8 → 启动 Signal Store 设计
   - knowledge.db 写入量 > 10万条/月 → 启动知识过期治理（v2 TTL）
   - 跨市场接入提案出现 → 启动多市场适配器设计
3. **将 "不做 Level 5" 的决策改为 "Level 5 在 Q3 前处于观察态"**，这样既明确冻结，又保留系统不因冻结而过时的安全性。

### 1.4 补充战略建议

| 建议 | 优先级 | 理由 |
|:-----|:------:|:------|
| Phase 0 结束前完成 Validation 测试桩 | P0 | 验证闭环不在 Phase 3 才构建，必须从第一天就开始 |
| Phase 1 策略重构采取"最小重写"原则 | P1 | 策略代码只需剥离交易依赖、改为输出 Signal，不重构逻辑本身 |
| Phase 2 knowledge.db 环境隔离与降级策略应在 Phase 1 并行 | P1 | 墨涵的环境隔离是墨萱红线 #2/5 的落地，不应在 Phase 2 才单独做 |
| 建立"冻结线文档化" | P2 | Level 1-3 vs Level 5 的具体范围差异写入 ADR-004 的决策日志，防止 1 个月后范围理解偏差 |

## 二、Signal Protocol v1 审阅

### 2.1 Schema 设计：ML/多市场扩展覆盖充分

**总体评价：Core 最小化 + Extension 预留的设计是当前阶段的最佳选择。**

#### 对 ML 扩展的覆盖

| ML 扩展需求 | 当前覆盖 | 评价 |
|:------------|:---------|:-----|
| 模型特征权重（feature importance） | ✅ `extras.ml.feature_importances` 预留 | 命名空间约定足够清晰 |
| 因子解释（factor PE ratio, etc.） | ✅ `extras.factor.*` 预留 | 预留 |
| 模型元信息（model version, ensemble weights） | ✅ `extras.ml.*` 预留 | 有余 |
| ML 特有的信号类型（如 "ml_signal"） | ⚠️ `signal_type` 枚举只有 `trend/reversal/grid` | ML 信号可能需要新的 `signal_type` 值，这需要 MAJOR 升级（见 §4.2 "枚举值新增必须 MAJOR 升级"）。但 v1 的 ML 信号很可能是以上三种类型的子集，当前策略可接受。如果 Q3 引入独立 ML 策略，建议在 MAJOR 升级到 2.0 时扩展 signal_type 枚举。 |

#### 对多市场扩展的覆盖

| 多市场扩展需求 | 当前覆盖 | 评价 |
|:---------------|:---------|:------|
| 跨市场扩展 | ✅ `extras.market.*` 预留 | 命名空间清晰 |
| symbol 不拼接市场前缀 | ✅ 规范明确规定 `symbol` 不含 `.SZ/.SH` | 正确设计——避免 schema 污染 |
| 多市场时序对齐 | ⚠️ 未覆盖 | `extras.market.session` 或 `extras.market.alignment` 可在 Q3 补充，v1 不做 |
| 期货/期权特有字段（contract month, strike, option type） | ⚠️ 未覆盖 | 由 `extras.market.*` 承载，v1 不做 |

**结论**：Schema 的 ML/多市场扩展覆盖度达 80-90%，未覆盖的剩余 10-20% 可通过 `extras` 承载。v1 阶段够用，Q3 ML 策略接入时补充即可。

### 2.2 生命周期状态机：合理但有一个缺口

**状态机设计（GENERATED → SERIALIZED → TRANSMITTED → CONSUMED → ARCHIVED / EXPIRED）** 基本完整，覆盖了信号从产生到消亡的全流程。

**亮点**：
- EXPIRED 状态设计合理——不是所有信号都能被消费（超时/过期），标记而不删除是正确做法。
- TRANSMITTED 阶段的重试机制（最多 60 分钟 → EXPIRED）阈值合理。
- 消费确认记录（consumed_{signal_id}.json）提供可追溯性。

**缺口：缺少 REJECTED（风控/规则拒绝）的显式状态。**

当前设计中，风控过滤被归入 CONSUMED 阶段的子状态（`decision: "FILTERED"`）。这有逻辑问题：
- 风控/规则过滤后的信号，严格意义上没有被"消费"，数据流被终端层中断。
- 长期来看，风控过滤的数据需要独立分析（什么类型的信号容易被风控拦截？），作为子状态记录在 consumed 目录下不适合做查询和统计。

**补充建议**：将 `FILTERED` 升级为独立生命周期状态 `REJECTED`，与 `CONSUMED` 并列；或者至少新增 `signals/rejected/` 目录专门存放被风控拦截的信号记录。

### 2.3 兼容性矩阵：充分

兼容性矩阵覆盖了所有核心场景：

| 场景 | 覆盖 | 评价 |
|:-----|:----:|:------|
| 同版本正常消费 (TC-01) | ✅ | 基本场景 |
| 跨 MAJOR 版本拒 (TC-02) | ✅ | 降级与拒绝边界清晰 |
| 同 MAJOR 未知 extras (TC-03) | ✅ | 降级策略 A（字段忽略）验证完整 |
| 空字段边界 (TC-04) | ✅ | 边界用例覆盖完整 |
| MINOR 版本降级 (TC-05) | ✅ | 向前兼容验证 |
| 降级优先级（A→B→C） | ✅ | 策略链清晰 |

**一个建议**：增加 **TC-06: MAJOR 降级（v2 Consumer 消费 v1 信号）**。当前 §4.1 兼容矩阵中有此场景（"降级消费-默认值填充"），但测试用例中没有对应的 TC。建议补充，确保 v2 Consumer 对 v1 信号有可执行的降级验证。

### 2.4 战略视角补充建议

#### 建议一：v1 阶段就预留 telemetry 接口

当前协议没有考虑**信号质量监控**——即如何事后衡量信号本身的预测质量（signal quality score）。Signal Protocol v1 的 CONSUMED 阶段的消费记录只记录了"信号是否被执行"，但没有记录"信号的最终预测误差是多少"。

当前可以不做，但建议在 `extras` 中预留一个 `quality.*` 键命名空间，在 parquet schema 中也预留一个可选字段。这样 Q3 引入信号质量监控时可以自然接入，不需要更改 Core。

```
extras.quality.predicted_price  → 信号预测的目标价格（若有）
extras.quality.signal_score     → 信号历史预测精度（回填）
```

#### 建议二：Signal ID 的引用追踪能力

当前 `signal_id` 只用于去重。如果未来（Q3）要分析"一个信号如何演化为最终订单"的数据链路，目前没有引用机制。建议在 extras 中预留 `trace.parent_signal_id` 键，用于关联衍生信号与父信号。

短期用不到，但预留命名空间不会增加当前成本。

#### 建议三：parquet schema 的兼容性风险

v1 将 `extras` 序列化为 `STRING(JSON)`，这保证了灵活性但也意味着：
- extras 中的嵌套结构在 parquet 中丢失了 schema 级约束
- 分析查询时需要解析 JSON 字符串，性能打折扣

这并非 bug，而是在灵活性与结构化之间做的合理取舍。但在 Q3 评估 Signal Store 时，**应重新评估是否需要从 `STRING(JSON)` 迁移到 `STRUCT` 或 map 类型**，尤其是当 signal 数量达到百万级时。
