## 问题 1：Q9 双层结构目录命名混用

### 现状分析

| 文档位置 | 使用名称 | 问题 |
|:---------|:---------|:-----|
| §3.3 标题 | "Q9: Failure Registry" | 未区分 a/b 层 |
| §2.3（Phase 0b 任务清单） | `q_failures`（目录名）+ `Q_FAILURES` | 仅指 Q9a |
| §1.6~1.7（Phase 1~2 任务清单） | `RESEARCH_FAILURES` + `research_failures_ext/` | 仅指 Q9b |
| §3.3 存储方案 | `research_failures/` 目录 | 命名与 Q9b 概念冲突 |

三个命名出现在同份文档中，读者容易混淆。

### 技术分析

**命名混用的实际风险**：
1. **代码结构不一致**：现有命名 `q_failures` vs `research_failures` vs `RESEARCH_FAILURES` 三个不同变体，导入路径、模块引用、文档索引都将面临选择困难
2. **引用关系模糊**：§1.7 提到"Q9b RESEARCH_FAILURES 与 Q9a Q_FAILURES 交叉引用查询"，但交叉引用的标准字段（strategy_id + failure_id）没有在表结构中体现
3. **Q9b 表结构独立**的决策正确，但目录名若用 `research_failures_ext/`，暗示这是"扩展附加品"，弱化了 Q9b 与 Q9a 的并列地位

**Owner 决策中的命名逻辑值得保留**：
- **Q9a Q_FAILURES** = 正式审计失败，用途是可信度治理 → 强调"失败"这个状态本身
- **Q9b RESEARCH_FAILURES** = 全量研究失败，用途是长期知识积累 → 强调"研究"这个来源
- 两个名称的语义差异是合理的，但**文档内使用需要统一**

### 修改建议

#### 建议 A1：文档内统一命名体系

| 建议名称 | 对应模块 | 目录名 | 用途 |
|:---------|:---------|:-------|:-----|
| **Q9a / Q_FAILURES** | 正式审计失败层（Phase 0b） | `q_failures/` | 可信度治理 |
| **Q9b / RESEARCH_FAILURES** | 研究扩展层（Phase 1~2） | `research_failures/` | 知识积累 + Meta Research |

#### 建议 A2：修改文档中的具体位置

1. **§3.3 标题**：`"Q9: Failure Registry（失败数据库）"` → `"Q9: 双层失败数据库（Q9a Q_FAILURES + Q9b RESEARCH_FAILURES）"`
2. **§2.3 任务 0b-1**：`q_failures/ 目录结构` 确认统一为 `q_failures/` 不变
3. **§1.6 任务 1.6**：`research_failures_ext/` → `research_failures/`（去掉 `_ext` 后缀），强调 Q9b 的独立地位而非"附加品"
4. **§3.3 存储方案**：目录树中的 `research_failures/` → 标注为 `Q9b RESEARCH_FAILURES 的存储位置`，避免与全称混淆
5. **§6.3 对比表**：`"代码库: Q1~Q9 + FR"` → `"代码库: Q1~Q8 + Q9a(Q_FAILURES) + Q9b(RESEARCH_FAILURES)"`

#### 建议 A3：交叉引用表结构补充

§3.3 Q9-A 记录规范的 `strategy_id` 和 `failure_id` 在 Q9a 和 Q9b 中各自存在（当前文档只列了一套字段）。建议明确：

```
Q9a Q_FAILURES 表: 字段如当前文档定义
Q9b RESEARCH_FAILURES 表: 字段同 Q9a + 额外扩展字段（如 yield_reference, external_source）
交叉引用：通过 (strategy_id, failure_id) 关联，支持跨层查询
```

### 实施影响评估

| 维度 | 影响 |
|:-----|:-----|
| **工时** | 0（纯文档修正） |
| **代码结构** | 中等——需统一 Q9b 目录名和模块导入路径 |
| **可读性** | 大幅提升——读者无需自行配对三个名字 |
| **一致性** | 提升——命名体系在文档、代码、注释三者间对齐 |

## 问题 2：ExistenceValidator C1 硬门禁 vs 主报告案例矛盾

### 现状分析

| 位置 | 内容 |
|:-----|:------|
| **§5.2** | `C1: 最小交易数 >= 30`，且标注为"硬门禁"（Hard Gate）：不到30笔直接 `exists=False`, `confidence=0.0` |
| **§2.2 任务 0a-3** | `601857 网格通过 ExistenceValidator`（作为第一个 Pass 案例） |
| **v3 报告案例** | 601857 网格的实际参数为：**84 天 / 2 笔交易** |

这是一个 **内在矛盾**：如果 601857 网格确实只有 84 天 / 2 笔交易，它连 C1 都过不了，也就无法成为"首个通过案例"。

### 技术分析

**矛盾根因**：601857 网格是当前研究的实际案例。它的真实交易笔数取决于回测参数设置。若全周期回测仅产生 2 笔交易，则 C1 硬门禁将其拦在门外。

**可能的实际场景**：
- 场景 A：601857 网格在全周期（如 3 年）回测中确实只有 2 笔交易 → C1 通不过 → 文档案例矛盾
- 场景 B：601857 网格的参数扫描中存在交易数 >= 30 的参数组合 → 可以通过 C1 → 文档表述不够精确
- 场景 C：601857 网格在**某个时间切片**中满足条件 → 可以通过但不代表整体

### 修改建议

#### 建议 B1：明确 601857 案例的实际状态

补充一段过渡说明，定位在 §2.2 任务 0a-3 附近或 §5.2 末尾：

```markdown
**关于 C1 硬门禁与当前案例的说明**：

根据当前数据分析，601857 网格（84 天 / 2 笔交易）确实无法通过 C1 硬门禁（>= 30 笔）。
此情况本身恰恰说明了 C1 的价值——在统计上，2 笔交易不足以构成 alpha 存在的证据。

两种可能的前进路径：

1. **参数优化后重新评估**：扩展回测参数空间，寻找交易频次 >= 30 笔的参数组合。
   若存在这样的组合，601857 可通过 ExistenceValidator 进入后续审计流程。

2. **作为"过渡期标本"**：承认当前版本通不过 C1，将其记录在 Q9a Q_FAILURES 中
   作为基线案例。随着研究参数集的扩展（长期回测、多品种验证），当其满足 C1
   条件时，重新提交 ExistenceValidator 评估。

> **设计启示**：C1 硬门禁不是为了"刁难"现有案例，而是为了防止"伪发现"污染后续分析。
  当前案例无法通过，正说明此案例需要更多工作才能证明其统计可信度。
  这也是 Q9a Q_FAILURES 存在的意义——记录失败的案例，追踪其后续变化。
```

#### 建议 B2：修正 §2.2 任务 0a-3 描述

当前：`"验证 601857 网格通过 ExistenceValidator"`
建议改为：

```markdown
| 0a-3 | 验证必选通过案例通过 ExistenceValidator | 墨衡 | 0.1天 | ✅ 已知通过案例 |
| 0a-3b | 验证 601857 网格的 ExistenceValidator 结果 | 墨衡 | 0.1天 | ✅ 记录当前状态至 Q9a |
```

或将其修改为 **"验证 ExistenceValidator 拦截已知失败案例"**，与 M0 验证条件对齐。

#### 建议 B3：M0 验证条件明确化

当前 M0 验证方法：`"3 个已知伪发现被拦截"`
建议补充：`"第 4 个为 601857 网格的实际状态验证——结果如实记录至 Q9a"`

## 问题 3：账本B B4 规则在小团队中的执行风险

### 现状分析

§4.4 B4 规则原文：

>❌ 同一研究者不得同时参与同一策略的账本A生产和账本B审计

这是"研究者不能自己给自己盖章"原则的落地。但在当前小团队结构中：

- **墨衡**：负责 P 层（账本A 生产工具、回测管线、参数扫描） + Q 层（Q1/Q3/Q8/Q9a/Q9b 核心审计逻辑）
- **墨萱**：负责 Q 层（Q2/Q4/Q5/Q7/G3 审计模块 + 数据库基础）
- **墨涵**：调度+文档

**现实**：墨衡既是 P 层核心开发者，又是 Q 层核心开发者。若严格执行 B4，墨衡不得参与他自己生产的策略的审计，但实际上他负责开发和维护大部分审计工具。

### 技术分析

**完全执行 B4 的障碍**：
1. **代码级分离困难**：墨衡写 P 层回测管线 → 也写 Q1 ExistenceValidator → Q1 直接处理 P 层输出。代码层实现分离需要接口隔离设计
2. **知识分离困难**：墨衡知道 P 层的设计细节和已知局限性 → 设计 Q 层检查项时无法"假装不知道"这些信息
3. **人力分离困难**：仅 2 人（墨衡+墨萱）承担技术实现，按照 B4 规则，任何策略审计都需墨萱独立完成账本B——但墨萱的技术能力覆盖 Q2/Q4/Q5/Q7/G3，并非全部

**不做缓解的风险**：
1. 审计独立性受到质疑——特别是主人视角，"同一个开发者的两套工具"是否足够可信
2. 墨萱的审计签署责任——若墨衡的代码有隐藏缺陷，墨萱作为审计方如何放心地签署

### 修改建议

#### 建议 C1：新增"小团队旅行条款"

在 §4.4 B4 禁止行为之后补充：

```markdown
#### B4a: 小团队旅行条款（适用条件：开发团队 ≤ 3 人）

当团队规模无法实现角色完全分离时，采用以下等效保证措施：

| 编号 | 措施 | 执行方式 | 责任方 |
|:----:|:-----|:---------|:------:|
| T1 | **接口隔离** | 账本A与账本B之间通过标准化的中间数据接口通信，任何开发者在账本B中仅接触接口输出（如 P 报告的 JSON），不直接操作账本A 的代码或数据库 | 墨衡实现，墨萱验收 |
| T2 | **交叉审查** | 账本B 的审计逻辑代码必须由另一人审查，且在 Q 层的 Audit Trail 中记录"代码审查者"字段。墨衡写的 Q 模块必须由墨萱审查通过 | 墨衡→墨萱，墨萱→墨衡 |
| T3 | **盲测机制** | 每季度选取 1-2 个策略，由墨萱独立编写"副本审计工具"对比 Q 层结果。若偏差 > 阈值，触发自动升级至主人 | 墨萱实施 |
| T4 | **审计日志透明** | 所有 Q 层判定（特别是硬门禁拦截），均记录审计者身份和代码版本。墨衡的判定和墨萱的判定可追溯、可区分 | 墨萱实施 |
| T5 | **签署责任声明** | 墨萱作为审计签署方，在每次 Q 层审计报告末尾注明："本审计结果基于团队最小分离原则（§B4a），存在代码级小团队依赖。审计独立性的等效保证措施见 Q 层审计日志。" | 墨萱签署 |
```

#### 建议 C2：B4 规则文本调整

当前 B4 表述为禁止行为（❌），建议改为**原则性目标 + 小团队例外条款**的结构：

```markdown
#### B4: 职责分离原则（理想状态） + 小团队旅行条款

**原则**：账本A 的生产者不得作为同一策略的账本B 审计者。

**小团队例外**：当前团队规模 ≤ 3 人时，执行 §B4a 等效保证措施以替代完全角色分离。
当团队规模 > 5 人时，应回归 §B4 的完全分离原则。
```

#### 建议 C3：审计日志结构补充

在 §9b 或在 Q 层审计日志中补充字段：

```json
}
```
