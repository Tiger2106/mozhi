## 二、影响度定级核验（逐项审查）

### 2.1 🔴 致命级（2项）

#### #1 无T+1交易延迟 — 🔴 致命 ✅ 维持

**核对**: `run_portfolio_backtest()` 调仓循环完全无 `lockup_period` 检查。标的可以在同一交易日买入后立即卖出。
- 日频调仓策略下，T+1 约束直接改变持仓周期
- 代码确认: 增量买入(57行)和卖出(65行)在同一次 `for date` 迭代中独立执行，无互斥锁
- **结论**: 致命级正确，无争议

#### #2 前视偏差(信号当日使用) — 🔴 致命 ✅ 维持

**核对**:
- `compute_signal_vector()` 使用 `sd["volume"]` → `calc_vol_rsi_std()` 计算当日 `vol_std`
- `classify_market_state()` 使用 `sd["close"]` 计算状态
- 生成的复合信号立即用于当日交易
- 数据流: `volume[t] + close[t] → signal[t] → trade at close[t]`
- **结论**: 致命级正确。这是回测中最严重的系统性偏差来源

### 2.2 🟠 重大级（4项）

#### #3 无滑点模型 — 🟠 重大 ✅ 维持

**核对**: 交易价格均为裸 `stock_data[t]["close"][idx]`，无任何价差调整
- **结论**: 重大级合理。对大盘标的(如600519.SH)影响有限，但系统性地高估收益率

#### #4 无涨跌停约束 — 🟠 重大 ✅ 维持

**核对**: 买入卖出逻辑均无涨跌停价格检查
- **注意**: DB 中已包含 `high`、`low`、`pre_close` 字段
  - 涨跌停价可直接由 `pre_close * 1.10`(主板) / `pre_close * 1.20`(创业板/科创板) 计算
  - 无需额外数据源
- **结论**: 重大级合理。但因DB已有基础数据，实现难度低于审计报告估计

#### #5 无交易量容量约束 — 🟠 重大 ✅ 维持

**核对**: 20万初始资金，分散至10只大市值标的，单只交易量占日成交比极低
- **结论**: 重大级合理，且审计报告已正确标注"对20万资金影响轻微"

#### #6 无停牌处理 — 🟠 重大 → 建议降为 🟡 轻微 ⚠️

**核对代码**: 发现引擎已存在隐式保护机制

```python
date_to_ticker_idx[date].get(t)
# 如果股票在该日期无数据 → idx is None → 股票不进入valid列表
# 不进入valid → 不产生买入/卖出操作
```

**具体分析**:
| 操作 | 停牌日行为 | 评价 |
|:----|:----------|:-----|
| 买入 | `date_to_ticker_idx[date].get(t) is None` → 不进入valid → 不买入 | ✅ 正确 |
| 卖出 | `positions[t] > target` 但 `idx is None` → `continue` → 跳过卖出 | ✅ 正确 |
| 持仓估值 | 使用停牌前最后收盘价估值 | ⚠️ 合理，会计上可接受 |
| 复牌首日 | 正常恢复交易，但无涨跌停约束 | ⚠️ 此问题由#4覆盖 |

**结论**: 代码的实际行为比审计报告描述的更好。停牌引起的特殊问题仅为：
1. 复牌首日涨跌停约束缺失（已被#4覆盖）
2. 停牌期间使用了暂停前的价格（会计上合理且无可避免）

**修正建议**: 降为 🟡 **轻微级**。真正的问题已被#4覆盖，停牌本身并不产生新的功能性缺陷。

### 2.3 🟡 轻微级（3项）

#### #7 仅收盘价交易 — 🟡 轻微 ✅ 维持

**核对**: DB 已包含 `open` 字段。当前代码仅使用 `close`。
- 当前定价逻辑对回测精度影响有限

#### #8 无印花税 — 🟡 轻微 ✅ 维持

## 三、漏项检查

### 3.1 重要漏项

#### 漏项1: 分红现金流未计入 ⚠️ 重大

**问题描述**: 审计报告只提到#9除权除息对K线价格的影响，但**未处理分红现金流的归属**。

**影响路径**:
- 除权除息日，股价下调但实际收到现金分红（现金股息）
- 代码中无分红收入入账逻辑 → 现金流被低估
- 对长期回测（2015-2025，10年）的累计影响不可忽略
- 尤其对于600519.SH等高分红的标的，每年现金分红可达数元/股

**量化估计**: 对10只大市值标的、10年回测期，累计分红漏计约为初始资金的 3-8%

**当前代码确认**: `run_portfolio_backtest` 中现金更新仅来源于买卖操作，无分红收入入账

#### 漏项2: 日期对齐歧义 — 2018熊市窗口 ⚠️ 重大

**问题描述**: `extract_window()` 使用 `start <= date <= end` 做字符串比较。熊市窗口定义为 `("20180101", "20181231")`，但：
- `trade_date` 在DB中为 `YYYYMMDD` 格式字符串
- 字符串比较在日期范围内**恰好可用**（YYYYMMDD按字典序与时间序一致）
- 但 `CRASH_2015 = ("20150601", "20160131")` 有**跨年**问题：20150601 ≤ 20151231 ✓，但需注意起始日是否精确对应交易日
- 需要注意: 窗口边界日期为非交易日时，`extract_window` 的 `start <= date <= end` 会截断非交易日两侧的数据，引擎需要确认窗口的精度要求

**风险**: 低风险但应标注。字符串日期比较YYYYMMDD格式恰好与时间顺序一致。

### 3.2 次要漏项

#### 漏项3: 信号系统中市值变迁未处理 🟡 轻微

标的池在2015-2025年间市值排名可能大幅变化。当前策略固定10只标的并等权持有，未考虑10年间龙头股的更替（如恒瑞医药、中国平安的市值变迁）。这不是回测引擎的缺陷，而是策略固定的假设。但审计报告作为引擎评估，可以提及此点。

### 5.2 争议项详细说明

#### ❌ 收盘价→开盘价交易（P2→P1）必须联动提升

**理由**:

前视偏差修复是P0级的逻辑变更。当前设计是 `signal[t] → trade at close[t]`。前视偏差修复后，信号变为 `signal[t-1] → trade at time[t]`。此时：
1. 如果用 close[t] 成交 → 产生新的时序错配（T-1信号在该日收盘执行，但信号在T日开盘已可用）
2. 正确做法应该是 `signal[t-1] → trade at open[t]`
3. **前视偏差修复和开盘价交易是同一件工作的两面，不能拆成两个优先级**

**就像换了引擎不换变速器**。

**建议做法**: 将 P2 的"收盘价→开盘价交易（2h→1h）"并入 P1，跟随前视偏差修复一起完成。

## 六、代码级额外发现

### 6.1 现金余额初始化的顺序问题 🟡 轻微

在 `run_portfolio_backtest()` 第40-53行:

```python
cur_equity = cash
# ... 计算持仓市值 ...

if di == 0:
    cur_equity = INITIAL_CAPITAL

equity_curve.append({"date": date, "total_equity": round(cur_equity, 2)})
```

- 对 `di == 0` 使用 `INITIAL_CAPITAL (200,000)` 覆盖 `cur_equity`
- 对非首日使用 `cash + 持仓市值`
- 问题在于: 调仓前记录的 `cur_equity` 包含了**前一日现金 + 当日持仓市值**，但每只股票的当日借以成交的close price**已经包含了当日涨跌幅**
- 也就是说，前一日现金 + 当日持仓 = 当日总权益，这是正确的结算逻辑
- **结论**: 此处不是BUG，但代码注释不够清晰

### 6.2 `valid` 列表仅检查了 `sig_arr[idx]`，未检查信号前向填充

代码对 `valid` 列表仅要求当日 `sig_arr[idx]` 不为NaN，但：
- `calc_vol_rsi_std()` 前 `rsi_period + std_period = 34` 天的信号为NaN
- 前34天中**不会产生任何交易信号**，但持仓估值仍进行
- **结论**: ✅ 行为合理，但文档中应说明信号启用延迟

### 6.3 交易策略只用做多信号，未分析做空场景

当前 C1-RWC 策略为**全仓做多**，信号反向(negative)意味着更强买入信号，正信号意味着较弱买入信号。分析中未涉及做空场景。
- **结论**: 不影响当前回测结论，策略设计层面不需要修改
