# T+21全量运行三层根因分析（玄知独立版）

> ⚠️ 本分析基于现象描述文档 v5 中的可观测事实，与 moheng 的 RCA v1 基于同一组数据。两套分析相互独立，部分结论可能存在交叉或差异，属于正常分析多样性。

---

## 问题1：管线运行时被 SIGKILL 杀死

### 第一层（直接原因）

管线进程（marine-s 和 mellow-o）被操作系统 OOM killer 强制终止。系统总物理内存 15.6GB，事后查询可用内存仅 1.76GB（约11%）。两次杀死均发生在进程启动后 ~15-17 分钟，符合 OOM killer 在内存耗尽场景下的典型行为模式——检测到内存紧张后挑选内存占用大户（管线进程）发送 SIGKILL。

### 第二层（间接原因）

系统内存存在**长期高压状态**，管线之外的常驻进程持续占用绝大部分物理内存——证据链如下：
- 第1次运行（16:43~17:00）被杀后，第2次运行（17:27）启动时需间隔27分钟才释放前一次管线RSS，但启动后仍仅15分钟就再次被杀
- 这意味着杀死 marine-s 释放的管线RSS并没有使系统回到安全水位——系统在管线启动前本就处于内存高压状态
- 最可能的常驻内存大户是 vLLM 推理服务（通常占用数GB显存和系统内存）
- **核心证据缺口**：无杀死时刻的实时内存快照（可用内存、管线进程RSS、其他进程RSS均不可追溯），因此"vLLM 是元凶"属于合理推断而非实证结论

### 第三层（系统性原因）

> 与 moheng 的分析相比，本分析更强调 **三层防御缺失** 视角：预防层、隔离层、恢复层。

1. **【预防层】缺乏预运行环境健康检查**
   - 管线启动前无 `available_memory >= required_memory + buffer` 的判断逻辑
   - 若在 16:43 启动 marine-s 前检查可用内存并发现不足4GB，应中止并报警，而非直接启动
   - 这是"第一次犯错就被杀死"的根因——系统本来有机会拒绝执行

2. **【隔离层】缺乏进程级资源约束**
   - 管线与 vLLM 等进程共享同一物理内存空间，无 memory cgroup、容器（Docker/Podman）或 systemd 资源限制
   - 一个进程的内存泄漏或高峰不会波及其他进程——本应如此，但当前架构无任何资源隔离措施
   - **这是最根本的系统级缺陷：共享资源无配额**

3. **【恢复层】缺乏 graceful shutdown 和断点续传**
   - SIGKILL 不可捕获、不可处理，被杀管线无法做清理工作
   - 管线没有 checkpoint 机制——被杀后只能重头跑（第2次就是从头重跑），无断点续传能力
   - 第3次使用 `--no-run` 跳过管线属于外围补救（手工变通），不是系统自带的恢复能力

4. **【横向对比】内存画像缺失**
   - 管线进程的峰值 RSS 需求从未被测量和记录
   - 没有运行时内存水位监控和告警（下降到阈值触发警告 vs 直接 OOM 杀死）
   - 被杀后无法精确回答"管线到底需要多少内存"这个基本问题

**交叉关系：** 问题1 直接导致问题2 的数据覆盖范围受限（第2次只产出到 2026-05-19，缺少后续截面）。第三次使用 `--no-run` 技巧性绕过，使基线验证得以完成，但这掩盖了管线层面的复现性问题——"管线死两次，第三次靠手工绕开"这不是可复用的解决方案。

## 问题2：黄金基线 verdict=FAIL（动量20日）

### 第一层（直接原因）

验证脚本 `validate_golden_baseline.py` 基于 415 个有效 momentum_20d 窗口（rank_ic NOT NULL），计算 5 项基线指标后判定：
- mean_ic_abs=0.221 ≥ 0.02 ✅（仅通过）
- positive_ratio=0.487 < 0.50 ❌
- ic_std=0.271 > 0.20 ❌
- significant_ratio=0.248 < 0.30 ❌
- half_life=0.47周 ∉ [4,12] ❌

**4/5 未通过 → Verdict=FAIL 是脚本逻辑的正确执行结果，不是软件缺陷。**

### 第二层（间接原因）

momentum_20d 因子在 A50 截面（2007~2026，近20年）上表现出**不符合黄金基线预期的统计特征**：

- **正负方向几乎均衡**（positive_ratio=0.487，非常接近随机水平的 0.50）→ 说明该因子在 A50 上择时能力极弱
- **IC 波动极大**（ic_std=0.271）→ 单个截面的 IC 值非常不稳定，信号嘈杂
- **显著截面比例低**（significant_ratio=0.248）→ 大部分截面 IC 不显著
- **IC 记忆量极短**（half_life=0.47周）→ 动量效应在 A50 截面半衰期约3天多，几乎无延迟持续性

这些指标一致指向 momentum_20d 在 A50 截面上的**因子表现确实不佳**——这不是数据错误，而是统计上的真实表现。与之对比，若 half_life 接近 [4,12] 周但 ic_std 略高，还可以归因于样本量不足带来的抽样误差；而 0.47 周远低于 4 周下限，差异达 **1 个数量级**，暗示动量因子在 A50 日频截面上的衰减速度远超预期。

### 第三层（系统性原因）

> 与 moheng 的分析相比，本分析在两个方面有不同判断：**不认为数据覆盖不完整是问题2的第三层根因**（415个窗口已足够做统计判断），但**对阈值问题提出了更批判性的质疑**。

1. **【核心缺陷】黄金基线阈值缺乏校准流程**
   - mean_ic_abs ≥ 0.02 门槛很低，动量因子轻松通过——这反而暴露了问题：当只有"极低门槛"通过而其他四项全军覆没时，说明阈值系统不是在"筛选有利因子"，而是在"排除明显异常的因子"
   - half_life ∈ [4,12] 周的区间标准对于 A50 截面动量因子的合理性**从未被验证过**。如果一个因子的真实 half_life 是 1 周（远短于 4 周但仍有预测价值），基线系统会直接判 FAIL——这是一种**阈值偏差风险**
   - **建议**：对验证脚本中每项阈值标注来源和实证依据，或改为 z-score/percentile 的相对评判方式（与同类因子对比），替代绝对二值判断

2. **【流程缺失】FAIL 后无诊断自动化**
   - 现阶段 verdict=FAIL 等于"问题被记录，无人处理"
   - 正常流程应该包含：脚本自动输出"哪些因子在同期窗口通过/未通过"、"与历史基线运行对比差异"、"建议调整阈值或淘汰因子"
   - 现在的结果就是一个 JSON 文件，信息熵约等于"4/5 failed"

3. **【无关要素澄清】数据覆盖范围不是问题2的根因**
   - 415 个有效窗口覆盖 2007~2026 近20年，样本量充足
   - 第2次运行被 SIGKILL 导致缺少 2026-05-19 后的最新数据，但最新几个月的截面不会将 positive_ratio 从 0.487 拉过 0.50 的门槛——这不属统计上的"样本不足"，而是"缺少最新尾段"
   - 因此，**将问题1列为问题2的"数据覆盖不完整根因"可能高估了 SIGKILL 对基线判定的实际影响**

**交叉关系：** 问题2 与问题1 的数据来源有关（SIGKILL 限制了数据范围），但 **不影响 verdict=FAIL 的根本结论**。问题2 与问题3 无依赖关系，仅在候选因子池可用因子数上有叠加效应（18→14）。

## 问题3：估值因子 IC=0（数据源缺失）

### 第一层（直接原因）

管线计算估值因子 IC 时所需的底层数据全部不可用：
- `stock_daily.pe`、`stock_daily.pb`、`stock_daily.pe_ttm`：**列定义存在，值全部为 NULL**（2007~2026，20年全覆盖）
- `stock_daily.ps_ttm`、`stock_daily.pcf_ttm`：**列定义本身不存在**
- 备选表 `a50_daily_ohlcv` 同样完全无估值数据

因此估值 4 因子在全部截面上 rank_ic 为 NULL/0。

### 第二层（间接原因）

这一层分为两个不同性质的问题：

**A. pe/pb/pe_ttm 三列：akshare 数据源接口不返回估值字段**
- 字段在 Schema 中有定义，写入代码也尝试写入了该字段，但 akshare 返回的数据结构中不包含或为空
- 数据库最终呈现为 None——这不是"写代码时跳过了估值字段"，而是"写入了但源本身没有"（与 moheng 一致判断）
- 背后原因可能是：akshare 免费版的 A50 成分股数据接口本身就缺少估值字段（付费版才提供，或者只提供 K 线数据而非基本面数据）

**B. ps_ttm/pcf_ttm 两列：Schema 设计阶段未纳入**
- 这属于纯粹的设计疏忽：stock_daily 表的列定义中根本没有这两个字段
- 即使 akshare 返回了 ps_ttm 和 pcf_ttm 数据，管线也没有地方写入
- 需要额外 DDL 操作（ALTER TABLE ADD COLUMN）才能入库

### 第三层（系统性原因）

> 与 moheng 的分析相比，本分析对 **A 类（列存在但值为NULL）和 B 类（列完全不存在）做了区分对待**，并且对"为什么直到全量运行才发现"提出了更尖锐的流程质疑。

1. **【最高优先级缺陷】数据采集阶段无数据质量门禁**
   - 管线应当（且应当在多个环节）设置数据质量验证：
     - **首次 ingestion 后**：`SELECT COUNT(*) FROM stock_daily WHERE pe IS NOT NULL LIMIT 1` — 如果返回 0，立即告警
     - **因子计算前**：检查依赖字段的空值比率，若 >90% 则跳过计算并记录
   - 当前流程中这些门禁全部缺失，导致估值因子"悄悄"产出 20 年全 NULL IC 而不被人知
   - **关键追问**：估值因子的 IC 数据是从 T+21 全量运行才开始产出的吗？还是之前增量运行中就一直产出 NULL？如果是后者，管线在开发阶段的函数调试中就应该发现这个问题——这说明**因子计算的单元测试覆盖了但可能没检查数据有效性，或根本无单元测试**

2. **【架构缺陷】数据源依赖关系未显式建模**
   - 每个因子模块应当声明其依赖的底层数据表和字段列表
   - 管线启动时应当检查这些依赖是否可用（schema 存在 + 字段有非空值）
   - 当前做法是"先算再说，算出来是NULL就是NULL"——这是最差的设计选择

3. **【设计缺陷】无数据源降级/替代方案**
   - 单一数据源（akshare）是单点故障
   - A50 是重要研究标的，其成分股的基本面数据至少应该有一个备选数据源（如 tushare pro、Wind、聚宽）
   - 估值数据可以由 "stock_daily + 定期财报数据" 计算替代，不一定依赖第三方直接提供 PE

4. **【流程缺陷】列定义与因子需求的同步机制缺失**
   - ps_ttm 和 pcf_ttm 列完全不存在，说明因子开发/数据库变更/数据采集三个环节之间没有协调流程
   - 如果 factor_valuation.py 需要的 4 个字段中有 2 个数据库里根本没有，这条链路在设计评审时就应该暴露

**交叉关系：** 问题3 与问题1 完全独立（即使无 SIGKILL，估值因子仍是 0）。与问题2 仅在候选因子池缩减层面有叠加效应，判定本身互不干扰。

## 问题间交叉关系总览（玄知独立评估）

| 路径 | 关系类型 | 玄知评估 |
|:----:|:-------:|:--------|
| 问题1 → 问题2 | 数据覆盖影响 | **有限但可接受**：SIGKILL 使数据截止于 2026-05-19（缺失尾段），但 415 个有效窗口跨越 20 年，统计量不因缺少数月新数据而反转。verdict=FAIL 在"全量"假设下仍然是 FAIL。 |
| 问题1 → 问题3 | 无影响 | 完全独立。即使管线完整运行，估值因子仍然产出 NULL。 |
| 问题2 → 问题3 | 无影响 | 独立因子集。但叠加效应：黄金基线 FAIL（因子池减1）+ 估值因子全灭（因子池减4）= 候选因子池从 18 个减少至 14 个。 |

### 修复优先级（玄知排序）

| 优先级 | 问题 | 理由 |
|:-----:|:----:|:----|
| P0 | **问题1：SIGKILL** | 阻塞所有后续全量运行。不修管线跑不完。 |
| P1 | **问题3：估值数据源** | 估值因子是 basic feature，4个因子集体归零意味着 22% 的候选因子池不可用。数据源修复周期通常最长，应尽早启动。 |
| P2 | **问题2：基线FAIL** | 这是"因子表现不好"而非"系统出错"——如果阈值确认无误，则决策是淘汰 momentum_20d 而非修复。但如果阈值本身需要校准（第三层分析建议），则 P2 升级为 P1。 |

### 对阈值校准的建议（玄知独立补充）

鉴于 half_life=0.47周 偏离下界 4 周接近一个数量级，我怀疑：
1. **4 周的 half_life 下界可能不适合日频截面**：如果截面频率是日频，半衰期以周计意味着 4 周 ≈ 20 个截面。动量效应在截面维度上通常衰减很快（日频更加显著），half_life ∈ [1, 4] 周可能更合理。
2. **建议检查同因子在类似资产上的历史 IC half_life 分布**：比对其他截面 IC 验证报告（如果有的话），确认 momentum_20d 的 half_life 是否异常。
3. **如果阈值经校准后仍 FAIL，则确实应淘汰该因子**。数据层面的 root cause 不是错误。

## 修补记录 v1-p1（2026-05-30T20:10+08:00）

> 基于 Claude 评估原文（18:44 + 18:59）的逐条审核，以下补充我原始 RCA 中遗漏/不足的内容。

### 修补1：问题2 → 数据血缘与去重说明

**原始 RCA 的不足**：使用了「415 个有效窗口」的数字，但未说明以下关键问题：
- 415 的来源：是从 541 行（222 + 428 - 109重叠）中过滤得来，还是仅来自某一次运行？
- 222 截面（第2次）和 428 截面（第3次/断点续跑）之间是否有交叉和去重？
- `--no-run` 出报告时基于哪份数据？是 541 行原始数据还是 415 个有效窗口？

**Claude 指出需要回答的 SQL**：
```sql
SELECT trade_date, COUNT(*) FROM a50_cross_ic_result WHERE factor_name='momentum_20d' GROUP BY trade_date HAVING COUNT(*) > 1;
```
**如果有重复行 → 黄金基线判定推倒重来**，需清库重跑。

**修补结论**：
我的原始 RCA 在问题2 层1中直接引用了 415 这个数字，**隐含假设了数据已去重且可用**，这是一个证据链条上的薄弱环节。正确的表述应该是：
> *「本分析基于现象描述文档所称的 415 个有效窗口。**但需承认，本人未独立执行数据完整性校验（SELECT 去重查询），无法百分之百确认这 415 个窗口不含跨运行重复截面。**如果数据确实含有重复，则 IC 统计量（positive_ratio 0.487 等）可能失真，黄金基线 FAIL 结论的置信度需重新评估。」*

此缺口不影响问题1 和问题3 的根因分析（它们不依赖此数字的正确性）。

---

### 修补2：问题1 → SIGKILL 并发关系和验证阶段细节

**原始 RCA 的不足**：提到了「第1次（marine-s）和第2次（mellow-o）」被杀，但没有说明：
- 两次运行是先后执行还是重叠执行？如果是重叠执行，两份数据是否有截面日期交叉？→ 这会影响问题2 的数据血缘判定
- mellow-o 被杀时处于「验证阶段」—— 这一阶段的具体含义不明确：是 IC 写库完成后的数据校验，还是管线内部的中间校验？→ 决定第2次运行产出的 428 个截面是否完整可信

**修补结论**：
- **并发性判断**：从时间线看（16:43 marine-s 被杀 → 17:27 mellow-o 启动，间隔 27 分钟），两次运行是**先后串行**的，不是并发。这一点可以基本确信。
- **验证阶段含义**：现象描述 v5 未明确 mellow-o 被杀时的具体阶段细节。**需要确认**：如果「验证阶段」在 IC 写库之后，则 428 个截面的 IC 数据已经完整写入（仅后续校验步骤未完成），数据可信；如果「验证阶段」是管线内部的中间校验（IC 尚未全部写出），则 428 个截面可能不完整。
- **此信息缺口需要在验证阶段主动补充**，本修补仅记录问题存在，不做预设立场。

---

### 修补3：纠正原始 RCA 交叉关系表对问题2 数据来源的表述

**原始 RCA 的表述**：
> 「问题1 → 问题2：SIGKILL 使数据截止于 2026-05-19（缺失尾段），但 415 个有效窗口跨越 20 年，统计量不因缺少数月新数据而反转」

**Claude 认为此表述有问题**：
问题不在「缺新数据」，而在于「现有数据本身是否可信」—— 541 截面 = 222 + 428 是否存在重复？在去重未确认之前，不能假定统计量可靠。

**修补为**：
> **问题1 → 问题2 的影响需要拆为两层**：
> - **如果经 SQL 校验确认无重复**：则 SIGKILL 的影响仅限于缺失尾段数据（2026-05-19 后），对 20 年跨度的统计量影响有限，verdict=FAIL 的前提下结论稳定。
> - **如果发现重复截面**：则问题2 的数据基础作废，黄金基线 FAIL 无法作为正式判定，需要清库重跑。
> 
> **此问题无需等数据出来再判断**——一条 SQL 即可确认：
