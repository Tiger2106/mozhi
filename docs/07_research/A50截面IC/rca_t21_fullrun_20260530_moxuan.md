# T+21全量运行三层根因分析（墨萱·独立分析）

## 方法论说明

本文为独立根因分析，与 `rca_t21_fullrun_20260530_v1.md`（墨衡版）分别基于同一份现象描述v5文档开展，形成独立判断。

三层根因结构中：
- **第一层（直接原因）**：触发该现象的可观测事件/条件
- **第二层（间接原因）**：使直接原因得以成立的缺失或前置条件
- **第三层（系统性原因）**：流程/架构/规范层面的根本缺陷——需要修复以消除同类问题

---

## 问题1：管线运行时被 SIGKILL 杀死

### 第一层（直接原因）

操作系统 OOM killer 判定内存耗尽，向管线进程发送 SIGKILL 信号强制终止。

**可观测证据**：
- 系统总内存 15.6GB，事后查询可用仅 1.76GB（约 11%）
- 两次杀死时间点均在启动后 ~15-17 分钟，符合 OOM killer 行为模式（进程持续申请内存直至超出限额）
- 第1次（marine-s，16:43~17:00）：写入 222 截面后被终止
- 第2次（mellow-o，17:27~17:42）：写入 428 截面（含追加的 206 个新截面）后被终止

**⚠️ 核心证据缺口（经Owner 18:59确认）：** 杀死时刻无实时内存快照。事后 1.76GB 不可代表死前时刻的内存状态。因此 OOM 推断是基于"事后低水位 + 行为模式匹配"的概率性结论，非实证性结论。这是 RCA 自身的置信度约束。

### 第二层（间接原因）

1. **系统内存被常驻进程大量占用**：事后 1.76GB 可用意味着系统中存在其他高频内存消耗进程（高度怀疑 vLLM 推理服务）。两次运行间隔 27 分钟虽足以释放前一次管线 RSS，但不足以使系统回到安全水位——说明内存瓶颈来自管线**之外**的持久进程。
2. **管线自身内存需求未知**：管线在处理 50 只成分股的截面计算时，峰值 RSS 未被记录或 profiling 过。不确定是管线本身内存需求大，还是可用内存太小，还是两者叠加。
3. **INSERT OR IGNORE 逐截面提交的设计意外挽救了数据完整性**：虽非有意为之，但每次截面写入后立即 commit 的机制确保杀死时已完成的数据不会丢失。这是一个"幸运的巧合"——如果管线采用批量写入（全部计算完再 commit），两次 SIGKILL 将导致 0 数据可用。

### 第三层（系统性原因）

1. **无预运行环境检查**：管线启动前不做可用内存/磁盘/CPU 检查。如果在启动时检查 `psutil.virtual_memory().available` 并判断可用 < 4GB 则拒绝启动，OOM 可被完全预防。
2. **无资源隔离机制**：管线与 vLLM 等进程共享同一物理内存空间，无 cgroup / Docker 资源限制。缺少隔离意味着任何进程的内存泄漏都会影响其他进程。
3. **无运行期监控/告警**：系统内存从初始水位持续下降至 1.76GB 的整个过程中无任何告警。OOM killer 是最后一道防线而非预警机制。
4. **无管线进程内存画像（profiling）**：死后无 RSS 快照可追溯——在 `memory_profiler` 或 `tracemalloc` 审计缺失的情况下，无法判断管线峰值内存需求是 2GB 还是 8GB，修复方向不明（是优化内存？还是加资源？）。
5. **无条件修复机制低效且无冗余**：进程被杀死后没有自动重试（带更小批次或更低内存模式的降级运行），只能手动重试。第3次改为 `--no-run` 跳过管线是从 "人脑决策" 而非 "自动容错" 出发的。

**与问题2的交叉关系**：SIGKILL 导致第2次运行仅产出到 2026-05-19 的截面数据，缺少最新截面。但 415 个有效窗口覆盖 20 年，缺失的最新数据不改变 verdict=FAIL 的统计结论。

## 问题2：黄金基线 verdict=FAIL（动量20日）

### 第一层（直接原因）

验证脚本 `validate_golden_baseline.py` 在 momentum_20d 因子的 415 个有效窗口（rank_ic NOT NULL）上计算 5 项基线指标，4 项未通过阈值：positive_ratio=0.487<0.50、ic_std=0.271>0.20、significant_ratio=0.248<0.30、half_life=0.47周∉[4,12]。Verdict=FAIL 是算法基于数据的正确输出。

**不可逆向归因：** 这不是 "管线跑错了" 或 "数据坏了"——脚本的计算逻辑、数据读取、去重处理、阈值判定均符合预期。FAIL 本身是对 momentum_20d 因子在 A50 截面上表现的客观评价。

### 第二层（间接原因）

1. **momentum_20d 在 A50 截面上的 IC 特征不达标**：
   - positive_ratio=0.487：略低于 0.5 随机线，说明在 A50 成分股中 20 日动量几乎不具备选股方向性
   - ic_std=0.271：IC 时间序列波动高，因子信号不稳定
   - half_life=0.47 周（≈3.3 交易日）：动量信号衰减极快，几乎是"周频失效"
   - 这些数字反映的是 A50 大盘股截面中动量因子的真实表现——大市值股票的价格动量普遍弱于小盘

2. **数据覆盖不完整但统计量充足**：415 个有效窗口覆盖 2007~2026 近 20 年。即使缺失最新截面（SIGKILL 所致），样本量足以支撑稳定的统计推断。16 个有效窗口的排除（rank_ic=NULL，num_stocks<30 或计算异常）经文档确认不影响结论。

### 第三层（系统性原因）

1. **黄金基线阈值缺乏实证校准**：
   - positive_ratio ≥ 0.50、half_life ∈ [4,12] 周等阈值来自 req_draft_v2 §5.3，但文档中未说明这些阈值对标了哪些成熟市场研究或 A 股实证。A50 是超大盘指数（平均市值超 3000 亿），动量因子在此类股票上的 IC 典型值本就低于全 A 股截面。
   - **具体疑点**：half_life=0.47 周是否意味着 momentum_20d 在 A50 上确实无效（需要淘汰该因子），还是阈值设置过于严格（0.47 离 4 周阈值很远，即使是放松标准也无法通过）？
   - 建议：补充同类研究报告参考数据，或对 A50 截面单独标定黄金基线阈值

2. **数据覆盖完整性预检缺失**：
   - 验证阶段没有先检查 DB 数据是否覆盖了预期全部日期范围（2007-01-05 ~ 最新交易日）。SIGKILL 导致的覆盖缺口（缺少 2026-05-19 后数据）未被自动检测和提示
   - 一个简单的 `SELECT MIN(trade_date), MAX(trade_date) FROM a50_cross_ic_result WHERE factor_name='momentum_20d'` 可验证范围

3. **无 FAIL 后的诊断/升级路径**：
   - 验证脚本判出 FAIL 后仅记录到 JSON，无任何后续动作（如：触发对照因子分析、检查同窗口期其他因子的通过率、或发出通知）
   - 缺乏 FAIL 后的下一步操作指引——问题2 的 verdict 应该流入因子池管理决策（淘汰 momentum_20d 作为黄金基线？还是调整阈值？）而非停在 JSON 中等待人工发现
   - 管线架构中缺少一个 "failover/remediation" 环节

4. **黄金基线因子选择本身值得审视**：
   - 为什么选 momentum_20d 作为黄金基线？在 A50 超大盘截面中动量效果历来偏弱（学术界共识），选择单一因子作为"门神"意味着如果该因子在 A50 上天然不work，整条管线就被阻塞

**与问题1的交叉关系**：SIGKILL 缩小了数据覆盖范围（缺少最新截面），但经确认不影响 verdict 稳定性。与问题3无依赖关系。

## 问题3：估值因子 IC=0（数据源缺失）

### 第一层（直接原因）

管线计算估值 4 因子（pe_ttm、pb、ps_ttm、pcf_ttm）的截面 IC 时，底层数据源的对应字段全部为空：

- **pe_ttm/pb**：`a50_daily_ohlcv` 表的 `pe` 和 `pb` 列定义存在，但值在 **2007~2026 全部年份全部为 NULL**
- **ps_ttm/pcf_ttm**：`a50_daily_ohlcv` 表中**根本不存在**这两列（Schema 缺失）

因此所有截面的估值因子 rank_ic 均为 NULL → 结果为 0。

### 第二层（间接原因）

按数据链追溯（从 pipeline 向上游逐层排查）：

1. **因子计算层**（pipeline IC 计算模块）：引用 `a50_daily_ohlcv.pe`、`a50_daily_ohlcv.pb`，但对 `ps_ttm` 和 `pcf_ttm` 引用不存在于 `a50_daily_ohlcv` 表中的列名。这意味着因子计算代码中直接用了 `ps_ttm` 和 `pcf_ttm` 作为列名去 `a50_daily_ohlcv` 中取值——该表无对应列，返回 NULL 或报错。
2. **ETL 层**（`etl_a50_daily.py`）：`FIELD_MAP` 中仅映射了 `pe_ttm → pe` 和 `pb → pb`，**根本没有 ps_ttm 或 pcf_ttm 的映射项**。这是源头遗漏——ETL 从 `market_data.db.stock_daily` 取数据时就跳过了它们。
3. **原始数据层**（`market_data.db.stock_daily`）：即使有映射，`stock_daily` 表中 `pe_ttm` 和 `pb` 列的值也全部为 NULL（2007~2026 全覆盖）。`ps_ttm` 和 `pcf_ttm` 列在 `stock_daily` 表中**不存在**（Schema 本身无定义）。
4. **数据源层**（akshare/tushare_pro）：akshare 接口返回的数据中**不包含** pe、pb 等估值字段的值（或返回空值）。这是最上游的缺失。

**💡 关键区别**（估值因子从同一起因分裂为两类问题）：

| 因子 | 问题类型 | 缺失层 | 严重程度 |
|:----:|:--------:|:------:|:--------:|
| pe_ttm / pb | 数据填充缺失 | 列存在但数据源无值 → 全部 NULL | ⚠️ 可修复（需补充数据源） |
| ps_ttm / pcf_ttm | Schema 设计缺失 | 列不存在于任何表 | ❌ 严重（连容器都没有） |

### 第三层（系统性原因）

1. **因子设计与数据源可用性完全脱钩（最关键缺陷）**：
   - `design_v2.md` §2.2.1 的因子清单中，pe_ttm 和 pb_lf 已明确标注 "⚠️ 字段存在，质量待验证"——**设计阶段就已识别风险**，但管线全量运行时**未先执行该验证步骤**。这相当于：知道路可能断了，但还是开过去。
   - ps_ttm 和 pcf_ttm 更严重——它们**不在设计文档的任何因子清单中**。意味着管线开发过程中，有人（可能是墨衡在编码阶段）自行添加了这两个因子名到计算逻辑中，但：
     - 未更新设计文档
     - 未确认对应的数据列是否存在
     - 未添加对应的 ETL 字段映射
     - 未创建对应的 DB Schema 列
   - 这是一个从设计→开发→ETL→验证全链条断裂的案例。

2. **无数据质量门禁（QA Gate）**：
   - 管线在计算 IC 之前，没有对输入数据做任何 "hydration check"（水合检查）：
     ```
     SELECT pe, pb FROM a50_daily_ohlcv WHERE pe IS NOT NULL LIMIT 1
     ```
     如果这个查询返回 0 条，管线应当在启动时立即 FAIL FAST，而不是默默算出全 0 IC。
   - 目前的状态是：管线正常运行，输出 18 个因子的 IC 值，其中有 4 个全为 0 ——没有报错，没有日志警告，没有质量标记。
   - **这是比其他两个问题更严重的系统性缺陷**——OOM 至少有一个明显的 "进程被杀死" 的可见信号，而估值因子的数据缺失是一个**完全静默的失败**。

3. **单一数据源依赖风险**：
   - 整个估值因子管线仅依赖 akshare（tushare_pro 标注）一个数据源。该数据源不提供 A50 成分股的历史估值截面数据，但没有替代方案。
   - 在 data_survey_report.md 中已发现 `turnover_rate`、`total_mv`、`circ_mv` 完全缺失，但 pe/pb 的缺失未作为阻塞项记录在摸底报告中。
   - 建议：建立多数据源策略，或至少对每个因子的输入列做 "data availability sign-off" 才能进入管线。

4. **Schema 设计审核流程缺失**：
   - `a50_daily_ohlcv` 表的 DDL 中没有 `ps_ttm` 和 `pcf_ttm` 列——但因子计算代码却引用了它们作为 column name。
   - 这说明管线开发到一半时，有人 "决定添加" 这两个因子，但只改了计算模块代码，没有同步更新：
     - DDL（数据库 Schema）
     - ETL（字段映射表 FIELD_MAP）
     - 设计文档（因子清单）
   - 需要一个 "Schema ↔ Factor 双向对齐" 的签核流程。

5. **摸底调查（Survey）的触发条件过窄**：
   - `data_survey_report.md` 只调查了已经从 stock_daily 存在的数据字段的缺失率，但**没有调查"因子需要的但 stock_daily 不存在的列"**。换句话说，摸底只查了 "表里有什么"，没查 "表里没有什么但因子需要"。
   - 如果摸底流程包含一步：**"列一份所有因子所需的输入数据列清单，与 stock_daily 的 Schema 做差集对比"**，ps_ttm/pcf_ttm 的缺失在摸底阶段就会被发现。

**与问题1/问题2的交叉关系**：**无依赖关系**。问题3 是独立的数据源问题，与 OOM 和动量基线结果无关。但三个问题叠加的效应是：

> **候选因子池从 18 个减少至 14 个（基线FAIL淘汰 1 个常用因子 + 数据缺失废掉 4 个估值因子）**
