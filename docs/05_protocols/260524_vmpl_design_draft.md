## 一、背景与动机

### 1.1 为什么要专用模板

EXP-001（NaN隔离防护验证试验）暴露了现有模板体系的缺口：

| 问题 | 表现 | 严重程度 |
|:----|:----|:--------:|
| **绩效指标不适用** | 验证试验无年化收益/夏普/回撤，只能手动替换为通过率/一致性 | 高 |
| **结论判定不匹配** | "实盘/观察/废弃"无法映射验证结论"PASS/FAIL" | 高 |
| **大量章节被删除** | TMPL-002 §2绩效归因、§3交易统计、§4市场敏感性全部不适用 | 高 |
| **风险类别全替换** | 过拟合/数据偏差 → 覆盖不完整/实现偏差/边界遗漏 | 中 |
| **元信息字段不适配** | strategy_name/backtest_period需手动改为experiment_name/validation_scope | 中 |
| **QA标准不匹配** | K线数据/涨停板/成交量校验 → 函数输出一致性/边界通过率校验 | 中 |

结论：不是"改几个字段能解决的"，需要**独立模板族**。

### 1.2 与"研究试验模板"的分界

| 维度 | 研究试验模板（EXP-系列） | 验证模板（VMPL） |
|:----|:-----------------------|:----------------|
| **目的** | 因子有效性/IC-IR验证 | 代码级功能正确性验证 |
| **产出** | 因子是否入选组合 | 函数/模块是否安全可用 |
| **指标** | IC/IR/夏普/分层收益 | 通过率/一致性/边界覆盖率 |
| **结论** | 实盘/观察/废弃 | PASS/FAIL/PASS_WITH_RESERVATION |
| **耗时长** | 数小时～数天（全量回测） | 秒～分钟级（VMPL 验证域专属超时阈值：5分钟） |
| **频率** | 中低频（每周/月级别） | 高频（每次代码变更可触发） |

研究试验模板（2605232224墨知回测试验模板.md）面向**金融因子研究**；VMPL 面向**代码级正确性验证**。两者互补不重叠。

## 二、模板数量决策

### 2.1 三方分歧回顾

| 提出方 | 方案 | 核心理由 |
|:------:|:----|:---------|
| **墨衡** | 2份：VMPL-SUMMARY + VMPL-REPORT | 保持与TMPL-001/002相同的summary/analysis分层 |
| **墨萱** | 1份：TMPL-Ex | 验证试验内容窄，一份足够 |
| **玄知** | 轻量模板族：1份核心 + 可选QA | 灵活，核心模板通用，QA按需引用 |

### 2.2 论证

**支持1份核心模板的理由：**

1. **受众单一**：验证试验的读者主要是技术团队（开发者+QA），没有"业务层面Owner和开发团队"的区分，故不需要「summary给Owner看、analysis给技术看」的分层

2. **内容规模小**：验证试验报告通常 3～10 页，远小于回测报告的 15～40 页。一份文档足够承载全部信息

3. **消除交叉引用**：TMPL-001/002 需要互相引用（summary §附录指向analysis path；analysis §A关联文档指向summary path）。一份文档无此问题

4. **降低维护成本**：1份模板维护成本约是2份的 40%（少一份元信息、少一份附录、少一份自检清单）

**支持2份模板的理由：**

1. **归档统一**：summary/analysis 分卷归档有利于长期查找——Owner只看摘要层，开发人员看详细层

2. **评审流程完整**：Stage 1 执行方填写 summary + analysis，Stage 2 QA 看 summary + 引用 analysis。多一份并不额外增加负担（模板骨架就 6-8 节）

**决策建议：1份核心模板（"决议"）**

> 理由：当验证试验的受众未来出现"管理层/执行层"分化时，可以再拆分为2份。当前覆盖 EXP-001 级别的验证（函数级/模块级），1份完全够用。若未来出现「需Owner决策是否采纳验证结论」的复杂场景，再引入 summary 拆分。
>
> 模板内部通过章节结构自然承载「执行摘要」（§2-3）和「技术详情」（§5-6）两个层次，无需物理分卷。

### 2.3 决策结论

```
方案选型：✅ 1份核心模板（VMPL-001）+ QA引用 TMPL-003（调整标准）
放弃方案：❌ 2份分卷 / ❌ 轻量模板族（在1份足够时不引入复杂性）
```

## 三、文件命名策略

### 3.1 编号方案对比

| 编号方案 | 格式示例 | 优点 | 缺点 |
|:--------|:--------|:----|:----|
| **VMPL-001** | `VMPL-001_verification_template.md` | 独立命名空间，清晰区分BT系列；"VP"暗示Verify Protocol | 新编号，团队需要适应 |
| TMPL-EX-001 | `TMPL-EX-001_experiment_template.md` | 复用TMPL前缀，保持体系感 | "EX"与研究试验EXP易混淆 |
| TMPL-004 | `TMPL-004_verification_template.md` | 延续已有编号 | 编号语义不清；未来增加类型时顺序编号不好用 |

### 3.2 建议

**VMPL = Verification Markup Protocol / Language**

> 类比 BT = Backtest，VMPL 命名空间使其在文件系统中的排列一目了然：

```
docs/03_templates/
├── TMPL-001_backtest_summary_template.md       # 回测摘要
├── TMPL-002_backtest_analysis_template.md       # 回测分析
├── TMPL-003_backtest_qa_checklist_template.md   # 回测QA
├── TMPL-004_...                                 # (保留)
└── VMPL-001_verification_template.md             # 验证模板（本草案产出）
```

若未来出现**性能基准测试模板**或**数据质量审计模板**，可继续使用 VMPL-002、VMPL-003 等编号。

### 3.3 QA 配套文件

| 文件 | 策略 | 说明 |
|:----|:-----|:-----|
| QA 标准 | **不新建文件**，复用 TMPL-003 但不引用背测试标准段 | TMPL-003 的§1-4 结构可复用（数据完整性/一致性/异常/复现），仅验证标准内容替换 |
| QA 产出 | 命名采用 `vqa_{task_id}.md` | "VQA" = Verification QA，与 `BT-QA-CHECKLIST-V1` 区分 |

## 四、完整章节结构（骨架级）

### 4.1 VMPL-001 总览

| § | 标题 | 定位 | 字数估算 |
|:-:|:----|:-----|:---------|
| §1 | 元信息 | 归档&检索 | 约 15 行 JSON |
| §2 | 验证结论与核心指标 | **决策者第一眼**：结论+判定+关键假设+核心指标一览 | 约 400 字 + 3-4 张表 |
| §3 | 验证配置 | 环境、数据、被测对象 | 约 2 张表 + 1 个 JSON |
| §4 | 详细验证结果 | **技术主力章节**：模式归因+敏感度分析 | 约 4-6 张表 |
| §5 | 风险评估 | 覆盖/实现/环境/局限性 | 约 1-2 张表 |
| §6 | 自检清单 | 执行方出具，通过性检查 | 约 8-12 项 |
| §7 | 审计日志 + 附录 | 追溯性+关联文档+环境信息 | 约 15-20 行 |


**总规模**: 6-8 节（实际 7 节，符合 6-8 节要求。验证结论与核心指标合并为§2快速阅读，详细分析作为§4技术深读，满足双层读者需求。）

### 4.2 各节详细内容定位

#### §1 元信息（Meta Info）

> 定位：档案级元数据，用于自动化归档检索、版本追溯、报告关联。
> 所有字段结构化（JSON block），便于程序化读取。

```json
{
  "template_id": "VMPL-001",
  "version": "1.0",
  "report_id": "<必填: 试验名_YYYYMMDD_HHMMSS>",
  "experiment_name": "<必填: 验证试验名称>",
  "validation_scope": "<必填: 验证范围（如 NaN隔离防护验证 / 多版本EMA输出一致性验证 ）>",
  "validation_tool_version": "<必填: 验证脚本/工具版本号>",
  "data_version": "<必填: 数据版本>",
  "code_version": "<必填: Git commit hash / tag 或 工作副本说明>",
  "executor": "<必填: 执行人>",
  "qa_reviewer": "[可选: QA验证人]",
  "completed_time": "<必填: ISO8601 +08:00>",
  "acceptance_criteria_version": "<必填: 验收标准版本>",
  "report_type": "verification",
  "verification_type": "<功能验证 / 一致性验证 / 安全边界验证 / 性能验证>"
}
```

#### §2 验证结论与核心指标（Core Conclusion & Metrics）

> 定位：***决策者第一眼***。结论 + 判定 + 关键假设 + 核心指标一览。

**结论字段：**

| 字段 | 说明 |
|:----|:-----|
| 一句话结论 | 类似 TMPL-001，但内容变为验证结论（如"4/4函数全部通过CORE-0~6"） |
| 最终判定 | PASS / PASS_WITH_RESERVATION / FAIL / INCONCLUSIVE |
| 判定置信度 | 高 / 中 / 低（基于验证覆盖面和重复性） |
| 判定依据 | 一句话说明核心理由 |
| 关键假设列表 | 与TMPL-001结构相同，但假设类型变为：验证条件是否覆盖生产场景、验收标准是否完整等 |

**核心指标表：**

| 指标名称 | 定义 | 取值范围 | 阈值 |
|:---------|:-----|:---------|:-----|
| **验收标准通过率** | 通过的验收标准数 / 总验收标准数 | [0,1] | =1.0 为 PASS |
| **多版本一致性** | 各实现版本在相同输入下的输出匹配度 | ✅ 完全一致 / ⚠️ 存在偏差 / ❌ 不一致 | ✅ |
| **边界测试通过率** | 通过的边界用例 / 总边界用例数 | [0,1] | =1.0 为 PASS |
| **模式覆盖率** | 已测试的输入模式 / 设计的测试模式 | [0,1] | ≥0.8 为 PASS |
| **复现性** | 相同条件二次执行的输出偏差 | 偏差值（无偏差时=0） | ≤1e-10 |

**与回测指标的映射关系：**

| 回测指标 | 验证替代指标 | 说明 |
|:---------|:-------------|:-----|
| 年化收益率 | 验收标准通过率 | 验证的"收益"是安全/正确 |
| 夏普比率 | 无直接替代（验证无风险收益权衡） | 验证关注正确性而非效率 |
| 最大回撤 | 边界测试通过率 | 极端条件下的生存能力 |
| 胜率 | 模式覆盖率 | 覆盖的测试模式比例 |
| 复现性偏差 | 复现性 | 两种模板共有 |

#### §3 验证配置（Verification Configuration）

> 定位：环境、数据源、被测对象列表。对标 TMPL-001 §3 策略配置摘要。

**内容：**
- 验证工具 / 引擎配置表（引擎名称、版本、精度、并行度、超时设置）
- 被测对象列表（函数名、源文件路径、实现语言/框架）
- 测试数据配置表（数据源、版本、构造方式、模式列表、边界用例列表）
- 验收标准版本号及来源

#### §4 详细验证结果（Detailed Analysis）

> 定位：***技术主力章节***。对标 TMPL-002 §2 绩效归因 + §3 交易统计 + §4 市场敏感性。

**子节结构：**

| 子节 | 内容 | 对标 TMPL-002 |
|:----|:-----|:-------------|
| **5.1 逐标准验证详情** | 每个验收标准在各被测对象上的通过情况 | §2 绩效归因 |
| **5.2 模式归因** | 各测试模式（正常/异常/边界）的输出特征分析 | §3 交易统计 |
| **5.3 敏感度分析** | 参数变化（如EMA窗口）对结果的影响 | §5 参数敏感性 |
| **5.4 一致性交叉验证** | 多版本/多实现间的输出对比详情 | §2.1 收益分解 |

#### §5 风险评估（Risk Assessment）

> 定位：验证的局限性。对标 TMPL-001 §5 风险评估。

| 风险类别 | 验证域专属定义 | 等级 |
|:---------|:--------------|:----:|
| **覆盖不完整风险** | 测试场景未覆盖全部生产使用模式 | 高/中/低 |
| **实现偏差风险** | 被测代码在不同环境/平台的行为差异 | 高/中/低 |
| **环境依赖风险** | 外部库版本、OS差异导致行为变化 | 高/中/低 |
| **边界遗漏风险** | 未测试的极端输入可能导致隐藏问题 | 高/中/低 |
| **参数泛化风险** | 仅验证特定参数组合，未确认通用性 | 高/中/低 |

#### §6 自检清单（Self-Check Checklist）

> 定位：执行方在提交前逐项确认。对标 TMPL-001 §4。

检查项示例：
1. ✅ 验收标准完整覆盖验证目标
2. ✅ 所有测试模式已执行
3. ✅ 边界用例已覆盖（空输入/极值/异常值）
4. ✅ 多版本一致性已验证（如适用）
5. ✅ 代码版本已标注（含无Git管理说明）
6. ✅ 环境信息已记录于附录
7. ✅ 验证可复现（种子锁定/确定性生成）
8. ✅ 审计日志完整

#### §7 审计日志 + 附录（Audit Log + Appendices）

**审计日志**: 同 TMPL-001 §7（操作时间/操作人/操作类型/操作描述）
**附录 A**: 关联文档（VQA 路径、验证脚本路径、原始日志路径）
**附录 B**: 数据构造补充说明（如需详细描述合成数据生成方式）
**附录 C**: 环境信息（OS/Python/依赖库版本，同 TMPL-002 附录C）

## 五、关键字段定义

### 5.1 核心指标体系

| 指标 | 定义 | 计算方式 | 阈值 | 说明 |
|:----|:-----|:---------|:-----|:-----|
| **通过率 (Pass Rate)** | 通过的验收标准数 / 总验收标准数 | 布尔校验汇总 | =1.0 | 核心指标，任何 FAIL 即不通过。若执行有效性≠✅，通过率标记为N/A |
| **边界覆盖率 (Edge Coverage)** | 通过边界用例数 / 总边界用例数 | 布尔校验汇总 | =1.0 | 边界=系统最脆弱处，强制全通过 |
| **多版本一致性 (Consistency)** | 各实现版本输出间匹配程度 | 两两对比偏差矩阵 | ≤1e-10 | 仅适用于对比验证场景 |
| **模式覆盖率 (Pattern Coverage)** | 已测模式数 / 设计模式总数 | 枚举核对 | ≥0.8 | 模式不全不否定结论但降低置信度 |
| **复现性偏差 (Reproducibility Gap)** | 二次执行与首次结果的差异 | 输出结果对比 | ≤1e-12 | 确定性系统应为 0 |
| **执行有效性 (Execution Validity)** | 验证脚本无崩溃/超时（超时阈值：5分钟） | 二进制结果 | ✅ | 基础条件 |

### 5.2 结论判定（取代"实盘/观察/废弃"）

| 结论 | 含义 | 后续行动 | 触发条件 |
|:----|:-----|:---------|:---------|
| **PASS** | 全部验收标准通过，边界用例通过 | 可安全合并/部署/使用 | 通过率=1.0 ∧ 边界覆盖率=1.0 |
| **PASS_WITH_RESERVATION** | 主要标准通过，存在已知但可接受的局限性 | 可合并，需在README/文档中注明局限性 | 通过率=1.0 ∧ 边界覆盖率=1.0 ∧ 模式覆盖率≥0.8 ∧ 存在非致命风险 |
| **FAIL** | 存在致命缺陷 | 不可合并/部署；需修复后重新验证 | 通过率＜1.0 ∨ 边界覆盖率＜1.0 |
| **INCONCLUSIVE** | 验证覆盖不足，无法得出结论 | 需补充测试模式或边界用例 | 模式覆盖率＜0.8 ∧ 通过率=1.0 |

**隐式判定优先级（排序规则）：**

判定按以下优先级依次检查，高优先级条件满足时直接判定，不再检查后续规则：

1. **先检查 FAIL**：通过率＜1.0 ∨ 边界覆盖率＜1.0 → 直接判定 FAIL
2. **再检查 INCONCLUSIVE**：模式覆盖率＜0.8 且 通过率=1.0 → 判定 INCONCLUSIVE
3. **检查复现性偏差**：复现性偏差超阈值（＞1e-12）→ 自动降级为 PASS_WITH_RESERVATION
4. **最后区分 PASS 与 PASS_WITH_RESERVATION**：无保留条件 → PASS；存在已知局限性且已记录 → PASS_WITH_RESERVATION

> ⚠️ 当多个条件同时满足时，**编号小的优先级更高**。例如：若通过率=1.0 且 模式覆盖率＜0.8 但同时复现性偏差超阈值，因条件1（FAIL）优先级最高，优先检查 FAIL 条件；若仅 INCONCLUSIVE 和 复现性条件同时触发，则以编号小的 INCONCLUSIVE（条件2）为准。

### 5.3 风险评估（取代"过拟合/数据偏差"）

| 风险类别 | 定义 | 评估示例 | 应对措施 |
|:---------|:-----|:---------|:---------|
| **覆盖不完整风险** | 测试场景未覆盖实际使用模式 | 仅验证window=25，未测试更大小 | 记录为已知限制，建议补充 |
| **实现偏差风险** | 不同实现路径的行为差异 | 未交叉验证多个等价实现 | 多版本交叉验证即可降低 |
| **环境依赖风险** | OS/库版本导致的差异 | 仅在Python 3.14测试 | 在CI/CD多环境跑一次 |
| **边界遗漏风险** | 极端输入未测试 | 未测试超长NaN窗口 | 新增对应边界用例 |
| **参数泛化风险** | 特定参数设置下结果不一定泛化 | 仅测试EMA_WINDOW=25 | 补充多参数验证 |

**等级标准**:
- **低**: 风险已被确认消除或验证充分覆盖
- **中**: 风险存在但在合理范围内，有针对性应对措施
- **高**: 风险可能导致结论不可靠，需进一步验证

## 七、QA 接口设计

### 7.1 与 TMPL-003 的依赖关系

> **设计借鉴，非运行时依赖**。
>
> VMPL 的 QA 章节在结构设计上借鉴 TMPL-003，但两者是**独立复制关系**：
> - VMPL QA（§7）的内容是 TMPL-003 结构调整后的**静态快照**
> - 未来 TMPL-003 版本更新**不会自动同步**到 VMPL QA 章节
> - 需更新时，由维护者手动将 TMPL-003 相应修改应用到 VMPL 文档
> - 两者之间无运行时引用或自动同步机制

### 7.2 QA 执行者角色定义

**执行人**：墨萱（独立QA角色）
- **角色**：与模板填写方（墨衡/开发者）**分离**，独立执行 QA 审查
- **能力要求**：
  - 理解验证域指标含义（通过率/边界覆盖率/复现性等）
  - 能审查测试覆盖是否充分
  - 能识别实现偏差风险
  - 无需金融策略背景（与回测 QA 的最大区别）
- **交叉审查**：若墨萱不可用，可由其他开发者交叉审查，但需在 QA 报告中注明"交叉"

### 7.3 设计原则

按团队共识：✅ **QA可直接复用 TMPL-003，调整引用标准即可**。

### 7.4 复用方案

| TMPL-003 节 | 复用方式 | 需要调整的内容 |
|:------------|:---------|:--------------|
| §1 数据完整性 | **结构复用** | 验证标准从"K线数据覆盖"替换为"验证输入数据完整性"；“数据库行数核验”替换为"测试覆盖率核验" |
| §2 数据一致性 | **结构复用** | 验证标准从"复权一致性/时间对齐"替换为"输入数据管道一致性/多数据源一致性" |
| §3 异常检测 | **结构复用** | 验证标准从"极端收益率/异常交易"替换为"极端输入/异常模式/输出稳定态检测" |
| §4 复现性 | **直接复用** | 标准几乎不变（种子锁定/二次运行偏差/环境记录） |
| §5 超时状态 | **直接复用** | 不调整。超时/资源约束两类通用 |
| §6 QA意见总结 | **结构复用** | 结论产出同格式，但"回测结论"替换为"验证结论" |
| §7 审计日志 | **直接复用** | 不调整 |

### 7.5 需要替换的验证标准

**数据完整性（替换前 → 替换后）：**

```
# 替换前（TMPL-003 §1.2）
1 | K线数据覆盖完整，无断点
2 | 成交量数据无缺失
3 | 涨跌停日数据正常处理
4 | 除权除息日数据正确
5 | 数据长度与配置一致
6 | 数据库行数核验

# 替换后（VQA 版）
1 | 验证输入数据完整，无意外截断
2 | 测试模式之间无数据污染（独立隔离）
3 | 合成数据确定性可复现（seed/构造算法已知）
4 | 所有预定义测试模式已执行完毕
5 | 实际执行的测试模式数与计划一致
6 | 验证覆盖率核验：检查断言覆盖全部验收标准
```

**数据一致性（替换前 → 替换后）：**

```
# 替换前
1 | 数据源版本唯一且可追溯
2 | 不同频率数据时间戳对齐
3 | 复权计算一致
4 | 异常值处理规则已实施
5 | 无前视偏差

# 替换后
1 | 所有被测函数使用同一输入副本
2 | 输入数据在调用链中不被意外修改
3 | 输入数据类型（List / np.ndarray / pd.Series）之间的隐式转换无误
4 | 数据管道无状态残留（单元测试之间的隔离）
5 | 无隐式全局变量污染

**异常检测（替换前 → 替换后）：**

```
# 替换前
1 | 极端收益率标记
2 | 异常交易行为检测
3 | 波动率突变识别

# 替换后
1 | 极端输入模式检测（NaN/Inf/空序列）
2 | 异常行为模式识别（输出稳定态偏移）
3 | 验证结果离群值分析
```

> **数据基准来源说明**：异常检测的基线值（阈值、判定标准、各指标的定义）定义于 §5.1「核心指标体系」。QA 执行时，以 §5.1 中定义的指标阈值为依据进行判定，无需另建独立基线。
```

### 7.6 VQA 产出文件命名

```
产出路径: {report_path_base}../qa/vqa_{task_id}.md
文件前缀: [VQA_{TEMPLATE_VERSION}]
示例:     [VQA_V1] EXP-001 NaN隔离防护验证 QA
```

**双向锚点规则：**
- VMPL 报告 **附录 A** 中必须包含 VQA 文件路径占位符：
  ```
  关联VQA: ../../qa/vqa_{task_id}.md
  ```
- VQA 文件中必须在头部注明**关联 VMPL 报告路径**：
  ```
  关联VMPL Report: {report_path_base}/VMPL-001_{task_id}.md
  ```
- 路径使用相对路径，便于归档和移动时保持可追踪性。
- 两个引用方向必须**同时填写**，不可遗漏任一方向。

## 八、应用范围

### 8.1 使用验证模板的场景

**✅ 强制使用 VMPL-001 的场景：**

| 场景 | 典型示例 | 判定标准 |
|:----|:---------|:---------|
| 代码级功能正确性验证 | NaN隔离防护、除零保护、空输入处理 | 无金融绩效指标产生 |
| 多版本一致性验证 | 不同实现（Python/NumPy/pandas）输出对比 | 核心产出是一致性矩阵 |
| 安全边界验证 | 异常值注入、极值输入、恶意输入 | 核心指标是边界通过率 |
| 重构/迁移回归验证 | 策略代码重构后行为不变性 | 输入输出对对比 |
| 数据管道验证 | 复权计算正确性、因子计算正确性 | 数据变换的数学正确性 |
| CI/CD 自动化验证 | 提交前自动触发的小型验证 | 秒级执行，通过/不通过 |

**推荐使用但不能通过泛化场景：

- 性能基准测试：虽然也可抽象为"验证执行时间≤阈值"，但涉及多轮统计，建议用 VMPL-002 覆盖
- UI/API 集成测试：与金融系统距离远，不在本模板范围

### 8.2 继续使用回测模板的场景

**必须使用 TMPL-001/002/003 的场景：**

| 场景 | 原因 |
|:----|:-----|
| 策略级绩效评估 | 需要年化收益/夏普/回撤等金融指标 |
| 因子 IC/IR 有效性检验 | 使用研究试验模板，非回测模板 |
| 组合/实盘前绩效审计 | 涉及风控、仓位、资金管理判断 |
| 多策略比较 | 基准对比、超额收益计算 |

### 8.3 决策树

```
本实验是否是代码级正确性验证？
├── 是 → 是否产生金融绩效指标（年化/夏普/回撤）？
│   ├── 否 → ✅ 使用 VMPL-001 验证模板
│   └── 是 → 使用 TMPL-001/002 回测模板（或手动适配）
└── 否 → 是否是因子 IC/IR 研究？
    ├── 是 → ✅ 使用研究试验模板（EXP-系列）
    └── 否 → ✅ 使用 TMPL-001/002/003 回测模板
```

## 九、附录：EXP-001 适配问题总结（经验教训）

### 9.1 适配工作量

| 章节 | TMPL-001 原始节数 | EXP-001 实际修改 | 工作量评估 |
|:-----|:-----------------|:-----------------|:----------|
| §1 元信息 | 8字段 | 5字段替换，1字段新增 | 中 |
| §2 核心结论 | 1表+假设 | 判定值替换，假设仍适配 | 低 |
| §3 核心指标 | 8指标+分阶段 | 8指标全部替换，分阶段删除 | **高** |
| §4 策略配置 | 1 JSON | 字段名全部替换 | 中 |
| §5 自检清单 | 10项 | 检查项内容替换 | 低 |
| §6 风险评估 | 4类别 | 4类别全部替换 | **高** |
| §7 后续建议 | 3种建议 | 内容全部重写 | 中 |
| §8 审计日志 | 4+行 | 仅替换内容 | 低 |
| 附录 | 代码版本+数据版本 | 保留并调整 | 低 |

> **结论**: EXP-001 适配 TMPL-001 约需 60% 的章节**全部或大部分**重写。TMPL-002 的状况类似。每次新验证实验都做一次适配 → **成本不可接受**。这直接推动 VMPL 专用模板需求。
