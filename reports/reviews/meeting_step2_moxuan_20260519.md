## 二、逐项技术审查

### 2.1 Layer Q 架构合理性 — 横向治理层

**审查结论**: ✅ 通过

**分析**:

Layer Q 被定位为 **Transverse Governance Layer（横向治理层）**，独立于 P 系列生产系统运行，技术上是**可行的架构模式**。其合理性依据：

1. **职责分离清晰** — Q层只读不写P系列数据，通过标准接口读取回测输出，不侵入P系列计算管线。这是正确的依赖方向（Q→P 单向依赖）。
2. **模式成熟** — 这对应企业架构中"跨领域关注点分离"（Cross-Cutting Concern Separation），与 OAUTH 认证、API Gateway 等横向治理层的设计理念一致。
3. **G1/G2/G3 三级门禁** — 从自动硬门禁→自动软检查→人工复审的递进设计合理，适应不同严重级别的问题。

**技术风险**: 低

- Q层需要 P 系列输出数据的版本一致性，这已在 AD-002 中记录为架构债务，需要在 Phase 0 启动前先锁定评分数据源版本基线。这个前提条件已在 Kill Switch KS6 中覆盖。

**建议**: Phase 0 实施前，先执行 AD-002 的基线锁定操作（创建 P 系列输出的版本快照），最小化版本漂移风险。

### 2.2 ExistenceValidator 6 项阈值

**审查结论**: ⚠️ **存在 P1 风险 — 需注意阈值初始化的参数敏感性和边界误判**

**逐项检查**:

| 检查项 | 阈值 | 判定 | 技术评价 |
|:-------|:----:|:----|:---------|
| C1: 最小交易数 | >= 30 | 🟢 合理 | 统计样本量底线，30为常见最小样本量要求，合理 |
| C2: 多Regime覆盖 | >= 2 | 🟢 合理 | 防止单一市场状态过拟合，可接受 |
| C3: 多年度覆盖 | >= 2年 | 🟢 合理 | 确保跨周期覆盖，合理 |
| C4: 非单段收益贡献 | < 40% | 🟡 边际 | 对波动率高的策略（如期货趋势跟踪），可能出现误杀 |
| C5: 信号密度 | >= 12 笔/年 | 🟡 边际 | 对低频策略（如周线级别）可能偏严 |
| C6: 样本分布 | 最长无交易间隔 < 30%总周期 | 🟡 边际 | 对季节性策略（如特定月份交易）可能误判 |

**P1问题 #1 — C4阈值对高波动策略误判风险**:

问题描述: 专家建议的C4阈值（单笔最大收益占比<40%）在统计学上是合理的，但**未提供针对策略类型的差异化处理**。对于高波动率策略（如期货CTA、事件驱动型），单笔大收益占比超过40%是策略本身的特性而非"一波流"伪信号。

- **影响范围**: 可能误杀部分有效的高波动策略
- **严重级别**: P1（中等，可在Phase 2校准窗口解决）
- **修复建议**: 
  - 在Phase 0a实施时，C4阈值保持40%作为初始硬阈值
  - 在Phase 0b/Phase 1期间，记录被C4拦截但有Q2/Q3通过的边缘案例
  - 在Phase 2阈值校准时，考虑增加策略类型差异化参数，允许高频策略C4放宽至60%

**P1问题 #2 — C5阈值对低频策略兼容性**:

问题描述: C5要求年均交易数>=12笔（月均至少1笔），这对日频网格策略是合理的，但**对周频甚至月频策略（如部分动量策略）可能偏严**。如果策略系统后续引入低频策略类型，C5需要调整。

- **影响范围**: 未来低频策略扩展时需调整
- **严重级别**: P1（非当前阻塞，但需记录架构债务）
- **修复建议**: 
  - Phase 0a当前保持12笔/年阈值
  - 在Q9 Failure Registry中新增策略类型字段，收集低频策略的"C5拦截"案例
  - Phase 2校准窗口时评估是否需要按策略类型差异化

**权重设计评价**:

C1权重0.30 + 硬门禁的设计合理。C2~C6各0.10~0.15的均匀分布也合理。但有一个**技术争议点**:

> 置信度计算权重总和为 0.30+0.15+0.15+0.15+0.15+0.10 = 1.00。但C1是硬门禁（不通过则exists=False），应该在逻辑上先于confidence计算。建议在代码实现中明确逻辑顺序：
> 1. 先检查C1（硬门禁），若失败则直接返回 exists=False, confidence=0.0
> 2. 再计算其余5项的confidence权重

**总结**: 阈值设定整体合理，但存在2项P1级别的边界误判风险，建议在Phase 2校准窗口处理。

### 2.4 双账本系统 — 数据隔离可行性

**审查结论**: ✅ 通过

**分析**:

账本A（P系列）和账本B（Q层）的数据隔离在技术上是**完全可行**的。其隔离方案：

| 维度 | 账本A（P/B/S/E/R/I） | 账本B（Q层） | 隔离方案 |
|:-----|:--------------------|:-------------|:---------|
| 存储 | P系列输出目录、数据库 | Q层审计报告、failure_registry表 | 物理路径分离 |
| 写入权限 | 墨衡（策略研究者） | Q层系统（自动）+ 墨萱（审计） | 通过Gate+权限控制 |
| 读取权限 | 所有人可读 | 所有人可读 | 只读不写约束 |
| 修改权限 | 墨衡 | 仅追加，不可修改已有记录 | 通过代码约束 |

**技术挑战**:

1. **"只读不写"约束需要代码强制** — 双账本规范第B4条禁止Q层修改P系列原始数据。但这份约束目前停留在"文档规范"层面，没有代码层的强制保障。如果墨衡在Q层代码中不小心写了一个修改P系列数据的操作，但没有代码审计阻止它，这个约束就会失效。

   **建议**: 在Q层代码模板中增加以下防护：
   - Q层模块的 `open()` 操作默认为 `r` 只读模式
   - 所有写操作仅限于 `research_failures/` 目录和 `failure_registry` 表
   - 在Gate代码层加入路径白名单检查

2. **"同一研究者不得同时参与同一策略的账本A生产和账本B审计"** — 当前团队只有墨衡和墨萱。墨衡负责Q1/Q2/Q3/Q6/Q7/Q8/Q9（7个模块），墨萱负责Q4/Q5（2个模块）。墨衡既是账本A的生产者，又是账本B的主要实现者，存在利益冲突风险。

   **建议**: 
   - 维护明确的**职责矩阵表**：每个策略在账本A的"生产者"和账本B的"审计者"必须不同人
   - 目前墨萱负责的模块（Q4/Q5）建议由墨衡做第二人交叉代码审查
   - 墨衡负责的模块（Q1/Q2/Q3/Q6/Q7/Q8/Q9）建议由墨萱做交叉代码审查

**综合判断**: 数据隔离方案技术上可行，但需要代码层的防护约束来确保规范不被绕过。

### 2.5 Failure Registry — 数据库设计和集成方案

**审查结论**: ✅ 通过（但有一个改进建议）

**分析**:

Q9 Failure Registry的设计合理：

1. **数据模型** — `failure_registry` 表包含全部必要字段，索引设计（failure_type/strategy_id/regime/timestamp）覆盖了主要查询模式 ✅

2. **文件+数据库双存储** — `research_failures/records/` 的JSON文件和 `failure_registry` 数据库表的双重存储有以下问题：

   ```
   ❌ 问题: 双存储意味着存在数据同步风险
   
   如果代码中的写入路径是先写文件再写DB（或反之），
   某个写入失败会导致两边数据不一致。
   ```

   **建议**: 采用**数据库优先**策略：
   - 以 `failure_registry` 表为主数据源
   - JSON文件仅作为缓存/导出格式（可通过查询引擎从DB生成）
   - 避免两个数据源互相作为权威

3. **门控集成** — Q1/G1/G2/G3自动写入Q9的设计完整，覆盖了所有Gate失败路径 ✅

4. **与KnowledgeBridge的互补关系** — 设计合理，形成"成功+失败"双向知识闭环 ✅

5. **查询能力** — 聚合统计+趋势分析+复发检测的覆盖完整 ✅

**工时估算**: 1.0天合理（0.2+0.2+0.3+0.2+0.1=1.0天），各子任务比例合理。

### 2.7 Kill Switch 条件

**审查结论**: ⚠️ **存在 P1 问题 — Kill Switch条件基本充分，但有一项缺失**

**Kill Switch 评估**:

| 编号 | 触发条件 | 触发阶段 | 技术判断 |
|:----:|:---------|:--------:|:---------|
| KS1 | Q7评分与人工偏差>0.10 | Phase 0验收 | 🟡 **时序问题**: Q7在Phase 1才实现，但KS1列在M0验收 — 详见2.3节分析 |
| KS2 | Q2对所有策略判为PEAK/PLAIN | Phase 1 | ✅ 合理 |
| KS3 | G3人审与Q层系统不一致 | Phase 1 | ✅ 合理 |
| KS4 | 校准后F级>70% | Phase 2 | ✅ 合理 |
| KS5 | Q层导致管线运行>10% | 任意阶段 | ✅ 合理 |
| KS6 | P系列版本分歧 | Phase 0前 | ✅ 合理 |

**P1问题 #3 — 缺失Kill Switch: 数据一致性校验失败**:

问题描述: 当前Kill Switch列表缺少一个**关键保护机制**：当Q层读取到的P系列计算结果与预期基线不一致时（例如：字段格式变更、计算方式变更导致Q层审计结果出错），系统缺少自动检测和停止机制。

- **具体场景**: 墨衡后期修改了P系列的某个计算函数（如夏普比率的计算方式），但没有同步更新Q1的读取逻辑。Q1读到了格式不同的数据，导致所有经由Q1的判断出错。
- **严重级别**: P1（在Phase 1期间可能出现，影响全部Q层审计）
- **修复建议**: 在Phase 0a/0b期间增加以下防护：
  - **KS7（新增）**: 每个Q模块输入数据与基线schema不一致时，自动暂停该模块并告警
  - 实现方式：创建P系列输出数据的schema定义文件，Q层各模块读取时先做schema校验

**时效性**: 建议在Phase 0a启动前在KS列表中添加此项。
