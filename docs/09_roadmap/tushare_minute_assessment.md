# 分钟级改用Tushare Pro — 可行性评估

> 作者：墨涵 | 日期：2026-05-22
> 背景：主人要求日频和分钟级数据源统一为 Tushare Pro

---

## 一、Tushare 分钟级接口

Tushare Pro 提供 `stk_mins` 接口（分钟K线）：

| 属性 | 内容 |
|:----|:-----|
| **接口名** | `stk_mins` |
| **输入** | ts_code, freq(1min/5min/15min/30min/60min), start_date, end_date |
| **输出字段** | ts_code, trade_date, trade_time, open, high, low, close, volume, amount |
| **是否为复权** | **否** — 接口返回不复权数据，需自行复权 |
| **历史深度** | **约3-6个月**（普通账号），付费升级后可回溯更长时间 |
| **每次调用消耗** | ~15分/次（单标的×单日） |

分钟级不复权是已知的重要约束（AKShare 同样问题），需额外实现复权处理。

## 二、积分消耗评估

### 历史回填（一次性）

| 场景 | 计算方法 | 预估消耗 | 能否在2000分/天完成 |
|:----|:---------|:-------:|:------------------:|
| **方案B分钟（4只×5年）** | 4只×~1200日×15分=72,000分 | **~72,000分** | ❌ 需分期36天 |
| **方案C分钟（40只×6年）** | 40只×~1440日×15分=864,000分 | **~864,000分** | ❌ 需分期>400天 |
| **方案B日频历史（12只×5年）** | 12只×~1200日×5分=72,000分 | **~72,000分** | ❌ 需分期36天 |

### 日常更新

| 场景 | 计算 | 每日消耗 |
|:----|:----|:-------:|
| **方案B日频（12只）** | 12次×5分 | ~60分/天 ✅ |
| **方案B分钟（4只）** | 4次×15分 | ~60分/天 ✅ |
| **方案C日频（40只）** | 40次×5分 | ~200分/天 ✅ |
| **方案C分钟（40只）** | 40次×15分 | ~600分/天 ⚠️ 可控 |

**关键约束**：历史回填的峰值消耗远超2000分/天上限。

**解决方案**：历史回填必须分批（每日分批跑一部分），或者升级Tushare账号提高积分上限。

## 三、分钟级复权问题

Tushare `stk_mins` 返回**不复权数据**。

**复权方案需要自行实现**，目前有两种路径：

| 方案 | 工作方式 | 工作量 | 可靠性 |
|:----|:--------|:-----:|:-----:|
| **A. 用日频复权因子推导** | 从 Tushare `adj_factor` 获取日复权因子 → 线性插值到分钟 → 换算分钟复权价 | ~3天 | ⚠️ 日->分钟插值存在精度误差 |
| **B. 用Tushare复权因子分钟表** | `stk_factor_min` 接口含复权因子，但与`stk_mins`需对齐 | ~5天 | ✅ 更准确但需要数据对齐 |

**两种方案均需要额外开发**，是分钟级管线不可省略的工程投入。

## 四、数据质量对比

| 维度 | Tushare `stk_mins` | AKShare `stock_zh_a_minute` |
|:-----|:------------------:|:--------------------------:|
| 数据源背景 | 券商行情接入 | 爬虫聚合 |
| 回拉深度 | ~3-6个月（普通） | ~5个交易日 |
| 历史回填 | **支持较长回溯（付费）** | 不支持 |
| 分钟级复权 | ❌ 无 | ❌ 无 |
| 断点率 | 低（商业数据源） | 中（爬虫不稳定） |
| 接口稳定性 | ✅ 稳定 | ⚠️ 偶发超时/空回 |
| 限频 | 200分/次 or 60次/min | 无官方限频 |

**结论**：Tushare 分钟级在**质量稳定性**上优于 AKShare，但**历史回溯能力受限**（普通账号仅3-6个月）。

## 五、对方案B Phase 1+2 的影响

| 决策 | 影响评估 |
|:----|:--------|
| ✅ Phase 1（12只×5年日频） | 无影响。日频已有Tushare采集器 |
| ⚠️ Phase 1 Tushare积分规划 | 日频历史回填需~72,000分，需做分批计划（~36天） |
| ⚠️ Phase 2 分钟级用Tushare | 分钟级回填同样需分批。普通账号仅能回溯3-6个月 → 分钟级方案B的实际回填长度需调整（3-6月而非5年） |
| ⚠️ 分钟级复权 | 额外3-5天工程，Phase 2需含此工作量 |

## 六、建议方案调整

1. **Phase 1 保持不变**：12只×5年日频（Tushare已有），历史回填做分批计划
2. **Phase 1 同期**：预先做Tushare积分分批策略，确保日频和分钟级历史回填都能在合理时间内完成
3. **Phase 2 分钟级调整**：
   - 分钟级使用Tushare `stk_mins`，但历史回填深度调整为**最近1-2年**（非5年）
   - 分钟级复权开发包含在Phase 2工程量中
   - 如果从Phase 1已有的日频复权因子推出分钟复权，可节省部分时间
4. **如需更长时间的分钟数据**：考虑升级Tushare账号（2000分→更高额度，或购买分钟级数据包）
