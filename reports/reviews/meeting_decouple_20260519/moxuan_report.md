## 1. 当前问题诊断（测试/集成视角）

从测试角度看，当前架构存在以下系统性痛点：

### 1.1 回测与交易耦合 → 测试边界模糊

- `strategy` 直接依赖 `position/account/order` — 这意味着回测逻辑里混入了交易执行逻辑。**测试时无法独立验证"信号生成是否正确"和"执行是否正确"，因为两者被绑定在同一函数路径中。**
- 如果要改一条 MACD 参数，必须跑完整交易流程才能看到效果。**单测覆盖率再高也救不了集成测试的困难，因为集成测试本身要把两个子系统一起跑。**
- 错误定位难：修持仓还是修信号？混合在一起后，一个 bug 可能同时影响两个子系统，排查成本翻倍。

### 1.2 无法独立验证"研究质量"

当前输出只有 `reports/research/*.md` — 这些是自然语言报告，**不是结构化测试断言**。我的验收工作变成了"人工读报告"，而不是"自动化断言校验"。
- 没有可断言的标准输出格式
- 回测结果没有评级、没有量化指标的直接数据库记录
- 即使墨衡做了策略，我也没有方法自动化验证这个策略"合格还是不合格"

### 1.3 测试范围模糊

目前要跑一个"完整测试"等于要跑：
> 数据加载 → 回测引擎 → 资金管理 → 订单执行 → 绩效计算 → 报告生成 → 飞书推送

整个链条混在一起。**任何一个环节出问题，整个 CI 都会失败。** 无法做增量测试、无法做分层测试。

### 1.4 回归测试成本高

修改一个回测参数逻辑，要重新跑完整交易流水线才能验证。没有快速回滚的边界。**这直接导致我不敢放权让墨衡频繁改代码，因为每次改动都是全量回归。**

## 2. 重构目标（可测试性层面）

我对解耦后应达成的测试目标提出以下要求：

| # | 目标 | 验证方式 | 优先级 |
|---|------|---------|--------|
| T1 | 回测系统可独立启动并运行 | `python run_backtest.py --config X` 成功退出 | P0 |
| T2 | 交易系统可独立启动并运行 | cron 各时点管线无回测依赖错误 | P0 |
| T3 | 回测结果可结构化断言 | `strategy_ratings` 表可被查询并验证字段完整 | P0 |
| T4 | 知识过期机制可测 | 插入过期记录 → 交易系统查询时不应使用 | P0 |
| T5 | knowledge.db 读写分离正确 | 交易系统写 knowledge.db 应被阻止 | P1 |
| T6 | 交易系统在回测系统崩溃时仍能运行 | 停掉回测进程 → 交易管线正常运行 | P1 |
| T7 | 参数扫描结果可自动化验证 | 全参数网格运行后 → 断言最优区间正确性 | P2 |
| T8 | OOS 验证流水线 | 拆分后应方便接入 out-of-sample 流水线 | P2 |

**核心原则：拆完之后，回测系统和交易系统各自要有独立的测试套件，且一套失败不影响另一套。**

## 3. 架构方案（接口与集成视角）

我认可墨衡提出的 **physical isolation, knowledge sharing** 原则。在此基础上，我从测试验收角度补充接口规范：

### 3.1 knowledge.db 的接口契约

knowledge.db 是整个拆分中**最核心的集成点**。它的接口是否干净，直接决定了测试的难易度。

**我要求将其视为一个"松耦合 API"来定义契约：**

```
回测系统 → knowledge.db (写入)
  - 契约：INSERT 操作必须满足完整字段约束
  - 校验项：
    * backtest_hash 唯一性
    * expires_at 不可为空
    * grade 取值限于 A/B/C/D
    * 每条记录必须关联至少一种策略

知识库 → 交易系统 (查询)
  - 契约：交易系统只能执行 SELECT，绝无 INSERT/UPDATE/DELETE
  - 校验项：
    * 所有查询使用 LIMIT + expires_at 过滤
    * 查询结果为空时的降级策略必须定义
```

### 3.2 建议增加的"测试桩"设计

为了让两个系统在互相隔离的情况下也能测试，我建议增加：

```
回测系统测试桩：
  - mock_knowledge_db.py — 模拟 knowledge.db 写入端
  - fake_market_data.py — 可注入任意行情数据

交易系统测试桩：
  - knowledge_db_stub.py — 模拟回测系统的查询结果
  - fake_signal_source.py — 手工注入交易信号
```

### 3.3 文件系统隔离的测试含义

墨衡的隔离方案（独立目录结构）对测试有利：
- 可以单独对 `backtest_tools/` 目录运行 pytest
- 可以单独对 `reports/` 和 `signals/` 目录运行测试
- 但需要确保 `reports/research/` 只读只写只在回测系统内

## 4. 核心模块与接口（我关注的测试点）

| 模块 | 所在系统 | 关键接口 | 测试关注点 |
|------|---------|---------|-----------|
| 回测引擎 | 回测 | `BacktestEngine.run(config) → Result` | 参数边界、空数据、异常数据 |
| 绩效计算 | 回测 | `CalcPerformance(trades, capital) → Metrics` | 精度（浮点数比较）、极端行情 |
| 参数扫描 | 回测 | `ParamSweep(config_grid) → DataFrame` | 网格边界、步长、参数组合数 |
| 结果写入 | 回测 | `ResultSaver: write_rating(...)` | 字段完整性、哈希冲突、过期时间 |
| knowledge.db | 共享 | `SELECT ... WHERE symbol, strategy` | 返回为空时的降级逻辑 |
| knowledge.db | 共享 | `INSERT ... WITH backtest_hash` | 重复记录过滤、数据校验 |
| 晨报管线 | 交易 | `pipeline_complete.json` | 信号格式契约、触发条件 |
| 仿真交易 | 交易 | `TradeExecutor: execute(signal)` | 滑点模拟、仓位限额 |
| 结算引擎 | 交易 | `SettlementEngine: settle() → PnL` | 精度、多账户汇总 |
| 运营日报 | 交易 | `DailyReport: generate() → .md` | 数据一致性 |

## 5. 优点（可测试性角度）

### 5.1 独立的测试套件

拆后可以建两套 `pytest`，互不影响：
- `pytest backtest_tools/` — 40+ 用例覆盖回测引擎、绩效、参数扫描
- `pytest tests/trading/` — 30+ 用例覆盖交易管线各节点

### 5.2 分层测试成为可能

| 测试层级 | 含义 | 例子 |
|---------|------|------|
| 单元测试 | 单函数/单类 | 夏普比率计算、信号排序 |
| 集成测试 | 模块间交互 | 回测引擎+绩效计算的链式调用 |
| 系统测试 | 整套管线 | 从参数输入到知识库写入全流程 |
| 隔离测试 | 子系统独立性 | 回测崩溃时交易系统的行为 |

### 5.3 测试数据可注入

回测系统独立后，可以构造各种"奇怪的行情数据"来测试：
- 空值行情
- 涨停跌停连续行情
- 分红除权行情
- 等持有期超过历史的情况

这些在耦合架构下几乎不可能做，因为数据流和执行路径绑在一起。

### 5.4 回归测试范围可控

修改回测引擎参数 → 只回归 `backtest_tools/` 套件
修改交易信号逻辑 → 只回归 `tests/trading/` 套件
**大幅降低每次改动的全量回归成本。**

### 5.5 知识库的测试断言性

有了 `strategy_ratings` 表，我可以自动化断言：
```python
assert db.query("SELECT grade FROM strategy_ratings WHERE strategy='trend'").iloc[0] == 'A'
```

而不是手动去读 markdown 报告。**这是质的飞跃。**

## 6. 缺点（测试复杂度增加）

### 6.1 新增一个"集成点"——knowledge.db

耦合架构下没有 database 交互的测试需求，拆后新增了：
- 写入接口测试（回测→知识库）
- 读取接口测试（交易→知识库）
- 读写分离验证
- 数据一致性校验

**需要增加至少 3 个测试文件**：`test_knowledge_db_write.py`、`test_knowledge_db_read.py`、`test_knowledge_db_integrity.py`

### 6.2 需要维护两套测试环境

回测系统和交易系统各自有独立的 `conftest.py`、independent fixture、数据依赖。虽然好处是互不影响，但维护成本翻倍。

### 6.3 跨系统问题的诊断难度增加

如果一个 bug 出现在"回测结果写入后、交易系统读取时"的交互路径上，日志追踪需要跨两个子系统。**需要设计统一的 trace_id 机制。**

### 6.4 知识库的时间敏感测试

`expires_at` 机制的测试天然有时序依赖：
- 测试"过期前应被使用"→ 插入当前时间戳
- 测试"过期后应被忽略"→ 插入过去的时间戳
- 测试清理机制 → 需要 mock 系统时间

**每次跑完整测试需要重置知识库状态。**

## 8. 长期扩展性（对报告质量验证的好处）

### 8.1 报告质量验证自动化

拆分后，回测结果以结构化数据（`strategy_ratings` 表）而非纯文本报告呈现。这对我的工作有两层质的提升：

**现在：**
```
人工读 MD 报告 → 是否有遗漏 → 是否有矛盾 → 给出 PASS/FAIL
```

**拆后：**
```
自动化断言评审标准 → 输出 PASS/FAIL → 人工查阅不一致时再介入
```

### 8.2 知识库驱动的验证流程

```
回测结果写入 knowledge.db
        ↓
墨萱验证 agent：查询知识库记录，自动化校验
  ├── 评级是否符合预期分布（A<B<C<D 合理性校验）
  ├── 参数建议是否在合理区间内（数值范围校验）
  └── 绩效指标是否在合理范围（Sharpe>0、MDD<X 等）
        ↓
输出：校验报告 → 合格/不合格
```

### 8.3 策略退化监控

长期累积的 `knowledge.db` 数据可以驱动策略退化分析：
- 同一策略、同一标的的不同时间段的绩效对比
- 过期记录的自动清理与警告
- 通过定期运行比较验证 OOS 稳健性

### 8.4 支持 A/B 测试

未来可以：
- 跑两套参数并比较结果（A/B 实验对比）
- 自动化跑参数扫描后，断言"最优参数集"的稳定性
- 这些都是我在测试报告中可以直接量化的

## 9. 风险

以下是我**最担心的问题**，按风险等级排列：

### 🔴 高风险：knowledge.db 成为新瓶颈

**问题：** 如果 knowledge.db 的读写没有做好隔离，表面上是"松耦合"，实际上成了一个"紧耦合的单点故障"。
- 写入时卡死 → 交易系统读不到，降级逻辑被执行 → 可能导致交易异常
- 表结构不兼容 → 回测系统新增字段 → 交易系统旧查询中断

**我的要求：** knowledge.db 必须做版本化迁移机制。每次修改表结构要写迁移脚本，且必须有向后兼容性。

### 🟠 中高风险：交易系统对知识库的"过度依赖"

**问题：** 设计稿里交易系统在 12:30 午间交易时"查询 knowledge.db → 策略经验参考"。如果这个查询成为交易执行的前置条件，**知识库空/慢 = 交易系统不工作**。

**我的要求：** 定义清晰的降级策略：
- knowledge.db 不可用时 → 交易系统应使用默认参数执行交易
- 知识库空 → 不应阻止交易执行
- 且降级行为必须可测试（见 T6）

### 🟠 中风险：测试环境与生产环境的数据隔离

**问题：** knowledge.db 的测试需要 mock 数据。如果测试时连到了真实知识库，会污染数据。

**我的要求：** 所有知识库操作必须支持环境切换：
```python
db = KnowledgeDB(env="test")   # 独立 test_knowledge.db
db = KnowledgeDB(env="prod")   # 生产库
```

测试框架必须默认使用 test 环境，且生产环境写测试数据要有警告机制。

### 🟡 中低风险：回测系统的"可启动性"退化

**问题：** 回测系统独立出来之后，如果依赖链没有清理干净（例如还隐式依赖交易系统的模块），可能会导致独立运行时 import error。

**我的要求：** 在 Phase 1 结束时，必须通过以下命令验证独立性：
```bash
python -c "from backtest_tools.engine import BacktestEngine; print('OK')"
```
而且在没有 `reports/morning/`、`signals/`、`trade_engine.db` 的情况下必须能正常工作。

### 🟢 低风险：测试用例维护滞后

**问题：** 52h 的测试编写量，如果开发（墨衡）和测试（我）不同步，可能出现"代码改了但测试没更新"的情况。

**我的要求：** 建立测试覆盖率门禁：
- 新增代码必须配套 test case
- 核心模块覆盖率达到 80% 以上
- 合并前 CI 必须通过
