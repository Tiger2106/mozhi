### 1.1 AKShare `stock_zh_a_hist()` 接口字段分析

通过源码分析确认，AKShare `stock_zh_a_hist()` 请求的 API 字段参数为：

```python
params = {
    "fields2": "f51,f52,f53,f54,f55,f56,f57,f58,f59,f60,f61,f116",
    #          ↓      ↓    ↓    ↓    ↓    ↓     ↓    ↓   ↓    ↓   ↓   ↓
    #         日期   开盘  收盘  最高  最低  成交量 成交额 振幅 涨跌额 涨跌幅 换手率 流通市值
}
```

| 索引 | API字段 | 中文列名 | 当前 column_map 已映射 | 备注 |
|:----:|:-------:|:--------:|:---------------------:|:-----|
| 0 | f51 | 日期 | ✅ → date | |
| 1 | f52 | 开盘 | ✅ → open | |
| 2 | f53 | 收盘 | ✅ → close | |
| 3 | f54 | 最高 | ✅ → high | |
| 4 | f55 | 最低 | ✅ → low | |
| 5 | f56 | 成交量 | ✅ → volume | |
| 6 | f57 | 成交额 | ✅ → amount | |
| 7 | f58 | 振幅 | ❌ 未映射 | 暂不需要 |
| 8 | f59 | 涨跌额 | ❌ 未映射 | 暂不需要 |
| 9 | f60 | 涨跌幅 | ❌ 未映射 | 暂不需要 |
| **10** | **f61** | **换手率** | **❌ 未映射 ← 正是本次需新增** | **P0** |
| 11 | f116 | 流通市值 | ❌ 被"股票代码"覆盖 | 注意:源码中用此列存symbol |

**关键发现：**
1. ✅ **f61（换手率）** 已在 API 响应中，当前 column_map 未启用 → 加一行即可读取
2. ⚠️ **f116（流通市值）** 也在 API 响应中，但当前代码已将末尾列重写为"股票代码"，**流通市值被覆盖丢失**
3. 若需方式B（计算法），需修改源码保留 f116 或改为从 `stock_zh_a_hist()` 请求独立字段

## ═══════════════════════ 3. 方式A vs 方式B 分析 ═══════════════════════

### 3.1 方案对比

| 维度 | 方式A：直接读取原生换手率 | 方式B：volume / 流通股本计算 |
|:-----|:------------------------:|:---------------------------:|
| 数据来源 | AKShare f61（换手率） | volume + circulating_shares |
| 代码改动 | 1 行 column_map 新增 | 需修改 column_map + 保留 f116 或另调用接口 |
| 精度 | **交易所原始值**（最精确） | 依赖于流通股本数据的时效性 |
| 股权变动处理 | ✅ 自动包含（交易所已调整） | ❌ 需手动处理增发/送股/解禁 |
| 复杂度 | 极低 | 中（需保留流通市值字段） |
| 历史回溯 | 全量补填（重跑数据管道） | 需历史流通股本序列 |

### 3.2 推荐方案：方式A（直接读取）

**理由：**
1. column_map 加一行即可，零额外风险
2. 交易所原始数据，精度最高
3. 自动包含股权变动调整
4. 当前 `market_daily.turnover_rate` 列已预留，直接写入即可

### 3.3 方式B 作为验证/兜底

方式B 的计算公式：
```python
turnover_rate = volume / (circulating_market_value / close) * 100
```

当前 AKShare 源码将 f116（流通市值）覆盖为股票代码，复用方式B 需：
- 修改 AKShare 响应解析逻辑，保留 f116 列
- 或使用独立接口获取流通股本

## ═══════════════════════ 4. 边界条件验证 ═══════════════════════

### 4.1 Size Effect（大小盘换手率差异）

基于 AKShare API 结构分析和 A 股市场规律，大小盘标的的换手率存在系统性差异：

| 市值分类 | 典型标的 | 日均换手率范围 | 示例 |
|:--------|:---------|:-------------:|:----:|
| 超大市值（>5000亿） | 工商银行、贵州茅台 | 0.1% ~ 0.8% | 工行 ~0.2% |
| 大市值（1000-5000亿） | 平安银行、宁德时代 | 0.5% ~ 3.0% | 平安银行 ~1.5% |
| 中市值（200-1000亿） | 科大讯飞、京东方A | 1.0% ~ 5.0% | 科大讯飞 ~2.5% |
| 中小市值（<200亿） | 题材股 | 3.0% ~ 15%+ | 题材股 ~8% |

**⚠️ 影响：** 跨标的的换手率绝对值不可直接比较。大盘蓝筹（0.2%）和小盘题材股（8%）的换手率差距可达 40 倍。

**建议：**
- 跨标的使用时做 **Z-score 标准化**（相对于自身历史均值和标准差）
- 在 Factor 元数据中注明 `normalization: "zscore"`（可选参数）
- 用于 Regime 分类时，建议使用换手率的变化值（Δ），而非绝对值

### 4.2 次新股换手率异常

| 上市天数 | 换手率特征 | 说明 |
|:--------|:----------|:-----|
| 1-30天 | 极高（30%~80%+） | 新股上市换手密集，投机性交易主导 |
| 30-60天 | 偏高（10%~30%） | 逐步回归正常水平 |
| 60-180天 | 偏高（5%~15%） | 仍高于成熟标的 |
| >180天（6个月） | 正常水平 | 换手率趋于稳定 |

**⚠️ 影响：** 上市不足 6 个月的标的，换手率不在正常分布范围内，会污染指标计算。

**建议：**
- 换手率因子中增加 **上市天数过滤器**：`min_listing_days=180`（默认参数）
- 在 Factor 计算前调用过滤器：`if days_since_listing < 180: return NaN`
- 上市天数可通过 `stock_zh_a_hist()` 返回的第一条日期推算

### 4.3 其他边界条件

| 条件 | 影响 | 处理方式 |
|:----|:-----|:---------|
| 停牌日 | volume=0，换手率=0 | 保持 0.0 或标记为 NaN |
| 股权登记日附近 | 流通股本变动 | AKShare 原生值已含调整 |
| 大宗交易数据 | 不纳入交易所换手率计算 | 无需处理 |
| 限售股解禁 | 流通股本突增，换手率骤降 | AKShare 原生值自动反映 |
| 港股通/陆股通 | A 股换手率仅计算 A 股流通部分 | 无需处理 |

## ═══════════════════════ 5. 与现有指标的关系 ═══════════════════════

### 5.1 现有 volume 族因子清单

| 因子名称 | 注册名 | 核心含义 | 与换手率关系 |
|:--------|:------|:---------|:------------|
| OBV | obv | 累积能量线 | 完全不相关（方向累积 vs 比率） |
| VWAP | vwap / anchored_vwap | 成交量加权均价 | 不相关（价格维度 vs 量维度） |
| 量比 | volume_ratio | 相对放量（当日/5日均量） | **部分相关但非冗余** |
| Volume Profile | volume_profile | 价格-成交量分布 | 不相关（分布 vs 比率） |
| Volume Flow | volume_flow | 资金流向 | 不相关 |

### 5.2 量比(VolumeRatio) vs 换手率

| 对比维度 | 量比(volume_ratio) | **换手率(turnover_rate)** |
|:---------|:-----------------:|:------------------------:|
| 公式 | volume / volume_ma5 | volume / circulating_shares |
| 含义 | 相对于自身的放量/缩量程度 | 实际流通盘交易活跃度 |
| 量纲 | 倍数（无量纲） | 百分比（%） |
| 跨标可比性 | ✅ 可（相对于自身历史） | ❌ 不可（大小盘差异大） |
| 信息维度 | 相对变化（时间序列） | 绝对水平（横截面） |

**结论：独立因子，非冗余。** 量比回答"今天比平时交易多/少了多少"，换手率回答"今天流通盘被交易了多少比例"——两个完全不同的信息维度。

### 5.3 与 market_data_adapter.py 现有函数的关系

`calc_turnover_rate()` 函数已存在，提供三种回退逻辑：
1. 有流通市值列 → `volume * close / circulating_cap * 100%`
2. 无流通市值 → `amount.rolling(120).sum() * 10` 估算
3. 兜底 → `volume / volume.rolling(5).mean() * 0.5`

**注意：** 试验验证阶段推荐**不调用此函数**，而是直接使用 AKShare 原生值。此函数作为兜底降级策略保留。
