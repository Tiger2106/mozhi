## 1. 当前问题诊断

### 架构层面的耦合危害

当前系统的核心病灶是 **「研究与执行」在代码层面没有边界**。从问题陈述中可以看出：

- **strategy 内直接依赖 position/account/order** — 这意味着 signal 的好坏被混入了仓位管理、资金管理的噪音。信号的胜率、赔率、波段特征根本无法独立测量。
- **signal 研究必须跑完整交易系统** — 这是最致命的。你想研究一个技术指标的有效性，必须先启动交易引擎、初始化账户、建立持仓。研究效率被锁死在交易系统的启动耗时上。

### 战略层面的耦合危害

1. **无法形成「研究操作系统」** — 你需要的不是"策略脚本集合"，你需要的是一套可以独立运行、独立验证、独立回测、独立部署的研究基础设施。当前的耦合使得它只能算一个"带研究功能的交易系统"，而不是一个研究平台。
2. **方法难以复用** — MACD 方法直接产订单，意味着同样的信号逻辑无法在不同策略框架、不同参数配置、不同时间框架下快速验证。研究资产在贬值。
3. **扩展性天花板** — 当你需要接入新数据源（另类数据、宏观指标）、新分析方法（ML 模型、NLP 情感）、新市场（期货、期权、加密货币）时，每次都需要改造交易系统。这种耦合意味着每项创新都背负交易层的技术债。

### 一句话总结

> **当前结构让"研究"成为了"交易"的子集。你需要的是一个让"交易"成为"研究"子集的架构。**

---

## 2. 重构目标

解耦后应达到的架构目标，按优先级排列：

### P0：信号与执行彻底分离

- Signal 层只输出「信号描述」，不关心执行细节
- Signal 是纯研究产物：时间戳、方向、强度、置信度、条件参数
- 执行层负责将信号翻译为具体订单

### P0：研究平台可独立运行

- 不启动交易引擎，不依赖账户/持仓状态，就能完成全流程研究
- 回测是一个研究工具，不是交易系统的子模式

### P0：Signal 可被"盲测"

- 同一信号能回放历史数据、独立评估胜率/赔率/夏普/最大回撤
- 评估结果不受仓位管理、资金管理策略污染

### P1：Signal 组合层独立存在

- 多信号融合 → 信号组合 → 权重分配 → 执行指令
- 组合层是研究到执行的桥梁，而非交易的附属品

### P2：异步双轨制

- 研究流水线和交易流水线在时间上解耦
- 研究结论 → 发布到信号仓库 → 交易系统异步拉取
- 不再要求"研究结束 = 立即执行"

---

## 3. 架构方案

我建议的架构布局为 **「四层管道 + 一个信号总线」**：

```
┌──────────────────────────────────────────────────┐
│                   数据层                          │
│  行情数据 · 另类数据 · 宏观指标 · 日历事件        │
│  输出：标准化时间序列 DataFrame                    │
└────────────────────┬─────────────────────────────┘
                     │
┌────────────────────▼─────────────────────────────┐
│                 研究层 (Research Engine)           │
│  技术分析模块 · 信号生成器 · ML 模型 · 回测引擎    │
│  输出：Signal 对象（纯描述，不含执行指令）           │
│  ★ 可独立运行，不依赖交易系统                      │
└────────────────────┬─────────────────────────────┘
                     │
┌────────────────────▼─────────────────────────────┐
│               组合层 (Combination Layer)            │
│  多信号融合 · 权重分配 · 仓位建议 · 风险过滤        │
│  输出：Trade Intent（交易意图，非订单）              │
│  ★ 可切换不同配置策略                              │
└────────────────────┬─────────────────────────────┘
                     │
┌────────────────────▼─────────────────────────────┐
│               执行层 (Execution Engine)             │
│  订单生成 · 仓位管理 · 资金管理 · 风控检查          │
│  输出：Order / Position / Account 状态              │
│  ★ 接收 Trade Intent，自身执行逻辑                  │
└──────────────────────────────────────────────────┘
                      │
          ┌───────────▼───────────┐
          │    信号总线 (Signal Bus)  │
          │   Signal Store / 信号仓库 │
          │   历史信号存档/回放/评估    │
          └───────────────────────┘
```

### 关键设计原则

1. **单向依赖**：数据 → 研究 → 组合 → 执行。上层不依赖下层。
2. **信号作为契约**：研究层产出 Signal 对象，其接口是上下层的契约。
3. **组合层为可选通道**：对于简单策略可跳过，直接 Signal → 执行。
4. **信号总线是跨时间桥**：研究层今天产的信号，交易系统明天拉取执行，时间解耦。

---

## 4. 核心模块

### 🧠 Signal Engine（信号引擎）— 战略级

我眼中的**最高优先级**模块。它应该是：

- 一个独立可运行的研究运行时
- 能够接收任意数据输入 → 运行分析逻辑 → 输出标准 Sigmal
- 内置信号评估框架（胜率、赔率、IC、IR、回撤统计）
- 支持信号注册/发现机制（未来可插件化）

### 🧵 Signal Store（信号仓库）— 战略级

- 所有历史信号的数据库
- 支持「信号回溯」：给定一个时间点，能查到所有层级的信号状态
- 支持「组合重算」：同一历史信号用不同权重重新组合，不需要重跑研究
- 这是从"策略脚本集合"到"研究操作系统"的关键特征

### 🔄 Backtest Engine（回测引擎）— 战术级

- 接受 Signal 输入，模拟执行
- 不耦合具体账户/持仓逻辑
- 支持参数扫描、敏感性分析

### 🎯 Combination Manager（组合管理器）

- 多信号加权融合
- 风险预算分配
- 动态权重调整

## 5. 优点

| 层面 | 好处 |
|------|------|
| **研究效率** | Signal 研究不再需要启动交易系统，从分钟级降到秒级 |
| **评估质量** | Signal 独立评估，胜率/赔率/IC 不被仓位管理噪音污染 |
| **可重复性** | 从研究到执行的每步都有独立记录，可完全重现 |
| **并行研发** | 研究团队和交易团队可并行工作，不互相阻塞 |
| **资产沉淀** | Signal 存入仓库，每一次研究都是可复用的资产 |
| **风险隔离** | 研究 Bug 不会直接产订单导致实盘损失 |
| **多市场扩展** | 新市场只需适配执行层，研究层和数据层无需改动 |

---

## 6. 缺点

| 维度 | 妥协说明 |
|------|---------|
| **延迟增加** | 信号经过总线再下达到执行层，比直接产订单增加一个异步间隙（可容忍，非敏感信号不受影响） |
| **复杂度增加** | 从两层变为四层 + 总线，需要更多的模块边界管理和接口维护 |
| **实时性折损** | 对于需要 tick 级响应的策略（高频/做市），多一层抽象可能延迟不够低，需要额外优化通道 |
| **过渡期阵痛** | 重构过程中老系统需保持运行，双轨运行增加运维负担 |
| **开发前期投入** | Signal 接口设计、Signal Store Schema、总线消息协议需要 upfront 设计，不能边做边改 |

---

## 7. 迁移成本

### 估算维度

| 类别 | 投入量级 | 说明 |
|------|---------|------|
| **Signal 接口定义** | 1-2 周 | 定义 Signal 对象的数据结构、评估标准、序列化格式 |
| **Signal Store 搭建** | 2-3 周 | 数据库 Schema、读写接口、查询 API |
| **回测引擎改造** | 3-4 周 | 从耦合到解耦的迁移，核心工作量 |
| **现有策略迁移** | 2-3 周/策略 | 将现有策略拆分为 Signal 层 + 执行层 |
| **总线 / 组合层** | 2-3 周 | 消息路由、组合逻辑、风控过滤 |
| **CI/CD 适配** | 1 周 | 新模块的测试框架、部署流水线 |

**总估：8-12 周（2-3 个月）核心开发 + 策略迁移并行**

### 渐进策略建议

- **Phase 1（前 2 周）**：定义 Signal 接口 + 搭建 Signal Store（最低 MVP）
- **Phase 2（第 3-5 周）**：改造回测引擎 + 迁移一个核心策略做验证
- **Phase 3（第 6-8 周）**：搭建组合层 + 总线 + 全部策略迁移
- **Phase 4（第 9-12 周）**：双轨运行、清理老系统、文档收尾

## 8. 长期扩展性

### 这个架构能支撑多久？

**3-5 年的战略框架，2 年内不需大改。**

判断依据：

1. **分层架构成熟**：数据 → 研究 → 组合 → 执行，是量化系统中经过验证的分层模式。主流量化平台（QuantConnect、WorldQuant、Blueshift）底层逻辑与此一致。
2. **Signal 契约化**：只要 Signal 接口保持稳定，任一层的内部重构不影响其他层。
3. **可插件化**：研究层支持新策略类型，只需要注册新 Signal 生成器。组合层支持新融合算法，只需要实现组合接口。
4. **多市场兼容**：新市场只需新增执行层适配器，研究层和数据层无需改动。

### 可能的扩展路径

- **+1 年**：引入 ML Pipeline（NLP 情感 + 另类数据）
- **+2 年**：支持多市场多币种（研究层统一处理，执行层分市场适配）
- **+3 年**：接入外部信号源（第三方研究/社区信号）
- **+3-4 年**：多策略并行 → 组合层升级为组合优化引擎（因子模型/风险平价）

### 何时需要再思考

- 当策略数量 > 100 时，Signal Store 需要考虑分片
- 当并行回测需求上升时，研究层需要做分布式改造
- 当延迟敏感度从「分钟级」变为「毫秒级」时，执行层可能需要直连通道跳过总线

## 9. 风险

### ⚠️ 我最担心的三个问题

#### 风险 1：过度设计

解耦重构最容易跌的坑是——在设计阶段把事情搞复杂了。

- **表现**：Signal 接口加了太多字段，总线协议过度泛化，组合层抽象层级过多
- **后果**：开发周期拉长，早期使用者（也就是你自己）觉得难用，回到旧习惯
- **缓解**：以实际策略驱动设计，先跑通一个完整 Signal → 评估 → 执行的 MVP 流程，再迭代

#### 风险 2：迁移途中的「断流期」

- **表现**：老系统被冻结，新系统还没完全准备好。研究产能断档 1-2 周。
- **后果**：错过市场机会，策略迭代停滞
- **缓解**：双轨运行策略，研究层优先上线（独立出 Signal），交易层后上。研究生产力不中断。

#### 风险 3：架构优雅但缺乏实际验证

- **表现**：设计做得漂亮，但搬到一个真正复杂的策略（多信号、多周期、多资产）时就暴露问题
- **后果**：架构需要打补丁，接口需要改，但改接口意味着所有层都需要改
- **缓解**：迁移初期选择 1-2 个最复杂的策略做"压力测试"，不要挑简单的梭哈

### 额外观察风险

- **用做研究平台的思路去设计交易系统** — 如果研究层的接口设计得过于灵活（允许自定义 Signal 类型、自定义评估函数），交易层会因此增加不必要的复杂度。需要明确边界：研究层可以灵活，执行层必须严谨。

## 附录：我作为宏观分析师视角的补充

我是宏观市场分析师（玄知）。虽然我的日常工作更多在板块轮动和趋势判断，但解耦重构对我同样重要：

1. **如果 signal 可以独立评估** — 我可以通过宏观信号库，直接评估"PMI 变化 + 社融数据"对某个板块的领先性，而不用等交易系统初始化
2. **研究操作系统** — 意味着我的宏观信号可以作为一层输入，与策略层的技术信号在组合层平行融合。这在当前耦合架构中是不可能的
3. **长期来看** — 宏观因子 + 技术信号 + ML 模型的融合，需要一个灵活的 Signal 总线。现在的耦合架构是这条路的天花板

换个角度说：解耦重构不仅是交易团队的技术债清理，它也是**扩展研究团队能力**的基础建设。当研究层能独立运行时，你才真正有了"多点并行研究"的能力。
