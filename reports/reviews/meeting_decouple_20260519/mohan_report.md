## 1. 当前问题诊断

从产品和知识管理角度看，当前系统的核心问题是**回测研究的产出没有积累效应**：

- **研究结果散落**：P系列报告、参数分析、回测结果全部以 markdown 文件堆在 `reports/research/`，没有结构化的知识库管理
- **知识无法复用**：墨衡每次做新研究都是从零开始，无法利用之前的研究成果
- **交易系统用不上研究成果**：晨报管线生成信号时，不会自动引用回测经验数据（因为不存在结构化存储）
- **ROI 不透明**：投入大量时间做回测研究，但研究成果对交易系统的影响不可追踪
- **"研究操作系统"缺失**：目前没有统一的研究平台，每次研究 = 写脚本 + 跑一次 + 出一份报告，没有迭代和积累

## 2. 重构目标

从产品视角，解耦后应达到：

- 回测研究可以**独立运行**，不依赖交易系统的任何组件
- 研究成果自动进入 `knowledge.db`，**一次研究，永久可用**
- 交易系统在生成信号时能**自动引用**回测经验（评级、参数建议、失败警示）
- 知识的有效性可管理（过期机制、置信度评级、版本回溯）

## 3. 架构方案

**三组件松耦合架构：**

```
┌─────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│   回测系统       │     │   knowledge.db    │     │   交易系统       │
│  (研究工具)      │ ──→ │  (知识桥梁)       │ ←── │  (自动化执行)    │
│                  │ 只写 │                  │ 只读 │                  │
│  墨衡/主人使用   │     │  策略评级         │     │  晨报管线        │
│  按需触发        │     │  参数建议         │     │  午间交易        │
│  不产生交易信号  │     │  失败记录         │     │  晚报结算        │
└─────────────────┘     └──────────────────┘     └──────────────────┘
```

**核心原则**：
- 物理隔离：不共享运行时文件
- 知识驱动：只通过 knowledge.db 交换
- 只读/只写分离：回测只写，交易只读

## 4. 核心模块

### 回测系统需要有的模块：
- **策略目录**：标准化策略注册（trend/reversal/grid 等）
- **参数管理器**：参数定义、范围、默认值管理
- **数据源适配器**：行情数据获取（akshare/东方财富/本地缓存）
- **回测引擎**：历史数据回放、仿真成交、滑点/手续费模型
- **绩效评估器**：Sharpe/MDD/胜率/盈亏比等指标计算
- **参数扫描器**：网格扫描/贝叶斯优化/参数稳健性分析
- **知识写入器**：回测结果 → knowledge.db 的标准接口
- **报告生成器**：可选，本地 md 报告

### 交易系统需要保留的模块（不变）：
- 晨报管线（7步 Agent 流水线）
- 午间交易调度 + pipeline_connector
- 晚报结算 + 运营日报
- Layer Q 治理层（Q1~Q9）
- 三个仿真账户

### 新增的桥接模块：
- **knowledge.db**：4表（strategy_ratings / parameter_recommendations / backtest_failures / strategy_summaries）
- **知识查询接口**：交易系统在 Step 1 墨衡分析时调用的 SELECT 接口
- **知识过期管理**：expires_at + 90天自动清理

## 5. 优点

- **研究效率提升**：回测独立运行，不再需要加载交易系统
- **知识积累**：每次回测结果自动入库，可追溯、可对比
- **信号质量提升**：交易系统分析时参考回测经验，决策更有依据
- **失败经验保留**：之前失败的参数组合也能入库警示，避免重复踩坑
- **降低复杂度**：两个系统各自聚焦，代码更清晰

## 6. 缺点

- **增加了中间层**：多了一个 knowledge.db 需要维护
- **知识同步延迟**：回测写完 knowledge.db 后，交易系统下次跑才能用到
- **初期工作量**：建库 + 迁移 + 接口开发需要时间
- **需要额外的质量门禁**：knowledge.db 的数据质量需要把关（墨涵的职责范围）

## 7. 迁移成本

从产品管理角度：

- **Phase 1 — 物理拆分 + knowledge.db 建库**：1.5 天
  - 目录分离 0.5d + DB schema + 写入接口 1d
- **Phase 2 — 回测系统独立**：2~3 天
  - 回测引擎 1.5d + 参数扫描 1d + 数据源适配 0.5d
- **Phase 3 — Layer Q 对齐**：1 天
- **Phase 4 — 交易系统接入 knowledge.db**：0.5 天
- **合计**：约 5~6 天

其中 Phase 1 和 Phase 2 可以并行。

## 8. 长期扩展性

- **策略插件化**：回测系统独立后，新策略可以通过简单注册即可加入
- **多数据源支持**：回测系统可以接入更多行情源
- **可视化**：回测系统未来可以加 Web UI
- **多人协作**：knowledge.db 支持多用户查询
- **知识图谱**：远期可将 strategy_ratings 扩展为知识图谱，发现策略之间的关联

## 9. 风险

- **拆一半卡住**：最大的风险是拆到一半发现依赖太深、退回原状
- **knowledge.db 数据质量**：如果写入的数据不准，交易系统引用了错误经验反而有害
- **知识过期无人维护**：90天过期后如果无人更新，交易系统将失去知识参考
- **资源分散**：同时维护两个系统，注意力被分散
