## 1. 当前系统的核心问题（测试/验收视角）

### 1.1 回测与交易耦合 → 测试边界模糊

`strategy` 模块直接依赖 `position/account/order` 模块。这意味着：

- **信号生成逻辑**与**交易执行逻辑**被绑定在同一个函数调用路径中，无法独立验证
- 修改一项参数（如 MACD 阈值），必须跑完整交易流程才能看到效果
- **单测覆盖率再高也没用**，因为集成测试必须同时依赖两个子系统
- bug 定位困难：一个问题可能同时来自信号逻辑和执行逻辑，排查成本翻倍

### 1.2 研究结果无法被自动化验收

当前回测输出只有 `reports/research/*.md` — 自然语言报告，不是结构化数据。

- **没有可供断言的标准输出格式**（无 schema、无字段约束）
- 回测结果没有量化评级、没有数据库记录供我查询
- 我作为验收方，**只能人工读报告来判断"合格还是不合格"** — 这根本不是可复现的验证流程
- 每次验收都要逐份阅读 Markdown 文件，效率低下且容易遗漏

### 1.3 测试范围模糊 → 无法分层测试

当前要跑一个"完整测试"，等于要跑整条链条：

```
数据加载 → 回测引擎 → 资金管理 → 订单执行 → 绩效计算 → 报告生成 → 飞书推送
```

- 任何一个环节出问题，整个测试都失败
- **无法做增量测试**：想验证"回测引擎的夏普比率计算是否正确"，不得不先过一遍全部上下游
- **无法做分层测试**：单元测试、集成测试、系统测试全都混在一起
- CI 管道退化成了一个"黑盒冒烟测试"

### 1.4 回归测试成本过高

修改回测参数逻辑，必须重新跑完整交易流水线才能验证效果。这意味着：

- **每次改动 = 全量回归**，无法做局部验证
- 这直接形成心理壁垒：我不敢放权让墨衡频繁改代码，因为每一次改动都是高风险、高成本的操作
- 没有快速回退边界，回滚需要重新跑整条链路验证

### 1.5 两个子系统之间没有明确的接口契约

当前架构中，回测和交易之间的数据传递是通过隐式依赖完成的，没有任何显式的接口定义：

- 回测生成的"信号"如何被交易系统使用？—— 没有契约文档
- 回测结果（评级、参数建议）如何传递给交易系统？—— 没有标准格式
- 两个系统之间是否存在隐式的共享状态（如全局变量、相同 DB 连接）？—— 无法审计
- 这导致我作为验收方**无法定义"什么算一次正确传递"**，因为没有"正确"的基准

### 1.6 系统之间的交互路径不透明

回测→交易的交互路径没有统一的 trace 机制：

- 一个 bug 从交易表现上发现，要回溯到回测阶段，中间经过至少 3 个模块
- 日志散落在不同文件里（回测日志、交易日志、飞书推送日志），缺乏关联 ID
- **跨系统问题的诊断完全依赖人工经验排查**，而不是自动化链路追踪

### 1.7 知识共享机制没有"过期"保障

当前架构中，知识库（如果存在的话）没有明确的过期机制：

- 半年前的策略评级今天是否仍适用？—— 没有校验
- 参数建议是否会过时？—— 没有时间戳约束
- 交易系统如何知道它使用的"知识"是否可靠？—— 无法判断
- **这意味着我无法对"知识使用是否正确"做任何测试断言**

## 2. 问题造成的后果

### 2.1 对测试覆盖的实际损害

| 问题 | 后果 | 量化指标 |
|------|------|---------|
| 测试边界模糊 | 单元测试与集成测试界线不清，高层级测试费力不讨好的情况下，单元测试也做不精 | 核心模块单测覆盖率 < 30%（估算） |
| 无法分层测试 | 每次测试都是"全链路验证"，无法做增量、分层验证 | 用例粒度不够细，一次完整测试耗时 > 15min |
| 无结构化输出 | 无法做自动化断言，验收等于人工审查 | 每次人工验收耗时 1-2h，且无法复现 |
| 缺接口契约 | 没有可测试的接口边界，所有测试都是"过程式"而非"契约式" | 接口回归测试数为 0 |

### 2.2 对质量保障的实际损害

- **漏测风险高**：耦合架构下，修改回测逻辑可能引入交易系统的 bug，反之亦然
- **没有"红/绿灯"**：我无法说"回测子系统本身是健康的"或"交易子系统本身是健康的"，因为两者根本分不开
- **知识可靠性无法验证**：交易系统在不知道知识时效性的情况下做决策，这是**质量盲区**
- **跨系统 bug 定位成本翻倍**：从客户反馈到根因确认，需要手动串联两端日志

### 2.3 对验收效率的实际损害

- **每次验收都是从头到尾的全量验证**，即使只是小的参数优化
- **验收结果不可自动化归档**：人工读报告 → 手写结论 → 没有数据库记录 → 验收记录无法追溯
- **验收标准不统一**：不同策略、不同时间段的"合格"标准没有统一量化指标，每次验收都要重新确定标准
- **无法做趋势分析**：没有结构化数据积累，无法对比"这个策略这个月比上个月退化了吗"

## 3. 三个最致命的问题

### 🔴 致命问题 #1：研究结果无法被自动化验收

**为什么致命：**
这是最根本的质量保障失灵。我作为测试验收方，**没有工具和方法来自动化判断一个策略是否合格**。当前验收流程是：
1. 墨衡提交策略报告（.md）
2. 我人工阅读 → 主观判断 → 输出结论
3. 结论无法自动化复现 → 下次验收又从零开始

这意味着：
- 验收效率完全受限于人工带宽（我一天能验收的策略数有限）
- 验收标准不统一（每个人的解读可能不同）
- 验收无法做回归：三个月后要看"同一个策略同一个参数的另一个回测结果"，只能重新读报告

**这是流程上的一级故障，不是代码层级的问题。**

### 🔴 致命问题 #2：回测与交易耦合 → 测试边界模糊

**为什么致命：**
这是架构层面的根本性缺陷。两个系统的耦合意味着：
- 回测引擎的错误定位器和交易执行错误定位器绑定在一起
- 没有"独立子系统健康证明"的概念
- 每次改动都是全量回归，不敢改 → 代码质量停滞 → 质量持续退化

更致命的是，这种耦合是**蔓延性**的：今天耦合的两个子系统，未来耦合的子系统只会越来越多。

**直接导致：测试无法做分层、无法做增量、无法做隔离。**

### 🔴 致命问题 #3：两个子系统之间没有明确的接口契约

**为什么致命：**
没有契约意味着：
- **"正确"无法定义**：我无法写一条断言说"回测结果传给交易系统时格式正确"，因为没有"正确"的参照系
- **接口变更不可控**：回测输出格式改了一行 → 交易系统可能隐式断裂 → 没人知道，直到生产环境出问题
- **降级行为不可测试**：当"回测→交易"这条通道中断时，系统行为无法断言

**这是验收的死角：没有契约的地方，就没有测试；没有测试的地方，就没有质量保障。**

## 4. 必须优先解决的 1-2 个问题

### 🥇 第一优先：研究结果无法被自动化验收

**为什么排第一：**
- 这个问题不依赖任何架构变更就能开始解决
- 只需要定义"回测结果的结构化输出格式"（schema），就可以从.md 输出改为带有结构化摘要的输出
- 这是一个**独立可做的改进**，不需要等系统拆分完成
- 解决后立刻能提升验收效率：从"人工读报告"变为"自动化断言校验"
- **这是我的核心工作流瓶颈，解决它就是解决了我的燃眉之急**

**建议解决方式（仅问题层面，不谈架构方案）：**
- 问题：当前输出只有自然语言报告，缺少结构化数据
- 需要：定义一套包含评级、绩效指标、参数、时间戳的标准输出 schema

### 🥈 第二优先：回测与交易耦合 → 测试边界模糊

**为什么排第二：**
- 这确实是更根本的架构问题，但解决成本更高（涉及系统拆分）
- 不过，这个问题**不解决，所有测试质量改进都有上限** — 只要两个系统绑在一起，测试永远做不精细
- 它的优先级仅次于"验收自动化"的原因，是两者的依赖关系：
  - 验收自动化可以在耦合架构下做（只改进输出格式）
  - 但"子系统的独立可测试性"必须在解耦后才能实现
  - 所以验收自动化可以先做，然后边进行系统拆分边解决耦合问题

**建议的逐步推进：**
1. **现在就能做**：定义结构化输出 schema（解决问题 #1 → 提升验收效率）
2. **解耦后做**：实现两个子系统的独立测试套件（解决问题 #2 → 提升测试覆盖）
