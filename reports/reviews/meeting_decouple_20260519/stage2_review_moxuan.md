## 1. 各方案测试验收评估

### 1.1 墨衡方案 — 可测试性评估

**评分：7/10**

#### 可测试性分析

| 子模块 | 可测试性 | 能否做自动化断言验收 | 难度评估 |
|--------|---------|-------------------|---------|
| common/signal_models | ✅ 高 | ✅ 可以。纯数据结构，字段类型、范围、必填项均可断言 | 低 |
| Signal 序列化/反序列化 | ✅ 高 | ✅ JSON/Parquet 读写可以断言序列化一致性 | 低 |
| 策略重构（Signal 产出） | ⚠️ 中 | ✅ 可断言每个策略产出 Signal 的字段完整性。但**需要先有一个基线回测结果作为对比** | 中 |
| 轻量 Simulator | ⚠️ 中 | ✅ 简单的盈亏计算可断言。但与 full Backtest 偏差需单独校准 | 中高 |
| SignalConsumer | ⚠️ 中 | ✅ 输入 Signal → 输出 OrderRequest，可断言映射正确性 | 中 |
| tech_signal_generator 合并 | 🔴 低 | ❓ 两套逻辑合并后，断言"计算结果完全一致"需要**逐子函数对比测试** | 高 |

#### 最难的验证模块

**1. 两套信号逻辑合并（Phase 1）** — 这是测试难度最高的模块。因为：
- 当前两套逻辑的代码路径不同，合并后断言"新旧一致"需要：
  - 对每个信号函数构造多组输入数据
  - 新旧两套代码分别跑结果
  - 断言结果匹配（含浮点精度容差）
  - 覆盖边界条件（空数据、异常值、极端行情）
- 若合并过程中有逻辑重写（不仅仅是代码合并），则失去直接对比的基线

**2. 轻量 Simulator vs Full Backtest Engine 偏差校准** — 墨衡自己也指出这是中高风险。我补充一个硬性要求：**偏差必须可量化，且每个策略有自己的偏差容忍阈值**。

#### 测试覆盖缺口

| 缺口 | 严重程度 | 说明 |
|------|---------|------|
| **knowledge.db 契约测试缺失** | 🔴 高 | 方案 12 天中未规划 knowledge.db 的接口测试，只能靠墨萱后补 |
| **回测引擎独立运行测试缺失** | 🟠 中 | Phase 2 新建 Simulator 后，旧的 BacktestEngine 是否还能独立运行未被验证 |
| **降级策略测试缺失** | 🟠 中 | "knowledge.db 不可用时交易系统仍能运行" 未被纳入测试 scope |
| **浮点精度断言标准未定义** | 🟡 低 | 绩效指标、夏普比率、PnL 的浮点对比需要统一精度规范 |

## 2. 方案之间的测试冲突与协调

### 🔴 冲突 1：测试节奏 — 契约先行 vs 后补测试

| 方案 | 倾向 | 对测试的影响 |
|:----|:-----|:-----------|
| 墨衡 | 先开发后测试 | 核心接口（Signal 格式、knowledge.db schema）如不先定，测试在 Phase 1 后期才能启动 |
| 墨萱（我） | 契约先行 | 要求 Phase 0 先定接口契约，Phase 1 开始测试与开发同步 |
| 墨涵 | 快速交付优先 | 隐含"先做出来再补测试"的态度 |

**我的立场不接受"先开发后测试"**。理由：

- Signal 格式和 knowledge.db schema 是被回测系统和交易系统**双方依赖**的接口
- 接口一旦定下来就拥有"API 惰性"——改接口成本很高
- 如果先开发再补测试，测试时发现接口设计有问题，**重构成本翻倍**
- 墨衡在我的 Stage 1 发言中也提到：核心接口需要 upfront 设计

**折中方案**：墨衡的 stage2 review 中提出的"Phase 0 共同定义接口契约"的方案我接受。核心接口（Signal 格式、knowledge.db schema）契约先行，非核心模块后补测试。

### 3.5 我的验收红线

以下是我作为测试验收方**不接受被突破**的底线：

#### 🔴 红线 1：Phase 0 结束时回测系统不能独立 import

```
python -c "from backtest_tools.engine import BacktestEngine; print('OK')"
```

如果 Phase 0 结束时这条命令失败，说明核心依赖未清理干净。验收结果为 **FAIL**。

#### 🔴 红线 2：Phase 2 结束时 knowledge.db 没有环境隔离

测试框架必须默认使用独立的 test_knowledge.db，不得与生产库共享。如果没有环境隔离机制，验收结果为 **FAIL**。

#### 🟠 红线 3：Phase 3 结束前双系统对账偏差超过 5%

旧系统与新系统的回测结果偏差率必须在 5% 以内（按 Sharpe Ratio 和总 PnL 两个指标）。超过则验收延后，直至修正或偏差原因被确认并接受。

#### 🟠 红线 4：两套信号逻辑合并后未通过一致性对比测试

合并后的信号生成逻辑，在 20 组不同参数 + 5 个不同标的上，新旧两套输出必须完全一致（浮点精度 ±1e-6）。任何不一致视为 **FAIL**，不进入 Phase 3。

#### 🟡 红线 5（预警而非硬性）：降级策略未定义

如果 Phase 3 结束时 knowledge.db 不可用时的降级行为没有定义和测试，验收意见中我会写入**"有条件通过"**，并注明降级策略缺失的安全隐患。
