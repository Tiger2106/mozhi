## 1. 算法逻辑正确性评估

### 1.1 diff 计算（`_calc_diff_pct()`）✅ 正确

公式 `|A-B| / max(|A|,|B|) * 100%` 合理。双零→0、一零一非零→100% 的边界处理已覆盖。

墨萱指出的死代码（第二个 `if denominator == 0` 不会被执行到）确认存在，**P3 优化项**，不影响逻辑正确性。

### 1.2 仲裁流程 ✅ 总体正确，详审见下

**主流程拓扑正确性：** Step 1→2→3→4→5→6 的仲裁树逻辑链完整：
- 单位检查通过 → 两源归一化比对
- 两源 ≤ 0.3% → 写入主表
- 两源 > 0.3% → 第三源仲裁
- 第三源无果 → 分钟聚合验证
- 全部失败 → REPORT

| 子步骤 | 判定 | 备注 |
|--------|------|------|
| Step 1 单位检查 | ✅ 合理 | 显式 unit 映射 + 量级推断双保险 |
| Step 2 两源比对 | ✅ 合理 | 阈值 0.3%（Owner 裁决）|
| Step 3 第三源仲裁 | ✅ 合理 | pairwise diff 找通过对 |
| Step 4 分钟聚合 | ⚠️ 见 §1.3 | **存在自举循环论证** |
| Step 5 审计日志 | ✅ 合理 | 字段完整度待提升（墨萱已提）|
| Step 6 违规不阻断 | ✅ 合理 | staging_raw + 主表 NULL 策略正确 |

### 1.3 ⚠️ 分钟聚合逻辑自洽性矛盾（玄知补充 P0/P1边缘）

**核心问题：分钟数据来源未定义，当前逻辑存在循环论证。**

算法描述 Step 4 的逻辑：
1. 从分钟数据聚合生成日线值
2. 计算该聚合值与源 A 日线的 diff
3. 如果 diff ≈ 0 → "分钟聚合与源 A 一致"

**但问题在于：**
- **如果分钟数据来自源 A** → 源 A 的分钟→源 A 的日线必然 ≈ 0%，这是**自洽性验证**（同一数据源的不同时间粒度的数据应该一致），不是**独立仲裁**。它不能证明源 A 的数据是"对的"，只能证明源 A 没发生分钟级错误。
- **如果分钟数据来自独立源** → 这实际上是"第四源"，应明确标注并在审计日志中记录来源。

**当前实现本质是自举**：用源A的分钟数据算出的日线值来"佐证"源A的日线值是对的 → 信息论上没有增加独立验证能力。

**要求**：
1. 明确分钟数据的来源策略
2. 如果来源≠三源中的任何一个 → 视为独立第四源，补充审计日志
3. 如果来源来自某源（最常见） → 不视为独立仲裁，降级为"同源自洽性检查"，不作为仲裁依据

**与墨萱的关系**：墨萱将这个问题归类为 P1（"分钟聚合来源未定义"）。我认为在实际实现中这是一个比 P1 更严重的问题——当前代码拿一个自洽性检查假装成独立的第三方验证是有误导性的。**建议提升至 P0/P1 边缘。**

## 2. 实现级技术风险

### 2.1 🔴 P0 — 单位映射方法A：归一化后数值仍可能漏检

**与墨萱完全一致**。以下案例演示漏检路径：

```
源A: {unit: 'hand', value: 10000}
源B: {unit: 'shares', value: 500000}

方法A: hand→shares 映射存在，ratio=100
  比值 = 10000 / 500000 = 0.02
  0.02 ≠ 1/100 → "数值不匹配换算" → return passed=True
  → 归一化后实际差异为 |10000*100 - 500000| / 500000 = 100%
  → 未检测到
```

**根本原因**：方法A在"有映射但比值不匹配"时直接放行，没有走归一化后的数值校验。

**修复方向**（与墨萱一致）：
```python
# 方法A 补充：有单位映射时，归一化后再做数值比对
if ratio is not None and ratio != 1.0:
    normalized_a = source_a.value  # hand 单位
    normalized_b = source_b.value * ratio  # shares 单位
    norm_diff = _calc_diff_pct(normalized_a, normalized_b)
    # 归一化后差异交给 Step 2/3，此处不阻断为单位错误
    return UnitCheckResult(passed=True, conversion_ratio=ratio)
```
**注意**：有单位映射时不应阻断为 UNIT_ERROR（单位已确认），归一化后的差异走 Step 2 正常流程。

### 2.2 🔴 P0 — trade_date/symbol 格式校验缺失

**与墨萱完全一致**。这不是数值交叉验证能发现的错位——两个源在同一天的不同日期字符串表示在数据库中不关联。身份错位的数据进入系统后，交叉验证比较的是不同日期的值。

**从实现角度看，这是 E-001 系统中最危险的盲区**之一：
- 日期格式未归一化 → `WHERE trade_date = '2026-05-20'` 会漏掉新浪的 `'20260520'`
- 代码格式未归一化 → `WHERE symbol = '600001'` 会漏掉 `'600001.SH'`

**要求**：`cross_source_validate()` 入口前增加 `_validate_identity_fields()` 函数。

### 2.3 ⚠️ P1 — `_normalize_to_standard_unit()` 依赖 `source.field_name` 但 `SourceValue` 无此字段

```python
def _normalize_to_standard_unit(source: SourceValue) -> float:
    ...
    standard_unit = conversion_map.get_standard_unit(source.field_name)
                                            # ⬆ SourceValue 没有 field_name 属性
```

对比 `SourceValue` 数据定义：
```python
@dataclass
class SourceValue:
    source_name: str
    value: float | None
    raw_value: Any
    unit: str
```

`field_name` 不存在。这是**运行时 AttributeError**，测试时必炸。

**修复方向**：两种方案任选其一：
- 方案A：`SourceValue` 增加 `field_name: str` 字段
- 方案B：`_normalize_to_standard_unit()` 增加参数 `field_name: str`

**建议**：方案A 更干净，因为 normalize 逻辑和 field_name 应该一起传递。

### 2.4 ⚠️ P1 — 分钟数据有序性约束缺失

`_aggregate_from_minute_data()` 中：
```python
minute_data[0]['value']      # open — 依赖按时间升序
minute_data[-1]['value']     # close — 依赖按时间升序
```

如果分钟数据无序，open 可能拿到盘中值、close 可能拿到盘前值。这是**使用正确数据得出错误结果**的典型实现级风险。

**要求**：在函数入口增加断言或排序：
```python
def _aggregate_from_minute_data(...):
    minute_data = sorted(minute_data, key=lambda x: x['timestamp'])
    ...
```

### 2.5 🟡 P2 — `_select_best_pair()` 实现过于简化

当前 `passing_pairs[0]` 的选取是任意的（依赖列表顺序）。如果三源中有两对通过，行为不可预期。

**建议**：按 `min(diff)` 排序后选最小差异对。

### 2.6 🟡 P2 — `_load_unit_conversion_map()` 每次调用加载 YAML 的性能隐患

当前写法是 `def _check_unit_mismatch()` → `_load_unit_conversion_map()` 每次调用都读 YAML。在 ETL 上下文中，每个字段的每次验证调用都读一次文件，对 volume/amount 等字段（数百万行）影响显著。

**建议**：改为模块级初始化加载 + 缓存。

### 2.7 🟢 P3 — 审计日志 `val_c` 命名不一致

`diff_ab/diff_ac/diff_bc` 使用下划线分隔，但 `val_c` 没有。建议统一为 `val_c` 或 `val_c` 保持现有命名。

### 2.8 🟢 P3 — `_calc_diff_pct()` 死代码（墨萱已提）

第二个 `if denominator == 0` 不会被执行到，同意移除。

### 2.9 🟢 P3 — 时区硬编码（墨萱已提）

同意改为系统时区或配置化。
