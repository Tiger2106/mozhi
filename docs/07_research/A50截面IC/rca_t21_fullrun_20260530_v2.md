## 补充1：关于"三问题独立、严禁混淆"的逻辑警示

**Claude原话**："两个独立问题被混在一起。工程问题（OOM/SIGKILL）和数据问题（黄金基线FAIL + 估值因子IC全零）是两个独立的问题，根因不同，修复路径也不同。必须分开处理。"

**v1的不足**：v1虽然将三个问题作为独立的章节展开，但问题2第三层第1点写道"SIGKILL（问题1）导致第2次运行只产出到 2026-05-19，缺少最新几个月的截面，但这不影响 20 年尺度的统计结论"——这句话隐含了一种因果关系假设（SIGKILL→数据覆盖不足→FAIL），但并未验证"数据覆盖不足→改变FAIL结论"是否成立。

**修补**：需要明确指出以下逻辑关系：

```
┌─────────────────────────────────────────────────────────┐
│ 严禁混淆的两种独立逻辑链路                                │
│                                                         │
│ 链路A（工程问题 → 验证工具链不可靠）：                     │
│   OOM/SIGKILL → 多次运行产生重复数据 + 缺失截面 →         │
│   黄金基线用到的a50_cross_ic_result数据是否干净存疑 →      │
│   ⚠️ 必须在确认数据完整性后，FAIL才能作为正式判定           │
│                                                         │
│ 链路B（因子本身特性 → 数据质量OK但指标不达标）：             │
│   假设数据干净 → momentum_20d的IC特征就是positive_ratio    │
│   ≈0.487/half_life=0.47周 → FAIL是因子本身的问题           │
│                                                         │
│ v1 默认走了链路B（假设数据没问题），但真正的路径可能走链路A    │
│ （数据有重复/缺失 → 指标失真 → FAIL不可信）。              │
│ 这是v1最严重的分析瑕疵。                                  │
└─────────────────────────────────────────────────────────┘
```

**修正结论**：在数据完整性（§补充2）得到确认之前，**不能**说"FAIL是验证脚本基于数据的正确判定"。v1问题2第一层的这个结论应当先挂起（pending）。

## 补充2：数据完整性深度分析（G2 — Claude最高优先级问题）

**Claude原话（18:44）**：
> "541个截面、9,711行数据的来源有问题：
> - 第2次运行产生222截面，第3次断点续跑产生428截面，两次独立运行的数据是否做了去重？
> - 如果没有，a50_cross_ic_result 里存在重复截面，IC统计结果是失真的。
> - --no-run 直接出报告，是基于哪些数据？是428截面还是541截面？
> - 541 = 222 + 428 - 重叠？还是有第三次完整运行的数据？"

### 2.1 数据溯源全景

```
运行记录：
  Run 1 (marine-s):   ??? 截面（被杀） → 数据可能部分写入
  Run 2 (mellow-o):   222 截面（被杀）→ 数据写入 DB
  Run 3 (断点续跑):    428 截面（完整完成）→ 数据写入 DB
  --no-run 生成报告:   最终产出 541 截面, 9,711 行

待解答的关键问题：
① 541 截面 = 222 + 428 - 重叠截面？还是另有完整第3次跑？
② 如果 run2(222) 和 run3(428) 的日期范围有重叠，这些重叠截面的 IC 值是重复写入的（后续没有去重）还是覆盖写入的（后续覆盖之前）？
③ run1 的数据是否也被写入且保留了？

验证方法：
  -- 检查 a50_cross_ic_result 表中 momentum_20d 的 trade_date 是否有重复
  -- 对每个 trade_date，COUNT(*) > 1 即存在重复
```

### 2.2 三种可能场景及影响

| 场景 | 描述 | 对黄金基线的影响 |
|:----:|:----|:--------------|
| **A** (最佳) | 541 = 第三次完整运行单独产出, run1/run2数据已清空或覆盖 | 数据完整，FAIL可采信 |
| **B** (中等) | 541 = 222 + 428 - 重复(≈11), 未去重; 但重复比例<2% | IC统计量受轻微影响，改变结论概率低 |
| **C** (最差) | 541 = 222 + 428, 且大量重叠截面重复或不完整 | IC均值/std明显失真，FAIL结论不可信 |

**关键阈值**：positive_ratio=0.487 距阈值 0.50 仅差 2.6%，重复数据偏差可翻转此结论。

- 修复优先级：低（不影响本次结论），但在重跑前需统一

## 追加修补G11：mellow-o"写到哪算哪"表述矛盾（Owner遗留矛盾，20:11）

**Owner原话**：mellow-o "写到哪算哪"的说法和实际覆盖2007-01-05~2026-05-19全量范围矛盾。需要澄清是刚写完最后一个截面还是写到接近末尾。

### 11.1 矛盾分析

| 证据 | 来源 | 支持"写到哪算哪" | 支持"完成全量" |
|:----|:----:|:--------------:|:-------------:|
| 覆盖范围2007-01-05~2026-05-19 | v5现象描述 | ✗ (不矛盾) | ✓ 覆盖约20年 |
| 产出222截面后被杀 | v5现象描述 | ✓ 仅222个窗口 | ✗ 与全量矛盾 |
| 被杀时在"验证阶段" | v5现象描述 | ✓ 未完成全量 | ✗ 仍在验证 |
| 428截面可生成(断点续跑后) | v5现象描述 | ✗ | ✓ 428≈2007~2026完整窗口数 |

### 11.2 可能解释

mellow-o 在被杀时已完成**IC计算**（所以数据写入了 DB 且日期范围覆盖全量），但被杀在**写库后验证阶段**——即222次IC计算已完成但仅222次写入DB的验证尚未完成。断点续跑检测到222截面已写入，从223继续算到428。

如果是这样，"写到哪算哪"的表述不精确——更准确的说法是：
> "mellow-o 完成了全量日期范围的计算，但在写库后验证阶段被杀。最终IC截面时间范围已覆盖2007-01-05~2026-05-19全量。被杀时实际仅完成222个截面入库验证，但后续断点续跑从第223截面继续补齐至428个。"

### 11.3 对RCA的影响

这一澄清对于判断问题1（SIGKILL）对问题2（黄金基线）的影响**至关重要**：
- 如果 mellow-o 确实已完成全量计算（只是死在验证阶段），则 a50_cross_ic_result 中已有全量数据 → 黄金基线数据基础没有因SIGKILL丢失
- 如果 mellow-o 仅算完222截面，则数据缺少时间序列后半段 → 基线数据基础有实质性缺口

**需要玄知确认此澄清。**
