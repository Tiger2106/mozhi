## 一、架构影响分析

### 1.1 单点故障评估

**集中化是当前阶段的正确选择，不构成重大风险。**

理由：
1. `mozhi_market_data.py` 当前是**零依赖的独立模块**（墨衡证实 `grep -r "market_data" src/` 无结果）。单点故障的前提是依赖节点集中，目前尚无外部依赖者。
2. 模块本身自带**多数据源降级链路**（东财→baostock/新浪），单点概率已被内部冗余抵消。
3. 未来若全系统都通过该模块获取行情，确会出现单点风险，但此时应通过**进程级隔离或独立部署**解决，而非放弃集中化。

**判断**：✅ 合理。集中化带来的维护收益 > 单点风险。

### 1.2 降级链路韧性

经过 Step1 实测数据和 Step2 审查，降级链路的实际韧性评估如下：

| 链路 | 韧性等级 | 原因 |
|:-----|:---------|:------|
| 日线：东财→baostock | 🟢 强 | baostock 实测可达且登录稳定（Step1 实测），降级后数据完整 |
| 分钟线：东财→新浪 | 🟢 强 | 新浪分钟线接口稳定，降级后数据完整 |
| 快照：新浪→东财 | 🟡 中 | **P1-02**：新浪实际是全量拉取后过滤（~75s），而非单只接口。东财才是高性能路径。当前降级优先级倒置 |
| 指数日线：东财→新浪 | 🟡 中 | **P2-01**：新浪降级仅返回当日快照，历史数据丢失 |
| 指数快照：东财 | 🟢 强 | 直连东财 `stock_zh_index_spot_em`，实测可达 |

**关键风险**：akshare 版本升级导致 API 名称变化（B2/B3 已在 Step1 暴露）。当前通过 `_try()` 重试机制可容忍偶发异常，但不能容忍 API 被移除/重命名。

**建议**：在 `diagnose()` 中增加 akshare 版本检测 + 接口可用性探测。

### 1.3 指数行情接口可用性

东财 `index_zh_a_hist()`（Step1 实测）：✅ 可用。
新浪降级 `stock_zh_index_spot_sina()`：⚠️ 仅返回当日单行快照，历史数据无法恢复。
指数快照 `stock_zh_index_spot_em()`：✅ 可用。

**问题**：指数日线的降级路径（新浪）在数据维度上不是"降级"而是"降质"——当日快照无法替代历史日线序列。如果东财指数接口长时间不可用，用户可能接收到残缺数据而不自知。

## 二、长期可维护性

### 2.1 Parquet 缓存策略

**现状**：`_ParquetCache` 基于 MD5 hash 的 key-based 缓存，TTL 过期后下次查询时惰性删除。

**膨胀风险评估**：
- 单只股票日线：约 50KB（按 600 交易日计算）
- 分钟线：约 5MB/只（按 1 年 5min 数据）
- 指数日线：约 50KB
- 若覆盖 100 只股票日线 + 50 只分钟线 + 10 个指数 ≈ **300MB**
- 默认目录 `data/market_cache/` 在项目工作目录下

膨胀速度可控，但缺少主动清理机制。若长期运行：
- 分钟线 TTL=3min，过期后每次惰性删除
- 文件碎片累积：Parquet 文件不可追加，每次 set() 写入新文件
- 永不访问的缓存文件（如历史回测使用后不再使用的标的）将永久残留

**建议**：增加 `max_cache_size_mb` 或最大文件数限制，达到阈值时 LRU 淘汰最旧文件。

### 2.2 NO_PROXY 域名演进

```
_NO_PROXY_DOMAINS = [
    "push2his.eastmoney.com",
    "datacenter.eastmoney.com",
    "*.eastmoney.com",
    "push2.eastmoney.com",
    "quotes.sina.cn",
    "finance.sina.com.cn",
    "hq.sinajs.cn",
]
```

**分析**：
1. `*.eastmoney.com` 通配符已覆盖绝大多数东财子域名，对版本演进的弹性很好。
2. `hq.sinajs.cn` 已包含，新浪行情全量覆盖。
3. 若 akshare 新增其他数据源（如网易财经、腾讯财经），需要手动追加域名。
4. 但此列表是**静态硬编码**的，无法随 akshare 版本动态更新。

**风险等级**：🟢 低。NO_PROXY 设置在模块加载时执行一次，后续假设新增数据源也只会增加域名而非需要移除。新增域名的人工维护成本极低（几行文本）。

### 2.3 接口抽象层可替换性

评估 `mozhi_market_data.py` 的抽象层是否能容忍 akshare → 其他数据源替换：

| 抽象维度 | 现状 | 可替换性 |
|:---------|:-----|:---------|
| 返回类型 | `pd.DataFrame` | ✅ 通用格式，任何数据源都可输出 |
| 降级策略 | 硬编码在 try/except 中 | 🟡 若更换数据源需重写降级链路 |
| 数据源定义 | 通过 akshare API 直接调用 | 🟠 强耦合：`self._ak.stock_zh_a_hist()` 等是 akshare 特有 API |
| 缓存层 | 独立于数据源 | ✅ 可复用 |

**判断**：模块对 akshare 是**强依赖**但**弱耦合**。替换 akshare 需要：
- 重写 `_daily_eastmoney`、`_minute_eastmoney`、`_snapshot_single_sina` 等方法
- 但缓存层、限速、重试、数据模型均无需变动
- 代码量约 150-200 行需要改写（占总量 ~25%）

**建议**：如果未来有计划替换数据源，建议在 `MarketDataClient` 和具体数据源方法之间加一层 `DataProvider` 抽象接口。当前引入成本约 30 行代码，但可节省未来 200 行重构。

## 四、与现有系统的关系

### 4.1 与 `data_source.py` 的共存策略

当前现状：

| 维度 | `data_source.py`（现有） | `mozhi_market_data.py`（新增） |
|:-----|:------------------------|:------------------------------|
| 定位 | 回测引擎专用数据源 | 统一行情库（通用） |
| 数据源 | 东财 → 新浪日线 | 东财 → baostock/新浪（多维度） |
| 缓存 | CSV+Parquet 混合，TTL=24h | Parquet 专用，TTL 可配置 |
| 返回格式 | 列名统一英文，index 无标准 | index=date(str)，6 列标准 OHLCV |
| 降级 | 东财→新浪日线（新浪降级） | 每条链路独立降级 |
| 并发安全 | 无特殊处理 | 同（无锁） |

**问题**：两者功能显著重叠 — 都提供日线获取、都带缓存、都从 akshare 获取数据。共存策略缺失：

1. **迁移路径不清**：何时用旧？何时用新？过渡期多长？
2. **维护负担翻倍**：两者各自维护缓存逻辑、降级逻辑、列名映射
3. **行为不一致**：返回的 DataFrame index 格式不同（`data_source` 是列存 `date`，`market_data` 是 `date` index），调用方切换时需适配

**建议**：制定迁移路线图：

```mermaid
Phase 1（当前）: 双轨并行
  - data_source.py 照常工作（不影响现有回测）
  - mozhi_market_data.py 投入新功能使用
  - 新代码（新回测、新策略）优先用 market_data

Phase 2（1-2 周后）: data_source.py 代理
  - data_source.py 内部委托给 MarketDataClient
  - 保留 data_source 接口签名兼容性
  - 逐步重写调用方（benchmark_data_source.py 等）

Phase 3（完成）: data_source.py 废弃
  - 确认无调用方
  - 添加 deprecation warning
  - 移除旧文件
```

### 4.2 对知识库 market_context 回填的影响

**不会显著简化回填逻辑**，但可提升数据质量：

- market_context 回填的核心复杂度在于**数据逻辑**（多周期合并、收益率计算、标的筛选），而非**数据获取**。
- `mozhi_market_data.py` 简化了数据获取层（稳定接口 + 自动降级），`get_daily_batch` 可直接用于批量回填。
- 缓存层对回填有利：日线 TTL=60min，回填任务在 1 小时内重复执行时无需重复请求。

**判断**：统一行情库是回填的必要基础设施，但不是充分条件。回填逻辑本身仍需单独实现。
