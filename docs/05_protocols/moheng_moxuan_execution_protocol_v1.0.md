# 墨衡-墨萱 任务执行协作协议 v1.0

> **背景**: 2026-06-01 全天流程试验发现任务拆分是瓶颈。Owner建议制定正式的执行协议。
> **定位**: 墨衡 = 主设计师/技术总负责。墨萱 = 质量师。
> **范围**: 仅覆盖墨衡↔墨萱之间的执行协作。调度方（墨涵）、技术把关（玄知）、Owner签署不在本协议范围。

---

## 1. 职责边界

| 阶段 | 墨衡（主设计师/技术总负责） | 墨萱（质量师） |
|:----|:-------------------------|:--------------|
| **P0 拆解方案起草** | 主责 — 自行拆解，粒度自控 | 不动 |
| **P0.5 拆解方案复核** | 被退回时修复 | **主责** — 审粒度/依赖/验收标准 |
| **Stage 1 编码** | 主责 | 不动 |
| **Stage 1.5 自检** | 主责（TMPL-006） | 不动 |
| **Stage 2 代码审查** | 被退回时修复 | **主责** |
| **再次被退回** | ≤2次修复，第3次告警Owner | — |

### 1.1 核心原则

- **墨衡设计，墨萱验证**。墨萱发现问题只说"这里不通过 + 原因"，不说"应该改成这样"
- **谁的问题谁修**。P0拆解被打回 → 墨衡修。Stage 2代码被打回 → 墨衡修。墨涵不代修
- **质量门不可绕过**。P0.5未通过不得进入编码。Stage 2未通过不得进入下一子任务

## 2. 流程总览

```
P0: 墨衡拆解方案起草
  │
  ├── self-review（墨衡自检）
  │
  ▼
P0.5: 墨萱复核拆解方案（≤10min）
  │
  ├── REJECT ──→ 退回墨衡修复 → 重回P0.5
  │
  └── PASS ────→ 锁定方案
                    │
                    ▼
              小循环（每子任务）：
              ┌─────────────────────┐
              │ Stage 1: 墨衡编码    │
              │ Stage 1.5: 墨衡自检  │
              │ Stage 2: 墨萱审查    │
              │   ├── FAIL → 墨衡修复│
              │   └── PASS → 下一子  │
              └─────────────────────┘
                    │
                    ▼
              全部通过 → Stage 3(玄知技术把关)
```

## 3. P0 拆解方案

### 3.1 墨衡起草规范

墨衡在收到任务后自行拆解，拆解方案包含：

```
task_id:          任务标识
meta:             总任务描述 + 上下文文件路径
task_level:       S(<30min) / M(30-90min) / L(90-240min) / XL(>240min)
sub_tasks:
  - id:          子任务唯一标识
    description: 单一逻辑操作
    estimated_min: ≤15min（编码子任务）
    affected_files: ≤3个文件（同模块优先）
    acceptance_criteria: 可观测标准（"输入X返回Y"，非"完成功能"）
    dependencies: [] 依赖的子任务ID列表
```

### 3.2 粒度规范

| 子任务类型 | 建议粒度 | 说明 |
|:---------|:--------|:-----|
| 编码 | 10~20min | 单一功能，≤3个文件 |
| 测试 | 10~20min | 单一测试模块 |
| 修复 | ≤10min | 按问题逐条修复 |
| 文档 | ≤10min | 单篇文档 |

**标注规则**：
- `estimated_min > 15` = 🟡 警戒，墨萱复核时会重点检查
- `estimated_min > 30` = 🔴 必须拆分
- 如果拆不出合理的子任务 → 说明任务理解未到位，暂停，重新理解任务

### 3.3 编码流程拆分示例（S/M/L/XL 分级）

以下是编码阶段（Stage 1）的标准拆分方案。墨衡在起草 P0 拆解方案时以此为基准参考。

| 子任务 | 内容 | 预估耗时 | 产物 | 前置依赖 |
|:------:|:-----|:--------:|:----:|:--------:|
| C1 编码实现 | 按需求编写核心代码逻辑 | ≤15min | 代码变更 | 无 |
| C2 黄金基线 | 跑黄金基线，生成快照 | ≤5min | golden_baseline_snapshot.json | C1 |
| C3 回归验证 | 与现有基线对比确认 IC/NAV 偏差 | ≤5min | 回归结果 | C2 |
| C4 自检 Q1+Q3 | 公式验证(6点边界) + 边界场景测试 | ≤5min | self_check_pass.json | C1 |
| C5 自检 Q2+Q6 | 代码影响范围分析 + 三角度一致性 | ≤3min | self_check_pass.json | C1 |
| C6 自检 Q4+归档 | 回归完整性记录 + .done 归档 | ≤2min | self_check_pass.json + .done | C3 |
| C7 信号写入 | 全部产物 read 验证 + 信号文件 | ≤2min | 信号文件 | C4+C5+C6 |

> **合计**: ~37min（含串行依赖等待）
> **C1→C2→C3 为串行核心链**，C4/C5 可在 C1 后与 C2 并行。

| 编码规模 | 标签 | 判断人 | 拆分要求 |
|:--------|:----:|:------|:--------|
| 单函数/公式变更（≤20行） | 🟢 S | 墨涵初见+墨衡校正 | C1+C4+C5 = 1个单元任务 |
| 模块级变更（50-100行） | 🟡 M | 墨涵初见+墨衡校正 | 全量拆分，C1→C2→C3 串联 |
| 多模块联调（200-300行） | 🔴 L | 墨涵初见→墨衡校正 | 拆4~5子任务；每子任务标注预估编码时间 |
| 跨系统改造（>300行） | ⚫ XL | 墨涵初见→墨衡校正→**Owner终审** | 拆≥6子任务+依赖关系；占用Owner时间需Owner先确认 |

**L 级触发条件**（满足任一即标L）：
1. 产出 ≥4 个独立产物
2. 涉及 ≥2 个不同领域/标的
3. 墨涵预估执行耗时 ≥10min
4. 墨衡自行评估认为应标L

**XL 级触发条件**（满足任一即标XL）：
1. 产出 ≥6 个独立产物
2. ≥3 个领域或跨流程
3. 总耗时可能 >15min
4. 存在高风险依赖
5. 墨衡主动申请升级

**XL 容量请求模板（墨衡填写→墨涵转发→Owner回复）**：

```
任务名称：[task_id]
总耗时预估：[X] 分钟（编码+测试+审查+复核）
子任务数：[N] 个
内存需求：[X] GB
主要风险：[列表]
是否阻塞其他工作：是/否（若阻塞，列出哪些）
能否缩减为更小版本：是/否（如果能，S级版本做什么）

Owner 签署：
[ ] 放行
[ ] 要求拆分更细
[ ] 暂缓
[ ] 当前系统容量不够（Owner告知可用容量，墨衡调整后重提）
```

**说明**：
- 墨衡在启动XL任务前**自行检查系统可用内存**（`Get-CimInstance Win32_OperatingSystem`），若`内存需求 > 系统当前可用`，向Owner报告："需要[X]GB，当前只剩[Y]GB"
- Owner杀掉无用进程释放内存后 → 墨衡继续
- 墨衡也能主动发现容量不足，不需要Owner提前查系统状态

## 4. P0.5 拆解方案复核（墨萱）

### 4.1 复核范围

墨萱用以下7项检查清单逐子任务复核：

| 编号 | 检查项 | 通过条件 |
|:---:|:------|:---------|
| Q1 | 完整性 | 所有子任务加总覆盖任务描述的全部维度 |
| Q2 | 正交性 | 子任务之间不重叠、不互相包含 |
| Q3 | 粒度 | 编码子任务 ≤15min（≥3个文件是红旗），测试 ≤20min |
| Q4 | 依赖 | dependencies DAG 无环、方向正确 |
| Q5 | 验收标准可测 | 标准是"输入X返回Y"而非"完成功能" |
| Q6 | 文件范围 | affected_files 存在且路径正确 |
| Q7 | 可测性 | 子任务完成后有明确的验证手段 |

### 4.2 快速评估流程（≤10min）

```
1. 总览（2min）: meta信息 + 总估算时间 + 子任务数量
   ↓
2. 逐项7项检查（5min）
   ↓
3. 依赖性验证（2min）: DAG闭环检查
   ↓
4. 结论（1min）: PASS / REJECT（附问题标注）
```

### 4.3 结论与反馈

- **PASS** → 锁定方案，进入编码
- **CONDITIONAL_PASS** → 小问题标注即可，由墨涵判断是否需进小循环
- **REJECT** → 退回墨衡修复。必须标注：
  - 是 Q1~Q7 哪一项不通过
  - 具体问题位置
  - 不产生"全部不行"的模糊结论

### 4.4 粒度过大的识别信号

| 信号 | 判定 |
|:----|:-----|
| `estimated_min > 15`（编码）或 `> 30`（测试） | ❌ 过大 |
| `affected_files` 跨多个模块 | ❌ 可能过大 |
| `description` 含"且/并/同时"等并列词 | ❌ 建议拆分 |
| `acceptance_criteria` 含多个独立可验证项 | ❌ 每项应归属独立子任务 |

## 5. 小循环（Stage 1 → 1.5 → 2）

每个 sub_task 独立走小循环，不跨子任务累积。

### 5.1 墨衡承诺

1. **编码粒度 ≤15min/子任务**，超出自动报告
2. **自检不走形式**：TMPL-006 逐项确认后再提交墨萱
3. **修复不修出新问题**：回归验证后重新提交
4. **修复标注**：重交时标注"第 N 次修复"（N≤2）

### 5.2 墨萱承诺

1. **10min 内给出审查结论**（复杂代码延至 15min，超时告警）
2. **审查结论存档**：每轮审查写入 `signals/reviews/{task_id}_subtask_{id}_review_{N}.json`
3. **只打回不代修**：说明原因和位置，不写具体代码
4. **连续 2 次 FAIL → 告警**：同一子任务第 3 次退回 → 通知 Owner

### 5.3 审查输出模板

墨萱审查分两步：**预检 → 三维度审查**。

#### 预检：文件路径合规

审查前请先执行路径合规检查，验证所有新产出文件位于正确目录：

```bash
# 列出本批产出文件路径
# 使用 path_checker 验证每一条
python -m src.compliance.path_checker --paths <产出文件路径列表>
```

**验证规则**：

| 检查项 | 合规要求 |
|:-------|:---------|
| 代码文件 | 必须位于 `mozhi_platform/src/` 下 |
| 报告/文档 | 必须位于 `mozhi_platform/docs/` 下 |
| 配置文件 | 必须位于 `mozhi_platform/` 下 |
| 信号文件 | 例外白名单：`mo_zhi_sharereports/signals/`、`.done`、`.failed` |
| 临时/工具脚本 | 必须位于 `mozhi_platform/scripts/` 下 |

**路径不合规 → 直接打回**，不进入三维度审查。

#### 三维度审查

路径合规通过后，墨萱审查锚定为三个维度：

| 维度 | 检查内容 |
|:----|:---------|
| 阻断问题数 | 影响功能正确性的问题 |
| 测试覆盖 | 是否覆盖正常 + 异常路径 |
| 静默失败路径 | 数据缺失/边界条件是否被优雅处理 |

三维度全 PASS 才通过。

### 5.4 预置拒因列表（减少重复书写）

90% 的拒绝理由在以下列表中：

| 拒因 | 处理方式 |
|:----|:---------|
| 验收标准模糊 | 重写为准入值/预期值 |
| 依赖有环 | 检查 direction |
| estimate 超标 | 重估或拆分 |
| 测试未覆盖异常路径 | 补充异常用例 |
| 代码超出范围 | 说明修改范围 |
| 文件路径违规 | 产出文件未按协议放置正确目录 |

## 6. 信号文件规范

所有阶段成果通过 `.done` 信号文件传递。

### 6.1 信号链

```
P0.5 复核:
  墨萱 PASS → signals/tasks/{task_id}_p0.5_pass.done
  墨萱 REJECT → signals/tasks/{task_id}_p0.5_reject.done + 问题描述

小循环:
  墨衡 Stage 1 → signals/tasks/{task_id}_sub_{sub_id}_s1.done
  墨衡 Stage 1.5 → signals/tasks/{task_id}_sub_{sub_id}_s1.5.done
  墨萱 Stage 2 PASS → signals/tasks/{task_id}_sub_{sub_id}_s2_pass.done
  墨萱 Stage 2 FAIL → signals/tasks/{task_id}_sub_{sub_id}_s2_fail.done + 问题描述
```

### 6.2 状态文件格式

```json
{
  "stage": "p0.5",
  "agent": "moxuan",
  "conclusion": "PASS",
  "timestamp": "2026-06-01T11:46:00+08:00",
  "issues": [],
  "reason": "方案合理，粒度合规，依赖无环"
}
```

### 6.3 小循环免审规则

极简单的子任务（如修文档拼写、只改注释），墨涵可裁量跳过墨萱审查。但必须有记录备查。

## 7. 退出与升级机制

### 7.1 退修规则

| 退修次数 | 行为 |
|:-------:|:-----|
| 第 1 次 | 墨萱说明问题 → 墨衡修复 |
| 第 2 次 | 加注具体修改要求 |
| 第 3 次 | **告警 Owner 裁决** |

### 7.2 墨萱主动升级

| 场景 | 升级路径 |
|:----|:---------|
| 同一子任务第 3 次不合格 | → Owner |
| 墨衡连续 3 个子任务均被退回 | → Owner |
| 墨衡超时无响应 | → 墨涵 → Owner |
| 发现修改超出范围 | → Owner |

### 7.3 墨衡升级路径

| 场景 | 升级路径 |
|:----|:---------|
| 墨萱审查结论不明确 | → 墨涵仲裁 |
| 墨萱审查标准不一致 | → 墨涵仲裁（查历史记录） |
| 墨萱超时未给结论 | → 墨涵启动熔断 |

### 7.4 拆解复核例外门

墨衡连续 2 次 P0.5 被墨萱 REJECT（因方案维度遗漏）：
→ 自动转入"拆解复核模式"，墨衡拆完即通知墨萱审，不等墨衡自检。

## 8. 故障归因与流程修订

### 8.1 归因清单

每次打回记录根因：

| 根因分类 | 示例 |
|:--------|:-----|
| 数据问题 | 数据来源有误/字段不存在 |
| 逻辑问题 | 推导错误/算法不适用 |
| 格式问题 | 文件格式/字段名不符合规范 |
| 超时问题 | 粒度超标导致 session 中断 |

### 8.2 流程修订触发

同类根因累计 ≥3 次 → 自动触发流程修订会议。
