# 编码原则讨论会 — 玄知发言稿（架构+数据角度）

---

各位好，我是玄知。以下从**架构把关**和**数据一致性**两个维度，对墨衡的13条编码原则发表意见。

---

## 一、架构完备性：总体到位，但有1个结构性缺口

13条原则覆盖面很广，三层分离（P-001）、三模块正交（P-005）、时间对齐契约（P-012/P-013）这三条是架构层面的核心骨架，我认可。

**但有一个重大缺失：没有原则覆盖"性能与精度的可配置化"。**

当前所有字段（close/open/high/low/adj_factor/pre_close/volume/price_limit_mask/suspend_mask）在 BacktestData 中是**固定宽度**的(n_days, n_stocks)。当标的量从10扩展到100甚至1000时：
- **内存消耗**：1000只 × 2500天 × 10个字段 × 8字节 ≈ 200MB，还在可接受范围
- **但 EXP-004 会引入因子数据**（如流通市值、PB、PE、动量因子、反转因子等），不是全量标的都需要全量因子

**我的建议**：补充一条P1或P2原则——**数据合约应支持可选的"附加字段通道"**（optional fields slot），让 EXP-004 可以注入因子矩阵而不破坏 BacktestData 的不可变性。当前 P-007 的 dataclass 是硬编码的，添加因子需要改合约定义本身。

## 二、约束优先级：停牌>涨跌停>T+1 架构合理性分析

从架构层看，这个排序是正确的。理由：

1. **停牌是绝对阻断**：标的在停牌日根本无法交易，无论涨跌停还是T+1，其前置条件已不成立。把停牌放在最前面是唯一合理的剪枝策略。

2. **涨跌停是价格条件**：在标的可交易的前提下（不停牌），检查当前价格是否超过限制。逻辑上位于第二层。

3. **T+1是时间条件**：依赖持仓时间戳和当前日期对比，不受当天价格影响。放在第三层是正确的，因为T+1限制的是历史买入量，与当天是否涨跌停独立。

**但有1个边界情况需要澄清**：涨停时虽然禁止买入、允许卖出（按原理），但如果该标的今日T+1锁仓（昨日买入），那么"涨停允许卖出"这个动作能否执行？按当前顺序：

```
停牌(通过) → 涨跌停(涨停→可卖不可买) → T+1(当日买入量不可卖)
```

逻辑上：T+1约束在涨跌停之后执行，所以"涨停状态下可以卖出"这个结论会进一步被T+1过滤。如果该标的今日确实有T+1锁仓量，卖出会被拦截。**这在实际执行中是对的**——但建议在文档或注释中显式注明这个case，避免实现时写错顺序。

**复牌日特殊规则设计合理**：跳过涨跌停检查保留T+1，逻辑自洽。

## 三、可扩展性：是否能支撑EXP-004？

### 能支撑的部分
- **三层分离 + 模块正交**：EXP-004只需要替换 `calc/` 模块的信号计算方法，或新增 `calc/` 下的函数
- **策略接口标准化（P-006）**：固定签名意味着EXP-004只需实现一个 `StrategyFn` 即可接入
- **BacktestData 数据合约（P-007）**：统一的数据接口，EXP-004不需要理解内部数据结构

### 需要关注的风险

**风险1：因子矩阵的集成路径未定义**

EXP-004的信号可能来自外部因子（如聚宽因子、自研因子），不是一个 `calc/` 函数能搞定的。当前架构假设"信号都在 `calc/` 里计算"，但EXP-004可能是在外部预计算好因子再注入。建议在 `runner.py` 层面预留一个"外部信号注入接口"，让EXP-004可以跳过高层 `calc/` 直接传入 `signal_matrix`。

**风险2：多标的扩展时的内存布局**

BacktestData 的数组布局统一为 `(n_days, n_stocks)`——这对10标的没问题。1000只时，`price_limit_mask` 的 shape 变成了 `(1000, 2500, 2)`，约40MB。如果再叠加12个因子字段，接近500MB。虽然不至于爆内存，但**建议在P-003中加上一条子规则**：当 `n_stocks > 200` 时，自动切换到 `float32` 并记录精度损失。

## 四、数据一致性：BacktestData 合约审计

结合我之前审计的 DB 字段，BacktestData 的字段清单基本完整，但有以下注意点：

### ✅ 已覆盖的关键字段

| 回测需求 | BacktestData字段 | DB源字段 | 状态 |
|:---------|:-----------------|:---------|:----:|
| 复权价格 | `close`, `adj_factor` | stock_daily.close, adj_factor | ✅ |
| 涨跌停判断 | `pre_close`, `price_limit_mask`（可预计算） | stock_daily.pre_close, stock_basic.price_limit | ✅ |
| 停牌判断 | `suspend_mask`（需预计算） | 需通过 `pre_close==open==close && volume==0` 推断 | ⚠️ |
| T+1 | 仅逻辑层，不需要数据 | N/A | ✅ |
| 成交量 | `volume` | stock_daily.volume | ✅ |

### ⚠️ 数据一致性问题：adj_factor 时间对齐

我之前审计注意到的关键问题：DB 中 `adj_factor` 是**后复权因子序列**，回测引擎用 `close / adj_factor` 做前复权。这意味着：

1. **历史复权价格会随最新分红事件而变动**——每次数据更新后，回测历史净值曲线会变
2. **同一份回测代码，跑在不同时间点数据上，产生不同结果**
3. 这直接违反 P-009（确定性执行）、P-011（黄金文件基线）的精神——黄金文件和不同时间的数据快照不匹配时会产生假报警

**我的建议**：在数据层（P-003/P-007）的实施中，补充一条：**每次回测运行时，在 `BacktestData` 中 `freeze` 一个 `as_of_date` 元数据字段**，标注数据截取的日期时间。黄金文件比对时，必须先校验 `as_of_date` 一致，再校验数值一致。否则"因为数据源更新导致的曲线变化"和"代码修改导致的曲线变化"无法区分，回归对比（P-008）会充满假阳性。

### ⚠️ suspend_mask 的数据来源不明确

当前设计 `suspend_mask` 由数据层预计算，但 DB 中没有直接的"停牌标记"字段。需要根据 `pre_close == open == close && volume == 0` 推断。建议：
- 在数据层（P-003实施）中将此逻辑标准化为 `compute_suspend_mask()` 函数
- 函数文档注释写明判断逻辑和边界条件
- 确认 DB 中停牌日 `volume` 字段是否为 0 还是 NULL，做好容错

## 五、架构把关人最担心的三点

按降低到高的顺序：

### 🟡 担忧3（中等）：约束引入回归的判定标准模糊

P-008 说"约束引入回归：新夏普 ≤ 旧夏普 + 0.05"。但约束对回测结果的影响是不对称的——有时约束后夏普反而更高（约束排除了虚假交易，资金曲线变平滑）。在这种情况下，判定是"通过"还是"告警但允许"？建议在实施时补充具体判定流程。

### 🟠 担忧2（较高）：前视偏差的回溯范围缺失

所有13条原则和审计报告聚焦于**时间轴上的前视偏差**（用当日数据预测当日）。但还有**信息集合的前视偏差**——比如用全市场数据计算信号时，若一个因子需要3000只股票的全截面数据，在当前标的仅10只的情况下，计算是否正确？扩展到EXP-004时，因子矩阵的计算范围与回测标的范围不匹配，也会引入隐性的前视/后视问题。

P-012 和 P-013 只覆盖了时间轴，没有覆盖截面因子计算的前视风险。**建议在 P-012 中补充一条"信息边界原则"**：信号计算时访问的数据矩阵的截面范围（列维度）不应超过回测标的集合。

### 🔴 担忧1（最高）：数据版本与回测结果的耦合

见第四节的讨论。当前架构中，回测结果对底层数据版本**没有显式依赖**。10只标的时，数据量小，通常从本地csv一次性读取，问题不大。但到EXP-004阶段，数据来源变为数据库/API/量化平台，每次拉取的数据版本可能不同。

**最糟糕的情景**：墨萱在做回归对比，墨衡在并行修改代码，二人拉的数据库数据版本差了一周，回归对比到处不一致，花半天排查发现是数据问题。这是纯粹的效率损失。

**最低成本的解决方案**：在 `BacktestData` 中增加一个 `data_fingerprint` 字段（SHA256: `sorted(symbols) + as_of_date + trading_dates[0] + trading_dates[-1]`），每次回测运行时计算并输出到日志和黄金文件。这样回溯时可以精确知道"当时跑的是哪个版本的数据"。

## 六、总结

| 维度 | 结论 | 建议 |
|:-----|:-----|:-----|
| 架构完备性 | ✅ 总体完整，1个结构性缺口 | 补充"性能与精度可配置化"原则（P1/P2） |
| 约束优先级 | ✅ 合理，1个边界需注明 | T+1和涨停的交互逻辑文档化 |
| 可扩展性 | ✅ 核心框架足以支撑 | 解决因子注入接口 + 大标的量下数据精度 |
| 数据一致性 | ⚠️ 字段完备，1个严重风险 | 增加 `as_of_date` 和 `data_fingerprint` |
| 最大担忧 | 🔴 数据版本变化影响确定性 | 最小成本方案：`BacktestData` 含数据指纹 |

**总体定性：原则体系设计质量优秀，架构骨架扎实。** 上述建议均为锦上添花的补充，不阻塞当前P0/P1修复的推进。我建议在实施顺序表（§8.1）的"黄金文件基线"步骤之后，增加一步"数据版本指纹集成"，确保确定性原则在数据层面有保障。
