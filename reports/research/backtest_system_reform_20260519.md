## 一、提案可行性评估

对提案中四项核心建议逐条评估：支持/反对（附理由）、本系统已实现程度、实施建议及优先级。

### 1.1 三层验证结构：存在性测试 → 稳健性测试 → 市场状态适配

**判定**: ✅ **支持**

**理由**:
- 该结构遵循了量化研究的行业最佳实践：先确认"策略是否存在基础alpha"，再检验"参数的稳定性"，最后验证"策略在不同市场状态下的适应性"
- 逻辑递进关系清晰：上一层不通过则下层无需执行，节省计算资源
- 与当前系统正处于"数据挖掘→研究有效性验证"的转型期高度契合

**本系统已实现程度**: **30%**

| 层级 | 当前状态 | 已实现内容 |
|:----:|:--------:|:-----------|
| 存在性测试 | 🔴 未实现 | 从未以"固定粗参数"+跨周期视角执行；所有分析都基于参数扫描后的最优解 |
| 稳健性测试 | 🟡 部分实现 | P3报告实现了参数热力图和Robustness Score（基于CV），但以单个84天窗口为基准，未做分段验证 |
| 市场状态适配 | 🔴 未实现 | 参数衰减分析虽然识别了问题，但未以验证结构的形式固化到回测管线中 |

**具体差距**:
- 存在性测试方面混淆了"验证策略"和"参数优化"——当前流程直接进入参数搜索，不先验证基础alpha
- 稳健性测试局限在单一时间窗口，未覆盖跨窗口稳定性
- 市场状态适配完全依赖人工解读，未自动化

**实施建议**:
1. 新建 `ExistenceTestValidator` 模块：固定粗参数（如n_levels=8, width_multiplier=1.5, lookback=20）直接跑3年/5年，输出基本绩效
2. 在回测管线中增加前置验证门（Pre-validation Gate）：存在性测试通过后才允许进入参数搜索阶段
3. 将P3的稳定性分析扩展为跨区间验证（3段独立周期），而非单窗口CV
4. 市场状态适配作为P3的补充维度，新增跨市场状态切片分析

**优先级**: P0 🔴（最高优先级，这是整个可信度体系的基础架构）

### 1.2 研究可信度评分体系（6维指标）

**判定**: ✅ **支持**

**理由**:
- 6维指标覆盖了过拟合检测的主要维度（交易量、参数、市场状态、样本外、时间、容量）
- 当前param_decay_analysis中的4维R评分（窗口长度、交易笔数、夏普衰减、参数稳定性）是最简版本，提案的6维更完整
- 评分标准化后，不同策略、不同标的之间可横向比较

**本系统已实现程度**: **25%**

| 维度 | 提案指标 | 当前状态 |
|:----:|:---------|:---------|
| Trade Count Score | 交易数量是否足够（≥30笔检验） | 🟡 已有基本计算（R_trades = min(n/30, 1.0)），但未标准化为独立评分单元 |
| Parameter Stability Score | 参数邻域稳定性 | 🟡 P3有热力图和CV计算，但邻域稳定性未量化为复合评分 |
| Regime Consistency | 跨市场状态一致性 | 🔴 完全未实现 |
| OOS Survival | 样本外存活率 | 🟡 P4 Walkforward已实现，输出IS/OOS对比表，但未纳入统一评分 |
| Temporal Stability | 时间稳定性 | 🔴 未实现 |
| Capacity Stability | 资金容量稳定性 | 🔴 未实现 |

**具体差距**:
- 已有评分框架仅4维（窗口、交易数、夏普衰减、稳定性），缺少Regime Consistency和Capacity Stability
- 各维度评分未标准化到同一尺度（当前R_trades使用max()归一化，R_window使用线性分段，不一致）
- 评分系统的数据和结果未存入数据库，无法跨报告回溯查询
- 横截面比较功能（不同策略/标的的评分对比）完全缺失

**实施建议**:
1. 新建 `ResearchConfidenceScorer` 模块，从已有P系列报告中抽取数据计算6维评分
2. 评分采用统一归一化机制（所有维度 [0,1]，使用行业标准阈值）
3. 新增维度：Regime Consistency（跨市场状态切片夏普相关性）、Capacity Stability（资金规模灵敏度测试）
4. 评分结果写入 `backtest_results` 数据库的 `reliability_scores` 表
5. 评分框架输出为API接口，供报告生成管线直接调用

**优先级**: P0 🔴（与三层验证结构互补，构成可信度体系的核心）

### 1.3 Research Confidence Rating（A~F六级评级）

**判定**: ✅ **支持**

**理由**:
- 将复合评分[R值]映射为可读性强的A~F评级，适合非技术决策者（如仓位配置判断）
- 已在param_decay_analysis报告中实际使用（R=0.02 → ⚫F），证明方案可行
- A~F评级的下钻机制清晰：评级背后可追溯到各维度分数，不丢失信息

**本系统已实现程度**: **40%**

| 方面 | 当前状态 |
|:----|:---------|
| 评级映射 | 🟡 param_decay_analysis已实现6级映射（A~F），但阈值确定（A≥0.80, B≥0.60, C≥0.40, D≥0.20, F<0.20）是临时设定的，未经过系统验证 |
| 评级输出 | 🟡 在分析报告末尾输出评级，但未纳入正式报告结构和数据库 |
| 评级展示 | 🔴 报告内仅有纯文本标注，未与P系列报告联动展示 |

**具体差距**:
- 评级阈值未通过历史回测数据校准（"A≥0.80"这个阈值是经验值，非数据驱动）
- 评级未纳入P系列报告的标准化输出段（每份P报告末尾应附带评级）
- 评级的条件解读缺失（如"C级"的具体含义：是参数不稳还是交易不足？）
- 评级的时间序列对比功能缺失（同一个标的参数在不同时间窗口的评级变化）

**实施建议**:
1. 将Research Confidence Rating集成到每份P系列报告的标准末段（至少P1/P3/P4/P6）
2. 评级阈值通过历史回测数据校准（对601857的全部432组回测结果做K-S检验，确定自然分布的阈值分位点）
3. 评级附带关键瓶颈维度标识（如"C-Stability"表示因稳定性不足导致C级）
4. 在六层报告结构（Layer 1~6）的"研究层"（Layer 6）新增评级时间序列展示

**优先级**: P1 🟡（在评分体系完成后自然集成，不独立立项）

### 1.4 Regime-aware Grid（震荡开/趋势关/动态缩网格）

**判定**: ⚠️ **有条件支持**

**理由**:
- 方向正确且价值极高：网格本质是均值回归策略，必须绑定市场状态识别才能存活
- 提案中四种状态（震荡/趋势上涨/趋势下跌/高波动震荡）的状态空间设计合理
- 但实施复杂度高，涉及策略引擎改造和市场状态分类器两个核心组件

**本系统已实现程度**: **15%**

| 方面 | 当前状态 |
|:----|:---------|
| 市场状态识别 | 🟡 TrendFollowSystem已有Regime划分（StateTransition），但仅用于趋势策略，非网格专用 |
| Regime状态持久化 | 🟡 RegimeContextBuilder已实现，可输出每日市场状态，但输出未与网格策略联动 |
| 网格策略自适应 | 🔴 完全未实现。当前所有网格策略都使用固定参数，无"根据市场状态调整"的逻辑 |
| 动态网格间距（ATR驱动） | 🔴 未实现 |

**有条件支持的原因**:
- **条件1**: Regime-aware Grid不应作为"回测可信度体系"的一部分实施，而应作为**独立的产品策略迭代方向**。将策略改进和研究质量评估混在一起会导致两者都无法完成。
- **条件2**: 需要先完成三层验证结构的市场状态适配层（第三层），让"跨市场状态策略评估"成为标准化流程后，再基于实际数据驱动网格自适应的设计。
- **条件3**: 实施前需要完整的跨周期回测数据支撑（至少3年），确认当前网格在不同市场状态下的实际行为边界。

**实施建议**:
1. 短期（Phase 0）: 将Regime-aware Grid纳入"研究层"（Layer 6）作为研究课题，输出策略设计文档
2. 中期（Phase 2）: 在动态边界网格（方案C/param_decay）验证有效后，启动Regime-aware改造
3. 长期（Phase 3）: 完成完整的Regime-aware Grid策略实现和回测验证

**优先级**: P2 🟡（作为产品方向，不阻塞可信度体系）

## 二、系统性差距分析

### 2.1 提案建议 vs 现有P系列工具链对比矩阵

| 能力维度 | 提案要求 | P系列现状 | 差距 | 严重程度 |
|:---------|:--------|:---------|:----|:--------:|
| **存在性验证** | 固定粗参数跨周期验证 | ❌ 全部基于参数扫描最优解 | 缺失完整的Pre-optimization Gate | 🔴 |
| **参数稳定性量化** | 参数邻域稳定性评分 | 🟡 P3有热力图+CV，但未归一化为评分 | 缺标准化评分输出 | 🟡 |
| **跨Regime一致性** | 市场状态切片验证 | ❌ 完全缺失 | 缺Regime切片框架（如P3分析在牛市vs熊市表现） | 🔴 |
| **OOS存活率** | Walkforward OOS/IS比率 | 🟡 P4已实现 | 未纳入统一评分体系 | 🟢 |
| **时间稳定性** | 滚动窗口验证 | ❌ 仅单窗口分析 | 缺时间切片框架（如P1分析在2020,2021,2022的独立性） | 🔴 |
| **容量稳定性** | 资金规模灵敏度测试 | ❌ 完全缺失 | 缺自定义资金规模下的回测对比 | 🟡 |
| **交易数阈值** | ≥30笔的统计显著性检验 | ⚠️ P6有TQ数据分析，但缺标准化阈值 | 缺预配置的统计显著性规则 | 🟡 |
| **评级标准化** | A~F统一评级输出 | 🟡 param_decay使用，未标准化 | 缺跨报告一致的评级引擎 | 🟡 |
| **数据持久化** | 评分存入数据库供回溯 | ❌ 纯文件系统 | 缺评分数据库表 | 🟡 |
| **API集成** | 报告管线直接调用评分 | ❌ 手动分析 | 缺评分调用管道 | 🟡 |

### 2.2 差距核心特征

1. **分析碎片化**：P1~P8的独立分析结论无法自动聚合为统一可信度评分
2. **验证流程缺失**：回测管线没有"研究质量门"（Research Quality Gate），参数扫描后直接出报告，无人验证结论的统计可靠性
3. **横截面功能缺失**：无法比较同一策略在不同标的、不同参数、不同时间窗口的可信度评分
4. **策略改进和研究评估不分家**：当前流程中"改进网格策略"和"评估参数可信度"是混合的，导致评估结果带上了策略开发者的偏见

### 3.2 组件说明

#### 组件A: ExistenceValidator（新建）
- **功能**: 验证策略是否存在基础alpha
- **输入**: 策略定义 + 固定粗参数 + 多周期数据（3年/5年/全量）
- **算法**: 固定参数直接跑多周期，检查夏普>0且非零交易
- **输出**: ExistencePass (bool) + BaselineMetrics (json)。不通过则不允许进入参数扫描
- **文件**: `src/analytics/research_confidence/existence_validator.py`
- **依赖**: 回测引擎（已有）

#### 组件B: RobustnessScorer（已部分实现，需重构）
- **功能**: 计算参数稳定性评分（参数邻域CV + 跨区间稳定性）
- **输入**: 参数扫描结果（P3输出）+ Walkforward结果（P4输出）
- **算法**: 
  - 参数邻域稳定性: CV of Sharpe in neighborhood
  - 跨区间稳定性: IS/OOS Sharpe ratio + cross-period Sharpe correlation
- **输出**: Parameter Stability Score [0,1] + OOS Survival Score [0,1]
- **文件**: `src/analytics/research_confidence/robustness_scorer.py`（重构P3/P4）
- **依赖**: P3扫描管道 + P4 Walkforward管道

#### 组件C: RegimeValidator（新建）
- **功能**: 验证策略在不同市场状态下的一致性
- **输入**: 回测结果 + Regime时间序列（来自RegimeAnalyzer，已有）
- **算法**: 按Regime切片计算Conditional Sharpe + Conditional WinRate，检查跨Regime绩效差异
- **输出**: Regime Consistency Score [0,1]
- **文件**: `src/analytics/research_confidence/regime_validator.py`
- **依赖**: RegimeAnalyzer（已有）+ 回测引擎

#### 组件D: CapacityValidator（新建）
- **功能**: 验证策略的资金容量稳定性
- **输入**: 回测结果 + 不同资金规模（10万/50万/100万/500万/1000万）
- **算法**: 按资金规模重算回测，检查收益/夏普对规模的敏感度
- **输出**: Capacity Stability Score [0,1]
- **文件**: `src/analytics/research_confidence/capacity_validator.py`
- **依赖**: 回测引擎（支持capital修改）

#### 组件E: ConfidenceScoreAggregator（新建）
- **功能**: 聚合6维评分为复合R值 + A~F评级 + 瓶颈标识
- **输入**: 6维评分 + 权重配置（默认或自定义）
- **算法**:
  - 复合R = Σ(w_i * score_i)，所有score_i∈[0,1]
  - 评级: A (R≥0.80) / B (R≥0.60) / C (R≥0.40) / D (R≥0.20) / E (R≥0.10) / F (R<0.10)
  - 瓶颈标识: 标记score_i<0.30的维度（如"瓶颈: Stability+OOS"）
- **输出**: 结构化评分JSON
- **文件**: `src/analytics/research_confidence/aggregator.py`
- **依赖**: B~D组件 + 权重配置

#### 组件F: CrossReportCompHub（新建）
- **功能**: 横截面比较 + 时间序列趋势
- **输入**: reliability_scores表 + 查询条件
- **算法**: SQL聚合查询 + 排序/对比/趋势图渲染
- **输出**: 比较表/图
- **文件**: `src/analytics/research_confidence/comparison_hub.py`
- **依赖**: 数据库表

#### 组件G: QualityGatePipeline（改造）
- **功能**: 在回测管线中嵌入质量门
- **位置**: pipeline/main_pipeline.py
- **门控点**:
  - G1 (Existence Gate): 参数扫描前，确认基础alpha存在
  - G2 (Robustness Gate): 报告生成前，确认参数稳定性达标
  - G3 (Review Gate): 报告生成后，标记Review状态
- **行为**: Gate不通过时写FAILED状态，不阻塞流程但标记不可部署

### 3.3 数据流

```
阶段1: 基础验证
```

## 五、Kill Switch 条件

若以下任一条件成立，改造方案后续Phase需暂停并重新评估：

| 条件 | 触发场景 | 应对 |
|:----|:---------|:------|
| P系列报告版本分歧 | Phase 0开始前仍未锁定评分数据源版本 | 暂停，先锁定版本基线 |
| 回测引擎修改破坏评分接口 | CapacityValidator或RobustnessScorer无法获取必要字段 | 隔离评分层，先修复引擎 |
| 评分阈值校准后产生"全部F级" | 历史432组回测中无任何一组获得C级以上评级 | 重新校准阈值或接受"当前所有策略均为低可信度"的结论 |
| 评级与人类直觉出现系统性偏差 | 人工评估认为某个策略有效，但评分始终为F | 暂停评分引擎，人工审查维度权重分配和阈值设定 |
