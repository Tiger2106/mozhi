## 三、关键路径分析

### 3.1 信号计算管线

信号采集 → 因子注册器 (FactorRegistry) → 因子计算 → 信号融合 (SignalFusionEngine)

- **FactorRegistry**：已实现 `FactorCache` 缓存机制，避免重复计算
- **瓶颈识别**：因子计算依赖 numpy/numba，CPU 密集；缓存命中时 O(1)

### 3.2 回测执行管线

数据加载 → 策略调度 → 信号融合 → 交易执行 → 损益归因

- **MultiInstrumentEngine**：已实现 `ThreadPoolExecutor` 并行多标回测
- **瓶颈识别**：单标的单策略回测为串行流水线，无冗余计算

### 3.3 报告生成管线

回测结果 → 分析 → 报告生成 (markdown/pdf)

- **瓶颈识别**：报告生成当前为同步阻塞，但执行时间 < 1s，无需异步化

### 4.2 当前性能基准

基于 r1_complete_e2e.json 的 E2E 测试数据：

| 标的 | 总耗时(ms) | 数据量(bar) | 因子数 |
|------|-----------|-------------|--------|

**平均 E2E 耗时：约 23.4 ms/标的**（含市场数据+因子+回测）

### 4.3 建议的优化方向（无需大规模改代码）

#### 优化 1：FactorRegistry 冷启动预热
- **问题**：首次启动时因子注册与初始化耗时约 4-5ms
- **建议**：模块加载时预热因子列表，而非首次 compute() 时 lazy init
- **优先级**：低（边际收益约 4ms/启动）

#### 优化 2：报告生成缓存
- **问题**：相同回测结果重复生成报告
- **建议**：按 backtest_config hash 缓存已生成的 report markdown，避免重复渲染
- **优先级**：低（当前报告生成 < 1s）

#### 优化 3：信号融合阈值判定短路
- **问题**：SignalFusionEngine 对所有因子都计算加权评分，即使大部分因子已失效
- **建议**：当有效因子数 < 3 时直接返回 HOLD 信号，跳过完整融合计算
- **优先级**：中（日常运行收益最高）

### 4.4 不需要优化的部分

| 场景 | 理由 |
|------|------|
| 报告生成异步化 | 当前同步 < 1s，异步增加复杂度无实际收益 |
| 数据库连接池 | 当前读写量极低，池化属于过度工程化 |
| 大规模并发处理 | 当前系统最多处理 5 个标的，单线程足够 |
| 内存池化 | Python 的 GC 足够应对当前数据量（< 1000 bars/标的） |

## 六、结论

- **可维护性**：良好，模块边界清晰
- **性能**：当前负载下无需优化，瓶颈预热已在架构层解决

- **建议**：保留当前性能基线，仅在下阶段新增大量标的（>20）时考虑并行扩展
