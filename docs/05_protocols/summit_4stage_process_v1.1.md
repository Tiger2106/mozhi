# 回测总结会4阶段流程 v1.2

## 总体结构

```
Stage 1           Stage 2a                   Stage 2b                    Stage 3        Stage 4
执行 ───→  QA验证（含§6填写）             技术评审 ──→ Owner签署 ──→ 归档入库
             ←→ 修复循环 ──→【§6闸门】──→ 环节串行                      墨涵
墨衡填模板      墨萱独立        墨涵确认   墨萱+玄知              ⬆ 墨衡开发
TMPL-001/002    TMPL-003 + §6  §6已完整   墨涵知识审查          ingest_analysis.py
```

⚠️ **发言顺序不可跳步**。每完成一步，主持人检查单标注 ✅ 后进入下一步。

## 退回重做机制

| 循环次数 | 操作 | 说明 |
|:-------:|:----|:-----|
| 第1次 | 打回墨衡修复 | 说明问题，要求重做 |
| 第2次 | 打回墨衡修复 | 仍未通过，加注具体修改要求 |
| 第3次 | **告警Owner介入** | 3次仍不通过，由Owner裁决 |

**关键规则：**
- 墨涵**只打回、不代修**。墨衡的产出质量问题由墨衡自己修复
- 超过2次不通过 → 可能涉及设计层面的问题，需Owner决策而非继续修复

## 三层质量检查制度

| 检查维度 | 检查方 | 关注点 | 工具 |
|:--------|:------|:------|:-----|
| **QA验证（含§6填写）** | 墨萱 | 回测数据正确性 + §6结论填写——数字对不对 + 结论有没有 | TMPL-003 QA清单 |
| **§6完成闸门** | 墨涵 | §6 QA结论完整性——§6是否全部填写，是否有空项 | 主持人检查单 |
| **技术评审** | 墨萱 + 玄知 | 架构合理性——设计好不好 | 评审意见表 |
| **知识审查** | 墨涵 | 知识完整性——该记的记了没 | 知识条目审查 |

**说明**：QA验证 → §6闸门 → 技术评审 → 知识审查 构成四层逐级过滤。§6闸门确保QA结论在进入技术评审前已填写完整，避免评审时发现§6空白。

**顺序不可逆：**
```
QA验证 → §6闸门 → 墨萱技术评审 → 玄知技术把关 → 墨涵知识审查
 数字    完整性    设计评估        架构+数据       知识归档
  核对    确认     ↑                ↑              ↑
```

## Stage 1 执行标准

### 墨衡产出要求
- Summary报告按TMPL-001填写
- Analysis报告按TMPL-002填写
- 代码版本标注：有Git时记录commit hash，无Git时标注 `未使用版本管理`/`工作副本（未提交）`
- 环境信息（附录C）：OS、Python版本、依赖库版本均为<必填>
- 审计日志：操作人、时间（ISO8601+08:00）、操作类型

---

## Stage 2a 评审标准

### QA验证（墨萱，TMPL-003）+ §6填写
核验范围：
1. 数据库行数核验（交易/持仓/净值表逐表比对）
2. 报告数据与数据库逐项交叉验证（年化收益/夏普/最大回撤/交易次数）
3. 确定性策略复现 → 偏差0.00%是必然结果，需标注说明

**§6 QA结论填写**：QA验证完成后，墨萱在报告中填写§6 QA结论（TMPL-003对应部分），包括每项核验结果、偏差说明、QA总体结论。

### 【QA§6完成闸门】墨涵确认

进入Stage 2b前，墨涵必须确认报告中§6部分已填写完整：

```
Stage 2a 墨萱QA验证（TMPL-003） → 在报告中填写§6 QA结论
    ↓
【QA§6完成闸门】墨涵确认§6已填写完整
    ├─ ✅ §6已填 → 继续
    └─ ❌ §6空白/不完整 → 退回墨萱补填
    ↓
Stage 2b 墨萱技术评审 + 玄知技术把关 + 墨涵知识审查
```

**闸门规则：**
- §6中存在任何空白项（如QA结论为空、待填、未完成等）→ 退回墨萱
- 墨萱补填后，墨涵二次确认 → 通过则继续，不通过则再次退回
- 此闸门不计入墨衡修复循环次数（只针对§6空白，不涉及报告内容质量）

### 超时策略
- 阈值：40分钟（从Stage 1开始计时）
- 超时后：自动终止，输出中间结果
- 超时标记记录在案，进入Stage 3由Owner评审
- **绝不默认通过**

### 发言时长表（含§6填写时间）
| 角色 | 阶段 | 内容 | 时长 |
|:----|:-----|:-----|:----|
| 墨衡 | Stage 1 | 填写TMPL-001/002报告 | 不限（会前完成） |
| 墨萱 | Stage 2a | QA验证（TMPL-003）+ 填写§6结论 | 600秒 + 200秒§6填写 = **800秒** |

| 墨萱 | Stage 2b | 技术评审 + 进入检查单 | 600秒 |
| 玄知 | Stage 2b | 技术把关 | 400秒 |
| 墨涵 | Stage 2b | 知识审查 | 300秒 |
| Owner | Stage 3 | 签署 | 300秒 |
| 墨涵 | Stage 4 | 归档入库 | 不限（会后完成） |

## Stage 2b 评审标准

### 墨萱技术评审 + 玄知技术把关 + 墨涵知识审查

核验内容同原流程（技术评审 + 技术把关 + 知识审查）：

1. **墨萱技术评审**：架构合理性评估，审查回测框架设计、参数选择、数据流水线是否合理
2. **玄知技术把关**：架构+数据一致性审查，确认回测结论与技术指标的一致性
3. **墨涵知识审查**：知识完整性审查，确认报告中的知识条目已归档

**顺序不可逆**，必须按以下顺序进行：
```
墨萱技术评审 → 玄知技术把关 → 墨涵知识审查
```
