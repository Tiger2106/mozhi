## 声明2（P3段）：Windows spawn 序列化约需 80-100MB/批，实际加速比 ~2-3x

**判定: ⚠️ 疑似 — 声明逻辑方向正确但量化精度无法验证**

### 验证分析

**可行方向确认:**
- ✅ 当前 Python 版本为 **3.14.3**（Windows），此版本 `multiprocessing` **必须使用 spawn**（fork 不可用）
- ✅ `run_exp_invfac002.py` 存在 `if __name__ == "__main__"` 防护（行934）→ spawn 基本条件满足
- ✅ Windows spawn 下大 numpy 数组 pickle 序列化确是已知痛点
- ✅ 负载不均的判断（高低波动样本量差距可达 10x）→ 无数据支撑但逻辑合理

**量化精度存疑:**
- ⚠️ **80-100MB/批** 的数据量估算：墨衡声明 `factor_values` 36MB + `all_stocks` 50MB + `market_states` 96KB。核对当前：
  - 该运行结果写入 `exp_results.json`（实际文件约 10-20MB），但运行时数组的 pickle 大小可能接近该量级
  - **数据是 12 只标的 × ~750 个交易日**（非 1000 天），实际体积可能略低
  - **精度声明**：此数值为墨衡基于"约1000天"的估算。考虑到数据实际长度为 ~750 天，实际约 **60-75MB/批**。误差在合理范围（~20%），不改变结论方向。
- ⚠️ **实际加速比 ~2-3x**: 含序列化开销(+1-2s) + spawn 启动(+1-2s) + 负载不均 → 纯理论推演，**无实测数据**。方向合理但量化不精确。

**结论**: 声明方向正确。量化数值可接受为合理估算（偏差≤25%）。不影响 P3 的"不建议当前实施"决策。

## 声明3（P4段）：numba 兼容性的主要风险是 numpy 版本约束

**判定: ✅ 通过 — 但需补充信息**

### 验证记录
| 检查项 | 结果 |
|--------|------|
| 当前 `numpy.__version__` | **2.4.4** |
| numba 0.59/0.60 支持范围 | numpy 1.22 ~ **1.26**（[numba 官方文档](https://numba.readthedocs.io/)） |
| numpy 2.4.4 vs numba 兼容范围 | ❌ **numpy 2.4.4 >> 1.26**，远超 numba 0.60 的上限 |
| Python 版本 | **3.14.3** — numba 0.60 支持的 Python 上限通常为 3.12 |

### 分析

墨衡声明"numba兼容性主要风险是numpy版本约束"——方向正确，但**实际风险比墨衡评估的更严重**：

1. **numpy 2.4.4** → 当前最稳定的 numba master（0.60.x）不支持任何 numpy ≥2.0 版本
2. **Python 3.14.3** → 最新的 numba 官方 release 也只支持到 Python 3.12
3. 这意味着在当前环境中要运行 numba，需要：
   - **降级 numpy** 到 1.26（影响项目其他部分是否兼容？需要验证）
   - 或 **降级 Python** 到 3.12 以下（环境调整大）
   - 或等待 numba 发布支持 numpy 2.x + Python 3.14 的版本（时间不确定）

**结论**: 墨衡的"numpy版本约束是主要风险"声明**正确**，但**低估了风险的严重程度**。实际需要 numpy 降级 + 可能的 Python 降级，实施成本从"低"提高到"中-高"。

## 声明4（P1段）：边界 p 值（~0.05）早停无收益，早停最多节省 20%

**判定: ⚠️ 疑似 — 逻辑合理但缺乏验证数据**

### 验证分析

**逻辑合理性:**
- ✅ **n=10000 的 bootstrap**：当前代码固定 10000 次迭代。在此规模下，早停的绝对收益上限是减少部分迭代。
- ✅ **边界 p 值（~0.05）确实是关键场景**：p 值恰好处于显著性阈值附近时，需要足够的样本来稳定估计 → 早停在此场景下的收益趋近于 0。
- ✅ **p值远离阈值时的节省**：如果真实 p 值 ~0.0001，可能 1000 次迭代即可判断 → 可节省 90% 时间。墨衡的"20% 最大节省"是针对**整体平均场景**（27 组合中部分可能被提前截止）。

**需要澄清的争议:**
- ⚠️ 墨衡在 P1 结论中写"早停最多节省20%"，但在综合建议表中写"0-25%"——**数值口径不一致**。前者可能指"在当前 bootstrap 规模下的整体流水线节省"，后者可能是"在最乐观场景下的节省上限"。需要在报告中统一口径。
- ⚠️ **20% 的逻辑来源**：墨衡的推理是"整体从 ~22s → ~13s（P4 加速后），P1 叠加可再节省 ~0-5s"。但在 P1 单独评估段与综合建议表中，20%/25% 的来源**没有明确的计算过程**。声明的可信度因此降低半级。

**结论**: 说法方向合理，墨衡的判断结论（"不优先实施"）也合理，但**20% 的量化数字缺乏推导过程**，判定为"疑似"而非"通过"。
