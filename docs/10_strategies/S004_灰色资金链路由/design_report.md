## §1 背景

### 1.1 策略动机
八部门整治的核心打击对象之一是灰色资金链——通过嵌套结构、跨境通道、虚假贸易等渠道游离于监管之外的资金流动。然而灰色资金链并非静态存在：当一级渠道被打击后，资金会迅速变形（借道新渠道、转换结构、缩短链路），形成"打一个换一个"的猫鼠游戏。

需要建立一套系统化的灰色资金路由追踪体系，自动识别、标注、跟踪灰色资金链，监测其变形后的新路由。

### 1.2 要解决的问题
- 灰色资金链可视化困难：多层嵌套结构难以手动梳理
- 渠道变形无追踪：已知渠道被封堵后，资金流向何处未知
- 标签体系不统一：不同分析手段对同一笔资金流产生不同标签
- 跨Agent协作困难：墨衡（数据层面）、墨萱（财务指标）、玄知（政策信号）需要统一的路由描述

### 1.3 与八部门整治政策的关系
S004 的"路由"指的是灰色资金在八部门整治压力下的路径选择。整治政策打击的渠道形成"灰名单"（S004 内部维护），新出现的变道路径自动进入监控。

## §2 设计目标

### 2.1 核心功能定位
建立灰色资金链的"三层打标 → 六字段接口 → 渠道变形跟踪"全链路路由追踪体系，实现资金流向的自动标注与变形监测。

### 2.2 预期产出

| 产出项 | 格式 | 说明 |
|:----|:----|:----|
| 路由拓扑图 | JSON(图结构) | 当前检测到的资金链路由图 |
| 三层打标输出 | JSON | 每条链路的Layer1/Layer2/Layer3标签 |
| 渠道变形警报 | JSON | 新出现/形态变化的渠道通知 |
| 灰名单渠道表 | JSON | 已知灰色渠道及其状态列表 |

### 2.3 使用场景
- **日常监测**：每日扫描资金流异常模式，更新路由拓扑
- **渠道变形触发**：某渠道被查封后，跟踪替代渠道的出现
- **政策联动**：八部门发布新打击方向后，快速在路由图中定位受影响节点
- **Agent协作**：向墨涵（投研产出）提供灰色资金链风险评估

## §3 技术方案

### 3.1 三层打标模型

**Layer 1 — 结构识别层**
检测资金流的传输结构是否具有灰色特征：

| 结构模式 | 示例 | 检测方法 |
|:----|:----|:----|
| 多层嵌套 | A→B→C→D→E，层级≥3 | 图遍历检测路径长度 |
| 快速穿透 | 资金进出时间<阈值 | 时序分析 |
| 分散聚合 | 多笔小额→单笔大额(或反向) | 聚类分析 |
| 跨境折返 | A(境内)→B(境外)→C(境内) | 外汇+账户关联 |

输出：结构标签（如 `nested_4layer`, `quick_penetration`, `scatter_gather`）

**Layer 2 — 渠道特征层**
识别资金流经过的具体渠道：

| 渠道 | 检测特征 | 烈度(1-5) |
|:----|:----|:----|
| 虚假贸易 | 报关金额与对手方财务数据不符 | 4 |
| 借壳通道 | 壳公司流水与业务规模不匹配 | 5 |
| 跨境支付平台 | 平台交易频率异常 | 3 |
| 加密货币桥接 | 法币↔虚拟币换入换出时间窗口 | 5 |
| 私募基金通道 | 产品结构异常嵌套 | 3 |

输出：渠道标签 + 烈度评分

**Layer 3 — 意图推测层**
基于行为序列推测资金意图：

| 意图 | 行为序列特征 |
|:----|:----|
| 规避监管 | 资金在稽查后迅速变更路径 |
| 套利套汇 | 资金短暂利用利差/汇差后回流 |
| 洗钱/转移 | 资金来源不明、多次穿透、最终出境 |
| 杠杆/配资 | 资金回流至A股盘口且与融资余额联动 |

输出：意图推测标签 + 置信度

### 3.2 六字段接口协议

所有路由信息通过统一的6字段接口传递：

```json
{
  "route_id": "R20260525_001",
  "layer1_tag": { "type": "nested_4layer", "confidence": 0.87 },
  "layer2_tag": { "channel": "借壳通道", "severity": 5, "confidence": 0.92 },
  "layer3_tag": { "intent": "规避监管", "confidence": 0.73 },
  "chain_json": "A→B→C→D(壳公司)→E→F(回收)",
  "timestamp": "2026-05-25T09:30:00+08:00",
  "signature": "sha256://hash"
}
```

**字段说明**：
| 字段 | 类型 | 必填 | 说明 |
|:----|:----|:----|:----|
| route_id | string | 是 | 路由唯一标识 |
| layer1_tag | object | 是 | 结构识别层标签 |
| layer2_tag | object | 是 | 渠道特征层标签 |
| layer3_tag | object | 否 | 意图推测层（置信度<0.5可不输出） |
| chain_json | string | 是 | 资金链JSON化描述 |
| timestamp | string | 是 | 检测时间 |
| signature | string | 是 | 数据签名（防篡改+去重） |

### 3.3 渠道变形跟踪

**核心问题**：已知渠道被封堵后，资金会变形为新的路由形态。

**实现方案**：

```
已知渠道C₁被封堵（来自S002/政策打击信号）
   │
   ▼
等待期（默认 48 小时，可配置）
   │
   ▼
扫描时段内新出现且无法匹配C₁的路由
   │  ┌──────────────────────────────────────┐
   ├─→│ 检查新路由拓扑特征与C₁的相似度      │
   │  │ Jaccard(节点集) + 图编辑距离综合评分 │
   │  └──────────────────────────────────────┘
   │  ┌──────────────────────────────────────┐
   ├─→│ 与Layer2渠道库核对是否为新渠道       │
   │  │ 若从未见过 → 标记为疑似变形渠道       │
   │  └──────────────────────────────────────┘
   │  ┌──────────────────────────────────────┐
   └─→│ 连续3日持续出现 → 正式加入灰名单     │
      └──────────────────────────────────────┘
```

## §5 验证标准

### 5.1 通过率指标

| 指标 | 目标值 | 说明 |
|:----|:----|:----|
| 打标准确率 (Layer1+Layer2) | ≥85% | 与人工标注比对 |
| 渠道变形召回率 | ≥70% | 变形后72小时内被检测到 |
| 路由完整性 | ≥80% | 已知灰色路由图节点覆盖 |
| 误报率 | ≤15% | 将正常资金流误标为灰色的比例 |
| 6字段接口可用率 | 100% | 输出始终符合字段规范 |

### 5.2 交叉验证方法
- 每周抽取50条路由由墨萱人工标注，与系统打标结果比对
- 渠道变形检测使用回测：以已知历史查封事件为起点，检验系统能否在事后检测到变形
- 误报率与S002热力图交叉校验（高监管区域的误报是可以接受的）

## §6 风险与边界

### 6.1 已知局限性

| 风险 | 描述 | 应对 |
|:----|:----|:----|
| 数据覆盖不足 | 灰色资金数据天然不完整 | 标注"数据盲区"标识 |
| 渠道边界模糊 | 某些通道介于灰色与合规之间 | 引入"灰度分级"标签 |
| 欺诈性变形 | 资金主动制造假路由误导监测 | 签名校验+交叉验证 |
| 移交风险 | Phase1→Phase2移交时知识丢失 | 完整移交文档+代码注释 |

### 6.2 边界条件
- 输入交易流连续24小时中断 → 标记"离线"并维持上次快照
- 打标置信度全面低于0.5 → 输出"无法标注"并延迟至数据丰富后更新
- 渠道数量超过100条 → 算法复杂度报警，触发简化处理

### 6.3 熔断/降级路径
1. 打标准确率监测→连续下降→触发告警→切换为"仅Layer1"模式
2. Phase1/Phase2移交失败→墨衡继续维护过渡期，不强制移交
3. 误报率>30%→暂停新渠道入库，仅维护已有渠道
