# 辩论赛标准流程（辩论模板 v1.2）

**来源**: 右侧交易辩论赛（2026-05-28 22:31~23:30）实践经验总结 + 辩论总结反思会（2026-05-29 15:56~16:34）集体建议
**适用**: 墨枢团队三方观点分歧时，需结构化辩论的场景
**归档**: `docs/05_protocols/debate_template_v1.2.md`

---

## 一、触发条件

- 三方对某个方向/方案存在明显分歧
- Owner认为"各自说各自的"信息不对称，需要直面交锋
- 需要做优先级/方向决策前
- 技术方案存在不可调和的路线分歧（如否决制 vs 兼容）

---

## 二、辩论前准备

### Step 0: 主持人判断是否启动辩论

**主持人（墨涵）** 在触发条件满足后：

1. 确认三方是否愿意接受辩论规则
2. 判定分歧是否足够结构化（非模糊分歧不必辩论）
3. 向Owner申请启动辩论并获确认

### Step 1: 下发思路卡（主持人）

- 汇总三方各自的核心立场/提议/论点
- 每个角色一个独立思路卡，标明"你是哪个"
- 包含全量对方信息（让每个人知道对方会说什么）
- **明确边界**: 只说思路不要方案（防止辩论变成设计会）

### Step 1.5: 会前预警通道（主持人收集）【v1.2 新增】

主持人提前收集各位参与者的 1-2 条 **"我知道有问题但暂时无法定量"** 的预警信息。
在辩论开始前全员可见，作为背景信息进入辩论环节。

**目的**：让线索性风险（如数据链路断裂预感、架构可扩展性隐忧）在辩论前就曝光，而不是靠辩论中的偶然追问才被发现。

**与思路卡的关系**：预警通道收集的是"不确定性"而非"立场"，是思路卡的补充而非替代。

### Step 2: "黑匣子清单"与《自我交叉审讯反思录》【v1.2 新增】

**每位参与者（含主持人自己在内）在提交思路卡的同时，提交一份"黑匣子清单"：**
- 列出自己论点中 ≤3 条**没有数据支撑的关键假设**
- 每条标注：假设内容 / 推测依据 / 预备的实证方法
- 如果假设被证明错误，对论点的影响程度（致命/重要/轻微）

**方案提出方（思路卡中的核心提案方）额外提交一份《自我交叉审讯反思录》：**
- 从另外两个角色的视角，向自己提 2 个逻辑点
- 回答格式："如果他们问我[问题]，我的回应是[回答]。这个回应的薄弱环节是[点]"
- 必须与思路卡同一时间节点提交，**不得在辩论结束后补写**（否则变成事后诸葛亮，失去压力前置的意义）

### Step 3: 准备时间（5分钟，严格计时）

**三方各自做三件事：**
1. 回顾自己立场，找出最弱环节（需防守）
2. 准备回答对方可能的质问
3. 准备向对方提 ≤3 个针锋相对的问题

**主持人**：倒计时结束后才可开始
**准备期间禁止沟通。**

## 三、辩论结构

### 角色

| 角色 | 职责 | 说明 |
|:----|:----|:----:|
| **主持人（墨涵）** | 计时、维持秩序、控制节奏、汇总结论 | 不参与观点交锋 |
| **辩手 x3** | 墨衡、墨萱、玄知 | 每人被两人提问，向两人提问 |
| **评审（Owner）** | 旁观全部辩论后做最终决策 | 全程沉默，不参与问答 |

### 轮次（六轮制）

| 轮次 | 提问方 → 被问方 | 问题数上限 | 方向说明 |
|:----:|:---------------:|:----------:|:---------|
| R1 | 墨萱 → 墨衡 | ≤3 | 技术实现质疑 |
| R2 | 玄知 → 墨衡 | ≤3 | 架构/数据一致性质疑 |
| R3 | 墨衡 → 墨萱 | ≤3 | 反击与澄清 |
| R4 | 玄知 → 墨萱 | ≤3 | 架构/数据一致性交叉质问 |
| R5 | 墨衡 → 玄知 | ≤3 | 理论前提追问 |
| R6 | 墨萱 → 玄知 | ≤3 | 技术实现追问 |

### 核心规则

1. **每人被** 另外两人提问，每人 **回答** 另外两人的问题
2. **每人向另一个人提问不超过 3 个**
3. 每轮 = 提问方抛出问题 → 被问方在下一轮回答
4. 回答可以简短（单点回击），但必须正面回答问题
5. **禁止不回答、转移话题、说"这个问题下次讨论"**
6. **强制响应条款**【v1.2 新增】：被质疑方必须给出"**接受 / 拒绝 / 延后举证**"的明确回复，不能回避或悬空
7. **每一问一答必须同步给不在场的第三方 + 推群给Owner**
8. **只说思路不要方案** — 辩论聚焦于"是否可行"而非"怎么做"
9. 主持人判定问题不合格（违反问题原则）有权阻止并要求重新提问

## 四、问题原则

| 好的问题 | 不好的问题 |
|----------|-----------|
| 针对对方论点中的逻辑缝隙 | 泛泛而问（"你怎么看？"） |
| 有具体切入点（数据/逻辑/假设） | 需要长篇大论的学术问题 |
| 对方必须回答才能过关 | "你觉得呢"式开放问题 |
| 暴露信息盲区或假设矛盾 | 承认自己不懂的问题 |
| 带数据或引用支撑 | 纯情感判断 |

每个问题应做到：**对方听到就知道你攻击的是他的哪个论点**。

### 重大认错处理

辩论中若一方主动承认自己的核心假设有误：
- 主持人记录该认错为辩论结论的关键输入
- 提问方不再继续追问该点（已获承认不再重复打击）
- 该认错进入最终结论摘要

## 五、辩论后流程

### Step 4: 主持人汇总结论

辩论结束后，主持人立即输出 **辩论结论摘要**，包含：
1. **三方达成共识的结论**
2. **已收敛的分歧**（谁认错了什么、谁让步了什么）
3. **仍未解决的分歧**（需Owner裁决）
4. **否决制状态**（是否触发否决、是否引入CONDITIONAL PASS灰度层）
5. **数据假设审计结论**【v1.2 新增】：围绕"黑匣子清单"中的假设，逐一记录是否已被质疑/验证/搁置
6. **预警通道跟踪**【v1.2 新增】：会前预警信息是否被覆盖或被遗漏

### Step 5: 定工作顺序

根据结论输出 **最终工作顺序表**：

| 优先级 | 工作项 | 负责 | 备注 |
|:-----:|:-------|:----|:-----|
| 1 | ... | ... | ... |
| 2 | ... | ... | ... |

原则：
- 优先级按"依赖链"排列（前置项先做）
- 每个工作项有明确Owner
- 标注时间约束（如T+30、T+7等里程碑）

### Step 6: 四种结果判定

| 结果 | 条件 | 后续动作 |
|:----|:-----|:---------|
| ✅ **全票通过** | 三方一致同意 | 记录结论，直接进入工作分配 |
| ⚠️ **CONDITIONAL PASS** | 存在保留意见但可接受 | 引入灰度层，设定验收条件 |
| 🔄 **退回重做** | 方案存在重大缺陷 | 退回设计阶段，附具体修改要求 |
| ❌ **否决** | 一方行使否决权 | 主持人记录否决理由，上报Owner裁决 |

### Step 7: 四方会签归档

| 签署方 | 签署条件 | 状态 |
|:------|:---------|:----:|
| 墨萱 | 技术合规确认 | ⬜ |
| 玄知 | 架构+数据一致性确认 | ⬜ |
| 墨涵 | 知识归档确认 | ⬜ |
| Owner | 业务方向确认 | ⬜ |

争议处理：若墨萱/玄知拒签，不得绕过，必须解决后再走会签。

### Step 8: 归档

- 辩论结论摘要 → `docs/07_research/{议题}/辩论总结_会签_v1.x.md`
- 会签记录 → `docs/05_protocols/archive_signoff/{议题}_signoff_owner.md`
- 知识条目激活 → 若辩论产生可固化的insight，由墨涵激活knowledge_entries
- "黑匣子清单"与反思录 → 附在会签记录之后，作为可追溯的假设记录

## 七、适用范围

- 📋 方向选择争议
- 📋 资源分配争议
- 📋 方案优先级争议
- 📋 Owner需加速决策时
- 📋 技术路线分歧（否决制触发场景）
- 📋 回测冲突结论需统一方向时

## 八、辩论后标准工作流

```
辩论前
   ↓ 预警通道 → 黑匣子清单 → 反思录
   ↓
辩论 — 六轮交叉提问（含强制响应条款）
   ↓
主持人汇总结论（含数据假设审计 + 预警跟踪）
   ↓
定工作顺序（优先级+Owner+里程碑）
   ↓
结果判定（通过/CONDITIONAL PASS/退回/否决）
   ↓
四方会签（墨萱→玄知→墨涵→Owner）
   ↓
归档 docs/07_research/{议题}/（含反思录+黑匣子清单）
```
