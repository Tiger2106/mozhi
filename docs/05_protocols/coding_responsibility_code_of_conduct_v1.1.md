## 三条规则

### 1. 红线：编码不越界

**墨涵（PO/质量控制）不得以任何理由代行编码职责。**

**"编码"的定义**：指直接修改主仓库中的 `.py` 文件逻辑、SQL 语句、测试用例、数据管线代码等影响系统行为的操作。不包括以下（属于PO正常职责）：
- 编写规约/协议文档（`.md` 文件）
- 编写调度脚本的 JSON 配置
- 创建目录结构、执行文件整理
- 管理 OpenClaw 配置文件（`openclaw.json` 等）
- 直接运行已存在的 Python 命令（如 `python -m pytest`）

**不可用时流程**：
- 墨衡被kill、超时、无响应 → 暂停推进，上报主人判断
- 不允许"先写着，后续review补救"

**唯一的例外**：主人明确指令的代行，且：
- 代行产物必须标注 `origin: fallback_mohan`
- 后续仍须经过墨衡 Review + 墨萱复核（按规则2）

### 2. 把关：双向代码审查

**所有进入主分支的代码必须经过独立的 Review + 复核。**

- 墨衡写的代码 → 墨萱 QA 复核
- 墨涵写的代码（含 fallback 例外）→ 墨衡 Review → 墨萱 QA 复核
- 发现问题 → 墨衡修复 → 墨萱复核 → **通过后方可合并**

**复核通过的客观标准（v1.1 新增）**：

| 标准 | 说明 | 触发动作 |
|:----|:-----|:---------|
| ① 阻断问题清零 | Review 发现的所有阻断问题已关闭或明确标记已修复 | 复核人逐项确认 |
| ② 测试覆盖率 | 新增代码至少跑通单元测试（不引入新的失败用例） | 展示测试输出 |
| ③ 无静默失败路径 | 代码中不存在"执行了但不报错但结果为空/错误"的路径 | Review 时逐条审查异常路径 |

Review 须发现具体问题并记录，不得走过场。

### 3. 排期：故障转移方案进下一 Sprint

**玄知提出的三层加固方案列为下个 Sprint 的优先事项。**

- 当前 Sprint：Phase 2 编码补完 + Phase 3
- 下个 Sprint：方案零（环境预检）→ 修正1（心跳探针）→ 修正2（熔断机制）
- 在故障转移机制就绪前，按规则1执行
