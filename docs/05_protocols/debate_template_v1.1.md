# 辩论赛标准流程（辩论模板 v1.2）

**来源**: 右侧交易辩论赛（2026-05-28 22:31~23:30）实践经验总结 + 辩论总结反思会（2026-05-29 15:56～16:34）集体建议
**适用**: 墨枢团队三方观点分歧时，需结构化辩论的场景  
**归档**: `docs/05_protocols/debate_template_v1.2.md`

---

## 一、触发条件

- 三方对某个方向/方案存在明显分歧
- Owner认为"各自说各自的"信息不对称，需要直面交锋
- 需要做优先级/方向决策前
- 技术方案存在不可调和的路线分歧（如否决制 vs 兼容）

---

## 二、辩论前准备

### Step 0: 主持人判断是否启动辩论

**主持人（墨涵）** 在触发条件满足后：

1. 确认三方是否愿意接受辩论规则
2. 判定分歧是否足够结构化（非模糊分歧不必辩论）
3. 向Owner申请启动辩论并获确认

### Step 1: 下发思路卡（主持人）

- 汇总三方各自的核心立场/提议/论点
- 每个角色一个独立思路卡，标明"你是哪个"
- 包含全量对方信息（让每个人知道对方会说什么）
- **明确边界**: 只说思路不要方案（防止辩论变成设计会）

### Step 1.5: 会前预警通道（主持人收集）

**新增（v1.2）**

主持人提前收集各位参与者的1-2条

## 三、辩论结构

### 角色

| 角色 | 职责 | 说明 |
|:----|:----|:----:|
| **主持人（墨涵）** | 计时、维持秩序、控制节奏、汇总结论 | 不参与观点交锋 |
| **辩手 x3** | 墨衡、墨萱、玄知 | 每人被两人提问，向两人提问 |
| **评审（Owner）** | 旁观全部辩论后做最终决策 | 全程沉默，不参与问答 |

### 轮次（六轮制）

| 轮次 | 提问方 → 被问方 | 问题数上限 | 方向说明 |
|:----:|:---------------:|:----------:|:---------|
| R1 | 墨萱 → 墨衡 | ≤3 | 技术实现质疑 |
| R2 | 玄知 → 墨衡 | ≤3 | 架构/数据一致性质疑 |
| R3 | 墨衡 → 墨萱 | ≤3 | 反击与澄清 |
| R4 | 玄知 → 墨萱 | ≤3 | 架构/数据一致性交叉质问 |
| R5 | 墨衡 → 玄知 | ≤3 | 理论前提追问 |
| R6 | 墨萱 → 玄知 | ≤3 | 技术实现追问 |

### 核心规则

1. **每人被** 另外两人提问，每人 **回答** 另外两人的问题
2. **每人向另一个人提问不超过 3 个**
3. 每轮 = 提问方抛出问题 → 被问方在下一轮回答
4. 回答可以简短（单点回击），但必须正面回答问题
5. **禁止不回答、转移话题、说"这个问题下次讨论"**
6. **每一问一答必须同步给不在场的第三方 + 推群给Owner**
7. **只说思路不要方案** — 辩论聚焦于"是否可行"而非"怎么做"
8. 主持人判定问题不合格（违反问题原则）有权阻止并要求重新提问

## 四、问题原则

| 好的问题 | 不好的问题 |
|----------|-----------|
| 针对对方论点中的逻辑缝隙 | 泛泛而问（"你怎么看？"） |
| 有具体切入点（数据/逻辑/假设） | 需要长篇大论的学术问题 |
| 对方必须回答才能过关 | "你觉得呢"式开放问题 |
| 暴露信息盲区或假设矛盾 | 承认自己不懂的问题 |
| 带数据或引用支撑 | 纯情感判断 |

每个问题应做到：**对方听到就知道你攻击的是他的哪个论点**。

### 重大认错处理

辩论中若一方主动承认自己的核心假设有误（如墨衡在右侧辩论中承认"截面IC在A50趋零"是理论假设非实证）：

1. 主持人记录该认错为辩论结论的关键输入
2. 提问方不再继续追问该点（已获承认不再重复打击）
3. 该认错进入最终结论摘要

## 五、辩论后流程（v1.1 新增）

### Step 3: 主持人汇总结论

辩论结束后，主持人立即输出 **辩论结论摘要**，包含：

1. **三方达成共识的结论**（如"情绪-流动性共振作为第一步"）
2. **已收敛的分歧**（谁认错了什么、谁让步了什么）
3. **仍未解决的分歧**（需Owner裁决）
4. **否决制状态**（是否触发否决、是否引入CONDITIONAL PASS灰度层）

### Step 4: 定工作顺序

根据结论输出 **最终工作顺序表**：

| 优先级 | 工作项 | 负责 | 备注 |
|:-----:|:-------|:----|:-----|
| 1 | ... | ... | ... |
| 2 | ... | ... | ... |

原则：
- 优先级按"依赖链"排列（前置项先做）
- 每个工作项有明确Owner
- 标注时间约束（如T+30、T+7等里程碑）

### Step 5: 四种结果判定

| 结果 | 条件 | 后续动作 |
|:----|:-----|:---------|
| ✅ **全票通过** | 三方一致同意 | 记录结论，直接进入工作分配 |
| ⚠️ **CONDITIONAL PASS** | 存在保留意见但可接受 | 引入灰度层，设定验收条件 |
| 🔄 **退回重做** | 方案存在重大缺陷 | 退回设计阶段，附具体修改要求 |
| ❌ **否决** | 一方行使否决权 | 主持人记录否决理由，上报Owner裁决 |

### Step 6: 四方会签归档

| 签署方 | 签署条件 | 状态 |
|:------|:---------|:----:|
| 墨萱 | 技术合规确认 | ⬜ |
| 玄知 | 架构+数据一致性确认 | ⬜ |
| 墨涵 | 知识归档确认 | ⬜ |
| Owner | 业务方向确认 | ⬜ |

争议处理：若墨萱/玄知拒签，不得绕过，必须解决后再走会签。

### Step 7: 归档

- 辩论结论摘要 → `docs/07_research/{议题}/辩论总结_会签_v1.x.md`
- 会签记录 → `docs/05_protocols/archive_signoff/{议题}_signoff_owner.md`
- 知识条目激活 → 若辩论产生可固化的insight，由墨涵激活knowledge_entries

## 七、适用范围

- 📋 方向选择争议
- 📋 资源分配争议  
- 📋 方案优先级争议
- 📋 Owner需加速决策时
- 📋 技术路线分歧（否决制触发场景）
- 📋 回测冲突结论需统一方向时

## 八、辩论后标准工作流

```
辩论结束
   ↓
主持人汇总结论（共识/分歧/否决状态）
   ↓
定工作顺序（优先级+Owner+里程碑）
   ↓
结果判定（通过/CONDITIONAL PASS/退回/否决）
   ↓
四方会签（墨萱→玄知→墨涵→Owner）
   ↓
归档 docs/07_research/{议题}/
```
