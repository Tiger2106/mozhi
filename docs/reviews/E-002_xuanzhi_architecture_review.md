## 一、架构设计（§2）— ✅ 通过，2 条建议

### 1.1 单入口 UnifiedIngestionPipeline 架构 — ✅ 合理

架构设计合理，分层清晰。核心判断：

- **Step 0→1→2→3→4→5 的串行流水线设计**符合单一职责原则，每个步骤职责正交
- **QCEngine 与 staging 逻辑分离**：校验与写入分离，QC Engine 只输出 verdict，staging 只负责根据 verdict 路由
- **E-001 作为可选增强**的"插拔式"设计合理，不增加 P0 核心复杂度

### 1.2 Staging 层 + batch_commit 原子写入 — ✅ 合理

**核验关键点**：

| 设计要点 | 核验结果 |
|:---------|:---------|
| 批次级 COMMIT（非逐行） | ✅ 正确。`_staging_commit` 中遍历全部行后统一 `conn.commit()`，异常时 `conn.rollback()` |
| QC REPORT 行写入 staging_raw + 主表 NULL | ✅ 正确。staging_raw 存完整 JSON 可追溯，主表写 NULL 保证查询面上可感知 |
| INSERT OR REPLACE 幂等性 | ✅ 正确。符合批量重灌场景 |

**建议 1（低优先）**：考虑将 staging_raw 的 `raw_json` 字段从 TEXT 改为外部压缩文件路径。当前设计单行存完整 JSON，单只标的单日数据约 2-3KB，全量重灌 16 只 × 5 年 ≈ 16×1200×3KB ≈ 58MB，可接受。但如果未来扩展到 500+ 标的，TEXT 存储效率会下降。建议在 P0 实现阶段先保持 TEXT，P2 时评估是否需要压缩归档。

### 1.3 batch_audit 审计日志表设计 — ✅ 合理

| 表 | 核验结果 |
|:---|:---------|
| batch_audit（主表） | ✅ 字段完整，batch_id UNIQUE、global_verdict、qc_config_hash 等关键字段齐全 |
| batch_audit_detail（明细表） | ✅ 逐日逐规则记录非 PASS 项，FOREIGN KEY 引用正确 |
| staging_raw | ✅ 与 E-001 复用同一表名 |

**建议 2（低优先）**：batch_audit_detail 中 `check_name` 字段建议改为 CHECK_NAME_ID（引用 QC 规则字典表），避免规则名称变更导致历史数据歧义。P1 阶段做。

### 1.4 与 E-001 的关系定义 — ⚠️ 需要修正（P1）

E-002 §1.3 的定义有两处不准确：

**问题 A**：E-001 的定位描述为"多源交叉验证系统"，但 E-001 实际是**完整的数据录入管道**（`E001IngestionPipeline`），包含 fetching → unit normalization → cross-validation → write → audit 全流程。E-002 的描述"E-002 是主框架，E-001 是其可选子模块"与实际不符——E-001 本身就是一个独立、完整的灌入程序。

**问题 B**：E-002 §2.1 架构图中标注"⬜ 可选: E-001 cross_source_validator"，但 E-001 的输出不是"增强 QC 置信度"，而是直接输出写入结果（包括自己写入 stock_daily 的能力）。E-002 调用 E-001 不应理解为"调用一个 validator 模块"，而应当是"E-002 作为主写入者，E-001 作为多源仲裁写入的替代路径"。

**建议修正**：在 §1.3 的关系定义中增加明确描述：

```
E-001 与 E-002 的关系修正：

1. E-001 是独立的数据录入管道（含多源获取 → 单位归一化 → 交叉验证 → 自写入 → 自审计）
2. E-002 是 E-001 的上层调度：当 enable_e001=False 时走 E-002 自有 QCEngine；
   当 enable_e001=True 时，E-002 调用 E-001 的执行结果，用 E-001 的验证结论替代 E-002 自有 QC
3. 两套审计表（validation_audit_log vs batch_audit）各自独立，E-002 的 batch_audit 是新增，不替代 E-001 审计
4. staging_raw 表两套程序共用，需确保写入逻辑不冲突
```

## 五、执行计划（§7）— ⚠️ 有条件通过（P1）

### 5.1 P0→P1→P2 串行顺序 — ✅ 合理

依赖链清晰：P2 需要 P0 的 batch_audit 表写入能力，P1 需要 P0 的 QCEngine 输出。串行顺序无误。

### 5.2 工时估算 — ⚠️ 有一条风险项

| 子任务 | 估算工时 | 评估 |
|:-------|:--------:|:-----|
| symbol_registry.yaml + SymbolRegistry | 0.5h | ✅ 合理 |
| QCEngine 13 条规则 | 1.5h | ⚠️ 存在低估风险 |
| staging 原子写入 | 0.5h | ✅ 合理 |
| batch_audit DDL | 0.5h | ✅ 合理 |
| audit log 写入 | 0.5h | ✅ 合理 |
| UnifiedIngestionPipeline 主类 | 0.5h | ✅ 合理 |
| E-001 集成点 | 0.5h | ⚠️ 取决于 E-001 §1.3 关系修正进度 |
| 测试 | 0.5h | ⚠️ 明显偏低 |

**风险评估**：

**风险 1（P1）**：QCEngine 13 条规则的 1.5h 估算是**仅写校验函数**的工时，不包括：
- 与真实数据联调（首次运行必然发现边界情况）
- adj_factor 语境感知检查（§3.4）的调试 — 涉及多日上下文的窗口滑动逻辑
- 涨跌幅检查中 `pre_close` 缺失时的降级逻辑测试

建议调整为 **2-2.5h**，或者标注"1.5h 为基础实现，额外 0.5-1h 为边界调测储备"。

**风险 2（P1）**：测试 0.5h 偏低。13 条规则 × 至少 PASS/REPORT 两种路径 = 26 个测试用例，加上 staging 写入路径（3 种 verdict）× 边界行数 = 30+ 用例。建议调整为 **1h**。

**总体工时重新估算**：

| 优先级 | 原始估算 | 调整后估算 | 差距 |
|:------:|:--------:|:----------:|:----:|
| P0 | 4-5h | **5-6.5h** | +1.5h（QCEngine + 测试）|
| P1 | 2h | 2h | 不变 |
| P2 | 3h | 3h | 不变 |

### 5.3 并行验证期 — ✅ 设计合理

2 个月并行验证期的设计合理。核验：

- ✅ 新旧程序写入不冲突（新程序以只读模式执行）
- ✅ 一致率阈值 99.9% 明确
- ✅ 延期机制（第 60 天未通过 → 延长 1 个月）

**安全边界的确认**：staging_raw 表在两套程序间共用，但不冲突。旧脚本不写 staging_raw，新程序写但老脚本不会读取。一致。
