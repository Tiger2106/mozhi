## 隐患 1：Signal Protocol 变"超级对象"

**裁决**: ✅ **全面接受**

### 承认错误

> Owner 说得对，我在 extras 的预留键上过于宽松了。
> 
> 第6条建议（quality.* / trace.parent_signal_id）应该连同红线一起提出，而不是只加预留键。只放行不进红线 = 边界在扩张。

**当前问题的根源**：§1.2 extras 使用规范的"v1 预留键"列表没有配合**禁止域**定义。只说了"可以放什么"，没明确"绝不可以放什么"——而人类架构经验告诉我们，`extras` 总是倾向于吸收所有未被显式禁止的数据。

### 修改方案

#### 修改 1a：§1.2 extras 使用规范 — 增加"禁止域"章节

在现有"3. 禁止行为"之后增加第4条：

```
4. **v1 禁止域**（以下4类字段不得出现在 extras 中，违反视为协议违规）：

   - ❌ **业务状态回写**：当前持仓、当前PnL、风控余额、订单状态、账户权益
     理由：Signal 是 Research 的一次性分析产出，不是 Trading 的状态回传通道。
     业务状态应通过 Trading 系统内的状态管理模块追踪，而非嵌入 Signal。

   - ❌ **大体积分析对象**：dataframe（序列化后 > 1MB）、K线序列（含OHLCV）、
     因子矩阵（超过 100×100）、原始行情快照
     理由：大体积对象应通过 knowledge.db 或信号存储层引用，而非嵌入 extras。
     如需要引用，使用引用 ID 替代原始数据。

   - ❌ **非幂等字段**：会被 Consumer 或下游系统动态修改后回写的字段、
     同一 signal_id 在不同时间点可产生不同值的字段
     理由：Signal 的 extras 应保证**时间不变性**——同一 signal_id 在任何时间点
     反序列化都产生相同的 extras 内容。非幂等字段会破坏审计链和重放能力。

   - ❌ **execution-only 字段**：broker_order_id、fill_price、slippage_real、
     realized_pnl、order_status、exchange_timestamp
     理由：执行层细节属于 Trading 系统内部记录，不应进入 Research 产出的信号中。
     执行结果在 `consumed_{signal_id}.json` 中记录，而非 Signal 本身。

**例外（允许进入 extras 的合理使用场景）**：

   - ✅ `ml.feature_importances`：ML 模型因子权重（体积小、幂等、研究侧属性）
   - ✅ `factor.pe_ratio`：因子解释值（研究侧分析结果）
   - ✅ `quality.predicted_price`：信号质量分数（非 PnL 回写，是研究侧预测值）
   - ✅ `trace.parent_signal_id`：信号引用追踪（固定值、幂等）
   - ✅ `analyst.notes`：研究员注释（人工标注、幂等）
   - ✅ `risk.annotation`：风险标注（研究侧评估，非风控执行结果）

**判断准则"三问规则"**（适用于边界案例）：

   问自己三个问题，全部回答"否"才可以放进 extras：
   1. ❓ 这个字段的值会随交易执行结果变化吗？
   2. ❓ 这个字段的值的完整性 > 1MB 吗？
   3. ❓ 这个字段只在 Trading 侧有意义，Research 侧看不懂吗？

```

#### 修改 1b：§1.5 序列化/反序列化验证规则 — 新增 V-13

在现有 V-12 之后追加：

```
| V-13 | extras 禁止域 | 检查 extras 中是否包含 v1 禁止的 4 类字段
                                     （业务状态 / 大体积对象 / 非幂等字段 / execution-only 字段） | 严格模式下 **拒绝**，
                                                                                                    宽松模式下 **警告+日志** |
```

**严格模式** vs **宽松模式** 的区分：
- **v1.0 阶段**：默认使用**宽松模式**（警告+日志），给现有代码迁移缓冲期
- **v1.1 升级**：切换为**严格模式**（拒绝），届时所有 Research 产出必须通过 V-13

#### 修改 1c：§5 附录 B JSON Schema — 更新

在 `extras` 属性的 JSON Schema 中增加 `"description"` 标注禁止域：

```json
"extras": {
  "type": "object",
  "default": {},
  "description": "扩展字段。禁止包含：业务状态回写、大体积分析对象(>1MB)、非幂等字段、execution-only字段。详见 §1.2。"
}
```

### 关于 Owner 的担忧

> Owner 说：否则会把 Trading 和 Research 糊回一起

**完全认同**。当年 SignalBridge 双向耦合就是这么死的——Signal 里夹带了 OrderRequest 的字段，OrderRequest 反向引用了 Signal 的字段。extras 不加红线，半年后必然重演这个历史。

**§1.3 direction + confidence 映射表的仓位权重字段不受此限制**，因为那是"语义映射建议"，不是"业务状态回写"——它不记录实际执行了什么，只解释 Research 的意图权重。

```
## 七、知识分类与 TTL 策略（v1 决议）

### 7.1 动机

knowledge.db 将承接 Research 侧信号生成过程中的所有中间数据、因子，
以及 Trading 侧消费后的归档信号。如果不做分类管理，3 个月后 knowledge.db 
将同时包含"每日运行快照"和"半年期的历史研究"，查询性能下降且存储成本
线性增长。

### 7.2 知识等级分类

所有写入 knowledge.db 的数据必须附带等级标签，由数据生产者负责归类。

| 等级 | 标签 | 含义 | 示例 |
|:-----|:-----|:------|:------|
| **transient** | `T0` | 临时研究 | 参数扫描中间结果、单次回测日志、调试输出 |
| **operational** | `T1` | 运行数据 | 日频 Signal 原始日志、Consumer 消费记录、仓位快照 |
| **reusable** | `T2` | 可复用知识 | 已验证的因子值、策略绩效归因结果、信号质量评分 |
| **strategic** | `T3` | 长期资产 | 策略核心参数、协议版本归档、审计所需完整轨迹 |

### 7.3 差异化 TTL 策略

| 数据类型 | 等级 | TTL | 到期行为 |
|:---------|:----:|:---:|:---------|
| 回测失败日志 | T0 | **30 天** | 自动清除（仅保留 `task_id` + `failure_reason` 摘要到审计表） |
| 参数扫描中间结果 | T0 | **30 天** | 自动清除 |
| 调试输出 / Profiling 数据 | T0 | **7 天** | 自动清除 |
| 日频 Signal 原始日志 | T1 | **90 天** | 自动归档至冷存储 |
| Consumer 消费记录 | T1 | **90 天** | 自动归档至冷存储 |
| 仓位快照 | T1 | **180 天** | 自动归档至冷存储 |
| 已验证因子值 | T2 | **永久保留** | 标记为不变版本 |
| 策略绩效归因 | T2 | **永久保留** | 标记为不变版本 |
| 信号质量评分 | T2 | **永久保留** | 标记为不变版本 |
| 策略核心参数 | T3 | **永久保留** | 版本化管理（可追溯历史变更） |
| 协议版本快照 | T3 | **永久保留** | 版本化管理 |
| 审计证据 | T3 | **180 天** | 到期前离线备份，备份保留 5 年 |

**TTL 执行规则**：

1. **数据写入时**：生产者必须指定等级标签 `classification: "T0" | "T1" | "T2" | "T3"`，否则默认标记为 `T0`
2. **日终清理任务**：扫描所有到期数据，执行清除或归档
3. **T2/T3 的版本化**：T2/T3 数据在第一次写入后视为"已发布版本"，
   后续变更必须产生新版本号（`v1 → v2`），不得原地覆盖
4. **清除不可恢复**：T0/T1 到期清除后不可恢复。
   如数据生产者认为某 T1 数据需要延长保留期，可在到期前主动升级等级
5. **TTL 重载**：若遇到监管要求、审计请求等特殊情况，
   可通过知识库管理员手动重载单条数据的 TTL

### 7.4 等级升级/降级流程

- **升级**：T0 → T1（发现临时研究中有价值结果）、
  T1 → T2（验证某个运行模式具有复现价值）
- **降级**：T2 → T1（发现可复用知识已过时）、
  T1 → T0（确认运行数据无审计价值）
- **审批**：升级/降级需经 墨涵（知识管理负责人）确认，
  若涉及协议或策略参数（T2/T3 间转换）需经 墨衡 或 墨萱 会签

### 7.5 与 Signal Protocol 的衔接

Signal 本身的数据属于两个等级：
- `signals/incoming/{date}/` 下的原始信号 → **T1（operational）**
- `signals/archive/{date}/` 下的归档信号 → **T2（reusable）** （已验证的信号记录）

Signal 的归档路径在 §3.2 ARCHIVED 中已描述，
TTL 按本节的 T2 规则执行（永久保留）。
```

## 隐患 3：缺"架构委员会"

**裁决**: ✅ **全面接受**

### 承认错误

> Owner 说得对，现在没有明确的架构裁决权归属。
>
> 我在当前架构中分别写了：墨萱定义质量门 → ✅ 已签 / 墨涵确认知识归档 → ✅ 签 / 墨衡起草协议 → ✅ 已签 / Owner 确认方向 → ⏳ 待签。
> 但"当墨衡和墨萱就某个字段该不该进 extras 产生分歧"时，没有仲裁机制。大家都签字各自的维度，但没有"最终裁决者"凝聚方向。

### 架构委员会的方案

#### 3a. 委员会定位

**架构委员会不是管理层，是技术仲裁机构。** 它不管理团队日常，只解决团队边界模糊时的架构决策冲突。

#### 3b. 组成与角色

| 角色 | 成员 | 席位 | 表决权 | 说明 |
|:----:|:-----|:----:|:------:|:------|
| **主席** | Owner | 1 | 1 票 + **一票否决权** | 业务方向最终把关 |
| **架构执行代表** | 墨衡 | 1 | 1 票 | 代表技术可行性与执行成本 |
| **质量门代表** | 墨萱 | 1 | 1 票 | 代表兼容性、测试覆盖、质量门禁 |
| **战略方向代表** | 玄知 | 1 | 1 票 | 代表长期路线图与扩展性 |
| **知识/产品代表** | 墨涵 | 1 | 1 票 | 代表知识管理和产品视角 |

**关于墨衡的定位**：

> Owner 问：墨衡作为主线执行方，是否可以成为委员会的一员而非委员会本身？

✅ **接受——墨衡是委员会成员，不是委员会本身。**

原因：
- 墨衡是架构的执行者，不是架构的拥有者
- 让执行者充当唯一的裁决者 = 没有人能审阅执行者的选择
- 墨衡在委员会中代表"技术可行性"和"执行成本"的声量，
  但决策需要多数票通过，不能墨衡一人说了算

**类比**：墨衡是"首席架构师"（有话语权），不是"架构独裁者"（无制衡）。

#### 3c. 裁决权范围

委员会只裁决以下事项（有限枚举），日常事务各 agent 自行决策：

| 可裁决事项 | 裁决门槛 | 谁可发起 |
|:-----------|:--------:|:---------|
| Signal Protocol 的新增字段是否被接受 | 简单多数 | 任何成员 |
| extras 红线范围的争议案例 | 简单多数 | 墨萱（质量门发现违规时） |
| 知识分类是否合理（等级争议） | 简单多数 | 墨涵（知识管理负责人） |
| 两个 agent 之间就设计原则无法达成一致的 | 简单多数 | 争议双方 |
| ADR 提案的批准/驳回（Owner 已审批的战略级除外） | 2/3 多数 | 提案人 |
| Level 边界争议（某功能是否超出当前 Level） | 简单多数 | 墨衡（scope 管理者） |
| 协议版本是否需要 MAJOR 升级 | 简单多数 | 墨萱（兼容性发现时） |

**不在委员会裁决范围内**的事项（各 agent 自行决策）：
- 日常执行顺序调整（墨衡自行决定）
- 测试用例的增删改（墨萱自行决定）
- 知识库日常维护操作（墨涵自行决定）
- 战略路线图的初稿方向（玄知自行决定）

#### 3d. 决策流程

```
触发：任一成员发起裁决请求（通过 ADR 提案或会议议题）
  │
  ▼
形式化：发起人填写《架构裁决请求表》（10 行以内的 Markdown）
  │
  ▼
通知：将请求表推送至委员会群（24 小时内至少 3 名成员回应视为生效）
  │
  ▼
讨论：各成员发表意见，以"理由 ≥ 结论"为原则（禁止无理由的 "支持/反对"）
  │
  ▼
投票：Simple Majority（4 票中 ≥3 票 = 通过）
      特殊情况可申请 Owner 一票否决权介入
  │
  ▼
记录：决议记入 ADR 档案（格式沿用 ADR 模板 appendix C）
  │
  ▼
执行：墨衡根据决议更新协议/规范/代码
```

#### 3e. 防"协议漂移"保障

委员会机制如何防止 Owner 担忧的"每个人都能改字段"：

1. **契约先行原则**（ADR-004 §4）：任何字段变更必须通过 ADR 流程
2. **委员会裁决权范围**：字段变更进入 extras、Core 字段新增/修改/废弃、
   extras 红线扩展——都需要委员会裁决
3. **每个变更都有 Owner 会签**：ADR 的最后一步需要 Owner 确认
4. **墨萱的质量门**：即使委员会通过了，墨萱仍有权在兼容性测试中发现
   问题并拒绝放行（**质量门独立于委员会决策**）
5. **触发门禁的保护**：
   - V-13 验证（extras 禁止域）
   - TC-01 ~ TC-06 兼容性测试
   - 会签制度（墨萱 → 墨涵 → Owner 三方签）

**简单理解**：委员会决定"能不能改"，墨萱决定"改得好不好"，Owner 决定"值不值得改"——三重保险。
