# 数据录入交叉验证规范 — 墨衡意见

## 0. 预估与置信度

- **预估耗时**：完整方案设计 1-2 小时；编码实施 3-5 小时（含单元测试）
- **置信度**：高（85%）— 核心原则明确，但部分边界细节需会议达成共识后确认

---

## 1. 规矩表述（建议措辞）

> **E-001 多源数据交叉验证规则**：
>
> 凡从两个及以上独立数据源获取的**同义字段**（如 stock_daily.volume 在东财和新浪中均存在），必须在写入前完成：
>
> 1. **单位归一化验证** — 各源数据换算至统一标准单位后，数值量级一致
> 2. **数值交叉比对** — 差异在可接受阈值内（默认 ±3%），或存在可解释的非技术性差异（如复权/不复权、前复权/后复权）
>
> 未通过验证的字段**禁止写入核心数据表**，应写入 staging 暂存区并由后续任务人工核查。

**关键措辞理由**：

- "同义字段"排除跨源无对应关系的字段（如不同 API 独有的扩展字段）
- "独立数据源"强调数据源 API 提供商不同（东财 vs 新浪），排除同一个上游的分发节点
- "写入前"明确检查时机在 ETL 环节，不是在消费端

---

## 2. 适用范围

### 必须执行（硬性）

| 类别 | 示例 | 理由 |
|------|------|------|
| 多源数值型时间序列 | volume, amount, open/high/low/close 价格 | 有明确单位约定，交叉比对成本低 |
| 多源计数型字段 | total_shares, negotiable_shares | 单位（股 vs 手）极易混用 |
| 跨货币字段 | 港股/美股换算后的 amount_hkd, usd | 汇率换算易出错 |

### 建议执行（软性）

| 类别 | 示例 | 理由 |
|------|------|------|
| 单源数值字段 | 部分财报衍生指标 | 无法交叉，但应做量级范围检查 |
| 多源文本/枚举字段 | stock_name, industry_category | 数值验证不适用，应做格式归一化 |

### 不适用

| 类别 | 示例 | 理由 |
|------|------|------|
| 内部计算派生字段 | ma_5, macd, rsi | 由上游原始字段推导，源头的正确性已被校验覆盖 |
| 单源、非关键字段 | 某些 API 独有的扩展元数据 | 成本 > 收益 |
| 日志/原始抓取快照 | API 原始返回的 JSON dump | 保留原始值正是为了后续校验 |

**总结原则**：**"多源且可交叉才强制，单源可验证范围，派生不重验"**

---

## 3. 验证标准

### 3.1 多源交叉验证

```
通过条件（满足任一即可）：
  条件A: 所有源归一化后数值差异 ≤ 阈值（默认 ±3%）
  条件B: 存在可解释差异 → 需在写入记录中附带 diff_reason 字段说明原因
          （如"新浪 vol=99999 手 vs 东财 vol=9999800 股 → 差异为 2 股，原因：东财含盘后大宗交易"）

阈值设定规则：
  - 价格类（open/high/low/close）：±0.5%
  - 成交量类（volume/amount）：±3%
  - 市值/股本类：±1%
```

### 3.2 单源合理性检查（降级方案）

```
若仅有一个数据源：
  - 必做：单位显式标注（单位附在字段名或元数据中：volume_shares / volume_lots）
  - 必做：量级范围检查（如 A 股日成交量 < 100 万手或 > 500 亿手 → 标记 WARN）
  - 不做：数值交叉比对（因为没有第二个源）
```

### 3.3 关键设计决定

> **❌ 不建议**要求"必须两个 API 一致"。理由：
> - 不同源有合法差异（复权方式、含不含盘后、延时级别不同）
> - 这会阻止合法数据写入，造成数据空洞
> - 有些字段本身就是单源的
>
> **✅ 建议**：**差异可解释即可通过**。核心是让"无解释的不一致"被捕捉，而不是强行追求数字完全一致。

## 4. 违规后果与补救

### 4.1 防写保护（第一道防线，自动触发）

```
验证未通过 → 自动行为：
  1. 字段数据写入 staging_raw.{table_name}（保留各源原始值）
  2. 在主表对应字段写入 NULL / 标记 'VALIDATION_FAILED'
  3. 在 validation_audit_log 中记录：
     - task_id, source_fields, diff_values, possible_reasons
     - 规则版本号 + 触发的具体规则
  4. 发送 WARN 告警至（配置的告警通道）
```

### 4.2 事后补救（第二道防线，手动触发）

```
发现已写入的错误数据 → 执行流程：
  1. 发出 RECALL 指令（含修正后的正确数据）
  2. 回填脚本修正所有受影响的表（主表 + 下游物化视图）
  3. 在 correction_log 中记录修正的版本号
  4. [可选] 重跑受影响的分析报告

避免的做法：
  ❌ 直接 UPDATE 覆盖 — 丢失原始错误记录
  ✅ INSERT 新版本 + 标记 deprecate_flag — 保留审计轨迹
```

### 4.3 长期改进

```
  - 每次违规写入应触发规则库更新
  - 统计违规频率，判断是否需要放宽/收紧阈值
  - 新接入的 API 源必须在 integration_test 中涵盖交叉验证用例
```

## 5. 执行成本评估

### 5.1 一次性投入

| 项目 | 预估工作量 | 说明 |
|------|-----------|------|
| 维护"字段-标准单位-验证规则"映射表 | 0.5-1h | YAML 配置文件，约 30-50 行核心规则 |
| 通用验证函数 | 2-3h | 支持单位归一化 + 交叉比较 + 阈值检查 |
| staging 暂存表 DDL | 0.5h | 几张 staging_raw_xxx 表 |
| 集成测试 | 1-2h | 覆盖东财 vs 新浪卷对、价格对等场景 |

**合计**：4-7 小时，首次实施一次性工作量

### 5.2 持续成本

| 项目 | 每接入一个 API 源 | 每次日常写入 |
|------|------------------|-------------|
| 额外时间 | ~10-20 分钟（映射表增加一条字段映射） | ~5-10ms（验证函数执行时间） |
| 平均来说 | 中低频（每月 ≤ 2 次新 API 接入） | 几乎无感（微秒级数值比对） |

### 5.3 成本收益比评估

```
正面收益（直接价值）：
  ✅ 消除"手 vs 股"类单位混用 — 本次触发案例
  ✅ 早期发现数据源异常变更
  ✅ 审计轨迹完善，便于排障
  ✅ 提升下游分析报告的可信度

负面影响（成本）：
  ⚠️ 开发初期需维护映射表
  ⚠️ 极少数 "合理差异" 需要人工覆盖
  ⚠️ 测试覆盖率要求提高

结论：纯收益正，建议执行。从 K 线数据关键字段开始实施，非关键字段标记 WARN 暂不阻写。
```

## 6. 实施路线建议（从执行方视角）

```
Phase 1（本周）：核心字段覆盖
  - volume / amount / open / high / low / close
  - 东财 vs 新浪 两组源
  - 单位归一化（股/手 → 统一为 股）

Phase 2（本月）：扩展字段
  - total_shares / negotiable_shares / market_cap
  - PE / PB / 股息率（注意财报日期差异）
  - 加入更多数据源（腾讯、通达信）

Phase 3（下季度）：运营制度化
  - 将验证规则纳入代码 review 检查清单
  - 新增数据源时，必须附带验证规则配置变更
  - 月度数据质量报告
```

---

## 7. 关键提醒

1. **避免过度设计**：核心是"单位归一化 + 数值差异可解释"，不是强行追求两个源一致。
2. **不要堵死写入**：验证失败 ≠ 禁止数据写入，而是写入 staging + 标记 → 待人工确认后提至主表。
3. **阈值可调**：初期宽松（±5%），运行一个月后根据统计分布收紧。
4. **与现有架构兼容**：建议在 ETL pipeline 的 `transform` 阶段插入一个 `validate` 步骤，不改变现有数据流拓扑。
