## 1. 各方案技术可行性评估

### 1.1 墨衡方案（执行方 / 我自己）

**技术可行性：✅ 高**

方案基于对当前代码库的直接审计，每一个模块和问题点都有代码级引证（文件路径、函数名、import 链）。核心改进路径——Signal 标准模型 → 策略重构 → 轻量 Simulator → SignalConsumer — 每一步都有明确的输入输出和前置依赖。

**问题点：**

- **估算偏乐观**：12 天包含 2 天缓冲，但 Phase 2 的"3 天策略重构"在面对 10+ 策略文件时可能低估了深层依赖扫描的工作量。实际可能需要 4-5 天。
- **未充分回应测试需求**：方案写了架构和技术细节，但没有为墨萱的测试体系规划时间。这 12 天是纯开发时间，墨萱的 58h 测试建设是额外工作量。
- **knowledge.db 细节不足**：方案聚焦于 Signal 层，对 knowledge.db 的 schema 设计、版本化迁移、环境隔离着墨较少——这些都是墨萱和墨涵的核心关注点。
- **长期扩展性优先于当前可交付性**：8.1-8.6 的扩展性讨论是好的，但当前最需要的是"确定能做完"而不是"做得漂亮"。

**关键风险**：Phase 2（策略重构）的依赖扫描深度，以及 Phase 5（双系统比对验证）的偏差不可控。

---

### 1.2 墨萱方案（测试/验收视角）

**技术可行性：✅ 高**

测试基础设施的依赖项清晰：conftest.py、fixture、mock、test DB、双 CI pipeline。每一个测试关注点都对应具体模块和接口。这是从"可测试性"出发的最完整方案。

**问题点：**

- **58h（7-8 天）全部是新增工作量**：与墨衡的 12 天无重叠。两者并行时总人力需要 15-16 天，且墨萱的许多测试需要等墨衡完成 Phase 1-3 后才能编写（如 knowledge.db 写入测试需要 schema 就绪后才能写断言）。不能完全并行。
- **对"降级策略可测试"的要求增加了开发复杂度**：要求交易系统在 knowledge.db 不可用时必须有降级行为且可测试——这本身需要设计降级逻辑，不是简单加个测试用例。
- **跨系统 trace_id 机制**：这是一个好的长期需求，但 Phase 0 就要开始做还是可以后期补？如果做，需要统一日志格式 + 请求 ID 生成器，至少额外 0.5 天设计。
- **环境隔离**：`KnowledgeDB(env="test")` 要求是合理的，但需要在 Phase 1 就纳入 common/signal 模块的设计中。

**与我的方案最大的冲突点**：她要求测试驱动开发节奏（先定契约再开发），我习惯任务驱动节奏（先完成功能再过测试）。需要折中：核心接口（Signal 格式、knowledge.db schema）先定契约、后开发；非核心逻辑（报告生成器等）可以后补测试。

---

### 1.3 玄知方案（战略/宏观视角）

**技术可行性：✅ 中高**

架构思路（四层管道+信号总线）是量化平台的标准模式，设计方向正确。但大多数建议落在"战略级"而不是"当前可执行"。

**问题点：**

- **估算偏差最大**：8-12 周 vs 我的 12 天 vs 墨涵的 5-6 天。差异原因很清楚——玄知方案包含了 Signal Store、组合层、总线等额外模块。但作为执行方，我需要澄清：**当前需要解决的问题是什么？** 如果问题是"代码耦合导致无法独立验证信号"，那 8-12 周的方案是过设计的。
- **Signal Store 的概念与 knowledge.db 的关系不明确**：两者都是"研究结果的持久化层"。是二选一？还是 Signal Store 是 knowledge.db 的超级？如果并存，谁来维护接口一致性？
- **"四层管道"中的组合层**：当前只有 4 个策略（trend/reversal/grid/factor），没有多信号融合需求。组合层是前置投资还是可后补延期？
- **对迁移途中断流期的担忧（§9 风险2）正确**，但缓解措施只说了"研究层先上线"——谁来做双轨切换的监控？谁定"切换回旧系统"的标准？这些执行细节缺失。
- **过度设计风险的自我识别（§9 风险1）说明玄知有自知之明**，但识别风险不等于不会犯。

**核心价值**：Signal Store 的概念对长期扩展非常重要，特别是多策略并行、多市场扩展、组合优化。但需要拆解为可执行的最小集：Phase 0 只需要 Signal 的标准序列化格式（文件级），Signal Store（数据库级）可以后延。

### 1.4 墨涵方案（产品/知识管理视角）

**技术可行性：✅ 高（但范围最小）**

5-6 天的方案是所有方案中最激进的，聚焦在最简单的三组件模型上：回测 ↔ knowledge.db ↔ 交易。

**问题点：**

- **scope 太窄**：方案只覆盖了物理拆分 + knowledge.db 建库，没有处理最核心的问题——**策略产出 OrderRequest 而非 Signal（§1.1）**。如果只做墨涵的方案，§1.1 和 §1.4 的致命问题并未解决。
- **Layer Q 对齐的需要 1 天**：但 Layer Q（Q1-Q9）本身不需要改动，只需要确认目录映射关系。0.5 天足够。
- **knowledge.db 4 表设计合理但缺乏接口规范**：strategy_ratings / parameter_recommendations / backtest_failures / strategy_summaries 的表结构需要与 Signal 标准模型对齐——如果 Signal 字段和 knowledge.db 字段不一致，后续会增加转换层。
- **"回测系统独立运行" 2-3 天**的估算中，数据源适配器（akshare/东方财富/本地缓存）如果独立包装，需要考虑网络依赖的 mock 测试——墨萱的工作。2-3 天不包括测试。
- **知识过期管理提及但未指定具体机制**："expires_at + 90天自动清理"谁来做？cron 作业还是查询时过滤？如果是查询时过滤，交易系统的 SELECT 逻辑需要统一加上 `WHERE expires_at > CURRENT_TIMESTAMP`。

**与其他方案的配合**：墨涵的方案可以作为"最小可交付"的起点（Phase -1），在此基础上叠加 Signal 标准化的 Phase 0，再进行策略重构的 Phase 1。5-6 天是**前提条件**而非完整方案。

## 2. 方案之间的冲突点与协调分析

### 🔴 冲突 1：Scope 差异 — 从"物理拆分"到"完整解耦"的连续谱

| 方案 | Scope | 估算 |
|:----|:------|:----|
| 墨涵 | 物理拆分 + knowledge.db | 5-6 天 |
| 墨衡 | 全部 6 Phase 含 Signal 标准化 + 策略重构 | 12 天 |
| 墨萱 | 墨衡方案 + 测试体系 | 12d + 58h ≈ 20 天 |
| 玄知 | 四层管道 + Signal Store + 组合层 + 全面迁移 | 8-12 周 |

**协调方案**：统一到一个 scope 分级模型中，分阶段执行：

```
Level 1（5-6 天）: 物理目录拆分 + knowledge.db 建库（墨涵核心）
Level 2（+5-7 天）: Signal 标准模型 + 策略重构（墨衡 Phase 1-2）
Level 3（+5-7 天）: SignalConsumer + 轻量 Simulator + 双系统验证（墨衡 Phase 3-6）
Level 4（并行）: 测试套件建设（墨萱 58h，对齐到各 Level）
Level 5（延期）: Signal Store + 组合层（玄知扩展）
```

**决策建议**：会议先确认 Level 1 是否就够？还是必须做到 Level 2/3？Level 5 明确延期。

---

### 🔴 冲突 2：knowledge.db vs Signal Store — 知识桥梁 vs 信号仓库

- **墨涵**：knowledge.db 是核心桥梁，4 张表存储策略评级
- **墨衡**：Signal 文件系统 + common/signal_models，knowledge.db 非核心
- **墨萱**：knowledge.db 是集成点，需要严格契约
- **玄知**：Signal Store（信号仓库），范围大于 knowledge.db

**决策需求**：两套数据存储（knowledge.db + Signal Store）是否合并？建议：

- **短期（当前）**：只用 knowledge.db 做策略评级和经验存储。Signal 采用文件系统（墨衡方案）
- **中期（3-6 个月后）**：如有多策略并行需求，将 Signal 文件 + knowledge.db 整合为统一的 Signal Store
- **不搞两套**：避免出现 "knowledge.db 存一套、Signal Store 存另一套"的双轨不一致

---

### 🔴 冲突 3：估算差异 — 谁更接近实际？

各方案估算差异的核心原因是**对"做完"的定义不同**：

- 墨涵的"做完" = 能物理拆开，能独立启动
- 墨衡的"做完" = 信号标准化 + 策略重构强 + 双系统验证通过
- 墨萱的"做完" = 以上 + 自动化测试覆盖
- 玄知的"做完" = 架构级解耦 + 信号总线 + 组合层

**从执行方角度，我的判断**：

1. **墨涵的 5-6 天不准**：因为核心问题 §1.1（策略产出 OrderRequest）不在此 scope 内。物理拆开后，策略仍依赖 OrderRequest → 没有实质改善。
2. **我自己的 12 天低估了测试时间**：若包含墨萱的测试配套，实际需要 15-18 天。
3. **墨萱的 58h 是纯测试工时**：与开发并行时会拉长总工期至 3 周。
4. **玄知的 8-12 周是正确但过度的**：在当前规模（4 个策略，单用户）下，这是 2-3 倍的设计。

**建议统一估算**：Level 1-3 共计 **15-18 天（3 周）**，含测试建设。Level 5（玄知扩展方案）**明确延期**，不纳入本次重构 scope。

---

### 🟡 冲突 4：开发节奏 — 契约先行 vs 功能先行

- **墨萱**：要求接口契约先于实现，测试驱动
- **墨衡（我）**：习惯先实现核心功能，再补测试

**这不是根本矛盾**，可以分层：

- **核心层（Signal 格式、knowledge.db schema）**：先定契约再实现 — 这两个接口被两个系统依赖，改造成本高，必须 upfront 设计
- **非核心层（参数扫描器、报告生成器）**：先实现后补测试 — 这些模块是独立的，改接口不影响上下游
- **中间层（回测引擎、绩效计算器）**：边开发边写测试 — 建议在实现的同时由墨萱同步测试

**折中建议**：
- Phase 0（第一天）：共同定义 Signal 标准模型 + knowledge.db schema
- Phase 1 起：墨衡开发功能，墨萱写测试，**日终跑一次双系统验证**
- 不用"先开发完再测试"，也不用"测试先通过再开发"——**同步推进**

## 3. 从执行方角度的风险加权评估

### 🔴 最高风险：scope 蔓延导致重构搁置

**严重程度：高 | 概率：中高**

我的 Stage 1 发言已经说过：最大风险是"重构做不完"。现在各方案叠加后，总 scope 从墨涵的 5-6 天膨胀到了玄知的 8-12 周。

**核心问题**：会议后的 Action Plan 如果包含所有方案的全部建议，实际执行人（我+墨萱）将在 3 周后仍看不到隧道尽头。

**缓解措施**：
1. 会议必须产出**明确的分级 scope**：当前阶段做 Level 1-3，Level 4-5 明确延期
2. 设定**固定时间盒**：3 周内完成 Level 1-3，不做额外功能
3. 每周 Level Review：每周末检查是否在一个 Level 的范围内。如果 Level 1 已经 7 天还没做完，先停顿、修复、再继续，不叠 Level 2

### 🟠 中高风险：knowledge.db 的成本被低估

**严重程度：中 | 概率：中**

核心原因是**三个 Agent 对 knowledge.db 的预期不同**：

| Agent | 对 knowledge.db 的预期 | 额外成本 |
|:-----|:---------------------|:--------|
| 墨涵 | 4 张表，简单写入读取 | 基准线 |
| 墨萱 | 版本化迁移 + 环境隔离 + 降级策略 + trace_id | +3-4 天 |
| 玄知 | 可能期望与 Signal Store 合并或对齐 | 待定 |

如果墨萱的全部要求被接受，knowledge.db 的建设成本将从墨涵估算的 1 天膨胀到 4-5 天。

**建议**：开会时明确 knowledge.db 的"最小契约"— 仅包含 schema 定义 + 基本读写 + expires_at 过滤。版本化迁移、环境隔离列为"必须做但可以后补"的 item。

### 🟡 中低风险：玄知方案可能过度设计

**严重程度：中 | 概率：低（可主动避免）**

玄知方案的架构思路是好的，但在当前规模下（4 个策略、单用户、日频交易）不适用。重点需要澄清：

- Signal Store：不建。当前用文件系统 + knowledge.db 足够
- 组合层：不建。当前没有多信号融合需求
- 总线：不建。当前只有回测→交易的单一通道

但玄知方案中的**"研究操作系统"理念**值得保留作为长期方向。
