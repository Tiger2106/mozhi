### 维度一：架构分层（评分：8/10）

#### 评估
4层划分（L1数据采集→L2计算引擎→L3决策判定→L4信号输出）+ L0调度层的设计逻辑清晰，依赖方向明确为从上到下：

```
L4 → L3 → L2 → L1 → DataBridge
L0 → 调度触发（不侵入计算引擎）
```

每一层均可独立 Mock 测试，层间传递标准 dict/DataFrame 结构。分层符合"关注点分离"原则。

#### 发现的问题

| # | 严重度 | 描述 |
|:-:|:------:|:-----|
| **A1** | 🔴 **高** | **L0 定位模糊**：SCL 在架构图（§1）中标注为"Layer 0"，§1.2 依赖规则表声明"L0 → 不依赖任何 L1-L4 模块"。但 SCL 的实际实现（§7.5）需要调用 `resonance_pipeline.run()`，而 pipeline 内部组装了 L1-L4 的所有模块。严格来说 SCL 对 L1-L4 有**间接依赖**（编译器/import 层面的依赖）。"不依赖"优于"不侵入"但前者过于绝对。建议修正为"不侵入内部实现，仅通过 pipeline.run() 门面接口调用，且不嵌入调度逻辑"。 |
| **A2** | 🟡 **中** | **RSM 跨层定位**：RSM 放在 L3（判定层），但其核心工作之一是共振强度计算（纯数学公式），属于 L2 计算引擎的职责。RSM 身兼"计算+判定"双重角色。虽然从模块责任角度可以接受（状态机天然需要计算结果），但分层逻辑上建议明确标注：RSM 定位在 L2.5（计算引擎之上、纯判定之下），或拆分为 RSM_Calculator (L2) + RSM_StateMachine (L3)。 |

### 维度二：模块职责（评分：8/10）

#### 评估
8个核心计算模块 + 1调度适配器 + 1管线组装，共10个核心文件~750行代码。模块间通过标准 dict/DataFrame 传递数据，无共享内存状态。每个模块有明确输入/输出/异常处理。

#### 发现的问题

| # | 严重度 | 描述 |
|:-:|:------:|:-----|
| **B1** | 🟡 **中** | **RSM 模块职责过重**（同 A2）：RSM 既承担"共振强度计算"（算法函数），又承担"状态机管理"（NONE→WARN→ACTIVE→DECAY→NONE）。这两个子职责变化频率不同（算法稳定，状态机可能扩充状态），建议在实现层面至少拆为两个内部方法：`_compute_strength()` 和 `_transition_state()`。 |
| **B2** | 🟢 **低** | **DSV 校验逻辑缺乏异常保护**：§2.4 中 Spearman 相关性计算可能在无数据变化时抛出异常（如序列为常数时 spearmanr 返回 NaN）。DSV 模块缺少针对特例（vol/z_turn 序列全相等、长度为0等）的 guard clause。建议补充 `if std == 0: return {"passed": False, "score": 0.0}` 的保护。 |

### 维度三：数据流（评分：6/10）

#### 评估
§3 覆盖了正常流程、日内触发流程、异常流程三种场景，从数据获取到信号生成为止的步骤链完整。

#### 发现的问题

| # | 严重度 | 描述 |
|:-:|:------:|:-----|
| **C1** | 🔴 **高** | **DSV 与 RSM 的上下游顺序在文档中存在多处矛盾**： |

具体矛盾点如下：

| 章节位置 | 描述 | 顺序 |
|:--------|:-----|:----:|
| §3.1 步骤链 | Step 5: DSV → Step 6: RSM | **DSV 先于 RSM** |
| §3.4 数据流图 | RSM → DSV → GKV | **RSM 先于 DSV** |
| §1.2 依赖规则 | "L3 (RSM) → L2 (ZNM, **DSV**)" | RSM 依赖 DSV → **DSV 先于 RSM** |
| §6.1 否决制图示 | GKV 接收来自 RSM + DSV 的并行输入 | **两者并行**而非串行 |

**影响分析**：
- RSM 的实际算法（§2.3）只使用 vol_zscore 和 turn_zscore（来自 ZNM），**不依赖 DSV 的输出**。
- DSV 的实际算法（§2.4）也只使用 z-score 序列作为输入，**不依赖 RSM 的输出**。
- 因此 **DSV 和 RSM 在数据流上是并行关系**，不应有上下游依赖。

**修复建议**：明确两者为**并行计算**关系，仅当 RSM 判定共振后，GKV 才同时引用 RSM + DSV 的结果进行否决判断。§3.1 Step 5/6 可改为：Step 5a: RSM 共振检测，Step 5b: DSV 双源校验（并行），Step 6: GKV 门控。

| **C2** | 🟡 **中** | **Lookback Buffer 的更新时机未在数据流中说明**：LB 是 20 日滚动缓存，§4.4 定义了其结构，但 §3 的数据流步骤中没有"更新 LB"的步骤。每次计算后是否更新？期初是否从 LB 读取历史数据？这些缺失。 |

建议在 §3.1 步骤链中增加：
```
Step 0.5: 从 LB 读取历史数据
Step 9.5: 更新 LB（追加当日结果，裁剪至20日窗口）
```

| **C3** | 🟡 **中** | **RSM 状态机状态持久化机制未定义**：RSM 的 NONE→WARN→ACTIVE→DECAY→NONE 状态转换需要跨多次 run 调用维护状态。文档只定义了 LB 存储 `resonance_state` 字段，但没有说明：（1）如何从 LB 恢复状态到 RSM；（2）多日未触发时状态转换的边界条件。 |

### 维度四：否决制 / CONDITIONAL PASS 位置（评分：9/10）

#### 评估
否决制和 CONDITIONAL PASS 的放置位置精确，层次关系清晰：

```
RSM/DSV → GKV(硬否决) → CPE(灰度放行) → SG(信号生成)
```

§6.3 明确区分了两者的本质："否决是能不能发信号的二元判定，条件放行是发了信号后怎么限制仓位的灰度判定"。墨萱红线（DSV 硬否决）已落实。

#### 发现的问题

| # | 严重度 | 描述 |
|:-:|:------:|:-----:|
| **D1** | 🟢 **低** | **0.6 阈值重复使用无显式协调**：GKV 否决基线 `strength < 0.6` 与 CPE 条件放行 `连续5日共振 > 0.6` 使用了同一阈值 0.6。如果未来某天调整 GKV 的否决阈值，需要同步调整 CPE 的条件放行阈值，但目前文档未用常量/配置参数统一管理。建议将 0.6 提取为配置常量 `RESONANCE_MIN_STRENGTH`，两个模块均引用此常量。 |
| **D2** | 🟢 **低** | **DSV partial=True 通过 GKV 后的行为不够明确**：DSV partial=True 通过 GKV 进入 CPE，但 CPE 的 CONDITIONAL_PASS 要求"双源校验通过"。DSV partial=True 不是"完全通过"，那在 CPE 中是否自动降级到 FULL_PASS？文档未明确说明。建议补充：`DSV partial=True → CPE 自动判为 FULL_PASS（不可触发 CONDITIONAL_PASS）。` |

### 维度五：调度层适配器设计（评分：8/10）

#### 评估
SCL 适配器提供了干净的抽象基类接口（run_once / force_run / on_event），PollingAdapter 是 Phase 0 默认实现，EventAdapter 作为 Phase 1+ 预留。`resonance_pipeline.run()` 明确为纯函数，调度层不嵌入计算逻辑。

#### 发现的问题

| # | 严重度 | 描述 |
|:-:|:------:|:-----:|
| **E1** | 🟢 **低** | **无重试/降级策略**：PollingAdapter._has_new_data() 可能抛出异常；resonance_pipeline.run() 如果中途失败（如网络中断），没有重试或降级机制。建议补充：`max_retries=1, fallback=skip_silently`。 |
| **E2** | 🟢 **低** | **force_run 参数格式未指定**：§7.2 `force_run(date: str)` 未注明 date 的格式（如 "YYYY-MM-DD" vs "YYYYMMDD"）。建议明确参数规约。 |

## 对自检报告的评审

### 自检覆盖评估

自检报告检查了 32 项（辩论约束8 + 架构原则6 + 显式需求10 + 隐式需求8），全部判定 PASS。作为静态检查，自检做得相当细致。

### 自检盲区识别

本评审发现的 C1（DSV↔RSM 顺序矛盾）在自检中未被检出，原因是：

- 自检按**检查项分类**（辩论约束/架构原则/需求）逐项验证，而不是按**跨章节交叉引用验证**
- DSV 在 §3.1 中顺序为 Step 5 → Step 6 RSM，在 §3.4 中图示为 RSM → DSV → GKV，这两个描述恰好相反，但自检没有做"数据流步骤 vs 数据流图"的交叉比对

**建议优化自检流程**：增加一项"跨章节一致性检查"——对同一概念在不同章节中的描述做交叉比对。

## 整体评价

### 优点
1. **辩论约束全覆盖**：6项核心约束全部精确实现，承诺清单（§0）从设计到实现链路清晰
2. **模块分离度高**：8个计算模块职责清晰，文件级解耦，每个模块 ~40-120 行
3. **调度层隔离干净**：SCL 抽象 + 纯函数 pipeline，调度机制可替换
4. **否决制层次合理**：硬门（GKV）→ 灰度门（CPE）的两层结构，墨萱红线位精确
5. **轻架构落地**：~750行/10核心文件，零新增外部依赖，复用现有 DataBridge/信号总线
6. **演进路线清晰**：§9 Phase 0→2 的演进路线明确，每个里程碑可交付

### 需修复的关键问题

**必须修复（C1）**：DSV 与 RSM 在数据流中的上下游关系需要统一为**并行关系**，消除 §3.1 与 §3.4 的矛盾。

**建议修复（C2、C3、A2/B1、A1）**：
- Lookback Buffer 更新 -> 在数据流步骤中补齐
- RSM 状态持久化 -> 定义序列化/反序列化策略
- RSM 跨层 -> 实现层面至少拆为内部子方法
- SCL 表述 -> "不侵入"替代"不依赖"

### 终审结论

```
评审结论: CONDITIONAL_PASS
评分:     7.5/10
条件:     修复 C1（DSV↔RSM 数据流矛盾）后进入实现阶段
建议:     修复 C2、C3、A2/B1 后进入实现阶段（非强制）
```
