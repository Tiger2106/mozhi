## Q1：U1"系统限制" — psutil 解决了还是绕过了？

**结论：psutil 是绕过，不是解决。**

"系统限制"指的是 WSL2 下 OOM 被杀时，系统层面没有留存进程内存快照（没有 core dump，没有 /proc/<pid>/smaps 快照）。这是 Linux 内核级行为的缺失，不是应用程序能直接控制的。

- **psutil 做的事**：在进程死**之前**定期采集内存指标（RSS、VMS、swap 等），写入日志或持久化存储。它在死亡点前保留了"监控视角"的快照，而非"事故瞬间"的内存完整状态。
- **绕过 vs 解决**：它绕过了"系统不主动留存快照"的约束，用应用层轮询填补了监控空白。但它**解决不了**事故时内存中正在处理的临界数据（比如刚好写到一半的 buffer、进程持有的锁状态）。psutil 能告诉你"OOM 前 RSS 涨到了多少"，但无法告诉你"OOM 时 buffer 里有什么待 flush 的写入"。

### WSL2 OOM killer 差异对复现可靠性的影响

**会显著影响复现结果可靠性。** 以下是两个影响点：

1. **WSL2 的 OOM killer 行为更接近 cgroup-level OOM**：WSL2 在中内存压力下倾向于触发 cgroup OOM killer（基于 memory.max/usage 限制），而非系统级 OOM killer。两者的触达路径不同——cgroup OOM 是按 memcg 优先级 kill 进程，系统 OOM killer 走 badness score 打分机制。前者行为更可预测（谁超限杀谁），后者有更多内核调度不确定性。
2. **主机对虚机施加的内存 limit 会干扰**：WSL2 本身受 Windows 主机的内存分配约束。如果主机关掉了 vm.swappiness、开启了 memory reclaim 等，即使 WSL2 内部配置一致，OOM 触发的精确时间窗口和 victim 选择也可能与原生 Linux 环境下不同。

**对复现的影响**：如果复现是在 WSL2 上做，而真实部署是原生 Linux，则"复现失败"并不能排除"原生环境下存在此类问题"的可能性——反之亦然。复现结论对 WSL2 有效，但不可直接推广到原生 Linux 环境。

## Q2：U2"需日志" — 设计缺陷还是执行问题？

**结论：两者兼有，但根因是设计缺陷。**

### 设计缺陷层面

`single source_version`（单一版本字段）意味着写入链路中多个来源共享同一个标识字段。这不是一个"忘了加标识"的问题——如果架构设计时就没有为"区分不同写入来源"预留字段或机制，那就是**架构层次的设计缺失**：

- `source_version` 只记录了一个版本号，但架构中没有规划 `writer_id`、`process_id`、`run_id` 这类追责字段
- 更根本的是，没有设计"写入链路追踪"机制——写入操作没有携带元数据（谁写的、什么时候写的、用的什么参数）
- 这等价于系统里没有一个**写入者签名**的原语

### 执行问题层面

非要说执行问题也可以，但"忘了加标识"本身就是设计未兜底的结果。如果架构强制要求每个写入操作携带写入者身份（类似数据库的 application_name 或分布式系统的 span_id），那么即使执行者"忘了"，基础设施层也会迫使它带上。

### 加上 run_id 重跑成功能否说"U2已确认是可解的"？

**可以，但有条件。** 可以确认的是：

- ✅ run_id 解决了"区分不同写入来源"这个具体问题
- ✅ 原始故障中"无法区分谁写入了什么数据"的**后果**可以被消除

**不能确认的是**：

- ❌ 原始故障的**根因**（OOM 导致写入中断）并没有被修复——只是我们增加了一层写入溯源，使得出了乱子能知道谁干的
- ❌ 如果未来出现新的写入来源（第三个进程），但忘记携带 run_id，问题会再次出现——除非 run_id 的传递是基础设施强制的

所以总结：**U2 的可观测性被解决了（能区分），但鲁棒性没有被解决（根因仍在）。**

## Q3：复现试验的架构风险

新增 `run_id` 列 + 重跑写入新数据的最大担忧，按优先级排列：

### 1. 新增列与历史数据的兼容冲突（P0）

- 如果表结构变更（ALTER TABLE ADD COLUMN），历史数据中该列全部为 NULL——下游消费逻辑是否都兼容 NULL？
- 如果原表设计不允许 NULL（NOT NULL constraint），迁移时需设默认值，默认值可能与实际 run 的含义冲突，需要额外清理标记
- 如果使用动态 schema（如 SQLite），历史数据没有这个列，按`SELECT *` 导出的结果集会缺失该列，影响下游ETL

### 2. run_id 的生成规则与稳定性绑定（P0）

- 重跑时的 run_id 生成规则是什么？时间戳？UUID？有状态计数器？
- 如果重跑时生成规则改变，同一批重跑数据的 run_id 不一致，会引入新的"伪混淆"
- 建议：run_id 必须**确定性派生**（如：复现 session 的 dispatch_id + 重跑轮次），否则"区分来源"的目标会被 run_id 本身的不一致性破坏

### 3. 重跑数据与历史数据的交集风险（P1）

- 重跑写入新数据时，历史数据并没有被清除或标记——意味着同一时间段内存在两套数据：`run_id=original` 和 `run_id=repro`
- 下游如果不感知 run_id 过滤，会同时消费两套数据，造成重复/冲突
- **向后兼容最大伤害**：如果下游已经假设时间戳 + source_version 是唯一键，新增 run_id 后唯一键会断裂

### 4. 业务逻辑层面的一致性（P1）

- 如果原始故障数据是不完整的（写入到一半 OOM），重跑写入的是完整数据——那么这两套数据在语义上就不等价
- 下游用量化策略读取时，"选哪套数据"本身就是个一致性决策，而不是简单的"run_id 过滤"

## Q4："不确定性"的架构价值

### 是否支持"通过复现来消除不确定性"？

**有条件支持。** 复现试验的核心价值是**验证假设的可观测后果**，它能够：

- ✅ 将一个"我们不确定 X 能不能解决"的命题，转化为"按照这个方法做了，我们看到的结果是 Y"
- ✅ 消除不确定性——信>疑的输出，给出决策依据

**但复现不是万能的**，它有天然的局限性：

- ⚠️ 复现环境和生产环境不等价（WSL2 vs 原生 Linux），复现"不出事"不意味着生产一定不出事
- ⚠️ 复现需要完整的输入条件——如果原始故障的 context 无法完全还原（比如 WSL2 的精确内存压力时序），复现结果可能产生假阴性或假阳性

### 更好的方法（补充方案，非替代）

**方案一：系统性故障注入（Fault Injection Testing）**
不依赖"复现一个特定故障"，而是主动构造各种故障场景（OOM、磁盘满、网络断等），观察系统行为，验证监控/溯源/恢复机制是否兜底。这比复现单个 case 覆盖面广得多。

**方案二：事故驱动的架构审查（Postmortem-Driven Architecture Review）**
在复现之前先做架构审查：
- 列出每个不确定项的"决策树"：如果这个不确定性被排除，具体能避免什么？
- 区分"能通过复现消除的不确定性"和"即使复现也无法完全消除的不确定性"（如环境差异）
- 对后者，直接做**防御性设计**（增加监控、引入事务日志、强制写入溯源）而不必等待复现结果

**方案三：形式化验证 + 监控闭环**
- 对"写入来源区分"这类问题，用形式化约束（比如 Schema 强制 writer_id 字段非空）来消除不确定性，而不是依靠复现来确认"原来执行者没忘"
- 对"系统限制"类问题，监控闭环（psutil + 日志 + 报警）是比复现更经济的选择

**建议的优先级**：
```
复现（针对U2可观测性验证）→ 架构审查（暴露环境差异风险）→ 防御性设计（对不可消除的不确定性做兜底）
```
