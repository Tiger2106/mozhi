## 一、规则声明

> **未形成团队共识的问题陈述前，不得进入方案设计阶段。**

---

## 二、背景

### 旧模式的缺陷

```
旧模式：问题 → 墨衡出方案 → 大家评审
```

这个模式的问题：
- 默认墨衡的理解是对的，未经团队验证
- 问题边界未经过团队共同划定
- 其他人只能"补漏洞"
- 容易陷入局部最优，方案越来越复杂但根因没解决

### 重构会议的成功经验

重构会议引入的"Stage 1：问题共识"模式被实践验证有效：
- 先收敛问题边界，再讨论方案
- 团队对"到底要解决什么"形成统一认知
- 方案的收敛速度反而因此加快

**R001 不是发明新流程，是把已经做对的事情固化为默认做法。**

## 三、问题等级制度

为避免流程膨胀，所有任务按以下等级划分：

| 等级 | 判定条件 | 示例 |
|:----:|:---------|:-----|
| **D级** | 涉及架构边界/数据协议/多Agent协作/长期影响 | 技术方案评审、模块间接口设计、新Agent接入 |
| **C级** | 涉及核心模块/回测正确性/数据质量/可复现性 | 回测引擎bug、数据源比对、报告生成器缺陷 |
| **B级** | 单模块修复/日常增量/时间敏感输出 | 单条管线参数调整、日常报表输出修复 |
| **Hotfix** | 线上阻断/数据中断/高风险错误 | 管线崩溃、数据管道阻塞 |

**等级判定由提案人自评，主持人确认。如果等级争议，主持人裁定。**

## 四、核心流程 — 四步问题定义法

### 运作模式

> **提案人出分析，团队负责质疑和确认**，不是每个人独立做四份再拼凑。
> 提案人负责 Step1-3 的分析输出，Step4 由团队共同完成。

### Step 1 — 描述现状（提案人）
- 当前发生了什么？
- 谁受到什么影响？
- **必须有数据或事实支撑，非猜测**

### Step 2 — 分析后果（提案人）
- 如果不解决，会怎样？
- 影响范围多大？紧急性如何？
- **注意：这是后果分析，不是根因分析。团队需在此确认"后果分析已收敛"后才进入Step3。**

### Step 3 — 寻找根因（提案人）
- 表面问题背后的本质原因是什么？
- **至少追问三层**。每层根因必须能回答"为什么上一层会发生"
- 排除第一层表象原因（"时间不够"、"资源不足"通常不是根因）
- **注意团队关系：墨萱独立质疑根因是否正确，墨衡可补充技术验证**

### Step 4 — 统一问题陈述（团队共同完成）
- 提案人抛出结论 → 其他人质疑 → 迭代直到全员认可
- 输出格式（一行话）：
  ```
  问题：[谁]在[什么场景]下，因为[根本原因]，导致[什么后果]
  ```
- **墨涵确认全员认可后，写入任务文档**

**争议裁决**：讨论超过30分钟无法达成共识 → 主持人（墨涵）或组长裁定。进入方案设计时附带"未达成一致的问题，按XX方案执行"的备注。

## 五、按等级的流程深度

| 等级 | 流程深度 | 提交要求 | 检查机制 |
|:----:|:---------|:---------|:---------|
| **D级** | 完整四步 | 提交即填3个标准节点 | **强制交叉复核**（独立方执行）+ 50%抽检 |
| **C级** | 三步（Step3免于正式讨论，提案人自行完成） | 提交即填3个标准节点 | 30%抽检（独立方执行） |
| **B级** | 一步（明确问题陈述即可） | 提交即填**节点1（现状）+ 节点3（问题陈述）**，节点2免填 | 自检 + 10%抽检 |
| **Hotfix** | 先修复，后补定义（见第六章） | 修复前写现象（1-2句，无格式要求） | 见闭环时限 |

### 3个标准节点
1. **一行话问题陈述** — 必须包含"对象+偏差+可观测指标"
2. **自洽性确认** — 根因与后果分析没有混用，数据与结论自洽
3. **风险等级披露** — 如实标注影响范围和紧急性

**模板格式**（标准化要求，单独附件，不堵主流程）：详见 `附件A_R001模板格式_v2.md`

## 六、紧急修复（Hotfix）特殊规则

| 阶段 | 要求 |
|:----|:-----|
| 修复前 | 写现象（1-2句，不设格式要求） |
| 48小时内 | 完成 Step1（现状）+ Step4（问题陈述） |
| 72小时内 | Step2 或 Step3 任意一项有实质内容即可关闭 |

**约束**：
- 热修复的问题定义不能消失在修复完成后——48h和72h两个时间节点均需正式记录
- 72小时节点只要求 Step2 或 Step3 任一项目有实质内容，不强求全部补全
- 逾期未补 → 自动触发复盘

## 七、检查与退回机制

### 检查主体
- 执行方自检（提案人自己完成）
- **独立方复核**（墨萱执行，不可取消）
- 抽检由独立方执行，非无人审计

### D级强制交叉复核
- 复核内容：检查3个标准节点是否存在且自洽
- 复核由墨萱（独立方）执行
- 重大缺陷 → **退回修改（上限2次）→ 仍不通过 → 升级墨涵/主持人仲裁**
- 允许提交者**标注修订继续流通**：退回只针对明显不可修复的缺陷，一般性问题可带标注修订并行推进

### 抽检率（风险加权）

| 等级 | 抽检率 | 说明 |
|:----:|:------:|:------|
| D级 | 50% | 强制交叉复核已覆盖全部，抽检作为冗余 |
| C级 | 30% | 中等风险，关注根因自洽性 |
| B级 | 10% | 低风险，发现系统性漏洞时触发升级 |
| A级 | 0% | 已走交叉复核或非核心任务，按需审计 |

## 八、版本化

**允许带标注修订**。核心原则：所有修订必须可追踪。

规则：
1. 每次修订注明**版本号、修订原因、修订人**
2. 修订后**重新触发展示问题陈述的全员确认**
3. 原定义保留，修订可见，不应静默修改
4. 格式示例：
   ```
   Problem Statement v1  — 原始定义
   Root Cause v2         — 修订根因（修订原因：方案设计中发现遗漏原因）
   Boundary Clarification v3 — 修订边界
   ```

这是有长期价值的资产——它会形成团队的**组织认知演化记录**。

## 十、墨涵职责（问题定义把关者）

1. **启动检查**：每次新任务启动前，检查问题定义是否完成
2. **等级仲裁**：判定提案人自评的等级是否正确
3. **会议主持**：引导团队完成四步流程，防止过早跳到方案讨论
4. **结论记录**：将最终问题陈述写入任务文档
5. **共识确认**：确保所有相关方对问题定义达成一致
6. **逾期追责**：Hotfix 超时未补 → 自动触发复盘
7. **版本维护**：跟踪问题定义的修订历史

---

## 十一、违规处理

| 违规行为 | 处理 |
|:---------|:-----|
| 跳过问题定义直接出方案 | 方案退回，补充问题定义 |
| 问题陈述未获全员认可 | 继续讨论，不进入方案阶段 |
| 两次以上同类问题跳过定义 | 暂停任务，复盘流程 |
| Hotfix 逾期未补定义 | 自动触发复盘 |

---

## 十二、适用边界（补充说明）

- **"日常运维不适用"的范围**：日常运维指 SOP 覆盖范围内的标准操作。运维中发现的系统性问题（如某模块持续异常报告），不属于日常运维，需按对应等级走问题定义流程。
- **报告管线任务**：使用简化版本（免 Step3 正式讨论，提案人自行完成）
