**关键空缺（战略级）：**

1. **market_context 空表**——aggregate_knowledge 无法按市场状态（趋势/震荡/波动）分类聚合，所有知识条目落入 `regime='any'`，失去了按市场状态区分差异的核心价值
2. **knowledge_run_links 空表**——关联表未填充，删除回测记录时知识条目的来源追溯将断裂，外键级联优势无法发挥
3. **下游消费端未接入**——KnowledgeExtractor 和 WeeklyReportExtractor 仍使用内存数据而非 knowledge.db，知识库的"消费"环节尚未打通

### 1.2 数据流闭环完整性

设计的闭环链路：
```
回测 → KnowledgeDB.store_run() → knowledge.db → aggregate_knowledge() → 
knowledge_entries → KnowledgeExtractor → 日报/周报 → 决策 → (反馈回路)
```

当前实现状态：
```
回测 → KnowledgeDB.store_run() → knowledge.db → aggregate_knowledge() → 
knowledge_entries
                                                    ↓ (断裂)
                                              日报/周报（未接入）
                                                    ↓ (断裂)
                                              决策反馈（未设计）
```

**结论：数据流仅完成"存储+聚合"的前半段，"输出→决策→反馈"的后半段完全缺失。** 知识库目前是一个只进不出的"黑盒"——数据入库了，知识也聚合了，但没有被任何日报/周报流程消费，更没有形成"从知识到策略调整"的反馈回路。

## 3. 后续方向建议

### 3.1 下一步最应该做的 3 件事

#### 建议 1 【P0】打通知识库→日报消费链路

**现状：** 知识库有 8 条聚合知识，日报流程中完全没有使用。
**目标：** 早报/日报的"历史对比"部分从 knowledge.db 读取数据。

实施路径：
1. 在 `weekly_extractor.py` 或日报生成器中加入 `KnowledgeDB().query_insights(symbol)`
2. 日报模板增加"历史知识对照"区块（如"过去某标的下网格策略平均夏普为 X.X"）
3. 由墨涵在日报草稿中引用知识条目

**预期价值：** 知识库从"存储"变为"被使用"，价值立现。用户能直观看到知识库的存在意义。

#### 建议 2 【P0】填充 market_context 表

**现状：** 市场状态字段全为 `unknown`，知识聚合无市场维度分类。
**目标：** 通过历史数据或外部 API 回填 market_context，使 `aggregate_knowledge()` 能按市场状态分类。

实施路径：
1. 对接 T6 市场状态分析（优先级最高）
2. 或降级方案：通过 `ta-lib`/`pandas_ta` 对标的代码计算简单的趋势动量（均线交叉）、波动率（ATR/布林带宽度）来标记市场状态
3. 先按「趋势市/震荡市/波动市」三档粗分类填充，后续精细化

**预期价值：** 知识聚合维度从 1D（symbol×strategy）提升到 2D（+market_regime），建议质量翻倍。"相同策略在相同标的不同市场状态下表现如何"——这是战略分析中最有价值的问题。

#### 建议 3 【P1】配置衰减 cron + 激活 knowledge_run_links

**现状：** 衰减逻辑已实现但未接入定时任务；关联表空表。
**目标：** 让知识库具备基本的生命周期管理能力。

实施路径：
1. 为 `daily_maintenance.py` Layer3 配置每日 02:22 cron
2. 在 `aggregate_knowledge()` 中补充 `sync_run_links()` 调用（当前缺失）
3. 回填脚本中增加关联表的历史数据回填

**预期价值：** 知识库进入"自动养护"模式，不再需要人工手动清理过期知识。run_links 的级联删除保护数据一致性。

### 3.2 与现有系统的集成路径

#### 早报/日报系统

```
weekly_extractor.py → 新增 query_insights(symbol, strategy, regime)
    → 返回 knowledge_entries 中的置信度最高的条目
    → 嵌入到 week_reference_{W}.yaml 的 "historical_context" 段

早报模板 → 增加 "历史对比" 区块:
  - "601857 网格策略历史平均夏普: X.X (样本量 N, 置信度 XX)"
  - "相比上次回测，胜率变化: +Y.Y%"
```

**集成点：** `week_reference_{W}.yaml` 新增字段+日报模板更新。

#### 调度器系统

```
scheduler → 调用 daily_maintenance.py --layer 3 (02:20-02:25)
    → aggregate_knowledge() 自动聚合新回测
    → decay_check() 自动标记过时知识
    → 生成 daily_doc_report.md
```

**集成点：** 调度器 cron 配置 + daily_maintenance 对接。

#### Step0 市场扫描

当前 Step0（玄知）输出 `market_scan_*.json` 包含市场状态判断，可以作为 `market_context` 的数据源：

```
Step0 → market_scan_{task_id}.json → 解析 market_regime 字段
    → 回填到 knowledge.db.market_context
    → 供后续 aggregate_knowledge() 按市场状态分类
```

**集成点：** 增量回填脚本，解析 market_scan 中的市场状态。

### 3.3 潜在盲点

#### 盲点 1：知识"重存轻用"的陷阱

当前系统在"存"的方面做得很扎实（六表 DDL、CRUD、聚合、备份），但"用"的方面近乎为零。这是一个值得警惕的模式——知识管理系统的价值 90% 在消费端，10% 在存储端。如果消费端一直不建设，随着回测记录从 689 增长到 2000+，知识库只会变成一个"越来越沉的包袱"而非"越来越智慧的大脑"。

**建议：** 优先建设 1 个最小可用消费场景（如早报中的"历史对照"），让知识库产生可见价值，获得继续投入的动力。

#### 盲点 2：market_context 的空表是"雪上加霜"问题

不仅市场状态表为空，而且当前也没有任何自动填充该表的计划（T6 尚未到位）。这意味着即使日报接入知识库，看到的聚合数据也是"全市场状态混在一起"的，其分析结论的粒度太粗，难以直接指导策略。

**建议：** 即使 T6 未完成，也应实施降级方案（均线+ATR 粗分类），让 market_context 至少有一层粗粒度数据，而不是永远 empty。

#### 盲点 3：知识质量缺乏持续评估

`validity_grade` 是单次回测的，"用历史数据验证知识结论"的环节未设计。一条 `knowledge_entry` 可能基于 5 次回测得到 "high" 置信度，但后续 20 次新回测的实际表现与知识结论不符——系统不会自动修正。

**建议：** 后续版本可考虑 `knowledge_entries` 增加 validity_tracking 字段，记录"出现后、后续回测是否符合预期"，实现知识的自我修正。

#### 盲点 4：SQLite 长期扩展性

当前 689 条回测记录下 SQLite 性能没问题，但设想 12 个月后（预计 2000-5000 条）的 `aggregate_knowledge()` 聚合查询，以及日报/早报频繁读取的 `knowledge_entries`，SQLite 在高频读+低频写的模式下表现尚可，但需关注 WAL 文件增长和 VACUUM 策略。

**建议：** 每月运行一次 `VACUUM;` 和 `PRAGMA wal_checkpoint(TRUNCATE);` 维护数据库健康。

### 通过依据

**设计质量优秀**——六表 DDL 精准对齐设计方案 v2.2，事务保护健全，连接管理规范，代码质量 4/5。

**实施质量可靠**——689 条历史数据回填成功率 100%，三策略即时写入全部验证通过，知识聚合/衰减/备份功能完整。

### 风险提醒

| 风险 | 严重程度 | 触发条件 |
|:-----|:---------|:---------|
| 消费端建设持续滞后 → 知识库成数据孤岛 | 高 | 7 天后仍未接入日报 |

| 知识反馈回路缺失 → 投入产出比难以衡量 | 中 | 3 个月后回顾系统价值时 |

### 总结

知识库系统在"存储层"（墨衡建设）达到了极高的完成度，在"质量层"（墨萱审查）确认了代码稳健性。但**战略层（本次评审）** 指出：知识库的完整价值链 = 存储 + 聚合 + **消费 + 反馈**，当前只完成了前 2 环。这不是墨衡的执行问题，而是系统演进的自然阶段——三阶段路线图本身也是从"建表"到"消费"到"反馈"递进的。

**核心战略判断：知识库值得继续投入，但下一个 sprint 必须聚焦消费端，让知识从"存进去"变为"用起来"。**
