### 1.1 共识确认详情

#### 共识1: 核心方向
> 当前第一优先级不是"堆新策略"或"继续调参数"，而是建设研究可信度基础设施。

**一致程度**: 完全一致
**理由**: 当前网格失效案例（R=0.02）揭示的是系统性的研究质量缺陷——不是某一个参数不好，而是系统本身无法判断"什么结论值得相信"。可信度基础设施将使所有未来策略（威科夫、VWAP、龙虎榜等）都受益。

#### 共识2: 七层（6+P+Q）架构
> 在Phase 4c定义的六层报告结构 + P系列分析工具 之外，新增Layer Q作为研究质量层。

**一致程度**: 经讨论后完全一致
**初始分歧**: 墨衡原提案将Research Confidence Engine放在Layer 6内部，墨涵（及专家第二版评估）要求Layer Q作为独立层。
**统一依据**: 见§2.2详细讨论。

#### 共识3: 三层验证结构
> 所有策略报告必须经过：存在性测试（基础alpha是否存在）→ 稳健性测试（参数平台是否存在）→ 市场状态适配测试（跨Regime是否一致）。

**一致程度**: 完全一致

#### 共识4: 参数地形判别
> 好策略的参数曲面是"岛/高原"（附近参数都赚钱），坏策略的参数曲面是"尖峰"（只有一两个点赚钱）。

**一致程度**: 完全一致

#### 共识5: 置信度评分标准化
> 6维评分 → 复合R值 → A~F六级评级 + 瓶颈维度标识。标准化后不同策略、不同标的、不同时间窗口可横向比较。

**一致程度**: 完全一致

#### 共识6: 不修改P系列
> Q层依赖P系列的输出但不修改其计算逻辑，P系列依然独立运行。Q层是构建在其之上的分析/审计层。

**一致程度**: 完全一致

#### 共识7: 与Phase 4c协同
> 改造方案作为Phase 4c的配套项目，命名"Phase 4c.5 Research Confidence Layer"，纳入同一实施时间线管理。

**一致程度**: 完全一致

## 二、争议点讨论与解决说明

### 2.1 争议点1: MarketStateFilter矛盾 —— TREND_UP才交易 vs 趋势应关网格

#### 问题陈述
当前系统存在一个**架构级矛盾**：
- **MarketStateFilter**（系统层面的交易开关）: 仅当市场为TREND_UP时才允许下单
- **Regime-aware Grid**（专家提案 + 墨涵支持）: 趋势上涨时应关网格
- 矛盾：MarketStateFilter要求"必须TREND_UP才交易"，但如果TREND_UP时网格应关闭，那么网格何时能交易？

#### 根因分析
MarketStateFilter的设计假设是"只有趋势向上时才值得交易"——这源自趋势策略（如TrendFollowSystem）的业务逻辑。但网格策略是均值回归策略，恰好**逆趋势**而赚。用同一套MarketStateFilter同时服务趋势策略和网格策略，必然产生矛盾。

这本质上是一个**策略路由问题**：MarketStateFilter不应该是一个策略无关的"单向门"，而应该根据当前激活的策略类型来路由交易信号。

#### 解决决议（双方同意）

**近期（Phase 0~2，不阻塞可信度体系建设）**:

1. **保留矛盾状态，不做顶层重构**
   - MarketStateFilter的TREND_UP逻辑暂时不变（不影响趋势策略运行）
   - 网格策略在可信度基础设施建设期间维持固定参数模式
   - 在可信度体系建设完成前，不对策略路由层做结构性修改

2. **文档化记录矛盾**
   - 在系统架构文档 `docs/architecture/market_state_filter.md` 中新增章节，记录此矛盾
   - 标记为 "Architectural Debt Item AD-001: MarketStateFilter strategy-unaware routing"

**远期（Phase 3，Regime-aware Grid实现时）**:

3. **MarketStateFilter升级为策略路由器**
   - 不再是一个简单的TREND_UP开关
   - 而是根据当前交易策略类型，动态决定路由规则：
     - 趋势策略（TrendFollowSystem等）→ 路由到TREND_UP环境
     - 均值回归策略（Grid策略等）→ 路由到OSCILLATION环境
     - 混合策略 → 按子策略分别路由
   - 架构模式参考：Strategy Pattern + Router Pattern

#### 实施影响评估
| 维度 | 影响 | 备注 |
|:-----|:----|:-----|
| 可信度评分 | 无影响 | MarketStateFilter改造不影响评分计算 |
| 三层验证 | 无影响 | 存在性/稳健性/市场适配验证与策略路由无关 |
| 报告管线 | 无影响 | 报告只展示结果，不影响路由逻辑 |
| Phase 3时间表 | +0.5天 | 需要额外设计策略路由API |

### 2.2 争议点2: Layer Q与六层结构关系 —— 独立层 vs 集成到Layer 6

#### 问题陈述
- **墨涵（+专家第二版）**：Layer Q应该是**独立的研究质量层**，不属于六层（P/B/S/E/R/I）中的任何一层，而是**横跨所有层的审计层**。
- **墨衡原方案**：将Research Confidence Engine作为**Layer 6（研究层）的核心组件**，认为质量评估是研究功能的一部分。

#### 根因分析
分歧源于对"Layer 6（研究层）"的定义理解不同：
- **墨涵理解**：Layer 6的功能是"**产出**研究结论"（synthesize），Layer Q的功能是"**审计/治理**研究结论"（govern）
- **墨衡原理解**：Layer 6包含"研究与质量评估"两个子功能

经过讨论，双方一致认为：**将质量审计放在同一个层内，会引入"自己审自己"的冲突问题**。研究层负责"产生好研究"，质量层负责"判断研究是否可信"——这两个职能需要分离。

#### 解决决议（双方同意）

**架构方案**：Layer Q作为**横向跨层审计层（Transverse Governance Layer）**

```
┌─────────────────────────────────────────────────────────────┐
│                     Layer Q (Quality)                        │
│             研究可信度治理层（横向跨越所有层）                  │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐      │
│  │ Q1-Q3:   │ │ Q4:      │ │ Q5-Q6:   │ │ Q7-Q8:   │      │
│  │ 基础验证   │ │ Regime   │ │ 容量+时间  │ │ 评级+归因  │      │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘      │
└──────────────────────┬──────────────────────────────────────┘
                       │ 审计所有层
       ┌───────────────┼───────────────┐
       ▼               ▼               ▼
┌─────────────┐ ┌─────────────┐ ┌─────────────────┐
│ Layer 1~5   │ │ Layer 6     │ │ P Series (P1~P8)│
│ (业绩/行为/  │ │ (研究层)     │ │ (分析模块)       │
│  结构/执行/  │ │             │ │                 │
│  风险)       │ │ →产出研究    │ │ →提供数据      │
│             │ │  结论        │ │                 │
└─────────────┘ └─────────────┘ └─────────────────┘
```

**关键设计决策**:

| 设计点 | 决策 | 理由 |
|:-------|:-----|:------|
| Layer Q定位 | **横向跨层审计层**，不是Layer 6的子模块 | 避免"自己审自己"的架构冲突 |
| Layer 6功能不变 | 仍为**研究结论综合分析层** | 将质量审计剥离后，Layer 6功能更纯粹——负责跨层交叉引用、研究结论生成 |
| Q层与六层的关系 | Q层审计结果**附加到**每份P报告和Layer 6报告中 | 遵循"不修改P系列逻辑"共识，Q层输出为报告末段的标准化附加段 |
| Q层输出格式 | 标准化JSON嵌入报告 + 独立Q-layer报告（可选） | 粒度可配置，最小干扰现有管线 |

#### 层级责任边界

| 层 | 核心职能 | 产出 | 角色 |
|:---|:---------|:-----|:-----|
| Layer 1~5 | 基础分析（收益/行为/结构/执行/风险） | 分层指标与结论 | 分析产出者 |
| Layer 6 | 研究结论综合与交叉引用 | 研究报告 | 结论生产者 |
| P1~P8 | 工具化分析 | 分析数据 | 数据提供者 |
| **Layer Q** | **质量审计与治理** | 置信度评分+评级+审计意见 | **审计者/质量门** |

> **一句话总结**: Layer 6负责"研究好不好"，Layer Q负责"研究能不能信"——两者是生产者和审计者的关系，必须分离。

### 2.3 争议点3: 实施顺序 —— 先做评分标准化还是先做ExistenceValidator？

#### 问题陈述
- **墨涵**：ExistenceValidator必须在所有其他模块之前实现。"验证存在"是整个可信度体系的逻辑前提。
- **墨衡**：Phase 0先做评分标准化（基于已有代码重构），Phase 1做ExistenceValidator。

#### 根因分析
这是一个"时间优先级" vs "逻辑优先级"的冲突：

| 视角 | 优先级观点 | 理由 |
|:-----|:----------|:------|
| 逻辑优先级（墨涵） | ExistenceValidator第一 | 验证策略"是否存在"是可信度的根基——如果不存在，评分就没有意义 |
| 时间优先级（墨衡） | 评分标准化第一 | 评分代码已有原型（param_decay），快速产出可见结果；ExistenceValidator需要新构建 |

#### 解决决议（双方同意）

**拆分Phase 0为两个并行子阶段，ExistenceValidator绝对优先**：

```
Phase 0a: ExistenceValidator MVP (0.5天) —— 绝对第一优先级
Phase 0b: 评分标准化 (2.5天) —— 与Phase 0a并行实施
```

**Phase 0a (Day 1): ExistenceValidator MVP**
| 子任务 | 工时 | 说明 |
|:-------|:----:|:------|
| 固定粗参数定义（n_levels=8, width_multiplier=1.5, lookback=20） | 0.1天 | 用已通过的粗参数 |
| 多周期回测执行（3年/5年/全量） | 0.2天 | 直接用回测引擎 |
| Pass/Fail判定逻辑（夏普>0.3 + 交易笔数>20） | 0.1天 | 简单阈值判断 |
| ExistenceGate集成到回测管线（G1门控） | 0.1天 | 在pipeline/main中添加前置门 |
| **关键**: 验证601857网格已实现此验证 | — | **首个Pass案例**

**Phase 0b (Day 1-3): 评分标准化（与Phase 0a并行）**
| 子任务 | 工时 | 说明 |
|:-------|:----:|:------|
| 重构param_decay评分为6维标准化 | 1.0天 | 4维→6维扩展 |
| 创建reliability_scores数据库表 | 0.5天 | schema + 迁移 |
| 实现ConfidenceScoreAggregator v1 | 1.0天 | 复合R + A~F + 瓶颈标识 |
| 集成到P系列报告末段 | 0.5天 | P1/P3/P4/P6标准化段 |

**并行关系验证**：
- ✅ 代码独立：ExistenceValidator是管线前置门，评分标准化是后置分析工具
- ✅ 数据独立：ExistenceValidator不需要评分数据，评分不需要ExistenceValidator输出
- ✅ 团队独立：可由两人（墨衡+墨萱）分别实施

**时间线收益**：
```
墨涵（ExistenceValidator MVP）: Day 1 完成  → Day 2-3 协助评分
墨萱（评分标准化）: Day 1-3 全时
                            ↓
Day 1结束时：ExistenceValidator已就绪（墨涵满意 ✅）
Day 3结束时：评分标准化完成（墨衡满意 ✅）
```

> **决议**: ExistenceValidator的**设计和启动**是Day 1的第一件事，满足"绝对优先"要求。评分标准化作为独立工作线并行推进，不拦路。

### 3.2 Q1~Q8 模块定义

#### Q1: Existence Test（策略存在性验证）

| 属性 | 定义 |
|:-----|:------|
| **核心问题** | 这个策略是否存在基础alpha？从统计上不是噪声？ |
| **输入** | 策略定义 + 固定粗参数 + 多周期数据（3年/5年/全量） |
| **算法** | 固定参数直接回测多周期，检查：① 夏普>0.3 ② 交易笔数>20 ③ 多周期均为正收益 |
| **输出** | `ExistencePass: bool` + `BaselineMetrics: {sharpe, trades, periods_positive}` |
| **门控行为** | 不通过 → 写入FAILED状态，标注"统计无效"，不允许进入参数扫描 |
| **文件** | `src/analytics/layer_q/q1_existence_validator.py` |
| **依赖** | 回测引擎（已有）+ 固定粗参数配置 |
| **实施工时** | 0.5天 |

#### Q2: Robustness Surface（参数稳定性曲面分析）

| 属性 | 定义 |
|:-----|:------|
| **核心问题** | 最优参数附近是否存在稳定平台？还是仅一个尖峰？ |
| **输入** | 参数扫描全结果集合（来自P3分析） |
| **算法** | ① 对每个参数维度计算邻域夏普CV（n_levels±2, width±0.3等）② 检测"参数岛"(低CV，高均值) vs "参数尖峰"(高CV，仅峰值点正收益) ③ 给出Plateau Score[0,1] |
| **输出** | `PlateauScore: float` + `LandscapeType: "PLATEAU"/"PEAK"/"RIDGE"/"PLAIN"` |
| **门控行为** | PlateauScore < 0.3 → G2 Warning（可继续但标记高风险）|
| **文件** | `src/analytics/layer_q/q2_robustness_surface.py` |
| **依赖** | P3报告输出（参数扫描结果数据已存在）|
| **实施工时** | 1.0天 |

**关于LandscapeType的补充说明**：
| 类型 | 特征 | 置信度 |
|:-----|:-----|:-------:|
| PLATEAU（参数岛） | 邻域稳定高收益，CV<0.3 | 高 |
| PEAK（参数尖峰） | 只有一个点高收益，周围为负 | 低 |
| RIDGE（参数脊） | 沿某一维度稳定但不全面 | 中 |
| PLAIN（参数平原） | 全部平坦（接近零收益或全部不佳） | 极低（无alpha） |

#### Q3: Regime Consistency（跨市场状态适配一致性）

| 属性 | 定义 |
|:-----|:------|
| **核心问题** | 策略在不同市场状态下的表现是否一致？是否存在"只在一种状态赚钱"？ |
| **输入** | 回测结果 + Regime时间序列（来自RegimeAnalyzer） |
| **算法** | ① 按Regime切片（TREND_UP/TREND_DOWN/OSCILLATION_LOWVOL/OSCILLATION_HIGHVOL） ② 计算每个切片的Conditional Sharpe + Conditional WinRate ③ 跨切片Sharpe相关性 ④ 极端差异检测（某一Slice Sharpe<0而另一>1） |
| **输出** | `RegimeConsistencyScore: float [0,1]` + `RegimeProfile: {slice: {sharpe, winrate}, ...}` + `RegimeDriftFlag: bool` |
| **门控行为** | 若存在某一Regime切片的Sharpe<-0.5 → 标记为"Regime Bounded"策略（仅在特定状态有效） |
| **文件** | `src/analytics/layer_q/q3_regime_validator.py` |
| **依赖** | RegimeAnalyzer（已有）+ 回测引擎按Regime切片的计算能力 |
| **实施工时** | 1.0天 |

#### Q4: Capacity Stability（资金容量稳定性）

| 属性 | 定义 |
|:-----|:------|
| **核心问题** | 策略在不同资金规模下的表现？是否因容量限制而显著衰减？ |
| **输入** | 回测结果 + 5档资金规模参数（10万/50万/100万/500万/1000万） |
| **算法** | ① 按不同Capital规模重算回测 ② 计算收益/夏普对Capital的弹性系数 ③ 检测衰减拐点 |
| **输出** | `CapacityScore: float [0,1]` + `DecayPoint: float（临界容量阈值）` |
| **文件** | `src/analytics/layer_q/q4_capacity_validator.py` |
| **依赖** | 回测引擎支持Capital参数注入（需小改造） |
| **实施工时** | 1.0天 |

#### Q5: Temporal Stability（时间维度稳定性）

| 属性 | 定义 |
|:-----|:------|
| **核心问题** | 策略在时间轴上的表现是否稳定？是否存在"前期赚钱、后期亏损"的时间漂移？ |
| **输入** | 回测结果的滚动窗口绩效（6个月/12个月滚动） |
| **算法** | ① 6个月和12个月滚动窗口夏普 ② 时间序列趋势检测（Mann-Kendall检验） ③ 最近X个月vs总周期绩效差异 |
| **输出** | `TemporalScore: float [0,1]` + `DriftDirection: "IMPROVING"/"DEGRADING"/"STABLE"` + `RecentDegradation: bool` |
| **文件** | `src/analytics/layer_q/q5_temporal_validator.py` |
| **依赖** | P1报告数据结构（含逐月收益明细） |
| **实施工时** | 0.5天 |

#### Q6: OOS Survival（样本外存活率验证）

| 属性 | 定义 |
|:-----|:------|
| **核心问题** | 策略在样本外表现是否与样本内一致？OOS存活率有多高？ |
| **输入** | Walkforward回测结果（来自P4分析）+ 滚动训练验证结果 |
| **算法** | ① OOS Sharpe / IS Sharpe比率 ② 多轮Walkforward的稳定性 ③ 检测"一窗现象"（只有某一特定窗口OOS表现好） |
| **输出** | `OOSScore: float [0,1]` + `SurvivalType: "CONSISTENT"/"DEGRADED"/"FAILED"` |
| **文件** | `src/analytics/layer_q/q6_oos_validator.py` |
| **依赖** | P4 Walkforward（已有） |
| **实施工时** | 0.5天 |

#### Q7: Confidence Score Aggregator（置信度评分聚合引擎）

| 属性 | 定义 |
|:-----|:------|
| **核心问题** | 综合所有维度，给出统一的可信度评分和评级 |
| **输入** | Q1~Q6的6维评分向量 + 权重配置 |
| **算法** | ① 复合R = Σ(wi × score_i)，所有score_i ∈ [0,1] ② 默认权重：Existence=0.25, Robustness=0.20, Regime=0.20, Capacity=0.10, Temporal=0.15, OOS=0.10 ③ 评级映射：A (R≥0.80) / B (R≥0.60) / C (R≥0.40) / D (R≥0.20) / E (R≥0.10) / F (R<0.10) ④ 瓶颈标识：标记score_i < 0.30的维度 |
| **输出** | `CompositeR: float [0,1]` + `Rating: "A"|"B"|"C"|"D"|"E"|"F"` + `Bottlenecks: [dimension1, dimension2]` + `DimensionScores: {dim: score}` |
| **阈值校准** | 对历史432组回测结果做K-S检验，确定评级的自然分布阈值分位点（反向修正[0.80,0.60,0.40,0.20,0.10]阈值） |
| **文件** | `src/analytics/layer_q/q7_confidence_aggregator.py` |
| **依赖** | Q1~Q6全部产出 |
| **实施工时** | 1.0天（含阈值校准） |

#### Q8: Failure Attribution Engine（失效归因引擎）

| 属性 | 定义 |
|:-----|:------|
| **核心问题** | 当策略评级较低（D/E/F级），失效的根本原因是什么？各维度贡献比例如何？ |
| **输入** | Q7的维度评分 + Q1~Q6的原始数据 |
| **算法** | ① 基于各维度得分与阈值（0.30）的距离，计算"失效贡献比" ② 自顶向下归因：从评级出发 → 瓶颈维度 → 各维度原始数据 ③ 输出结构化的归因链 |
| **输出** | `FailureChain: [{"factor": "窗口过拟合", "contribution": 45%, "detail": "84天窗口仅2笔交易"}, ...]` + `TopSuggestion: string` |
| **文件** | `src/analytics/layer_q/q8_failure_attribution.py` |
| **依赖** | Q7输出 + Q1~Q6原始评分数据 |
| **实施工时** | 0.5天 |

### 6.2 集成接口定义

#### 接口1: Stage 1 (Q → Phase 4c) — 报告末段标准化段

**方向**: Q层输出 → Phase 4c报告生成
**格式**: Q层审计结果以标准化JSON段附加到报告末尾
**实现**: Phase 4c的报告生成管线在报告末尾插入 `{research_confidence: {...}}` JSON块
**时机**: Q层计算完成后（报告生成管线中定位在最后一步）

#### 接口2: Stage 2 (Phase 4c → Q) — 报告产出数据管道

**方向**: Phase 4c产出 → Q层评分
**格式**: Q层读取P系列报告的 `output_summary.json`（大版本重构后）或当前P报告的 `summary` 段
**实现**: Q层模块调用已存档的P报告数据；必要时新增P报告的`export_summary()`方法
**时机**: 报告生成完成后，Q层按需触发

#### 接口3: Stage 3 (Q → 数据库) — 评分持久化

**方向**: Q层输出 → reliability_scores数据库表
**格式**: 结构化JSON存入DB
**设计**: 与Phase 4c共享同一个数据库连接，但独立于Phase 4c的schema

### 6.3 依赖关系时间表

| 阶段 | Phase 4c进展 | Phase 4c.5进展 | 依赖风险 |
|:----:|:------------|:---------------|:---------|
| Phase 0 | 执行中 | 并行推进 | ✅ 无依赖——Q层评分基于现有P系列报告输出 |
| Phase 1 | 持续推进 | 并行推进 | ✅ 无依赖——Q层验证模块独立于报告结构 |
| Phase 2 | 多数模块完成 | Q层集成接口实现 | ⚠️ 低风险——集成接口设计不影响核心评分逻辑 |
| Phase 2后 | Layer 1~6全部上线 | Layer Q作为横向层附加 | 🟡 中等——需要Phase 4c报告的`output_summary.json`格式来对接 |

**风险缓解**: Phase 2中的"Q层+Phase 4c集成接口设计"任务已在两个项目完成前定义，确保接口兼容性有足够的缓冲时间。
