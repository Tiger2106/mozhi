## §1 背景

### 1.1 策略动机
八部门整治政策的执行效果不是线性的。"堵"（查处通道、冻结资金）与"疏"（合规化引导、试点放开）往往同时发生，且力度此消彼长。市场参与者需要在政策博弈中判断当前阶段——是"堵大于疏"还是"疏大于堵"——进而调整资产配置策略。

然而，单一政策信号（如一篇新闻通稿）容易被过度解读。需要一个系统化的监测框架，将零散的政策信号、资金行为、市场反应整合为可量化的"堵疏同步指数"。

### 1.2 要解决的问题
- 政策信号碎片化：不同部门（央行、证监会、金监总局、外汇局）的政策方向可能不一致
- 堵疏力度难以量化：无法简单用一个二进制指标判断"松"或"紧"
- 历史可比性缺失：当前政策力度与历史哪个阶段可比？缺乏参照系
- 跨Agent协同困难：墨萱（基本面）、玄知（政策文本）、墨衡（量化）需要统一的语言

### 1.3 与八部门整治政策的关系
S002 是整个策略系统的**核心监测层**。它是 S001（资金回流情景推演）的数据输入源，也是 S003（政策概率校准）和 S004（灰色资金链路由）在时间维度上的参考基准。

## §2 设计目标

### 2.1 核心功能定位
建立政策堵疏同步性监测体系，通过同步指数、热力图和历史案例库三重工具，实现整治政策执行力度的实时量化与可视化。

### 2.2 预期产出

| 产出项 | 格式 | 说明 |
|:----|:----|:----|
| 堵疏同步指数 | 数值[-1, +1] | +1=完全疏导，-1=完全封堵，0=均衡 |
| 堵疏热力图 | 二维矩阵 | 部门×工具，颜色编码力度大小 |
| 历史相似案例库 | JSON | 当前状态与历史阶段的相似度打分 |
| 三线格式统一视图 | 三线图+文本 | 墨衡/墨萱/玄知三线同步解读 |

### 2.3 使用场景
- **每日早盘**：输出当前堵疏状态总览，作为当日策略基调判断依据
- **政策发布后**：实时更新同步指数，评估新政对堵疏平衡的影响
- **周报/月报**：展示堵疏热力图变化趋势，识别政策拐点
- **Agent同步**：为墨衡（量化）、墨萱（财务）、玄知（政策）提供统一政策量化坐标系

## §3 技术方案

### 3.1 核心算法：堵疏同步指数

**指数构成**：

```
同步指数 SI = Σ (wᵢ × sᵢ)
```

其中：
- wᵢ：各维度权重（动态调整，基于该维度当前敏感度）
- sᵢ ∈ [-1, +1]：维度得分

**五大监测维度**：

| 维度 | 权重范围 | 数据源 | 打分依据 |
|:----|:----|:----|:----|
| 政策信号 D₁ | [0.2, 0.4] | 玄知·政策文本NLP | 措辞严厉程度(-1)~鼓励程度(+1) |
| 资金流监测 D₂ | [0.15, 0.3] | 墨萱·资金面 | 特殊渠道资金净流出(-1)~净流入(+1) |
| 监管行动 D₃ | [0.15, 0.25] | 行政处罚公告 | 惩处频次+力度(-1)~放行频次(+1) |
| 合规成本 D₄ | [0.1, 0.15] | 合规咨询费用/通道费率 | 费率上升(-1)~下降(+1) |
| 市场预期 D₅ | [0.1, 0.15] | 期权隐含波动/舆论情绪 | 恐慌(-1)~乐观(+1) |

### 3.2 堵疏热力图

**结构**：行 = 监管部门（央行、证监会、金监总局、外汇局、公安部等8部门），列 = 政策工具类别（发文约束、现场检查、行政处罚、放行审批、合规豁免、试点开放）

**数据流**：

```
玄知·政策分类器 ─→ 标签化(部门, 工具, 力度)
         │
         ▼
   频率统计 → 归一化 → 热力图着色
         │
         ▼
   每日快照 + 7日滚动均值
```

### 3.3 历史案例库

- **收录范围**：2018年资管新规以来的主要监管周期
- **案例标签**：政策类型、力度值、持续时间、市场影响、后续演变
- **相似度计算**：以当前同步指数 + 热力图 → 余弦相似度匹配历史案例
- **输出**：Top-3 相似历史阶段 + 该阶段后续 1/3/6 个月资金回流轨迹

### 3.4 三线格式统一（墨衡/墨萱/玄知对齐）

**问题**：三个Agent各自从不同视角出具报告，格式不一致导致合并困难。

**统一格式**（简称"三线格式"）：

```yaml
timestamp: <ISO8601>
line:
  - agent: moheng       # 量化线
    title: <一句话标题>
    index_value: <堵疏指数>
    indicators:
      - <关键指标名>: <值>
    direction: <堵 | 疏 | 均衡>
    confidence: <高|中|低>
  - agent: moxuan       # 基本面线
    title: <一句话标题>
    key_data: <关键数据点>
    view: <堵 | 疏 | 均衡>
    confidence: <高|中|低>
  - agent: xuanzhi      # 政策线
    title: <一句话标题>
    key_signal: <关键政策信号>
    view: <堵 | 疏 | 均衡>
    confidence: <高|中|低>
summary: <三线综合判断>
```

### 3.5 Agent协作分工

| Agent | 职责 | 交付物 | 频率 |
|:----|:----|:----|:----|
| 墨衡 | 同步指数计算、热力图生成、三线格式定义 | 同步指数值+热力图+三线主模板 | 每日 |
| 墨萱 | 资金流维度D₂评估、基本面线撰写 | 资金流监测数据+基本面线内容 | 每日 |
| 玄知 | 政策维度D₁打分、政策线撰写 | 政策NLP结果+政策线内容 | 每日 |

## §4 接口协议

### 4.1 内部输入接口

**接口名称**：`inputs/policy_signals`（来自玄知）

| 字段 | 类型 | 说明 |
|:----|:----|:----|
| signals | object[] | 政策信号数组 |
| signals[].dept | string | 发布部门 |
| signals[].type | string | 工具类型 |
| signals[].severity | float[-1,1] | 力度(负=堵,正=疏) |
| signals[].text_hash | string | 去重标识 |
| signals[].timestamp | str(ISO8601) | 发布时间戳 |

**接口名称**：`inputs/fund_flow`（来自墨萱）

| 字段 | 类型 | 说明 |
|:----|:----|:----|
| special_channel_net_flow | float(亿元) | 特殊渠道资金净流量 |
| channel_fee_index | float | 渠道费率指数 |
| regulatory_action_count | int | 近期监管行动数 |
| timestamp | str(ISO8601) | 时间戳 |

### 4.2 外部输出接口

**接口名称**：`outputs/sync_index`

| 字段 | 类型 | 说明 |
|:----|:----|:----|
| sync_index | float[-1,1] | 堵疏同步指数 |
| heatmap | object[][] | 8×6热力图矩阵 |
| historical_match | object[] | Top-3历史相似案例 |
| confidence | enum[高,中,低] | 指数置信度 |
| three_line_report | object | 三线统一格式报告 |

### 4.3 与其他模块接口
- → **S001**：提供 `policy_block_index`（即同步指数取绝对值并映射到[0,1]）
- → **S003**：提供同步指数历史时间序列（用于政策概率校准的时序特征）
- → **S004**：提供渠道级别热力图（用于判断哪些灰色渠道当前受关注程度）

## §5 验证标准

### 5.1 通过率指标

| 指标 | 目标值 | 说明 |
|:----|:----|:----|
| 同步指数方向命中率 | ≥80% | 指数正负方向与事后专家评议一致 |
| 热力图敏感度 | ≥0.7 | 政策发布后24小时内热力图同区域着色变化匹配 |
| 历史案例命中率 | ≥60% | Top-3 推荐案例中包含实际最相似时期 |
| 三线一致性 | ≥75% | 墨衡/墨萱/玄知三线方向同向的比例 |

### 5.2 交叉验证方法
- 每周邀请一位领域专家（独立第三方）对当周堵疏状态进行评议，与同步指数比较
- 使用滚动窗口（每季度重新校准权重 wᵢ）
- 三线不一致时（>1条线方向不同），自动触发"分歧标注"，由墨衡人工确认

## §6 风险与边界

### 6.1 已知局限性

| 风险 | 描述 | 应对 |
|:----|:----|:----|
| 政策信号滞后 | 文件公布后实际执行有延迟 | 引入执行率修正系数 |
| 部门间博弈 | 政策口径不一致时指数失真 | 加权方案取置信度权重而非等权 |
| 样本偏差 | 历史案例库无法覆盖新政策类型 | 标注"低相似度"不输出案例 |
| NLP模型短板 | 玄知政策文本解析可能遗漏隐性信号 | 设置人工复核阈值 |

### 6.2 边界条件
- 当日无任何政策信号输入时，指数沿用上一日值，置信度降一档
- 单个部门严重缺位（连续5日无属于该部门的信号），临时移除该部门维度
- 三线格式中任一条线缺失（对应Agent未交付），合并视图变更为"两线视图"并标注缺失

### 6.3 熔断/降级路径
1. **Level 1**：置信度降至"低"后，指数输出值保留但附加"需人工确认"标签
2. **Level 2**：置信度连续3日"低"，停止向 S001 输送数据，降级为 S002 独立运行
3. **Level 3**：任一Agent连续2日未交付三线内容，触发通知，暂停三线合成为两条
