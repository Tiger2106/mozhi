# Q1：Signal 是什么

当前共识：

> 结构化对象，而非 BUY/SELL 字符串

这是绝对正确的。

因为：

BUY/SELL 只是：

- 最终动作

- execution action

而不是：

- research conclusion

Research 应输出的是：

> “对未来市场状态的判断”

而不是：

> “请立刻下单”

这是非常重要的哲学区别。

---

## 墨衡的核心价值

### core + optional

这是成熟系统标准做法。

否则：

- Signal 会无限膨胀

- 所有策略互相污染

建议：

```python
class SignalCore:
    direction
    horizon
    confidence
    timestamp
    signal_type
```

扩展：

```python
extras = {
    "regime": ...,
    "evidence": ...,
    "factor_values": ...
}
```

这会非常稳。

---

## evidence_hash

这个建议非常专业。

因为：

未来：

- 回测

- 实盘

- Layer Q

- KnowledgeBridge

都需要：

> “这个Signal当时到底依据什么产生？”

evidence_hash 本质是：

> Signal可追溯性锚点

这是研究治理系统的重要基础。

---

## signal_type

这个补充也非常好。

否则：

BUY/SELL 会混淆：

- 建仓

- 加仓

- 减仓

- 平仓

- 风险调整

建议正式枚举：

```python
ENTRY
EXIT
ADJUSTMENT
WARNING
INVALIDATION
```

未来会非常有价值。

# weight 的处理

你们的讨论非常成熟。

“建议仓位”

- 自己映射

这是对的。
