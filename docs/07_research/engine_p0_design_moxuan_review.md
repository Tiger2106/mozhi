## P0-1：T+1 交易延迟 → ✅ PASS

### 设计方案正确性

买入日追踪逻辑正确，设计采用"仅全仓新建时记录 buy_date"策略：

```python
if target > current:
    if current == 0:  # 仅从零到一时记录
        buy_date[t] = di
```

**追踪时序验证**（代码交叉确认）：
- `positions` 在每交易日结束时通过 `positions = new_positions` 更新
- `current = positions.get(t, 0)` 取的是 **交易日开始时** 的持仓量
- 若 `current == 0` 且 `target > 0`，表示当日从零仓重建 → 符合"全仓新建"语义 ✅

### T+0卖出拦截逻辑

```python
if current > target:
    last_buy_idx = buy_date.get(t, -1)
    if last_buy_idx >= 0 and (di - last_buy_idx) < 1:
        new_positions[t] = current  # 保持原仓位，跳过卖出
        continue
```

逻辑验证：
- `(di - last_buy_idx) < 1` 等价于 `di == last_buy_idx`，即同日
- 只有同日买入 → 卖出被拦截 ✅
- 隔日（至少T+1）卖出不受影响 ✅

### 已知简化说明

**简化点：仅追踪最后一次性"从零到一"的买入，不追踪部分加仓。**

实际A股规则：**按批次锁仓**。如有100股（T-1持有），今日再买50股（共150股），今日最多可卖100股（T-1的部分），不可卖50股（今日加的）。

当前实现（全仓级别锁）：
- 若 `current > 0`，加仓买入不会更新 `buy_date`
- T日可卖出全部当前持仓（上限无区分）
- **影响**：对"今日加仓 + 今日卖出"边缘场景放宽了限制，但在此策略中（日频全仓调仓，每日几乎全部换手）极少出现。

**建议**：在代码注释中明确标注此简化。不要求修改。

### 验证标准

✅ 回测交易日志中无同日买卖同一标的记录  
✅ T+0卖出操作被跳过  
✅ 验证通过

## P0-3：分红现金流 → ❌ FAIL（致命）

### 核心问题：方向错误 + 公式错误

**数据验证（实际数据库 `market_data.db` 交叉核对）**：

> 验证方法：对 600519.SH 的 `stock_daily` 表按日期排序，提取所有 `adj_factor` 变化事件，与已知茅台分红日期和金额逐项比对。

**发现：Tushare 的 `adj_factor` 在除息日 **上升**，而非下降。**

```python
# 实际数据（600519.SH，2024年分红）：
# 2024-06-18  adj_factor: 7.8576
# 2024-06-19  adj_factor: 8.0204  ← 上升，非下降！
# 上升比例: 1.020719 (≈2.07%)
```

**所有26次adj_factor变化事件均为上升（ratio > 1.0），无一次显著下降。** 设计文档中描述"`adj_factor`在除权日发生向下跳变"与实际情况完全相反。

### 修正后的正确公式

测试了两种公式，与实际已知分红金额对比：

| 分红日期 | 已知分红 | 正确公式 | 设计公式 |
|:--------:|:--------:|:--------:|:--------:|
| 2016-07-01 | 6.171 | 6.17 ✅ | 42.35 ❌ |
| 2022-12-27 | 21.91 | 21.90 ✅ | 166.71 ❌ |
| 2024-06-19 | 30.876 | 30.88 ✅ | 244.36 ❌ |
| 2025-06-26 | — | 27.59 | 226.63 ❌ |

**正确公式**（已验证匹配已知分红数据）：

```python
div_per_share = close_prev * (1 - adj_factor[t-1] / adj_factor[t])
# 或等价：
div_per_share = close_prev * (adj_factor[t] - adj_factor[t-1]) / adj_factor[t]
```

其中 `close_prev` 为 **除息前一日收盘价**（而非除息日收盘价）。

### 设计公式错误分析

设计公式 `div_per_share ≈ (adj_factor[t-1] - adj_factor[t]) * close[t]` 存在两个错误：

1. **方向完全相反**：adj_factor 上升（`adj[t] > adj[t-1]`），导致 `adj[t-1] - adj[t] < 0` → 公式始终输出负值
2. **缺失除以 `adj_factor[t]`**：差值需要被当前 adj_factor 归一化
3. **使用 `close[t]` 而非 `close[t-1]`**：虽为近似，但除息日close可能已偏离较大

由于错误(1)和(2)同时存在，设计公式的符号和量级都完全错误，会产生 6×~8× 错误放大的无效值。

### 额外问题

**缺少 `load_stock_data` 修改说明**：设计文档中 `compute_dividends` 函数引用 `sd.get("adj_factor")`，但当前 `load_stock_data()` 的 SQL 查询只加载 `trade_date, close, volume`，**未加载 `adj_factor`**。这是设计中的遗漏。

**无法区分现金分红与股票送转**：`adj_factor` 上升同时反映现金分红、送股转增、配股等所有事件。当前公式无法区分，对纯现金分红会**高估**。

### 修复方案修正建议

建议将 `compute_dividends` 修正为：

```python
def compute_dividends(stock_data):
    dividends = {}
    for ticker, sd in stock_data.items():
        adj = sd.get("adj_factor")
        close_arr = sd["close"]
        dates = sd["dates"]
        if adj is None or len(adj) < 2:
            dividends[ticker] = []
            continue
        ticker_divs = []
        for i in range(1, len(adj)):
            if adj[i] is None or adj[i-1] is None:
                continue
            ratio = adj[i] / adj[i-1]
            if ratio > 1.001:  # adj_factor 上升 → 除权事件
                div_per_share = close_arr[i-1] * (1 - adj[i-1] / adj[i])
                if div_per_share > 0.01:  # 过滤噪声
                    ticker_divs.append((dates[i], round(div_per_share, 4)))
        dividends[ticker] = ticker_divs
    return dividends
```

同时需修改 `load_stock_data` 增加 `adj_factor` 字段加载。

### 验证标准建议

PASS条件应为：
1. ✅ 选定标的（600519.SH, 601318.SH, 000858.SZ）的已知分红日期与 computed_dividends 输出日期一致
2. ✅ 计算出的每股分红金额与已知金额偏差 < 5%（以已知数据为基准）
3. ✅ 资金曲线在除息日出现正向脉冲（现金入账）
4. ✅ 10年累计分红金额与年报累计分红基本一致
