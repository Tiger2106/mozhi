# ADR-004: 研究系统与交易系统解耦重构 — 架构迁移决议

**决议编号**: ADR-004
**主题**: 研究系统与交易系统解耦重构的架构迁移方案
**决策日期**: 2026-05-20
**状态**: ✅ 批准（Owner裁决）
**涉及Agent**: 墨衡（执行）、墨萱（测试）、墨涵（迁移治理）、玄知（战略）
**汇总人**: 墨涵

---

## 一、为什么要迁移？

### 1.1 当前系统的根本问题

当前架构中，研究系统（Research）与交易系统（Trading）存在**代码级耦合**，而非逻辑层耦合。具体表现为：

| # | 问题 | 影响 | 严重程度 |
|:-:|:----|:-----|:--------:|
| §1.1 | 策略代码直接依赖交易类型：`strategy.py` import `OrderRequest/OrderSide/OrderType`，超10个文件直接引用交易层类型 | 修改交易层接口必然级联修改所有策略代码 | 🔴 致命 |
| §1.2 | SignalBridge 双向耦合：`signal_bridge.py` 同时进口研究端（IndicatorEngine）和交易端（OrderRequest/OrderSide/OrderType） | 修改任一方向都会击穿另一方向 | 🔴 致命 |
| §1.3 | 回测引擎与交易概念混淆：PositionManager/CapitalManager/FeeModel/SlippageModel 编译在回测引擎（~400行 + 8+子模块） | 回测无法独立于交易上下文运行 | 🟠 严重 |
| §1.4 | 技术分析研究重复：`tech_signal_generator.py`（1100+行）与 `strategies/` 下存在冗余信号逻辑 | 修改信号逻辑需同步手动更新两处，几乎不可能做到 | 🟠 严重 |
| §1.5 | 数据流混乱：Signal/Factor Score/Indicator Value 无标准化序列化输出管道 | 中间产物无法独立验证、回放或被下游系统消费 | 🟡 中 |

### 1.2 不迁移的后果

- 新增策略必然继承 OrderRequest 依赖——**每多一个策略就加固一次耦合**
- 无法独立验证研究结果的正确性——研究层永远需要"带着交易系统跑"
- 交易层接口升级（如订单类型扩展）时，**重构范围无法控制**
- 长期来看，系统复杂度会指数级增长而非线性增长

**结论：当前耦合状态不可持续。迁移不是可选项，是必要条件。**

## 二、为什么选 Level 3 做迁移冻结线？

### 2.1 Scope 分级回顾

论证会中，各Agent提交的方案Scope差异巨大：

| 级别 | 内容 | 估算 | 提出方 |
|:----:|:-----|:----:|:------:|
| Level 1 | 物理目录拆分 + knowledge.db 建库 | 5-6天 | 墨涵 |
| Level 2 | + Signal 标准模型 + 策略重构（剥离交易依赖） | +5-7天 | 墨衡 |
| **Level 3** | **+ SignalConsumer + 轻量Simulator + 双系统验证** | **+5-7天** | **墨衡** |
| Level 4 | + 测试套件建设（对齐各Level） | +60h | 墨萱 |
| Level 5 | + Signal Store + 组合层 + 消息总线 | 8-12周 | 玄知 |

### 2.2 为何不选 Level 1？

**Level 1 是"物理拆分"，不是"架构解耦"。**

>
> 将代码从一个文件夹移到另一个文件夹，不改变 import 链、不改变依赖方向、不改变数据流——这不是解耦，只是搬家。
>

Level 1 完成后，策略仍产出 OrderRequest，SignalBridge 仍双向依赖，数据流仍混乱。3个月后，随着新策略的增加，代码会**以新的目录结构重新耦合到与现在相同的复杂度**。问题只是换了位置，没有解决问题。

**结论**：Level 1 治标不治本。只做 Level 1 等于不做。

### 2.3 为何不选 Level 2？

**Level 2 让 Research 独立了，但系统不完整。**

Level 2 完成后，策略产出标准的 Signal，Research 可以不依赖 Trading 独立运行。但：

- 交易侧没有 Signal 消费层 → Signal 产出后如何进入执行系统？
- 没有轻量 Simulator → 研究员修改 Signal 后如何快速验证效果？
- 没有新旧系统对比 → 怎么确保新系统的结果与旧系统一致？

Level 2 做到了"可产出"，但没有做到"可验证"。没有验证闭环的系统在迁移途中时**最危险的**——你会不知道新系统是否更糟。

**结论**：Level 2 方向正确但风险不可控。没有验证闭环的迁移不应投入生产。

### 2.4 为何选 Level 3？

**Level 3 是"最小完整架构"——首次形成完整链路：**

```
Research → Signal → Consumer → Simulator → Validation
```

每个组件都有明确的职责边界：

| 组件 | 职责 | 输入 | 输出 |
|:-----|:-----|:----|:----|
| Research | 策略分析、信号生成 | 行情数据 | Signal |
| Signal | 标准数据格式 | Research产出 | Consumer消费 |
| Consumer | 信号→订单映射 | Signal | OrderRequest |
| Simulator | 快速验证信号效果 | Signal | 绩效指标 |
| Validation | 新旧系统对比 | 双系统输出 | 偏差报告 |

Level 3 的完整链路保证了：

1. **可独立验证**：研究结果可以在 Simulator 中独立验证，不需要加载交易上下文
2. **可灰度切换**：新旧系统并行运行，Validation 持续对比直到偏差在阈值内
3. **可回滚**：如果新系统偏差超阈值，回切旧系统无影响

**结论**：Level 3 是迁移所需的最小完整闭环。低于 Level 3，系统不完整；高于 Level 3，在当前阶段过度设计。

## 三、为什么不做 Level 5？

### 3.1 优先级判断：Rescue > Expansion

Level 5（玄知方案：Signal Store + 组合层 + 消息总线 + ML集成）属于 **Architecture Expansion**——在现有架构基础上做扩展和增强。

当前系统的需求不是"不够强"，而是"不够对"（正确性问题 > 能力问题）。

>
> 在一个耦合的架构上叠加组合层和信号总线，不会让系统更可维护。它只会让原本需要修复 5 处耦合的问题，变成在修复 5 处耦合的同时还需要维护 3 个新模块。
>

### 3.2 实际需求不匹配

Level 5 的假设前提是：当前系统有**多策略并行、多市场覆盖、多信号融合**的需求。但实际情况：

| Level 5 覆盖 | 当前状态 | 匹配度 |
|:--------------|:---------|:------:|
| 多市场适配器 | 仅A股日频交易 | ❌ 不需要 |
| 组合层 | 仅4个策略，无信号融合需求 | ❌ 不需要 |
| 消息总线 | 仅 Research→Trading 单一通道 | ❌ 不需要 |
| ML 集成 | 无 ML 策略 | ❌ 不需要 |

### 3.3 正确顺序

先做 **Architecture Rescue**（Level 1-3），再做 **Architecture Expansion**（Level 5）。

- Phase 0-3 的目标：**让当前系统能稳定运行、独立验证、可控迁移**
- Q3 再评估：当策略数 > 10 或有跨市场需求时，启动 Signal Store 等扩展

**结论**：Level 5 方向正确但时机不对。现在做属于过度设计，会浪费 2-3倍的时间在当前不需要的功能上。

## 四、为什么契约优先？

### 4.1 当前问题的根源就是"没有契约"

研究系统和交易系统的耦合，本质上不是因为代码放在一起，而是因为**没有定义清晰的接口契约**。

- Signal 没有标准格式 → 研究生产的信号和交易消费的信号格式不一致 → 需要 SignalBridge 做转换
- SignalBridge 的转换逻辑没有正式化 → 成为新的耦合点
- 策略没有明确的输入输出约定 → 策略代码自然延伸到交易层

### 4.2 "边写边改"的历史教训

当前系统的每个模块都是在"先实现再改协议"的模式下开发的。结果：

1. `OrderRequest` 被12个文件 import，导致修改 OrderRequest 需要改12个文件
2. `SignalBridge` 从最初的"简单桥接"膨胀到包含业务逻辑
3. `tech_signal_generator.py` 和 `strategies/` 的重复逻辑源于"没有约定哪套是权威"

### 4.3 契约先行的正确顺序

```
Signal Protocol v1 定义
    ↓
Interface Freeze（接口冻结，非必要时不解封）
    ↓
测试桩开发（基于冻结接口，可并行）
    ↓
功能实现
```

这种顺序保证了：

1. **接口稳定性**：修改接口需要通过正式变更流程（新ADR），而非随意改动
2. **并行开发**：墨衡做实现、墨萱做测试桩、墨涵做knowledge.db接入，三方基于同一份契约独立推进
3. **可验证性**：测试用例的断言与接口定义对应，接口变更直接反映在测试用例上

### 4.4 契约不覆盖全部

并非所有模块都需要契约先行。以下策略已被接受：

| 适用契约先行 | 适用边写边测 |
|:------------|:------------|
| Signal 标准模型（两个系统依赖） | 参数扫描器（独立模块） |
| knowledge.db schema（接口共享） | 报告生成器（无下游依赖） |
| Consumer 接口（输入输出规范） | 绩效计算器（内部逻辑） |

**结论**：核心共享接口必须契约先行。这是防止"信号膨胀、边界回退、隐式依赖重生"唯一可靠的手段。

## 七、知识分类与 TTL 策略

### 7.1 分级定义

| 等级 | 分类 | 内容 | TTL | 存放位置 |
|:----:|:-----|:-----|:---:|:---------|
| **T0** | 系统运行关键数据 | 信号文件、消费记录、current signal mapping | **90 天** | signals/live/、signals/archive/ |
| **T1** | 高价值分析数据 | 因子绩效统计、策略回测结果、验证报告 | **30 天** | knowledge.db / reports/ |
| **T2** | 中等价值参考数据 | 日常分析快照、中间计算结果、标准化日志 | **7 天** | knowledge.db / temp/ |
| **T3** | 低价值/临时数据 | 调试日志、临时试跑结果、冗余存档 | **24 小时** | temp/ 或直接丢弃 |

### 7.2 等级升降级流程

| 操作 | 条件 | 执行者 | 记录 |
|:----|:-----|:-------|:-----|
| **自然衰减（降级）** | T0 → T1 → T2 → T3，到达 TTL 后自动降一级 | 日终归档任务 | 写入 knowledge.db 日志 |
| **主动升级** | 跨两次以上分析被重复引用，可升级一级 | 研究员申请 / 墨涵审查 | knowledge.db tag = `promoted` |
| **主动归档** | 研究员将特定信号/报告标记为高价值，延长 TTL | 研究员手动标注 | knowledge.db tag = `archived` |

### 7.3 与 Signal Protocol 衔接

| Signal 实体 | 适用分类 | TTL | 归档行为 |
|:------------|:--------:|:---:|:---------|
| ARCHIVED 状态信号文件 | T0 | 90 天 | 到 TTL 后移入冷存储 |
| 消费记录 consumed_{signal_id}.json | T1 | 30 天 | 到 TTL 后自动清理 |
| extras 中 `debug.*` 键 | T2 | 7 天 | 归档时自动剥离 debug 数据 |
| extras 中 `ml.trial_*` 键 | T1 | 30 天 | 试运行结束后移入知识库 |

## 八、架构委员会（ARB）

### 8.1 组成

| 角色 | 身份 | 票权 | 说明 |
|:-----|:-----|:----:|:-----|
| **主席** | Owner | 一票否决权 | 业务方向最终确认 |
| **架构执行** | 墨衡 | 1 票 | 负责技术方案可实施性 |
| **质量门** | 墨萱 | 1 票 | 负责兼容性与测试 |
| **战略** | 玄知 | 1 票 | 负责长期方向 |
| **治理/记录** | 墨涵 | 1 票 + 会议记录 | 负责归档与落地监督 |

### 8.2 裁决权范围

1. **Signal Protocol 接口变更** — 对已冻结的 Core 字段做任何新增/删除/类型变更
2. **ADR 层级决策** — 跨越 Level 3 的 scope 变更（新增组件、变更架构方向）
3. **知识分类等级变更** — 跨级升级申请（如 T2 → T0）
4. **重大兼容性争议** — Consumer + Producer 冲突且无法通过 §4.3 降级策略解决

### 8.3 决策流程

```
1. 提案人提交 ADR 草案（含影响分析）
2. 14 天审查期（各委员独立审核）
3. 投票：至少 3 票通过 + Owner 不否决
4. Owner 一票否决仅限"对运行系统存在不可逆风险"的情况
5. Owner 行使否决权需在 24h 内书面说明理由
```

### 8.4 防漂移保障

- **季度回顾**: 每季度检查 ARB 是否超出权限范围
- **否决权节制**: Owner 一票否决使用次数限制在每季度 ≤ 2 次
- **会议记录**: 所有决策必须有书面会议记录（墨涵负责）
- **自动休眠**: 连续两个季度无表决事项 → 委员会休眠 → 有提案时自动激活
- **轮值回避**: 若提案涉及委员自身利益（如玄知提案涉及研究系统重构），该委员在该议题上自动回避
