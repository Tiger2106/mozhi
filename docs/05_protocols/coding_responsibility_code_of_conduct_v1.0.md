# 墨枢编码职责规约（v1.0）

> 基于 2026-05-31 越界事件的实际链条。
> 越界发生 → Review发现真实问题 → 返工修复 → 复核通过。
> 该链条验证了三方各司其职的有效性，本规约将其固化为常态约束。

---

## 三条规则

### 1. 红线：编码不越界

**墨涵（PO/质量控制）不得以任何理由代行编码职责。**

- 编码是墨衡的专属职责范围
- 墨衡不可用时（被kill、超时、无响应）：
  - 暂停推进，上报主人判断
  - 不允许"先写着，后续review补救"
- 唯一的例外：主人明确指令的代行，且：
  - 代行产物必须标注 `origin: fallback_mohan`
  - 后续仍须经过墨衡 Review + 墨萱复核

### 2. 把关：双向代码审查

**所有进入主分支的代码必须经过独立的 Review + 复核。**

- 墨衡写的代码 → 墨萱 QA 复核
- 墨涵写的代码（含 fallback 例外）→ 墨衡 Review → 墨萱 QA 复核
- Review 必须发现具体问题，不得走过场（参见 2026-05-31 墨衡对 daily_basic_collector.py 的审查）
- 发现问题 → 墨衡修复 → 墨萱复核 → 通过后方可合并

### 3. 排期：故障转移方案进下一 Sprint

**玄知提出的三层加固方案（心声探针 + 熔断 + 预检）列为下个 Sprint 的优先事项。**

- 当前 Sprint：Phase 2 编码补完 + Phase 3
- 下个 Sprint：拆分实现方案零（环境预检）、修正1（心跳探针）、修正2（熔断机制）
- 在故障转移机制就绪前，按规则1执行
