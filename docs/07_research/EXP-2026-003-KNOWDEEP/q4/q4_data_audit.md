## 验证项 1: 2018熊市窗口 "insufficient data"

**结论: BUG**

### 现象
Q4报告显示2018熊市窗口（2018-01-01 ~ 2018-12-31）的 `extract_window()` 返回 `{"trading_days": 0, "error": "insufficient data"}`，但C1-RWC equity_curve 中实际包含 **243个2018年交易日**（从20180102到20181228），10只标的全部从2015年起有连续数据。

### 根因
**日期格式不匹配**（`extract_window()` 中的字符串比较问题）:

- `extract_window()` 使用 `BEAR_2018 = ("2018-01-01", "2018-12-31")`（YYYY-MM-DD 格式带连字符）
- equity_curve 中的日期格式为 `"20180102"`（YYYYMMDD 无连字符）
- 字符串比较时：`"20180102" <= "2018-12-31"`  
  在第5个字符位置：`'0'` (ASCII 48) vs `'-'` (ASCII 45) → `'0' > '-'` → **条件失败**
- 因此 2018年所有日期（`2018xxxx`）均大于 `"2018-12-31"`，被过滤器排除

### 为什么2015股灾窗口正常
`CRASH_2015 = ("2015-06-01", "2016-01-31")` 跨年度范围:
- `"20150601" <= "2016-01-31"` 在第4个字符位置：`'5' < '6'` → 条件通过。跨年范围因年份字符的不同避开了格式冲突。

### 修复方向
```python
# extract_window() 中统一日期格式
# 方案1: 将 equity_curve 中的日期转为 YYYY-MM-DD
# 方案2: 将窗口范围转为 YYYYMMDD
# 方案3: 使用 dateutil.parser 或 datetime 对象比较
```

## 验证项 2: 年化波动率 16,631,097%

**结论: BUG**（根因：验证项5的现金双倍扣除问题）

### 现象
C1-RWC 报告的 `annual_volatility_pct = 16,631,097.15%`，显然错误。

### 根因分析
`compute_metrics()` 中的波动率计算:
```python
eq_vals_safe = np.maximum(eq_vals, 1.0)  # 将负数裁切为1.0
daily_ret = np.diff(eq_vals_safe) / eq_vals_safe[:-1]
ann_vol = np.nanstd(daily_ret) * np.sqrt(250.0)
```

由于验证项5的现金双倍扣除BUG，净值曲线在正负值之间剧烈振荡。典型日收益率:

| 场景 | 前日权益 | 昨日权益(裁切后) | 今日权益(裁切后) | 日收益率 |
|------|---------|----------------|----------------|---------|
| 正→正 | 51,175.94 | 51,175.94 | 5,392.78 | -89.46% |
| 负→正 | -375.89 | 1.00 | 198,851.03 | **19,885,003%** |
| 正→负 | 198,851.03 | 198,851.03 | 1.00 | -99.9995% |
| 负→负 | -10,283.63 | 1.00 | 1.00 | 0% |

`np.maximum(eq_vals, 1.0)` 仅缓解了负数分母问题（避免除零），但无法解决振荡驱动的极端日收益率。这些极端值的标准差 × sqrt(250) 产生了数百万%的波动率。

### 正确值应为多少
用基准（等权持有）对比：基准的年化波动率约 25%（同期 A50 蓝筹的实际波动率水平）。C1-RWC 的波动率应接近基准水平。

## 验证项 3: 最大回撤 112.74%

**结论: BUG**（根因：验证项5的现金双倍扣除问题）

### 现象
C1-RWC 报告的 `max_drawdown_pct = 112.74%`，纯多头组合最大回撤不能超过100%。

### 根因分析
`compute_metrics()` 中的回撤计算:
```python
peak = np.maximum.accumulate(eq_vals)
dd = np.where(peak > 0, (peak - eq_vals) / peak, 0.0)
```

计算公式**本身正确**，但输入数据（equity_curve）因验证项5的BUG而错误:
- 峰值: 200,000.00（初始资金，从未超过）
- 最低点: -25,489.01（20150603，由现金双倍扣除导致）
- MDD = (200,000 - (-25,489.01)) / 200,000 = **112.74%**

### 正确值应为多少
基准等权持有组合的最大回撤为 **47.36%**（2015股灾期间）。C1-RWC 的实际回撤应该类似水平，因为持有的10只标的与基准相同。

## 验证项 4: 交易次数与费用一致性

**结论: BUG**（费用数据被正确计算，但交易次数因验证项5的BUG被严重扭曲）

### 数据分析
| 指标 | 数值 |
|------|------|
| 总交易次数 | 22,873 笔 |
| 年均交易次数 | 2,138 笔/年 |
| 日均交易次数(单边) | ~8.5 笔/日 |
| 总费用 | ¥130,060.07 |
| 费用/初始资金 | **65.03%** |
| 平均费用/笔 | ¥5.69 |
| 平均交易金额(推算) | ¥3,671 |

### 一致性分析
- 10只标的 × 日均8.5笔 ≈ 每个标的不到1笔买/卖调整 → 对于每日全仓重平衡策略，日均8.5笔合理
- ¥3,671 平均交易金额 / ¥20K 预期仓位 = 18%仓位调整 → 合理范围
- **关键问题**: 费用¥130,060 远超初始资金¥200,000（65%）。11年累计200K费用意味着平均每年费用消耗3万美元

### 费用累加机制
费用在 `run_portfolio_backtest()` 中按每次买卖计算:
```python
fee = trade_val * FEE_SINGLE  # FEE_SINGLE = 0.00155
```

具体计算逻辑正确。但由验证项5的现金BUG导致的**虚假振荡重平衡**产生了大量不必要的交易:

- 当权益变为负数（如-¥25K）→ 所有持仓被清偿 → 10笔卖出产生 ~¥155 费用
- 次日权益可能变为正数 → 重新建仓 → 10笔买入产生 ~¥155 费用
- 这种全额清仓/重建循环在11年中反复发生

### 低本金阶段的费用
当 NAV 降至接近0（例如 equity=¥1,732）时，系统仍在交易:
- ¥1,732 / 10只 = ¥173/只
- 若股价¥50 → `shares = int(173/50) = 3`股
- 3×50 = ¥150交易额 × 0.155% = ¥0.23费用
- 2年242笔×10只 ≈ 2,420笔 → ~¥500-1,000费用

虽然单笔费用在NAV降低后减少，但交易次数仍维持高位，导致总费用率（相对初始资本）异常高。

## 验证项 5 (根因): `run_portfolio_backtest()` 现金双倍扣除

**结论: BUG** — 这是Q4所有异常指标的**根本原因**

### 位置
`scripts/exp003_knowdeep/run_exp003_q4.py`, `run_portfolio_backtest()` 函数

### 代码流程（错误路径）
```python
# ── 步骤A: 计算分配金额 ──
allocated = sum(target_shares × price)        # 所有目标持仓成本

# ── 步骤B: 先扣分配 ──
remaining_cash = cur_equity - allocated        # 错误：从总权益中扣除了持仓成本

# ── 步骤C: 再按交易扣一次 ──
for t where target > current:
    trade_val = buy_shares × price
    remaining_cash -= trade_val + fee          # 错误：trade_val 已在B中扣除

for t where target < current:
    trade_val = sell_shares × price
    remaining_cash += trade_val - fee          # 错误：trade_val 已在B中计入
```

### 双倍扣除示例
**场景**: 初始¥200K，从空仓到10只标的各建仓¥20K

| 步骤 | 操作 | remaining_cash | 说明 |
|------|------|---------------|------|
| 初始 | - | ¥200,000 | - |
| B | cur_equity - allocated | ¥0 | 正确：将全部资金分配给持仓 |
| C(买入) | -¥200,000 - ¥310 | **-¥200,310** | **错误：交易额被双倍扣除！** |

正确结果应为: ¥200,000 - ¥20,000×10 - ¥310 = **-¥310**（仅费用扣减）

**场景**: 持有10只股票（¥200K市值），调整为9只（¥180K市值，卖出¥20K）

| 步骤 | 操作 | remaining_cash | 说明 |
|------|------|---------------|------|
| 初始 | - | €0 | - |
| B | cur_equity - allocated | €20,000 | 正确：释放了¥20K卖出部分的现金 |
| C(卖出) | +¥20,000 - ¥31 | **€39,969** | **错误：卖出金额被双倍计入！** |

### 对净值曲线的具体影响

C1-RWC equity_curve 显示典型的"高-低交替"振荡模式:
```
20180103: 50,852.67  ← 卖出日(高)
20180104:  5,392.78  ← 买入日(低)
20180105: 51,175.94  ← 卖出日(高)
20180108:  5,517.68  ← 买入日(低)
...
20180808:    -98.44  ← 最低点(负值)
```

这种模式不是实际投资回报，而是现金核算错误的**产物**。

### 修复方向
```python
# 正确方案：以实际现金为基准，仅处理增量交易
remaining_cash = cash  # 使用上一日实际现金余额

# 买入：仅扣除增量买入金额和费用
for t where target > current:
    buy_shares = target - current
    remaining_cash -= buy_shares * price + fee

# 卖出：仅记入增量卖出金额（扣除费用）
for t where target < current:
    sell_shares = current - target
    remaining_cash += sell_shares * price - fee
```

## 验证项 6: 最小交易单位（A股100股/手约束）

**结论: BUG**（严重度较低）

### 现象
`run_portfolio_backtest()` 中:
```python
shares = int(target_value / price)
```

仅使用 `int()` 截断至整数股，但A股实际交易需遵守100股/手的整数倍规则。

### 影响分析
- 当股价较低时（¥20-50区间），`int(20,000/50) = 400`股 = 4手 ✓
- 当股价较高时:
  - 假设 600519.SH（贵州茅台）历史高价¥2,600 → `int(2,000/2,600) = 0`股 ✗（无法买入）
  - 但Q4使用行业权重（全仓10只），实际上各标的权重动态分配
- 2015年时茅台约¥200 → `int(20,000/200) = 100`股 = 1手 ✓
- 当标的权重降低时（某只下跌较多），可能出现非整百股

### 修复方向
```python
# 近似修正方案
shares = int(target_value / price / 100) * 100
# 或更精确的 A 股适配
```
