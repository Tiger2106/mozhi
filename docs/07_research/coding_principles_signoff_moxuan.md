## 三、P-014/P-015 暂不纳入的接受理由

### P-014（断言密度）— 有条件接受

**接受理由**:
1. 当前 13 条原则已覆盖架构、性能、验证、时序四大维度，实施负担已较重
2. 断言密度偏向"实现质量"而非"架构原则"，更适合在实施手册中补充而非编码原则
3. P-009（确定性执行）+ P-010（交易日志审计）+ P-013（运行时检测）的组合已提供了部分错误暴露能力

**有条件接受**:
- 实施阶段必须达成一项最低约定：**所有 calc/ 和 sim/ 模块的函数入口，禁止静默吞异常**
- 如果实施过程中发现因"边界未处理"导致的回归失败，应立即补充断言密度规则
- 建议 P1 修复阶段结束时重新评估此原则的必要性

### P-015（测试覆盖率）— 有条件接受

**接受理由**:
1. 覆盖率门限是质量指标而非编码原则，更适合作为项目的 Code Review 规则或集成测试规范
2. 当前 P-008（回归对比）+ P-011（黄金文件基线）已提供修改层面的验证保障
3. 覆盖率目标的首次设定需要先建立测试基础设施，当前阶段不具备条件

**有条件接受**:
- 约束组合（停牌+T+1、涨跌停+T+1、复牌首日、同日买卖）的集成测试是**必须的**，不应等到覆盖率规则设立才补
- 建议在回归对比框架（P-008 步骤 5）部署完成后，立即补充测试覆盖率规则作为 P2 项

### 关于中间层回归检查点

这是我最不放心的遗漏。P-001+P-012 同步实施意味着"三层分离"和"时间对齐修复"同时改变结果——一旦回归失败，无法快速定位问题归属。Owner 已决定一并做，我尊重此决策。作为补偿措施：

1. **我将**在 P-013 测试套件中构建一个"中间层验证辅助函数"——对比三层分离前后计算层输出的信号矩阵，单独隔离"分层导致的差异"
2. 实施时建议：先做 P-012 时间对齐修复 + P-013 TimeAlignmentGuard，再分离三层（或反之，分两个 PR 提交），而不是一个 PR 内全部做完

## 六、我的保留意见（会签条件）

以下意见不阻塞会签，但要求在后续阶段重新评估：

### 保留1：中间层回归检查点（Priority: 中）

P-008 当前仅覆盖终值回归（L1）和时间序列回归（L2），缺少中间层回归（L3：计算层输出矩阵对比）。**三层分离重构和 P-012 前视偏差修复同步进行时**，终值回归只能回答"结果是否变了"，不能回答"哪一层变了"。

**后续动作**: 我会在 P-013 测试套件中提供中间层验证辅助工具，如有效，建议补充进 P-008 作为 L3 层级。

### 保留2：断言密度规则（Priority: 中高）

我仍然认为缺乏断言密度规则是当前编码原则体系中**最危险的质量缺口**。回测引擎中最隐蔽的缺陷并非架构错误或性能问题，而是"边界条件静默失败"——如交易日少于计算窗口、数据长度为0时的除零隐式传播。当前原则体系没有任何规则覆盖这一风险。

**后续动作**: 建议在 P1 修复阶段结束前，至少就"禁止静默吞异常"达成团队约定。如实施中发现此类问题，应立即补充至原则体系。

### 保留3：数据版本与回测结果的耦合（Priority: 低）

玄知在技术把关中提出的 `data_fingerprint` 建议我认同——但作为 P2 遗留而非当前 P0 阻塞项，因为当前 10 只标的 + 一次性加载的架构下，数据版本漂移的风险可控。
