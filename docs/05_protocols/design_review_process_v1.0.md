# 设计方案评审会流程 v1.0

## 总体结构

```
Step 0（会前）           Step 1      Step 2        Step 3        Step 4         Step 5        Step 6
├── 墨衡拟制方案报告    ─→ 汇报方案 ─→ 技术审查 ─→ 战略审查 ─→ 归档确认 ─→ 汇总纪要 ─→ Owner批准/打回
├── 墨涵入场检查               ↑       │  无P0      │  无重大风险                 │            │
└── 路径合规才进入               └── P0返回 ←─────┘ 重大风险返回 ←───────────────┘            ↓
                                                                                        结论入库
```

## 各步骤说明

### Step 0：会前准备（墨衡 + 墨涵）

**墨衡产出：**
- 方案报告（覆盖 §1 背景/§2 设计目标/§3 技术方案/§4 接口协议/§5 验证标准/§6 风险与边界/§7 实施计划）
- 产出文件清单（*.py、*.json schema、README.md 等）
- 开源/依赖声明

**墨涵入场检查：**
- 方案报告文档存在且章节完整（§1~§7 无空白）
- 产出文件路径已声明（输出目标目录、文件名规范）
- README.md 已更新（接口协议、依赖关系、主建人标注）
- ✅ 全部合规 → 通知全员进入会议
- ❌ 不合规 → 退回墨衡补全，补全后重新检查

### Step 1：墨衡汇报方案（15分钟/模块）

- 陈述设计目标与核心逻辑（5分钟）
- 关键技术路径与依赖关系（5分钟）
- 接口协议与输出规范（3分钟）
- 风险与边界说明（2分钟）

### Step 2：墨萱技术审查（10分钟/模块）

**核验范围：**
1. 技术可行性——方案在现有框架内可实施
2. 数据依赖——所需数据源已覆盖
3. 接口完整性——输入/输出字段定义完整无歧义
4. 性能边界——计算量预估不超限
5. 回测兼容——与现有回测引擎对接无冲突

**判定：**
- ✅ **无P0问题** → 进入 Step 3
- ❌ **有P0问题** → 返回 Step 1，墨衡修改后重新汇报

**P0问题定义（触发返回条件）：**
| 类型 | 判定标准 | 示例 |
|:----|:--------|:-----|
| 技术不可行 | 方案依赖的技术在当前框架内不可实现 | 需突破性算法但无实现路径 |
| 数据断供 | 核心输入数据源不存在且无法获取 | 依赖付费API无订阅、字段无上游 |
| 接口崩溃 | 输入/输出字段与上下游Agent接口冲突无法调和 | 字段名冲突、类型不兼容 |
| 性能不可接受 | 单次执行时间超过模块允许时间上限 | 全标的全频扫描超过30分钟 |

### Step 3：玄知战略审查（10分钟/模块）

**核验范围：**
1. 业务逻辑自洽性——方案前提假设与市场认知一致
2. 模块间稳定性——本方案输出不与其他模块产生矛盾信号
3. 风险敞口可控——极端情景下不会输出错误信号
4. 知识积累价值——方案产出值得被系统长期记录

**判定：**
- ✅ **无重大风险** → 进入 Step 4
- ❌ **有重大风险** → 返回 Step 1，墨衡修改后重新汇报

**重大风险触发条件：**
| 类型 | 判定标准 | 触发阈值 |
|:----|:--------|:--------|
| 逻辑矛盾 | 方案核心假设与已有回测结论/市场认知方向相反 | 无合理解释 |
| 信号冲突 | 本方案输出信号与另一已上线方案信号方向相反 | 未设优先级裁决规则 |
| 极端风险 | 在边界条件下方案输出决策反直觉 | 未定义熔断/降级路径 |
| 知识冲突 | 方案产出与现有 knowledge_entries 中条目矛盾 | 偏差超出置信区间 |

### Step 4：墨涵归档确认（5分钟/模块）

确认最终产出路径：

| 确认项 | 内容 |
|:------|:-----|
| 产出文件路径 | 方案报告 → `docs/10_strategies/{模块}/` |
| 知识条目入口 | 哪些环节产出触发 knowledge_entries draft |
| 触发机制 | 依赖哪个 cron/pipeline 步骤执行 |
| 负责Agent | 主建人和协作人标注清晰 |

### Step 5：墨涵汇总纪要（会中整理，5分钟）

输出标准格式会议纪要：

```markdown
## 设计方案评审会议纪要

### 会议信息
- 日期：YYYY-MM-DD
- 评审模块：S00X-模块名称
- 主建人：XXX
- 参会：墨涵（主持）、墨衡、墨萱、玄知、Owner

### 评审结论
- 墨萱技术审查：PASS / FAIL（含P0问题清单）
- 玄知战略审查：PASS / FAIL（含重大风险清单）
- 墨涵归档确认：PASS / FAIL

### 修改要求
| # | 问题描述 | 提出人 | 类型 | 责任人 | 要求完成时间 |
|:-:|:--------|:------|:----|:------|:----------|

### Owner裁决
- 评定：批准 / 打回
- 意见：
```

### Step 6：Owner批准或打回（2分钟/模块）

- **批准** → 墨涵更新 version_status = "评审通过"，实施计划启动
- **打回** → 标记 version_status = "已终止"，原因归档，按失败归档流程

## 退回规则

| 循环次数 | 操作 | 说明 |
|:-------:|:----|:-----|
| 第1次 | 打回墨衡修改 | 说明P0/重大风险问题，要求重做 |
| 第2次 | 打回墨衡修改 | 仍未通过，加注具体修改要求 |
| 第3次 | **告警Owner介入** | 设计层面问题，需Owner裁决而非继续修改 |

**不代修原则：** 墨涵/墨萱/玄知发现问题时，只打回、不代修。墨衡的产出质量问题由墨衡自己修复。
