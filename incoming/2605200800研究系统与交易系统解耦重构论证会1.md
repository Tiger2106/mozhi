“Stage 1：问题共识”是整个论证会最关键的一步。  
很多技术讨论失败，不是因为没人会设计，而是：

> 大家讨论的其实不是同一个问题。

有人在解决：

- 工程复杂度

有人在解决：

- 回测性能

有人在解决：

- 研究自由度

有人在解决：

- 实盘一致性

最后方案一定混乱。

所以：

# Stage 1 的目标只有一个：

# “统一病因定义”

不是讨论方案。

不是讨论代码。

不是讨论谁对。

而是：

# 确认：

# “当前系统到底哪里出了问题？”

---

# 一、你这次会议的 Stage 1 应该怎么开

建议：

# 会议主持人（你）

先宣布：

```text
今天 Stage 1 禁止直接提出架构方案。

只讨论：
1. 当前系统的核心问题
2. 问题造成的后果
3. 哪些问题最致命
4. 哪些问题必须优先解决
```

这句话非常重要。

否则墨衡会直接开始讲架构。

墨涵开始讲知识治理。

玄知开始讲未来。

会议会立刻发散。

---

# 二、Stage 1 的正确结构（推荐）

建议分成：

# 4轮

---

# 第一轮：现状描述（What）

目标：

# 不带方案，只描述问题。

---

## 你可以这样提问：

```text
请每位成员回答：

1. 当前系统最大的结构问题是什么？
2. 最影响研究效率的问题是什么？
3. 哪个问题会在未来放大？
4. 你最不满意哪里？
```

---

# 正确回答示例（墨衡风格）

```text
当前回测系统与交易系统强耦合。

signal 逻辑直接依赖 position/account/order。

导致：
- 新方法开发需要完整交易上下文
- 无法单独验证 signal 质量
- 难以建立统一研究层
```

---

# 错误回答（要制止）

```text
我建议采用事件驱动架构……
```

这是 Stage 2 内容。

要打断。

---

# 第二轮：问题后果（Why）

目标：

# 让大家意识到问题严重性。

---

## 你可以问：

```text
这些问题：
- 目前已经造成什么影响？
- 未来会阻碍什么？
- 哪些研究现在根本做不了？
```

---

# 这一步非常重要

因为：

有些问题只是“不优雅”。

有些问题是：

# “系统演化阻断点”。

比如：

---

## 例子

### “不能单独研究signal”

这不是小问题。

它会导致：

```text
无法建立：
- Layer Q
- Signal质量研究
- Meta Labeling
- 因子评估
- Regime研究
- Ensemble
- AI评分
```

这就是：

# 战略级问题。

---

# 第三轮：问题归因（Root Cause）

这是最关键的一轮。

目标：

# 不讨论表象。

# 讨论根因。

---

## 你应该问：

```text
为什么会变成现在这样？

是：
- 初期快速开发遗留？
- 分层缺失？
- 生命周期混乱？
- signal和execution边界不清？
- strategy职责过大？
```

---

# 这一轮极重要

因为：

如果根因没找对。

后面所有方案都会：

# “头痛医头”。

---

# 例如你当前真正根因可能是：

---

## 根因1：Signal与Execution没有边界

strategy 同时负责：

- 看行情

- 生成信号

- 管仓

- 下单

- 风控

导致：

# “上帝对象”。

---

## 根因2：系统从“单策略脚本”演化而来

而不是：

# “研究平台”

所以：

研究层根本没被正式定义。

---

## 根因3：研究对象不是signal，而是trade

这会导致：

- 无法研究技术分析本身

- 无法验证因子

- 无法做研究可信度

---

# 第四轮：形成“统一问题陈述”

这是：

# Stage 1 最终成果。

必须形成：

# 一份统一病历。

---

# 三、最终要产出什么（非常关键）

建议最终形成：

# 《问题共识文档》

格式：

---

# 1. 当前系统核心问题

例如：

```text
P1：
研究系统与交易系统强耦合

P2：
signal 无法脱离交易上下文独立研究

P3：
strategy职责过大，缺少分层

P4：
回测目标偏向“交易模拟”而非“研究验证”

P5：
Layer Q 缺乏独立审计对象
```

---

# 2. 问题后果

```text
C1：
新方法开发成本高

C2：
无法建立signal质量体系

C3：
难以做多Regime研究

C4：
研究结果难复现

C5：
研究与执行无法解耦演化
```

---

# 3. 根因

```text
R1：
Signal与Execution生命周期未分离

R2：
系统从脚本演化，未经历平台化重构

R3：
研究层缺失正式架构定义
```

---

# 4. 本次重构目标

这一步很关键。

否则后面方案会漂。

例如：

```text
G1：
建立独立Signal Research Layer

G2：
实现研究/交易解耦

G3：
支持Layer Q可信度治理

G4：
支持未来多策略/多Regime研究

G5：
保持旧系统渐进兼容
```

---

# 四、你作为主持人最重要的职责

不是：

“提出最强方案”。

而是：

# “防止讨论失控”。

---

# 五、Stage 1 最容易出现的三个问题

---

# 问题1：过早进入方案讨论

例如：

```text
我们应该用事件驱动……
```

立刻打断：

```text
先不讨论怎么解决。
先确认问题是什么。
```

---

# 问题2：讨论代码细节

例如：

```python
run_backtest.py 应该拆……
```

这太低层。

要拉回来：

```text
这是代码层。
我们先讨论系统层问题。
```

---

# 问题3：问题太泛

例如：

```text
系统太乱了
```

这是无效问题。

你必须追问：

```text
具体哪里乱？
为什么乱？
造成什么后果？
```

---

# 六、我建议你在Stage 1引入“问题评级”

非常有价值。

例如：

| 问题                | 严重度 | 未来影响 | 是否阻断演化 |
| ----------------- | --- | ---- | ------ |
| Signal耦合Execution | S级  | 极高   | 是      |
| 目录混乱              | B级  | 中    | 否      |
| 命名不统一             | C级  | 低    | 否      |

这样：

# 会议不会陷入“所有问题都重要”。

---

# 七、你这次会议真正要完成的事

不是：

# “决定用什么架构”。

而是：

# “统一：

# 我们到底在重构什么。”

这一步做对了。

后面：

- RFC

- 分层

- Layer Q

- ResearchRunner

- Signal Engine

都会顺很多。
