#### 🆕 新增发现

| # | 严重度 | 章节 | 描述 |
|:-:|:------:|:----:|:-----|
| **X-A1** | 🔴 **高** | §1.2 | **依赖声明存在结构性错误：RSM 对 DSV 的依赖为误声明** |

**详细分析**：

§1.2 依赖规则声明：
```
L3 (RSM) → L2 (ZNM, DSV)
```

这一声明表示 **RSM 依赖 DSV**。但验算 RSM 的实际算法（§2.3）：
- RSM 输入 = `vol_zscore` + `turn_zscore`（均来自 ZNM）
- RSM 算法不涉及 DSV 的任何输出（vol_corr, turn_corr, passed）

同时验算 DSV 的实际算法（§2.4）：
- DSV 输入 = z-score 序列（来自 ZNM）
- DSV 算法不涉及 RSM 的任何输出（resonance_state, strength）

**结论**：RSM 和 DSV 是**并行关系**，均以 ZNM 为共同输入源，互无依赖。§1.2 的错误声明可能导致实现时人为引入不必要的串行等待。

**修正建议**：

```
§1.2 应修正为:
L4 (SG) → L3 (CPE → GKV → RSM)    # GKV 依赖 RSM 的结果
L3 (GKV) → L2 (DSV)                # GKV 依赖 DSV 的校验结果（非RSM依赖DSV）
L3 (RSM) → L2 (ZNM)                # RSM 仅依赖 ZNM 的 z-score 输出
L2 (ZNM, DSV) → L1 (DCM, LQM)      # ZNM 和 DSV 均依赖 L1 原始数据
L1 (DCM, LQM) → DataBridge
L0 (SCL) → 不侵入计算引擎
```

> **与 C1 的关系**：墨萱的 C1 揭示了 §3.1/§3.4 中 DSV 与 RSM 的执行顺序矛盾。本评审的 X-A1 揭示了同一问题的**根源**——依赖声明错误。C1 是"症状"，X-A1 是"病因"。

### 维度二：数据流完整性（评分：6/10）

#### 评估
§3 三种流程场景（正常/日内触发/异常）+ §3.4 数据流图的覆盖面基本完整。但数据流在处理**跨时态状态管理**方面存在多处缺失。

#### 确认墨萱发现

| # | 原编号 | 严重度 | 描述 | 本评审 |
|:--|:------:|:------:|:-----|:-----:|

| **C2-✅** | C2 | 🟡 中 | Lookback Buffer 更新时机未在数据流中说明 | ✅ 确认。同时补充：期初读取、期末回写均需在步骤链中明确定义。 |
| **C3-✅** | C3 | 🟡 中 | RSM 状态机状态持久化机制未定义 | ✅ 确认。详见以下 X-B1 补充。 |

#### 🆕 新增发现

| # | 严重度 | 章节 | 描述 |
|:-:|:------:|:----:|:-----|
| **X-B1** | 🟡 **中** | §2.3 / §4.4 | **RSM 状态机缺少"读→判→写"闭环设计** |

**详细分析**：

RSM 状态机（NONE→WARN→ACTIVE→DECAY→NONE）是一个**跨日程的状态机**，其核心需求：

```
每次 run() 调用:
  1. 从 LB 读取 {前一次状态, 持续天数, 历史强度序列}
  2. 根据当日强度 + 历史状态计算状态转移
  3. 写入 LB 更新后的 {当前状态, 持续天数, 新强度}
```

但当前设计：
- §2.3 只定义了当前日的共振检测算法，未说明如何**输入前次状态**
- §3.1 的 10 步流程中完全没有"读取 LB"和"写入 LB"的步骤
- §4.4 LB 存储结构定义了 `resonance_state` 序列，但未定义 RSM 如何与之交互

**影响分析**：如果没有明确的"读→判→写"闭环，RSM 状态机每一次 run() 都相当于"失忆"——它不知道昨天的状态是 WARN 还是 ACTIVE，无法正确地做 `WARN→ACTIVE`（需持续 2 天）或 `ACTIVE→DECAY`（需衰减 3 天）等时序判定。

**修复建议**：
1. §3.1 步骤链中插入：
   ```
   Step 0.5: 从 LB 读取历史状态（前次 state + 持续天数 + 历史强度序列）
   Step 6.5: 将最新状态写入 LB
   ```
2. §2.3 RSM 接口增加 `previous_state` 参数
3. LB getter/setter 设计需支持"读取上个交易日状态"

**与墨萱 C3 的差异**：C3 指出了"状态持久化机制未定义"，X-B1 进一步揭示了"缺少跨日程读→判→写闭环设计"这一更根本的问题。C3 是问题声明，X-B1 是设计重构建议。

| # | 严重度 | 章节 | 描述 |
|:-:|:------:|:----:|:-----|
| **X-B2** | 🟡 **中** | §3.1 | **数据流步骤缺少"期初状态恢复"和"期末状态持久化"操作** |

**具体缺失**：

| 步骤 | 当前 §3.1 | 应补充 |
|:----|:----------|:-------|
| Step 0 前 | (空) | 从 LB 读取历史 z-score 序列（用于 20 日滚动计算） |
| Step 0.5 | (空) | 从 LB 读取 RSM 前次状态（用于状态机转移判定） |
| Step 6 后 | (空) | 将 RSM 新状态写入 LB |
| Step 9 后 | (空) | 将当日 z-score/hv/turnover 追加到 LB，裁剪至 20 日窗 |

**修复建议**：在 §3.1 步骤链中新增以下步骤：

```
Step 0: SCL 触发 run_once()
  │
├─ Step 0.5A: 从 LB 读取历史数据（z-score 序列 + 前次 RSM 状态）
  │
Step 1: DataBridge 获取 A50 日线数据（最近 40 日）
  │
... (现有 Step 2-9) ...
  │
├─ Step 9.5A: 更新 LB（追加当日数据，裁剪至 20 日窗）
├─ Step 9.5B: 更新 LB（写入当日 RSM 状态 + 持续天数）
  │
Step 10: 输出 → signals/signals/
```

#### 确认墨萱发现

#### 🆕 新增发现

| # | 严重度 | 章节 | 描述 |
|:-:|:------:|:----:|:-----|
| **X-C1** | 🟡 **中** | §5.3 | **DataBridge 耦合风险未文档化** |

**详细分析**：

§5.3 声明 DataBridge 为共振框架复用"morning_pipeline 的数据管线"，并注明"读操作不修改原数据"——耦合评级为低。这一判断在**正常运行**场景下成立。但在以下场景中有潜在风险：

| 风险场景 | 影响 |
|:---------|:-----|
| morning_pipeline 修改 DataBridge 接口（参数变更、返回格式变化） | 共振框架静默异常 |
| morning_pipeline 数据源切换（如 Tushare→AKShare） | 共振框架数据口径不一致 |
| morning_pipeline 调度时间变化 | 共振框架数据就绪时间偏移 |
| DataBridge 缓存策略变更（如减少历史缓存天数 < 40） | 共振框架 20 日滚动窗数据不足 |

**修复建议**：
1. 在 §5.3 增加明确的**依赖契约**：DataBridge 对共振框架的 API 承诺（接口签名、返回值 schema、缓存最低天数 40 日）
2. 增加一个简单的**集成测试**：验证 DataBridge 返回的数据字段对 DCM/LQM 模块的适配性
3. 文档备注"当 morning_pipeline 有结构变更时，需同步验证共振框架的 DataBridge 适配"

| # | 严重度 | 章节 | 描述 |
|:-:|:------:|:----:|:-----|
| **X-C2** | 🟢 **低** | §2.4 / §5.1 | **DSV 辅源可用性信号路径未定义** |

**详细分析**：

DSV 的缩量策略定义：当辅源不可用时（如 Parkinson HV 需要 OHLC 数据而 DataBridge 只返回 close），降级为单源+置信度衰减。

但设计中缺少 DataBridge → DSV 的"数据维度声明"机制：
- DataBridge 返回的数据中是否包含 high/low 字段？
- DSV 如何知道辅源可用/不可用？
- 当前的设计隐含假设 "2026-05-29 的 A50 数据包含 OHLC"，但这一假设未显式声明

**修复建议**：
1. DataBridge 返回的数据对象增加 `available_dimensions: List[str]` 字段（`["close", "high", "low", "volume", "free_float"]`）
2. DSV 初始化时检查 `available_dimensions`，若缺少 high/low 则跳过 Parkinson HV 支路，自动降级

| # | 严重度 | 章节 | 描述 |
|:-:|:------:|:----:|:-----|
| **X-D1** | 🟢 **低** | §2.6 / §6.2 | **CONDITIONAL_PASS 定义缺少"信号可执行性"维度** |

**详细分析**：

当前 CONDITIONAL_PASS 的输出只有仓位上限（`position_cap: 0.5`）。但从实际交易执行角度看，CONDITIONAL_PASS 还应该定义：
1. **信号生效窗口**：条件放行的信号在多长时间内有效？（如 T+1 日内有效？T+1 开盘 30 分钟内有效？）
2. **仓位递减规则**：50% 上限是一次性执行还是分步执行？（如首次 30%，确认后追加至 50%）
3. **与现有风控的关系**：如果整体风控已有 30% 的仓位限制，CONDITIONAL_PASS 的 50% 是追加后的上限还是受风控约束的上限？

这些问题在 Phase 0 可以暂不实现，但建议在文档中标注"待后续完善"以避免理解歧义。

### 维度五：模块间依赖关系（评分：7/10）

#### 评估
依赖图整体**无环**、方向明确（L4→L3→L2→L1→DataBridge）。但从依赖声明到实际算法实现存在不一致（X-A1）。DataBridge 的对外依赖未文档化（X-C1）。本维度与维度一/二有重叠，以下仅列出未覆盖的新发现。

#### 🆕 新增发现

| # | 严重度 | 章节 | 描述 |
|:-:|:------:|:----:|:-----|
| **X-E1** | 🟢 **低** | §8 | **~750 行估算未包含 DSV 辅源管线代码** |

**详细分析**：

§8.1 代码行数估算表中，DSV 为 ~80 行。但 DSV 的辅源（Parkinson HV）需要额外实现：
- 如果 DataBridge 返回 OHLC：Parkinson HV 计算 (~30 行)
- 如果辅源是 5 日滚动 HV：额外滚动计算 (~20 行)
- Spearman 相关性计算（scipy.stats.spearmanr 或自行实现，~20 行）
- 缩量降级逻辑 (~15 行)

实际 DSV 模块可能在 100-120 行，略高于估算。建议更新估算值以保证实现阶段的计划合理性。

## 架构层面总体评估

### 优势
1. **4层+L0 分层合理**：层级职责明确、依赖方向一致、无环，每层可独立 Mock 测试
2. **调度层隔离干净**：SCL 适配器模式 + 纯函数 pipeline，调度机制可替换
3. **否决制层次合理**：硬门（GKV）→ 灰度门（CPE）→ 信号生成（SG），墨萱红线精确
4. **系统集成策略务实**：复用现有 DataBridge/信号总线，`resonance_` 前缀命名空间隔离
5. **辩论约束全覆盖**：6 项核心约束精确映射到独立模块，承诺清单（§0）可追溯

### 关键风险

| 风险 | 等级 | 描述 | 缓解 |
|:----|:----:|:-----|:-----|
| **依赖声明误导** | 🔴 | §1.2 的错误声明可能导致实现时引入不必要的串行等待或错误的状态管理 | 修复 X-A1 |
| **状态机失忆** | 🟡 | RSM 跨日程状态管理未闭环，首次 run() 后状态可能丢失 | 修复 X-B1/X-B2 |
| **DataBridge 静默失联** | 🟡 | 共振框架对 DataBridge 的依赖无契约守卫，上游变更可能导致沉默故障 | 修复 X-C1 |
| **DSV 辅源路径不清晰** | 🟢 | DSV 何时降级、如何感知数据维度不明确 | 修复 X-C2 |
