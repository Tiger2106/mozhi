# 回测试验方案评审会流程 v2.1

## 总体结构

```
Stage 1      Stage 2        Stage 3        Stage 4
  修改  ──→    方案评审  ──→   会签  ──→     归档
               ↺退回修订
               (≤2轮)
```

## 流程说明

本流程覆盖试验方案执行前的评审阶段（模板 §0 ~ §0.5）。执行完成后的结果评审走已存在的《回测总结会4阶段流程 v1.1》。

---

### Stage 1：修改

**起草人：** 墨衡
**主持人（R001）：** 墨涵
**参与者：** 全员（按需）

**产出：** 模板§0~§7填写完整的试验方案草案
**门禁：** 草案状态标记为 [DRAFT_READY]，文件路径写入 experiments/{EXP_ID}/signals/modify_done.json

**两个前置条件：**

1. **R001 问题定义已共识**——墨涵主持四步流程（描述现状→分析后果→找根因→统一陈述），全员确认后写入模板§0
2. **模板§1~§7已填写完整**——无空白、无缺段；失败案例库自查已执行（cold_start可放行）

---

### Stage 2：方案评审（新增）

**性质：** 评估讨论 —— 问"行不行"，允许质疑、讨论、修订
**主持：** 墨涵
**参与：** 全员（墨萱技术评估、墨衡自评、玄知架构把关、Owner观察）
**模式：** 自动触发，分级响应

#### 2.1 分发与触发

墨衡 §0~§7 全部 READY → 自动触发方案评审

- **L1 异步评审（默认）：** 各Agent独立审阅方案，15分钟内提交评审意见
- **L2 联席评审：** 触发条件之一满足时，升级为飞书群同步讨论
  - 任一Agent的L1意见标注"建议联席"
  - 墨涵或Owner主动发起
  - 涉及跨Agent策略交互或架构突破
- **L3 紧急打回：** 发现阻塞性缺陷时，Owner 2小时内决策是否终止

#### 2.2 12项核验表

评审阶段使用12项核验表作为评估工具，发现具体问题：

| 核验项 | 对应节 |
|:------|:-----|
| 标的排除标准合规（上市≥2年、free_float≥0.5、复权质量） | §2.1 |
| 数据精度检验规则明确 | §2.3 |
| 因子权重配置无循环依赖 | §3.1 |
| A股硬约束已启用 | §3.2 |
| 停复牌运行规则已定义 | §2.2 |
| IC/IR 验收阈值已设定 | §4.1 |
| P2 自洽性执行时间已嵌入 | §6 |
| 数据窗口划分合理 | §2 |
| 失败案例库已自查 | §1 |
| 参数扫描范围已明确 | §3 |
| H0/H1 判定规则已写清 | §1 |
| 环境版本已锁定 | §5 |

#### 2.3 评审产出（双输出）

**① 评审纪要（存档）：**
- 参与人及其角色
- 各人意见（逐条记录，标注提出人）
- 会议结论（PASS / GREEN / YELLOW / RED）

**② 修改清单（追踪）：**
每条含：序号 | 问题描述 | 提出人 | 责任人 | 要求完成时间

verdict 4级门禁信号：

| verdict | 含义 | 后续行为 |
|:--------|:-----|:---------|
| PASS | 全员同意，无修改要求 | 直接进入Stage 3会签 |
| GREEN | 有修改但已全部完成 | 正常进入Stage 3会签 |
| YELLOW | 有修改未完成或存在分歧 | 延期，需Owner裁定后再进入会签 |
| RED | 方案被否决 | **流水线终止**，不进入会签 |

#### 2.4 退回修订（子循环）

当 verdict = YELLOW 时触发子循环：

1. 墨涵将修改清单写回 experiments/{EXP_ID}/signals/revise_{EXP_ID}.json
2. 墨衡按清单修订 §0~§7
3. 修订完成后返回 2.2（仅审修改部分，不重审全量）

**退回规则（附修改要求）：**
- 第1次退回：定位缺陷 + 规则引用 + 修改方向（三要素缺一不可）
- 第2次退回：加注更具体修改要求
- 第3次仍不通过 → 强制触发 Owner 裁决
- **退回时间上限：** 每次退回修改≤2个工作日

---

### Stage 3：会签

**性质：** 确认承诺 —— 问"确认吗"，只回答是/否
**原则：** 顺序固定，不可并联。只确认、不讨论、不新增修改项。

#### 第1轮：墨萱技术签

确认12项核验表已评审通过、修改清单已清零。签字即承诺"方案在回测框架内可复现"。

#### 第2轮：墨涵知识签

**核验范围：**
- R001 问题陈述完整（对象+偏差+可观测指标）
- 模板§0已填写（问题记录、根因分析、全员确认签名）
- 方案评审阶段的各方意见已闭合
- 修改方向符合要求

**退回规则（不代修原则）：**
- 不合格 → 退回Stage 2.4，注明不合格范围 + 修改方向
- 覆盖范围：知识层面问题（非技术细节）
- 格式/结构错误可顺手修正

#### 第3轮：Owner 终签

**判定：**
- **准予执行** → 进入 Stage 4
- **终止** → version_status 标记为"已终止"，按失败归档流程处理

---

### Stage 4：归档

**执行人：** 墨涵

**标准步骤：**

1. version_status 从"评审中"推进至"准予执行"
2. 试验编号 EXP-{YYYY}-{XXX} 正式激活
3. 会签结果写入 `backtest_run.notes`
4. 方案文档归档至试验目录

## 两种会议模式

根据R001阶段的性质，区分两种启动模式：

### 草案评审模式（标准）
适用于已有完整试验草案需要评审：Stage 1(修改)→Stage 2(评审)→Stage 3(会签)→Stage 4(归档)

### 议题研讨模式（简化）
适用于空议题/问题描述需要先讨论：Stage 0(议题发起)→Stage 2(研讨)→Stage 1(修改)→Stage 2(评审)→Stage 3(会签)→Stage 4(归档)

两种模式在流程图上表现为同一个 Stage 2 出现在不同位置，但核心逻辑一致：**评价始终在确认之前。**

---

## 快速通道

当以下条件全部满足时跳过 Stage 1 直接进入 Stage 2（方案评审）：
- R001 已共识且有书面记录（含至少两层层根因追问）
- 核心逻辑不变（非"仅数值调整"——明确排除条件变更/设计重构）
- 无跨团队依赖项

**快速通道下的补偿检查：**
- 墨萱抽检 R001 书面记录三要素合规性（对象+偏差+可观测指标）
- 墨涵确认根因至少追到第二层
