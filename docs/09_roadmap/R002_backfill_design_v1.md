## 1. 回填目的

用系统历史上真实发生的故障/风险事件，对 R002 评分框架（六维评分 × 四级行动 × 红名单）进行回溯验证，回答三个核心问题：

1. **有效性**：如果当时应用 R002，能否提前识别这些风险？
2. **分级恰当性**：六维总分对应的四级行动等级是否合理？
3. **框架缺陷**：是否存在 R002 无法覆盖的风险类型？

## 3. R002 六维评分模板

每个事件需填入以下模板：

```json
{
  "event_id": "R002-BF-001",
  "event_name": "<事件名称>",
  "description": "<简要描述>",
  "scores": {
    "impact_scope": {        // 影响范围：0-10
      "score": 0,
      "rationale": "影响模块、用户数、数据量级"
    },
    "irreversibility": {     // 不可逆性：0-10
      "score": 0,
      "rationale": "数据能否修复、恢复成本"
    },
    "uncertainty": {         // 不确定性：0-10
      "score": 0,
      "rationale": "根因是否明确、复现难度"
    },
    "verification_cost": {   // 验证成本：0-10
      "score": 0,
      "rationale": "修复后验证需要的人力/时间"
    },
    "cascading_risk": {      // 级联风险：0-10
      "score": 0,
      "rationale": "引发其他组件/数据连锁损坏的可能性"
    },
    "observability": {       // 可观测性（反向）：0-10
      "score": 0,
      "rationale": "能否及时发现、是否有监控覆盖"
    }
  },
  "total_score": 0,
  "action_level": "NONE|OBSERVE|WARN|IMMEDIATE",
  "redlist_triggered": false,
  "retrospective_assessment": "如果当时用了 R002，能否提前识别？"
}
```

### 评分刻度说明

| 维度 | 0-3 (低) | 4-6 (中) | 7-10 (高) |
|------|---------|---------|---------|
| **影响范围** | 单模块内部 | 跨2-3模块 | 全系统/数据全量 |
| **不可逆性** | 可回滚修复 | 需手工修补 | 数据永久性损坏 |
| **不确定性** | 根因明确 | 需排查分析 | 原因不明/间歇性 |
| **验证成本** | <1h | 1-4h | >4h |
| **级联风险** | 无连锁效应 | 影响相邻模块 | 引发链式故障 |
| **可观测性(反向)** | 有完整监控 | 部分可观察 | 完全黑盒 |

### 四级行动阈值（参考）

| 总分 | 行动等级 | 含义 |
|-----|---------|------|
| 0-15 | NONE | 容忍，不处理 |
| 16-30 | OBSERVE | 监控观察，设告警 |
| 31-45 | WARN | 列入修复计划，本周内处理 |
| 46-60 | IMMEDIATE | 立即停机修复 |

## 4. 各事件评分要点

### BF-001: data_source 单位切换 bug (volume 手/股混用)

| 维度 | 评分 | 依据 |
|-----|------|------|
| 影响范围 | 8 | 影响所有成交量计算指标、VWAP、流动性因子 |
| 不可逆性 | 6 | 历史数据可重算但成本高 |
| 不确定性 | 3 | 根因明确（代码假设不一致） |
| 验证成本 | 5 | 需对比全量历史数据 |
| 级联风险 | 7 | 污染下游因子库和策略信号 |
| 可观测性(反向) | 4 | 极端值可发现但常规检查无覆盖 |
| **总分** | **33** | **OBSERVE（实际应WARN）** |
| **红名单** | 触发（数据质量类） |

**回溯评价：** 如果使用 R002，在代码审查阶段可识别 "手/股" 元数据标注缺失 → 评分为 WARN → 推动增加单位校验前置拦截。R002 可以有效提前发现。

---

### BF-002: daily_basic 表缺失导致 turnover_rate = 0

| 维度 | 评分 | 依据 |
|-----|------|------|
| 影响范围 | 5 | 仅影响换手率相关因子 |
| 不可逆性 | 2 | 补全表即可恢复 |
| 不确定性 | 2 | 表缺失原因明确 |
| 验证成本 | 1 | 补全后重新计算即可 |
| 级联风险 | 3 | 换手率因子稀疏性影响有限 |
| 可观测性(反向) | 2 | 空值容易被发现 |
| **总分** | **15** | **NONE→OBSERVE** |
| **红名单** | 未触发 |

**回溯评价：** R002 评分偏低（15分），但实际影响比表面大（turnover_rate=0 会影响哪些策略未获知）。建议增加 **缺失值传播影响评估** 作为额外权重。

---

### BF-003: VWAP 被 volume 污染

| 维度 | 评分 | 依据 |
|-----|------|------|
| 影响范围 | 8 | VWAP 是核心成交价参考，广泛使用 |
| 不可逆性 | 5 | 可重算但需确认污染区间 |
| 不确定性 | 4 | 根因追踪需要代码审计+时间线对比 |
| 验证成本 | 6 | 需验证修复后与替代源一致性 |
| 级联风险 | 7 | 污染传递至价差分析、订单簿回测 |
| 可观测性(反向) | 5 | 无 VWAP 交叉验证监控 |
| **总分** | **35** | **OBSERVE→WARN** |
| **红名单** | 触发（核心定价数据类） |

**回溯评价：** R002 识别为 WARN 等级是合适的。如果当时有 R002 的事件预检 + 红名单，可能在 volume 单位 bug 修复时就评估其会对 VWAP 产生级联污染，从而做预防性检查。

### BF-004: TECH-2 fail-closed bug

| 维度 | 评分 | 依据 |
|-----|------|------|
| 影响范围 | 6 | 影响技术指标插件的可用性 |
| 不可逆性 | 1 | 修改 fallback 逻辑即可 |

| 验证成本 | 7 | 需要通过故障注入测试 |
| 级联风险 | 5 | 触发后导致下游指标为空 |
| 可观测性(反向) | 6 | 无 fallback 路径告警 |

| **红名单** | 未触发 |

**回溯评价：** R002 评为 OBSERVE（30分），但实际为架构层面的设计缺陷，建议 **红名单扩容**：将 "服务降级/熔断/容错" 类风险纳入红名单触发条件。

### BF-005: write_conn 重试过短

| 维度 | 评分 | 依据 |
|-----|------|------|
| 影响范围 | 7 | 影响所有入库管道写入 |
| 不可逆性 | 3 | 写入失败后数据可重拉 |
| 不确定性 | 2 | 根因明确（配置值不合理） |
| 验证成本 | 2 | 改配置重测即可 |
| 级联风险 | 6 | 频繁重试打挂连接池 |
| 可观测性(反向) | 3 | 日志有重试记录但未聚合告警 |
| **总分** | **23** | **OBSERVE** |
| **红名单** | 触发（基础设施类） |

**回溯评价：** 23分 OBSERVE 合理。如果 R002 框架增加 **"配置类风险"专项检查**，可以在上线前识别该问题（重试间隔 < 数据库连接池超时时间）。

### BF-006: 漏启动通信问题

| 维度 | 评分 | 依据 |
|-----|------|------|
| 影响范围 | 5 | 影响服务间 RPC 通信 |
| 不可逆性 | 1 | 重启可解决 |
| 不确定性 | 4 | 时序依赖导致偶发失败 |
| 验证成本 | 4 | 需要搭建集成环境复现 |
| 级联风险 | 6 | 一个服务挂起阻塞整条链路 |
| 可观测性(反向) | 5 | 无启动顺序告警 |
| **总分** | **25** | **OBSERVE** |
| **红名单** | 未触发 |

**回溯评价：** R002 评为 OBSERVE，实际应在初期就识别为 **架构层风险**（缺乏健康检查探针和启动序依赖声明）。建议增加 **"架构级依赖"维度** 或补充为红名单判断条件。

|-----|------|------|

| 不可逆性 | 2 | 清理日志即可恢复 |

**回溯评价：** 16分处于阈值临界。这类问题 R002 评分偏低，但实际运维成本高。建议将 **"基础设施最佳实践缺失"** 类问题调整为下限 15 分起步。

### BF-008: 前复权 vs 股息率冲突

| 不确定性 | 5 | 涉及复权算法与财务数据口径匹配 |
| 验证成本 | 7 | 需要对照交易所源数据逐条验证 |
| 级联风险 | 5 | 污染持仓收益表和因子库 |
| 可观测性(反向) | 7 | 两种数据源交叉比对后才暴露 |
| **总分** | **34** | **OBSERVE→WARN** |
| **红名单** | 触发（定价源类） |

**回溯评价：** R002 评为 WARN（34分）合理。这也是红名单类问题的典型案例。如果在引入前复权数据时就触发 **"跨源数据一致性检查"** 的 R002 流程，可以提前发现。
